## 2. 核心结论

AI Coding 已进入可用阶段，但其价值不是替代工程判断，而是通过制度化的使用方式（前置 Spec、后置 Test、分层验证、明确 agent 行为）把机器产出的不确定性纳入可控流程。衡量 AI 在工程中是否可用的关键指标是“一次成功率”——一次交互是否能产出可推进的计划或初步产物。要把关注点从“AI 能否写代码”转变为“如何通过 SpecDD（规格驱动）和 TDD（测试驱动）两道红线约束 AI 的输入与输出”，并在项目中建立 spec / agents / tests 的落库与回写机制。工具选择应优先考虑与模型原厂或有稳定 harness 的方案（如 Codex、Cloud Code），并在 CI/CD 中结合分层测试（冒烟、局部、回归）实现质量防控。最终工程实践强调：以人负责架构、以团队维护 spec 与测试资产、以自动化把变更风险降到可接受范围。

## 3. 知识地图

| 主题 | 关键观点 | 落地价值 |
|---|---:|---|
| 一次成功率 | 把“一次交互产出可推进计划/产物”作为可用性衡量标准 | 快速判断模型/Agent 是否适配项目，避免无效迭代 |
| SpecDD（规格驱动） | Spec 是可执行、可审阅的契约型文档，优先于代码 | 规避直接侵入既有代码、驱动测试与回归，便于审查与复用 |
| TDD（测试驱动） | 先写测试（或明确测试样例），把测试作为验收标准 | AI 产出的行为可被自动化验证，回归套件长期沉淀质量资产 |
| Plan/Agent 模式 | 用 plan 模式进行多轮澄清，生成 spec 与执行前讨论 | 降低误解，形成可落库的设计与执行计划 |
| 工具选型 | 优先原厂/有 harness 的模型工具（Codex、Cloud Code）；谨慎使用第三方 IDE 插件 | 保证模型版本与工程集成稳定性，降低长期维护风险 |
| 测试策略 | 冒烟 → 局部 → 回归，多层验证并以测试失败暴露变更 | 用测试作为变更探测器，避免盲目追求“全部通过” |
| Spec/test 管理 | 在仓库中建立 spec/、agents.md、docs/ 等约定目录并保持回写 | 形成项目级记忆与检索入口，便于 agent 与人工复用 |
| 团队治理 | 明确交付流程、agent 修改范围、plan 存档与更新责任 | 降低线上风险，保证可追溯与审计 |

## 4. 关键方法论

按主题分节说明方法论的背景、步骤、适用场景与注意事项。

- SpecDD（规格驱动开发）
  - 背景：PRD 偏场景与体验，缺少实现细节，容易导致实现与验收不一致；AI 更擅长在明确规范下工作。
  - 步骤：
    1. 先撰写初始输入：背景、场景、核心用法、用户画像、输入/输出、边界条件。
    2. 用 plan 模式或让 Coding Agent 代写并多轮提问完善 spec。
    3. 要求 agent 输出“执行前讨论”（plan），并标注变更点与示例代码。
    4. 在确认前不允许直接修改代码；把最终 spec 落库（spec/ 文件夹）。
  - 适用场景：新增功能、模块扩展、需与现有系统耦合的变更。
  - 注意事项：spec 要能驱动测试，且是动态文档；实现后必须把事实回写到 spec。

- TDD（测试驱动）在 AI 协作中的实践
  - 背景：测试不仅为通过而存在，而是发现变更并暴露影响；AI 可快速生成测试但无法判断设计变更是否合理。
  - 步骤：
    1. 基于 spec 定义测试用例（单元/集成/场景）。
    2. 先写测试（红），再实现（绿）；AI 可协助生成测试用例。
    3. 将通过的局部测试纳入回归套件；在 CI 中执行分层测试（冒烟、局部、回归）。
    4. 回归失败：agent 给出初步定位，人工判断是改实现还是改测试/更新 spec。
  - 适用场景：所有面向长期维护的工程变更，尤其是影响公共逻辑或多系统交互的改动。
  - 注意事项：TDD 理想但“反人性”，实际可采用“先明确测试样例并优先实现关键测试”的折衷。

- Plan 模式与 Agentic Workflow
  - 背景：复杂任务需拆解、澄清边界，避免 AI 盲改代码或坚持首版实现。
  - 步骤：
    1. 用 plan 模式从模糊需求生成多轮澄清问题与执行计划。
    2. 要求 agent 在 plan 中给出示例代码与变更文件位置。
    3. 按步骤拆解成可验证的小任务，每步要有明确输入/输出与测试点。
  - 适用场景：初始需求模糊、跨团队协作、需要多轮决策的实现。
  - 注意事项：plan 模式产物必须落库为 spec；若要 agent 修改代码，须明确修改范围与回写规则。

- Agent 行为约束与范围限制
  - 背景：不受约束的 agent 可能修改超出范围的文件或坚持其初始实现。
  - 步骤：
    1. 在指令中明确指定要修改的文件路径与“不修改其他文件”的约束。
    2. 在仓库中放置 agents.md / cloud.md，定义 agent 行为模板（先写测试、不得改动外部接口等）。
    3. 在执行后要求 agent 把变更与实现摘要回写到 spec。
  - 适用场景：任何让 agent 直接写/改代码的任务。
  - 注意事项：使用 @ 引用文件只是辅助，不能替代明确的路径约束；若修改范围涉及架构层面，人工必须介入。

- 测试分层策略（冒烟 → 局部 → 回归）
  - 背景：不同粒度的测试承担不同职责：冒烟保证基本可运行，局部覆盖本次改动，回归检测全局影响。
  - 步骤：
    1. 合并前跑冒烟测试（启动/主要流程是否可运行）；失败直接打回。
    2. 执行本次改动对应的局部测试（单元/集成），纳入回归套件。
    3. 在 CI 中定期或每次合并执行回归套件，回归失败触发人工评估（改实现/改测试/更新 spec）。
  - 适用场景：持续集成与持续交付环境。
  - 注意事项：把回归失败当作暴露变化的信号，而非单纯追求“全通过”。

## 5. 工程实践清单

| 实践动作 | 目的 | 输入 | 输出 | 风险控制 |
|---|---:|---|---|---|
| 在仓库创建 spec/ 与 agents.md | 集中管理规格与 agent 行为约定 | 项目 README、初始需求 | spec 文档、agent 模板 | 强制 agent 读 spec、避免盲改 |
| 定义一次成功率标准 | 评估模型适配性 | 代表性任务与验收标准 | 一次通过率统计 | 若低则更换模型/调整流程 |
| Plan 模式生成并落库 spec | 澄清需求、避免直接改代码 | 模糊需求描述 | 可审阅的 spec + plan | 不落库则计划会丢失，必须回写 |
| 先写测试（或明确测试样例） | 以测试作为验收标准 | spec | 单元/集成测试用例 | 强制测试通过或人工判定回归失败原因 |
| CI 中分层测试（冒烟/局部/回归） | 自动化质量防控 | 代码变更、测试套件 | 测试报告、回归结果 | 失败触发人工评审与修复流程 |
| 指令中明确修改范围 | 限定 agent 行为 | 指令 + 文件路径 | 仅目标文件被修改 | 防止超范围变更 |
| 将 plan/spec/测试回写到仓库 | 保证事实同步 & 可追溯 | 实施后变更信息 | 更新后的 spec、变更日志 | 防止文档与代码脱节 |
| 模型适配性评估 | 判断模型是否能用于工程 | 多次代表性任务 | 成功率数据 | 低成功率停止微调 prompt，考虑替换模型 |
| 让 agent 先尝试再补资料 | 避免资料一开始被滥用 | 补充材料文件夹 | 更有效的 agent 产出 | 若失败再提供材料避免信息滥用 |

## 6. 典型场景与案例

- “Todo API” 规格澄清示例
  - 问题触点：是否有截止日期、是否支持撤销、默认排序、是否为个人或团队使用、关键词检索等。很多细节在 PRD 中不明确，但会直接影响实现与测试。

- webcoding 与工程化的差别
  - webcoding（demo / 氛围编程）适合快速验证想法并发布到 GitHub，但不能直接作为长期维护的工程产物。工程化需要补齐监控、审查、CI、测试和维护流程。

- 回归失败的真实价值（配置覆盖顺序案例）
  - 小改动（配置覆盖顺序修复）引发回归失败，测试暴露问题；团队需要判断这是应该接受的新行为还是实现错误，进而选择修复实现或更新测试/spec。

- YC 创业公司代码生成现象（课程引述）
  - 有创业团队大量使用 AI 生成代码（课程中提及的观察），但这提示职责分工与质量控制的必要性：更多精力转向审查、架构、测试与维护。

- AI 容易产生单脚本倾向
  - 大量代码由 AI 生成时，容易演化为长期难以维护的单文件、大脚本，需要在早期用架构化思想拆分模块。

## 7. 对软件团队的启发

从多个角度的要点凝练：

- 研发流程
  - 将 SpecDD + TDD 写入团队工作惯例：启动任务先写 spec、并伴随测试设计；把这些约定加入 PR/Issue 模板与项目行为文档。
  - 在 CI/CD 中实现分层测试，将测试失败当成“变化信号”而非简单“阻塞”。

- 测试
  - 把测试视为长期资产：局部测试应入池成为回归套件，任何新特性都应通过回归验证其对历史约定的影响。
  - AI 可生成测试草案，但人工需判断测试是否覆盖关键边界。

- 重构与架构
  - 在 AI 写大量代码时，主动介入架构评审与模块划分，防止单脚本化。复杂与跨模块的架构决策仍由有经验工程师主导。

- 协作
  - 设定 agent 行为协议（agents.md），把 plan/spec 存档与更新责任明确到人，保证多人协作时模型执行的一致性与可追溯性。
  - 在团队中培训“如何看懂 AI 产物”：关键是判断 spec 是否满足业务需求、测试是否覆盖关键点。

- 产品/业务理解
  - 产品经理应学会用最小可行的 spec 与示例推动 AI 产出 demo，同时保留工程上的 spec 与测试约束，避免 demo 与工程实现脱节。

- 组织分工
  - AI Coding 的落地改变了一部分“体力活式编码”角色，但不会消灭工程师。工程师的重点转向设计、审查、测试治理与长期维护。

## 8. 可直接复用的行动计划（面向工程团队的 30 天落地计划）

目标：在 30 天内为一个中小型项目建立 SpecDD + TDD 的 AI Coding 流程，并完成一次从 spec 到 CI 回归验证的闭环。

- 第 1-3 天：准备与约定
  - 建立仓库约定：创建 spec/、docs/、agents.md、README（说明模型工作目录与团队约定）。
  - 安装与测试工具链：VSCode + Codex/Cloud Code/Copilot（或 Cursor），确认账号订阅与插件登录可用。
  - 约定一次成功率衡量方法并设计代表性任务（用于后续模型评估）。

- 第 4-8 天：样例规范与模版化
  - 在 spec/ 下写一个 spec 模板（背景、key actions、API/behavior、测试设计、边界）。
  - 把 agents.md 写入团队约定（先写测试、不得改动其他文件、回写 spec、plan 落库等）。
  - 用 plan 模式对一个小功能（如 Todo API）生成 spec，并存档。

- 第 9-15 天：测试与实现闭环
  - 基于 spec 让 agent 生成 PyTest（或语言对应测试框架）测试用例，审阅并在本地运行（先让测试失败，再实现）。
  - 完成实现使测试通过，并把实现变更、测试与更新后的 spec 一并提交到仓库。
  - 在提交模板/PR 中加入“已对照 spec 与测试清单”的复核项。

- 第 16-20 天：CI/CD 与分层测试
  - 在 GitHub（或现有 CI）中配置基本流水线：合并触发冒烟测试、局部测试纳入回归套件。
  - 设置 CI 测试报告产出并保存到 docs/，并让 agent/CI 在回归失败时生成初步失败报告。

- 第 21-25 天：模型适配与一次成功率评估
  - 对代表性任务运行若干次（不同 prompt/同一 spec），记录一次成功率。
  - 若一次成功率低，评估是否更换模型或调整 agent 行为模板。

- 第 26-30 天：沉淀与培训
  - 将 plan、spec、测试样例与 CI 报告整理成团队实践手册，放入 docs/。
  - 对团队进行短培训：如何使用 plan 模式、如何审阅 agent 产物、如何在 PR 中核对 spec 与测试。
  - 规划下一阶段（如：将更多功能纳入 spec/test 库、与更复杂 CI/CD 集成、运行性能/安全检查）。

交付成果：项目中有明确的 spec 文档与测试，CI 能跑分层测试并产出报告，团队有 agent 行为约定与一次成功率基线数据。

## 9. 局限与注意事项

- 本笔记基于课程视频口述整理，内容来自讲师的经验分享与演示。课程中提及的工具（如 Codex、Cloud Code、Copilot、Cursor、DeepSeek、pywrite 等）为讲师演示与推荐，工具可用性与具体表现随厂商与版本变化而异，需在实际项目中验证。
- 课程强调的方法论（SpecDD + TDD、plan 模式、分层测试、一次成功率）为工程实践指导原则，具体在团队中落地时应结合已有 CI/CD、代码基线、团队能力与安全合规要求做调整。
- 关于模型选择与“一次成功率”评估：建议用团队代表性任务做量化试验，再决定模型与 agent 的生产化路径；不要只依赖少量交互判断。
- Plan 模式产物必须落库并持续更新；若没有明确的回写机制，计划与 spec 很容易过时，对模型产生误导。
- AI 生成的测试或实现需要人工审查，尤其在涉及安全、合规、关键业务逻辑与架构层面的改动时，不能放行自动化批准。
