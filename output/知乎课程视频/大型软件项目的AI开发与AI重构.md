## 2. 核心结论

在大型或长期演进的软件项目中，AI 可显著提高代码读写与重构效率，但必须用规范化的工程化流程来约束 AI 的行为、保证安全性与可回溯性。Spec‑Driven Development（specDD）强调把“具象化的 spec 文档”作为长期资产存档与演进，TDD 则通过测试反向驱动实现，两者互为补充：spec 提供可检索的全局缩影与验收标准，TDD/特征测试把这些验收标准落实为可自动执行的门禁。AI 在工作中常处于“plan 模式”（会产出临时计划），但 plan 模式并不等同于落地的 spec；因此团队的工作重心从“写代码”转向“审验与治理 AI 产物”（spec、测试、AI 报告）。对遗留/大型代码库，要先由人划定骨架/模块边界与契约，然后把可验证的小模块任务交给 AI，配合分层测试与预生产验证（stage）小步迭代，结合 Git/CI 门禁与回滚策略，才能保证平滑可控的 AI 驱动重构与交付。

## 3. 知识地图

| 主要主题 | 关键观点 | 落地价值 |
|---|---:|---|
| SpecDD vs TDD | SpecDD：把 spec 写成具象化、可存档的文档库；TDD：用测试驱动实现 | 将需求/验收标准变为长期资产，可复用、可追溯 |
| Plan 模式与 Agent | Plan 模式是会话内的临时计划，不是最终 spec；agent 可做短期拆解与执行 | 避免把临时产物误当长期规范；用于任务拆解与快速探索 |
| Spec 管理与状态化 | Spec 需版本化、状态流转（草稿→审查→锁定/发布→废弃） | 保证 spec 与 release 对齐，便于审计与追溯 |
| 上下文工程 | 用高质量 spec/索引补偿模型的上下文窗口限制 | 让 AI 在局部任务中获得“缩影式”全局视角，减少盲改风险 |
| 遗留代码重构策略 | 先定义骨架/契约，再模块化拆解；锁定关键行为 | 降低大改风险，保证线上行为不被破坏 |
| 特征测试（Feature Tests） | 把系统外部行为锁定为黑盒断言，验证重构前后兼容性 | 为重构提供“不可破坏”的验收门禁 |
| 分层测试与发布 | 本地单元 → 开发集成 → Stage/预生产 → 正式上线 | 多层验证，防止 AI 变更在高仿真场景下出问题 |
| 运行时框架与 AST 扫描 | 框架负责任务分发与质检，AST 用于识别耦合热点 | 提高模型输出质量与定位重构重点 |
| CI/CD 对接与门禁 | 在 CI 中运行测试、生成/发布 spec、状态更新与回滚管控 | 把 AI 产物纳入自动化质量保障链 |

## 4. 关键方法论

- Spec‑Driven Development（specDD）
  - 背景：AI 无法一次性“看全”大型仓库，上下文工程通过 spec 提供浓缩全局认知。
  - 步骤：
    1. 把设计方案写成具象化文档（spec.md 或 spec/ 文件夹内一组文件），标明职责、关键流程、文件/类索引与不可变行为。
    2. 版本化并在仓库中存档（并在 release 时锁定）。
    3. 对每次变更要求 agent/开发者回写并生成 diff，人工审查后合入。
  - 适用场景：长期演进项目、需要多人协作与跨版本追踪的系统。
  - 注意事项：plan 模式产物不能直接替代 spec；spec 要有状态机并由负责人维护。

- Spec → TDD / 特征测试协作流程
  - 背景：spec 描述期望效果，TDD/特征测试把效果转为可运行断言。
  - 步骤：
    1. 从 spec 中提取关键效果/验收点。
    2. 为每条效果写对应测试（单元/集成/feature test）。
    3. 把测试放入 tests/ 目录并在每次变更后运行（CI 门禁）。
    4. AI 在改动后运行测试并产出测试报告（测试总数、通过数、error/warning）。
  - 适用场景：重构、引入 AI 自动修改或生成新功能时的安全门。
  - 注意事项：AI 生成的测试与 spec 必须有人审计；测试覆盖关键场景比数量更重要。

- 遗留代码的渐进式 AI 引入（骨架先行）
  - 背景：直接让 AI 改动“神仙代码”风险高。
  - 步骤：
    1. 人先定义架构骨架、模块边界与对外契约。
    2. 编写或补齐每个关键模块的 spec（职责、索引、风险点）。
    3. 拆解重构任务为模块级小任务并配套最小可验证测试（baseline）。
    4. 让 AI 基于这些测试完成小批量改动，执行回归并逐步扩大覆盖。
  - 适用场景：大型/遗留仓库、线上运行代码不得一次性大改场景。
  - 注意事项：保留回滚路径（Git 分支/备份），在 Stage 环境进行高仿真验证。

- 运行时框架 + AST 扫描的重构诊断
  - 背景：单模型直接运行效率与质量有限，需要框架控制 effort 与质检层。
  - 步骤：
    1. 在运行时框架（Cloud/Clout Code）中下发任务卡（action card）。
    2. 框架要求模型通读并用 AST 扫描引用关系/依赖图，找出高耦合/大文件。
    3. 输出 Markdown 报告与架构图，生成行动卡与分步重构计划，由框架控制每步质检深度。
  - 适用场景：需要全面诊断、并行模型对比与可视化审查时。
  - 注意事项：高质量的中间推理（chain of thought / reflection）能提高可靠性，但成本与时间更高。

- 特征测试（Feature Tests）与黑盒验证
  - 背景：特征测试用于锁定系统对外行为，确保重构兼容。
  - 步骤：
    1. 识别核心行为并把模块当黑盒。
    2. 准备受约束的假数据或轻量数据库（例如 SQLite）模拟外部系统。
    3. 编写并运行外部行为的断言性测试，作为重构的可执行规范。
  - 适用场景：重构、接口变更、跨模块业务链路验证。
  - 注意事项：特征测试应覆盖外部副作用场景，并与 CI 集成。

- 小步安全重构（小不安全）
  - 背景：一次只改一个项，便于监控、审查与回滚。
  - 步骤：
    1. 制定每次小改的目标与回退条件。
    2. 在分支上实现、运行全部特征测试。
    3. 人工审查合格后合并到主干。
  - 适用场景：所有对关键路径的重构工作。
  - 注意事项：每次改动都应有 commit、diff、PR 审查与可回滚方案（优先 Git）。

## 6. 典型场景与案例

- Plan 模式 vs Spec：Plan 模式是 AI 在会话内的“临时计划/工作过程产物”，不等于已落地的 spec；必须把重要结论写成 spec 并存库。
- Spec 指向性：在 prompt 中显式告知模型 spec 的路径或把路径写入 agent 的项目记忆，避免模型自行扫描并猜测。
- 遗留 “大类/神仙代码” 示例：某 order class 超长（数百到上千行），承担税务、优惠、运费等多个职责，应先用特征测试锁定行为，再拆分为“扣钱中心”等抽象模块。
- AI 误重构案例：对 core 目录重构时未明确上下文，AI 创建大量空文件夹并用兼容性填充，反而使结构更乱；教训是必须先写清 spec、锁定不可变行为并提供回滚方案。
- 多版本入口混乱：出现 V1/V2/V3 多个入口并存的情况，通过架构图一眼能发现异常入口数并决定对齐策略。
- 特征测试与轻量 DB：使用受约束的假数据或 SQLite 模拟外部系统输入，进行黑盒特征测试以保证外部行为稳定。
- Tooling 示例：使用 AST 扫描识别耦合热点、用 architecture diagram 把文本渲染成可视图帮助审查、在运行时框架（Cloud/Clout Code）中分配任务与质检。

## 7. 对软件团队的启发

- 研发流程：流程应从“写代码”向“产物治理（spec、测试、AI 报告）”转变。所有重要设计都应具象化为 spec 并纳入版本管理。
- 测试：测试成为门禁而非补充，特征测试用来锁定外部行为，分层测试（单元→集成→stage）保障不同层面的安全。
- 重构：采用“先有骨架 → 模块化拆解 → 小步安全重构”的策略，保证兼容性与可回退性。重构先保障对外接口对齐，再做内部结构化。
- 协作：Prompt/行动卡作为可复用协作产物，形成团队共享的 prompt 库；同时要明确谁负责 spec 的维护与签收（owner 制度）。
- 产品/业务理解：技术改动需以业务语义为准则，重构决策应先追问“为什么存在该副作用或复杂度”，由业务与技术共同决定优先级。
- 组织分工：人负责高阶判断（架构、契约、审查）、AI 负责执行性工作（代码/测试生成、重复性重构），运行时框架负责任务调度与质检；分工明确有助于降低事故率。

## 8. 可直接复用的行动计划（30 天落地计划）

目标：在 30 天内为一个大型/遗留仓库建立 specDD + TDD 的 AI 协作链路并完成首轮安全重构试点（包含 CI 门禁、特征测试与小步重构）。

周 0（准备与沟通，Day 0）
- 召开启动会议：明确目标、负责人（spec owner、测试 owner、CI owner、AI 工具管理员）。
- 确认可用资源（Git 仓库、Stage 环境、AI 工具访问、运行时框架是否可用）。

第 1 周（Day 1–7）：现状评估与基线建立
- Day1–2：仓库扫描与高层架构图
  - 产物：初版架构图、文件/模块索引、候选重构雷点清单（AST 扫描或手工）。
- Day3–4：定义 spec 文件夹结构与状态管理流程
  - 产物：spec/ 目录模板、状态流（Draft→Review→Locked→Deprecated）、issue/PR 模板。
- Day5–7：为关键路径编写 baseline 特征测试草稿（黑盒断言）
  - 产物：tests/feature_baseline/ 中的基础测试用例、假数据或 SQLite 模拟脚本。

第 2 周（Day 8–14）：AI 流程与工具链搭建
- Day8：准备 prompt/行动卡库（把常用 prompt 整理为可复用卡片）
  - 产物：prompt_library/ 文档与示例卡片。
- Day9–11：把一条小重构任务（非关键路径）交给 AI（Plan 模式探索 → 产出草案）
  - 产物：AI plan 文档、临时执行记录（勿直接合并）。
- Day12–14：在本地/CI 中集成测试运行步骤（若示例为 Python，可用 pytest 报表）
  - 产物：CI 流水线草稿（GitHub Workflow 示例任务：checkout → install → run tests → report）。

第 3 周（Day 15–21）：试点执行与审计
- Day15–16：把 AI 生成的“实现 + 测试”进行人工审查（按 spec 与测试断言）
  - 审核点：是否覆盖 spec 中关键断言、是否引入不可接受副作用。
- Day17–18：在分支上执行小步重构（一次只改一项），运行所有特征测试与单元测试
  - 产物：PRs（每次 PR 对应一小步改动）、测试报告、回滚记录（若失败则 revert）。
- Day19–21：在 Stage 环境运行高仿真回归（部分关键场景）
  - 产物：Stage 回归测试结果与问题清单。

第 4 周（Day 22–30）：固化流程与扩展
- Day22–24：根据试点结果修订 spec 模板、prompt 卡与 CI 流程
  - 产物：更新的 spec 管理规范、prompt 库改进、CI 流程文档。
- Day25–27：把已通过的 spec 与测试合并到主分支并在仓库中归档（标注版本）
  - 产物：spec/ 已锁定文档、tests/ 中的稳定用例。
- Day28–29：培训与 handover（对团队进行 1–2 小时实操培训）
  - 内容：如何使用 prompt 卡、如何审查 AI 产出、如何在 CI 中查看测试报告。
- Day30：复盘与下一步规划
  - 产物：复盘报告（已完成/未完成/改进点/下阶段候选模块），确定接下来 60 天重点模块。

注：每天应留有审查窗口，不把 AI 产物直接合并到主分支。所有重要变更必须在 CI 通过且由负责人签核后合入。

## 9. 局限与注意事项

- 课程口述内容与脱离项目的验证：
  - 课程中口述的原则、方法卡片、工具示例（如 Codex、Cloud Code、Hedex、DeepSeeker、Viz Pro）和实践方法为通用工程思路与示例，适用于建立流程与验证思路，但具体工具能力、集成细节与性能需结合实际项目环境验证。
  - 课程提供的 prompt / 行动卡片是可直接参考的模板，但不能盲用；必须理解目的并按业务场景调整后再执行。
- 需要在项目中验证的点：
  - AI 生成 spec、测试与代码的准确率、稳定性与重现成本（token/调用成本）需要在真实仓库与数据上评估。
  - 某些工具/agent 在读取超大仓库、多媒体资产或长上下文时的表现差异需要实际测量：包括模型的上下文窗口限制、AST 扫描对特定语言的支持程度等。
  - Stage 环境的搭建成本与能否复现真实外部依赖（脱敏数据、第三方服务模拟）需要项目组评估。
- 风险提示：
  - 不要把 plan 模式或 AI 会话内的产物直接等同为最终规范；必须落地 spec 并由人审核签收。
  - 在没有充分测试与回滚策略前，不要允许 AI 对关键路径进行大范围自动化修改。
  - Spec 与测试的质量直接决定 AI 改动的安全性与可控性，低质量 spec 会造成更大的技术债。
- 责任分配建议：
  - 将 spec 的最终锁定与关键发布决策交给明确的负责人（如技术 Leader / 模块 Owner）；AI 与 agent 仅为辅助执行者。
