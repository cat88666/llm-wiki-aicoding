# 概念：上下文工程与活文档

## 一、定义

上下文工程不是把全仓库塞给 AI，而是让它在当前任务里看到必要、准确、足够少的信息。

```
大项目代码近似无限
  └── 模型上下文有限
        └── 需要可检索、可分层、可更新的上下文
```

活文档是上下文工程的长期形态：让 spec、结构摘要、状态地图随着代码一起演进。

## 二、三种上下文

| 类型 | 内容 | 作用 |
|---|---|---|
| 入口上下文 | public API、调用方、路由、命令 | 防止改错边界 |
| 结构上下文 | 模块职责、依赖图、状态地图 | 防止只看局部 |
| 历史上下文 | 可疑行为、事故、兼容约定 | 防止顺手破坏历史 |

## 三、AI 先读不先改

大项目 AI 的第一份交付物经常不是代码，而是结构理解：

```
只读交付
├── public API 和入口清单
├── 职责板块
├── 状态地图
├── 隐性耦合
└── 可疑但不能先改的行为
```

这份交付物要被人审。审通过后，它才成为后续 spec 和特征测试的输入。

## 四、活文档的维护

| 文档 | 什么时候更新 |
|---|---|
| spec | 需求、接口、边界、验收标准变化时 |
| 模块摘要 | public API、职责、依赖变化时 |
| 状态地图 | 数据库、缓存、全局变量、消息副作用变化时 |
| 行动卡 | 流程经验、失败模式、门禁规则变化时 |

如果文档只在项目启动时写一次，它很快会变成噪音。

## 五、和 AI 记忆的关系

```
仓库内活文档
├── 对人可审
├── 对 AI 可检索
├── 和代码一起版本化
└── 能通过 PR 维护
```

不要把关键工程规则只留在聊天记录里。会话内 plan 可以丢，仓库内 spec 不能丢。

## 六、一句话

```
上下文工程的目标，是让 AI 每次只读对的那部分，并且读到的是最新事实。
```
