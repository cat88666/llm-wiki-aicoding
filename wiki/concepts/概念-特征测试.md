# 概念：特征测试

## 一、定义

特征测试（Characterization Test）先记录系统当前行为，再用测试锁住它：

```
不先判断现状对不对
  └── 先保证重构前后不能悄悄变
```

它最适合遗留系统和大项目重构。

## 二、和普通测试的区别

| 维度 | 普通测试 | 特征测试 |
|---|---|---|
| 核心问题 | 行为是否符合正确需求 | 改完后是否仍等同于现状 |
| 遇到疑似 bug | 写期望正确行为 | 先锁住当前行为 |
| 主要阶段 | 新功能、修 bug | 重构前、迁移前 |
| 价值 | 验证正确性 | 限制重构爆炸半径 |

疑似 bug 可以修，但要另开任务。重构 PR 里顺手修 bug 是风险源。

## 三、要锁哪些行为

不要只锁返回值。

| 行为面 | 示例 |
|---|---|
| 返回值 | 金额、状态码、错误结构 |
| 数据副作用 | 库存、订单、审计记录、缓存 |
| 外部副作用 | 事件、消息、通知、日志 |
| 异常路径 | 失败时是否回滚、是否吞错 |
| 公共入口 | import 路径、public API、命令行参数 |
| 性能边界 | 大输入下是否超时、是否爆内存 |

## 四、四步流程

```
1. 只读梳理
   ├── public API
   ├── 职责板块
   ├── 状态地图
   └── 可疑行为

2. 写现状 spec
   └── 明确哪些行为先不改

3. 写特征测试
   └── 用黑盒断言锁住行为

4. 小步重构
   └── 每一步都跑测试，红了停
```

## 五、典型翻车点

| 翻车 | 本质 | 处理 |
|---|---|---|
| AI 顺手清理历史分支 | 把“看起来错”当成“可以改” | 先锁现状，修复另开 PR |
| 测试不稳定 | 时间、随机数、全局状态、数据库污染 | 先稳定测试，再重构 |
| 只测 happy path | 隐性耦合没被覆盖 | 补副作用、异常、公共入口 |
| 一次抽太多模块 | 无法定位回归来源 | 回到单职责小步 |

## 六、一句话

```
特征测试不是证明旧行为正确，而是让 AI 重构不能偷偷改掉旧行为。
```
