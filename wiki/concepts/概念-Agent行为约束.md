# 概念：Agent 行为约束

## 一、定义

Agent 行为约束是把团队希望 AI 遵守的工程纪律写成显式规则：

```
修改范围
测试顺序
禁止事项
回写要求
审查门禁
```

AI 没有人类工程师的隐性刹车，所以规则必须外显。

## 二、约束什么

| 约束 | 示例 |
|---|---|
| 路径范围 | 只改 `src/billing/` 和对应测试，不碰 public API |
| 开工顺序 | 先读 spec，再写测试，再实现 |
| 禁止项 | 不做格式化全仓、不重命名公共入口、不顺手修 bug |
| 验证项 | 必须跑局部测试和 lint，失败贴出日志 |
| 回写项 | 实现事实、测试证据、兼容影响写回 spec 或 PR |

## 三、常见载体

| 载体 | 管什么 |
|---|---|
| `AGENTS.md` | 仓库级行为规范 |
| spec | 模块意图、接口、边界 |
| issue 模板 | 任务类型、范围、验收 |
| PR 模板 | 测试证据、回退方案、兼容影响 |
| workflow | 自动化硬门禁 |
| skill / action card | 可复用操作流程 |

## 四、和人工责任的边界

```
AI 适合
├── 读代码整理结构
├── 生成测试草案
├── 机械重构
├── 根据失败日志修正
└── 写 PR 摘要

人负责
├── 定架构边界
├── 定业务语义
├── 判断疑似 bug 是否修
├── 决定不可逆操作
└── 最终合并
```

AI review 可以辅助，但不能成为最终门禁。

## 五、最小可用规则

```
1. 开工前确认路径、分支、任务范围
2. 先读相关 spec 和测试
3. 没有测试保护，不做重构
4. 一次只改一个可验证目标
5. 所有 AI 产出走同一条 CI / PR 门禁
```

## 六、一句话

```
Agent 不是需要鼓励的新人，而是需要明确边界和硬反馈的执行器。
```
