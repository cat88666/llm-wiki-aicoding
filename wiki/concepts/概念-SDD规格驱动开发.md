# 概念：SDD 规格驱动开发

## 一、定义

SDD（Spec-Driven Development）把规格作为首要产物：

```
先定义“什么算对”
  └── 再让代码、测试、契约、文档沿着规格展开
```

在 AI Coding 里，SDD 的价值不是多写文档，而是给概率模型一组可审查、可验证、可回放的边界。

## 二、规格写什么

| 模块 | 要写清楚什么 | 不要写什么 |
|---|---|---|
| 使用场景 | 谁在什么情况下使用 | 空泛愿景 |
| 输入输出 | 字段、类型、约束、默认值 | 模糊描述 |
| 行为规则 | 成功路径、失败路径、边界条件 | “尽量合理” |
| 明确不做 | 删除、分页、持久化、鉴权等排除项 | 给 AI 留自由发挥口子 |
| 验收方式 | 测试、类型约束、示例、性能指标 | 不可验证的口号 |

## 三、好规格的四个标准

```
好规格
├── 聚焦怎么用，不先讨论怎么实现
├── 人能几分钟审完
├── 每条都能变成测试或约束
└── 明确划出不做的范围
```

不可验证的规格等于没有规格。

| 弱规格 | 强规格 |
|---|---|
| 性能不能太差 | P99 延迟不超过 200ms |
| 标题要合理 | 去空白后非空，长度 1-100 |
| 出错要友好 | 缺少 title 返回 422，错误字段包含 title |
| 支持完成待办 | `POST /todos/{id}/done` 幂等，重复调用仍返回 200 |

## 四、在 AI Coding 里的位置

```
需求 / 口头描述
  └── SDD：规格
        ├── 人审：业务语义是否正确
        ├── TDD：把规则转成测试
        ├── API 契约：OpenAPI / SDK / 类型
        └── Agent 任务：按边界实现
```

SDD 让审查前移。人不再最后审几百行实现，而是在开工前审十几条可对照的规则。

## 五、和 Plan 模式的区别

| 维度 | Plan 模式 | SDD |
|---|---|---|
| 生命周期 | 会话内临时计划 | 仓库内长期资产 |
| 主要作用 | 澄清任务和拆步骤 | 固化契约和验收标准 |
| 审查对象 | 做法是否合理 | 行为边界是否正确 |
| 后续要求 | 可丢弃 | 必须随实现回写 |

Plan 可以生成 SDD 的草稿，但不能替代落库后的规格。

## 六、常见误区

| 误区 | 风险 | 收敛方式 |
|---|---|---|
| 把 PRD 当 spec | 场景有了，边界缺失 | 补字段、错误、边界、明确不做 |
| 把架构方案当 spec | AI 提前陷入实现 | 先写调用样例和行为规则 |
| spec 写得太厚 | 人不审，AI 也抓不住重点 | 保留关键约束，细节外链 |
| 实现后不回写 | spec 变成过期误导 | 每个 PR 检查 spec 与实现是否一致 |

## 七、一句话

```
SDD 不是“先写一份文档”，而是先制造一个可审查的正确性源头。
```
