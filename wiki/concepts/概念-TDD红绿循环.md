# 概念：TDD 红绿循环

## 一、定义

TDD（Test-Driven Development）用测试驱动实现：

```
红：先写失败测试
  └── 绿：写最少实现让测试通过
        └── 重构：在测试保护下清理结构
```

在 AI Coding 里，TDD 把“做对了吗”变成机器能持续执行的判定。

## 二、为什么 AI 时代重新重要

传统 TDD 难坚持，是因为人写测试慢，且容易跳过红阶段。AI 改变了两件事：

| 变化 | 含义 |
|---|---|
| AI 写测试成本低 | 原本不愿补的边界测试可以批量生成 |
| AI 需要明确反馈 | 红绿结果比自然语言批评更稳定 |
| AI 生成速度快 | 没有测试，错误也会高速进入主干 |

TDD 对人有点反直觉，对 Agent 正合适。Agent 不需要成就感，需要明确目标和判定器。

## 三、TDD 接 SDD 的方式

```
SDD：规格
├── title 不能为空
├── title 去空白后不能为空
├── title 长度 1-100
├── 不存在 id 返回 404
└── 完成待办接口幂等

TDD：测试
├── test_create_empty_title_422
├── test_create_blank_title_422
├── test_create_long_title_422
├── test_done_missing_id_404
└── test_done_is_idempotent
```

规格是源头，测试是规格的可执行形态。

## 四、分层验证

| 层级 | 工具或动作 | 拦截什么 |
|---|---|---|
| 静态检查 | 类型检查、lint | 导入错误、类型不匹配、低级结构问题 |
| 自动测试 | 单元、集成、特征测试 | 行为回归 |
| 示例运行 | example、demo、冒烟路径 | 使用方式是否真的跑得通 |
| 人工复核 | diff、业务规则、架构边界 | 测试无法判断的语义 |

测试不是人审的替代品，而是把低层正确性先拦住。

## 五、结构化失败回喂

AI 修测试失败时，不要只说“fix bug”。应该回喂：

```
失败测试名
输入
期望结果
实际结果
相关断言
最近 diff
```

失败信息越结构化，Agent 越可能做最小修正。

## 六、适用边界

| 场景 | TDD 方式 |
|---|---|
| 新功能 | 规格先行，按边界写红测试 |
| Bug 修复 | 先写能复现 bug 的红测试 |
| 遗留重构 | 用特征测试先锁现状，再重构 |
| UI 体验 | 用截图、可访问性、交互冒烟补充，不只靠单测 |

## 七、一句话

```
TDD 把规格里的“应该”变成 CI 里反复执行的“必须”。
```
