# 综合：30 天落地路线

## 目标

为一个中小型或遗留项目建立 AI Coding 的最小工程闭环：

```
spec 可审
tests 可跑
Agent 有边界
CI 有门禁
PR 可回退
经验能沉淀
```

## 第 1 周：建立基线

| 时间 | 动作 | 产物 |
|---|---|---|
| Day 1 | 确认试点项目、负责人、工具 | 项目范围和责任人 |
| Day 2 | 梳理 public API、关键路径 | 入口清单 |
| Day 3 | 建 spec 模板和目录 | spec 初版结构 |
| Day 4 | 建 AGENTS 或行动卡 | Agent 行为约束 |
| Day 5 | 选一个小功能或非关键重构点 | 试点任务 |
| Day 6-7 | 写第一份可审 spec | spec 草稿和审查意见 |

## 第 2 周：测试闭环

| 时间 | 动作 | 产物 |
|---|---|---|
| Day 8 | 从 spec 提取测试点 | 测试清单 |
| Day 9-10 | 写红测试或特征测试 | 失败测试 |
| Day 11-12 | 让 AI 做最小实现 | 初版实现 |
| Day 13 | 本地跑 lint 和测试 | 测试日志 |
| Day 14 | 回写 spec 和测试说明 | spec / tests 同步 |

## 第 3 周：CI 与 PR 门禁

| 时间 | 动作 | 产物 |
|---|---|---|
| Day 15 | 加 PR 模板 | 测试证据、回退方案字段 |
| Day 16-17 | 配 lint + test workflow | CI 初版 |
| Day 18 | 设置 required check | 红灯不能合 |
| Day 19 | 用试点 PR 跑完整流程 | 可审 diff |
| Day 20-21 | 修正失败和流程缺口 | 稳定门禁 |

## 第 4 周：固化和扩展

| 时间 | 动作 | 产物 |
|---|---|---|
| Day 22-24 | 复盘试点失败模式 | 规则和测试补充 |
| Day 25-26 | 把行动卡写成团队模板 | 可复用流程 |
| Day 27 | 做一次模型 / Agent 一次成功率评估 | 选型依据 |
| Day 28 | 培训如何审 spec、测试、AI diff | 团队共识 |
| Day 29 | 选下一批任务 | 扩展计划 |
| Day 30 | 锁定流程版本 | 稳定基线 |

## 最小验收标准

```
30 天结束时必须有
├── 1 个已落库 spec
├── 1 组能跑的测试
├── 1 条 CI 门禁
├── 1 个通过门禁的 PR
├── 1 份 Agent 行为约束
└── 1 份复盘后的行动卡
```

## 风险控制

| 风险 | 处理 |
|---|---|
| 一开始想全仓治理 | 只选一个试点模块 |
| 测试太慢 | 先做冒烟和局部测试 |
| spec 太厚 | 保留验收边界，细节外链 |
| AI 改太多 | 限定路径和单职责 |
| CI 推不动 | 先 required test，再逐步加安全和性能 |

## 一句话

```
30 天目标不是“全面 AI 化”，而是跑通一条可复制、可审计、可回退的闭环。
```
