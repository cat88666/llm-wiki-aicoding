# 综合：翻车模式与处置

## 一、总表

| 翻车模式 | 根因 | 立即处置 |
|---|---|---|
| AI 自行补需求 | spec 缺边界 | 停实现，补明确不做 |
| AI 漏调用方 | 上下文不足 | 补 public API 和调用面 |
| 重构改掉历史行为 | 缺特征测试 | 回退，先锁现状 |
| 测试全绿但业务错 | 测试断言错 | 人审 spec 和测试语义 |
| PR 巨大 | 任务没拆 | 拆成单职责小 PR |
| workflow 红还想合 | 门禁不硬 | required check 阻断 |
| 文档过期 | 无回写机制 | PR 增加 spec 回写项 |

## 二、按阶段处理

### 1. 规格阶段

```
症状：AI 产出看起来合理，但和真实需求偏了
原因：规格只写目标，没写边界、错误、排除项
处理：把“怎么用”和“不做什么”写成可验证条目
```

### 2. 测试阶段

```
症状：测试不稳定或覆盖不到关键路径
原因：时间、随机数、全局状态、数据库污染，或只测 happy path
处理：先稳定测试夹具，再补异常和副作用断言
```

### 3. 实现阶段

```
症状：AI 顺手重命名、抽象、格式化全仓
原因：没有显式路径范围和禁止项
处理：回到最小 diff，重申只改目标文件和目标行为
```

### 4. 重构阶段

```
症状：代码变漂亮，但行为变了
原因：没有特征测试，或把疑似 bug 当成可修项
处理：撤回重构；先锁现状；bug 修复另走任务
```

### 5. 协作阶段

```
症状：PR 看不动、主干不稳
原因：每人一个 Agent 后产出速度超过审查能力
处理：issue 划边界，PR 模板要测试证据和回退方案，required check 设硬门禁
```

## 三、排查顺序

```
1. 先看 spec 是否明确
2. 再看测试是否覆盖该行为
3. 再看 diff 是否超范围
4. 再看 CI 日志
5. 最后判断是改实现、改测试、还是更新 spec
```

不要让 AI 在没有证据的情况下连续猜修。

## 四、把事故变成资产

每次翻车至少沉淀一个东西：

| 事故类型 | 沉淀 |
|---|---|
| 漏边界 | spec 模板新增字段 |
| 漏测试 | 回归用例 |
| 超范围修改 | AGENTS 新增禁止项 |
| CI 未拦住 | workflow 新增检查 |
| 人审漏过 | PR 模板新增核对项 |

## 五、一句话

```
AI 翻车不可怕，真正的问题是同一种翻车没有变成下一次的规则、测试或门禁。
```
