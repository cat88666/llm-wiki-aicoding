# 综合：AI Coding 工程闭环

## 一、总图

```
需求
  └── SDD：可审规格
        └── TDD / 特征测试：可执行约束
              └── Agent：在边界内实现或重构
                    └── CI / PR：确定性门禁
                          └── 人审：业务语义和架构裁决
                                └── 回写：spec / tests / AGENTS / workflow
```

这不是一条工具链，而是一套协作协议。底层工具可以换，协议不应该随意变。

## 二、三类任务的闭环

| 任务 | 开工前 | AI 做什么 | 验收 |
|---|---|---|---|
| 新功能 | 写 SDD，明确不做 | 写测试、实现、示例 | 测试绿、契约一致、人审语义 |
| Bug 修复 | 复现并写红测试 | 最小修复 | 红转绿、无额外 diff |
| 遗留重构 | 只读梳理，写特征测试 | 单职责小步重构 | 行为不变、入口不变、可回退 |

## 三、人和 AI 的分工

```
人
├── 定方向
├── 定边界
├── 定标准
├── 定不可逆操作
└── 做最终裁决

AI
├── 读代码整理结构
├── 生成测试草案
├── 机械实现
├── 根据失败日志修正
└── 生成变更说明
```

人的价值不在少打一段代码，而在提高每个 AI 任务的约束质量。

## 四、产物流

| 阶段 | 产物 | 下一步怎么用 |
|---|---|---|
| 规格 | spec | 生成测试和任务边界 |
| 测试 | unit / integration / characterization | 判断实现是否合格 |
| 契约 | OpenAPI / 类型 / public API | 传导到前端和调用方 |
| 变更 | commit / PR / diff | 审查和回退 |
| 结果 | 测试日志 / CI 报告 | 回喂修正 |
| 经验 | AGENTS / action card / workflow | 下次复用 |

## 五、最小闭环模板

```
任务开始
├── 路径 / 分支 / 范围确认
├── 读取相关 spec / tests / public API
├── 写或更新规格
├── 写红测试或特征测试
├── 实现最小改动
├── 运行验证
├── PR 记录测试证据和回退方案
└── 回写长期资产
```

## 六、判断是否该让 AI 做

| 条件 | 判断 |
|---|---|
| 行为能写成规格 | 可以 |
| 验收能自动化 | 更适合 |
| 范围能收窄 | 可以小步执行 |
| 缺业务语义 | 人先补边界 |
| 不可逆且无回退 | 不应直接交给 AI |

## 七、一句话

```
AI Coding 的工程闭环，是把不确定的生成能力接到确定的规格、测试、门禁和人审上。
```
