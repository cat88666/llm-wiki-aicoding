# 主题：大项目 AI 安全重构

## 一、核心问题

大项目里用 AI 的风险集中在三件事：

```
看不全
  └── 上下文窗口有限，隐性依赖看不到

改不敢
  └── 遗留行为没人说得清，漂亮重构可能改变线上语义

管不住
  └── AI 产出太快，PR 和主干门禁跟不上
```

解决方式不是简单换更强模型，而是把大改拆成可理解、可锁定、可回退的小步。

## 二、重构前交付物

| 交付物 | 必须回答 |
|---|---|
| public API 清单 | 外部依赖哪些入口、路径、命令 |
| 职责地图 | 计价、库存、风控、通知等职责散落在哪 |
| 状态地图 | 数据库、缓存、全局变量、消息如何读写 |
| 可疑行为清单 | 哪些像 bug，但重构前必须先保留 |
| 特征测试清单 | 当前行为如何被黑盒断言锁住 |

第一步只读，不改。

## 三、安全重构流程

```
1. 只读梳理
   └── 入口 / 职责 / 状态 / 可疑行为

2. 现状 spec
   └── 把当前外部行为写清楚

3. 特征测试
   └── 锁返回值、副作用、异常、公共入口

4. 小步重构
   └── 一次只抽一个职责块

5. 每步验证
   └── 测试红就停，不继续堆改动

6. PR 合入
   └── CI 通过 + 人审 + 可回退
```

## 四、重构纪律

| 规则 | 做法 | 禁止项 |
|---|---|---|
| 保入口 | public API 和 import 路径先不动 | 一上来改调用方 |
| 小步走 | 单职责、单 PR、单回退点 | 巨型 diff |
| 分任务 | 重构、bug、新功能分开 | 顺手修历史行为 |
| 先验证 | 每一步跑特征测试 | 红灯继续改 |
| 留证据 | commit、diff、测试日志、回退方案 | 只在聊天里说“已验证” |

## 五、隐性耦合排查

代码表面拆了模块，不代表结构安全。优先查：

```
隐性耦合
├── 全局变量
├── import 副作用
├── 缓存读写
├── 数据库共享表
├── 启动顺序
├── 多版本入口
└── 日志 / 事件消费者依赖
```

发现隐性耦合后，先补状态地图和特征测试，再谈抽模块。

## 六、常见翻车处理

| 翻车 | 停在哪一步 | 处理 |
|---|---|---|
| AI 漏关键入口 | 只读梳理 | 追 public API、调用方、import 面 |
| 特征测试不稳定 | 写测试 | 查时间、随机数、全局状态、数据库污染 |
| AI 顺手修 bug | 小步重构 | 撤回；修 bug 另开 issue |
| diff 过大 | PR 前 | 拆成单职责小步 |
| workflow 红 | CI | 以日志为准做最小修复 |
| PR 混合任务 | 审查 | 拆 PR |

## 七、一句话

```
大项目 AI 重构的顺序是：先理解，再锁住；先小步，再验证；先入 Git，再进主干。
```
