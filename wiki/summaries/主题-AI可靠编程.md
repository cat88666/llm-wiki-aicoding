# 主题：AI 可靠编程

## 一、核心问题

AI 写代码已经进入日常，但“写得快”不等于“写得对”。

```
AI 生成速度上升
  └── 审查、架构、调试、测试、安全、维护仍在人
        └── 需要把正确性从感觉判断变成工程约束
```

这条路线的核心是：

```
SDD 定义什么是对
  → TDD 把对变成机器可判
    → 契约把约束传到下一层
      → CI 和人审兜住主干
```

## 二、知识地图

| 主题 | 核心页 | 解决什么 |
|---|---|---|
| 规格 | [SDD](../concepts/概念-SDD规格驱动开发.md) | 开工前定义正确性源头 |
| 测试 | [TDD](../concepts/概念-TDD红绿循环.md) | 把规则转成红绿反馈 |
| 行为约束 | [Agent 行为约束](../concepts/概念-Agent行为约束.md) | 防止 AI 自行发挥 |
| 门禁 | [AI 质量门禁](../concepts/概念-AI质量门禁.md) | 让产出进入确定性管线 |

## 三、标准流程

```
1. 写规格
   ├── 输入 / 输出
   ├── 成功 / 失败
   ├── 边界条件
   └── 明确不做

2. 人审规格
   └── 审业务语义和边界，不审实现细节

3. 写红测试
   └── 每条规格至少有一个可执行断言

4. AI 补实现
   └── 只追求红转绿，不扩大范围

5. 契约传导
   └── OpenAPI / 类型 / SDK 成为下一层规格

6. 门禁合入
   └── lint + test + PR + 人审
```

## 四、Todo API 示例抽象

| 规格点 | 测试点 | 契约影响 |
|---|---|---|
| `title` 必填 | 空字符串返回 422 | 前端输入框必填 |
| 去空白后非空 | `"   "` 返回 422 | 前端 trim 规则一致 |
| 长度 1-100 | 101 字返回 422 | UI 限制最大长度 |
| 完成接口幂等 | 重复完成仍 200 | 前端可安全重试 |
| 不存在 id 返回 404 | `id=999` 返回 404 | UI 显示资源不存在 |

关键不是 Todo 本身，而是“哪一层产物能当下一层 spec”。

## 五、工具选型维度

| 维度 | 为什么重要 |
|---|---|
| spec 遵从度 | 复杂任务里比“自由发挥”更重要 |
| 自主探索 | 适合陌生代码阅读和 UI 探索 |
| 中文文案 | 影响文档和团队协作材料 |
| 成本与速度 | 决定是否能长期规模化使用 |
| harness 稳定性 | 决定能否接入仓库、测试、CI |

具体工具会变，但选型顺序稳定：先看是否能按规格工作，再看探索能力、成本和体验。

## 六、最小落地清单

```
AI 可靠编程最小闭环
├── 一份可审规格
├── 一组红绿测试
├── 一个可运行示例
├── 一条 CI 门禁
├── 一个 PR 模板
└── 一次人工语义审查
```

## 七、一句话

```
可靠 AI Coding 的关键，不是让模型永不犯错，而是让错误被规格、测试、契约和门禁连续拦截。
```
