# 总结：大项目里的 AI — 特征测试做安全重构，GitHub 打通协作管线

> 原文：`output/大项目里的 AI：用特征测试做安全重构，用 GitHub 打通协作管线/`
> 模块 7「AI Coding 带来的范式变革」· 第 2 课

---

## 核心判断

大项目里用 AI，核心不是“模型会不会写代码”，而是：

```
AI 只能看到局部
  └── 大项目的风险来自它没看到的依赖、状态和历史约定

老代码没人敢动
  └── 不知道当前行为是否必须保留，所以不能直接重构

AI 产出速度太快
  └── 每人一个 agent 后，PR 会大量涌入，原来的人工协作节奏失效
```

### 问题总览

| 风险 | 本质 | 不能靠什么解决 | 必须靠什么解决 |
|---|---|---|---|
| 看不全 | 上下文窗口有限，只能读到局部 | 只换更强模型 | 上下文工程、结构摘要、活文档 |
| 改不敢 | 遗留行为没人说得清 | 直接让 AI 重构 | 特征测试先锁住现状 |
| 管不住 | AI 产出进入主干太快 | 只靠口头规范 | GitHub workflow、required check、人审 gate |

## 角色变化

```
人
├── 定架构边界
├── 定模块职责
├── 定接口契约
├── 定验收标准
└── 做最终裁决

AI
├── 读代码并整理结构
├── 补特征测试
├── 做机械重构
├── 根据失败日志修正
└── 生成 PR 级别的变更说明
```

| 角色 | 过去重点 | AI 时代重点 |
|---|---|---|
| 人 | 亲自实现细节 | 定边界、定规则、定验收 |
| AI | 辅助补代码 | 在规则内批量实现、测试、修正 |
| 团队 | 谁写谁负责 | workflow + PR + 人审共同兜底 |

**定边界比填实现更难，也更值钱。**

## 安全重构流程

```
① 读懂遗留代码
    └── 只读，不改
          ├── public API / 入口清单
          ├── 职责板块
          ├── 状态地图
          └── 可疑行为清单

② 特征测试锁住现状
    └── 不判断行为对不对，只判断改前改后是否一致
          ├── 返回值
          ├── 数据库副作用
          ├── 事件 / 日志
          ├── 异常路径
          ├── public import 路径
          └── 性能边界

③ 小步重构
    └── 一次只抽一个职责块
          ├── 保留 public 入口不变
          ├── 重构和修 bug 分开
          ├── 每步跑特征测试
          └── 测试红先停下来定位原因

④ GitHub workflow 门禁
    └── 所有 AI 产出必须进同一条工程管线
          ├── issue
          ├── branch
          ├── commit
          ├── PR
          ├── CI：lint + test
          ├── 人审
          └── merge
```

### 1. 读懂遗留代码

| 交付物 | 要回答的问题 | 用途 |
|---|---|---|
| public API 和入口清单 | 外部到底依赖哪些入口？ | 防止改坏调用方 |
| 职责板块 | 计价、折扣、税、运费、库存、风控散落在哪？ | 找到可切分的重构单元 |
| 状态地图 | 数据库、全局变量、缓存怎么读写？ | 识别隐性耦合 |
| 可疑行为清单 | 哪些看起来像 bug，但重构前必须保留？ | 防止 AI 顺手“修掉”历史行为 |

### 上下文工程

```
上下文工程
├── 按需检索
│     └── 只放当前任务需要的文件
├── 分层摘要
│     └── 先读模块摘要，再下钻到具体文件
└── 活文档
      └── spec 跟代码一起更新
```

### 2. 特征测试锁住现状

| 维度 | 普通测试 | 特征测试 |
|---|---|---|
| 核心问题 | 行为对不对？ | 改完和改前一不一样？ |
| 遇到现状 bug | 写正确行为 | 先锁住当前 bug，修复另开任务 |
| 目标 | 验证业务正确性 | 给重构提供安全网 |
| 适用阶段 | 新功能、修 bug | 遗留代码重构前 |

### 重构前必须回答

```
重构前检查
├── 为什么要重构？
├── 哪些行为必须不变？
└── 出问题怎么回退？
```

**铁律：没有特征测试，不许 AI 重构这块。**

### 3. 小步重构

| 规则 | 做法 | 禁止项 |
|---|---|---|
| 保入口 | public API / import 路径不动 | 一上来改调用方 |
| 小步走 | 一次只抽一个职责块 | 一个 PR 同时抽多个模块 |
| 先验证 | 每抽一块跑一次特征测试 | 测试红还继续改 |
| 分任务 | 重构、bug、新功能分开 | 混在一个 PR |
| 可回退 | 每步都能从 Git 回到上一步 | 巨型 diff 一次性提交 |

### 难点：隐性耦合

```
代码表面已经拆模块
  └── 但内部仍可能靠这些东西耦合：
        ├── 全局变量
        ├── 近循环依赖
        ├── import 副作用
        ├── 缓存读写
        └── 启动顺序

处理顺序：
  └── 先画依赖图和状态地图
        └── 再写 spec 和特征测试
```

### 4. GitHub workflow 门禁

```
issue
  └── branch
        └── commit
              └── PR
                    └── CI：ruff check + pytest -q
                          └── required check 通过
                                └── 人审
                                      └── merge
```

| 门禁 | 拦截什么 | 为什么不能省 |
|---|---|---|
| lint | 风格、静态错误、低级问题 | 确定性强，适合机器拦截 |
| pytest | 行为回归 | 给 AI 产出建立硬边界 |
| required check | 红灯不许合并 | 防止“先合了再说” |
| PR 模板 | 变更范围、测试证据、回退方案 | 让审查有结构 |
| 人审 | 架构、业务语义、不可逆风险 | AI review 不能替代最终责任 |

**AI review 只能辅助，不能当门禁。**
原因：AI review 的标准会随模型漂移；workflow 的 lint、测试、required check 是确定性裁判。

## Spec 状态账本

```
spec/
├── governance/
│     └── 长期规则：命名、目录、边界、禁止项
├── planned/
│     └── 已设计、未落地
├── implemented/
│     └── 已落地，带实现锚点
└── archived/
      └── 废弃或搁置
```

### 改动完成标准

| 维度 | 必须一致 |
|---|---|
| 实现 | 代码已经落地 |
| 测试 | 特征测试 / 单测 / 集成测试覆盖对应行为 |
| 文档 | spec 状态和代码一致 |
| 示例 | example 或使用方式能跑通 |
| 兼容信息 | public API、迁移说明、回退方案清楚 |

**一个改动不算完，直到实现、测试、文档、示例、兼容信息、spec 状态讲的是同一个故事。**

### Issue 分流

| Issue 类型 | 处理方式 |
|---|---|
| 局部 bug | 走最小修复，避免扩大改动面 |
| 架构边界变化 | 先停下来写 spec，再让 AI 实现 |
| 重构 | 先补特征测试，再小步提交 |
| 新功能 | 先定义验收标准，再进入实现 |
| 不确定行为 | 先记录现状，不让 AI 自行判断对错 |

## 质量飞轮

```
AI 生成
  └── spec / 特征测试 / lint / pytest / PR 模板验证
        └── 失败信息结构化回喂给 AI
              └── AI 修正
                    └── 人做最终裁决
                          └── 规则沉淀进 AGENTS.md / 模板 / workflow
```

| 环节 | 产物 | 作用 |
|---|---|---|
| 生成 | 代码、测试、文档 | 提高产出速度 |
| 验证 | lint、pytest、特征测试 | 把质量判断机器化 |
| 回喂 | 失败日志、断言、diff | 让 AI 根据证据修正 |
| 裁决 | 人审意见 | 判断架构和业务语义 |
| 固化 | AGENTS.md、issue 模板、PR 模板、workflow | 把一次经验变成团队制度 |

### AI 比人更需要制度

```
靠谱工程师的隐性刹车
├── 线上压力
├── 代码审美
└── 对历史坑的敬畏

AI 没有这些隐性刹车
  └── 所以规则必须写进：
        ├── issue 模板
        ├── PR 模板
        ├── AGENTS.md
        ├── lint
        ├── pytest
        └── branch protection
```

## 常见翻车处理

| 翻车 | 先停在哪一步 | 处理 |
|---|---|---|
| AI 架构梳理漏关键入口 | 读懂遗留代码 | 追问 public API、调用方、import 面，不继续生成测试 |
| 特征测试不稳定 | 特征测试 | 查数据库、全局状态、时间、随机数；不稳定不许重构 |
| AI 顺手修了疑似 bug | 小步重构 | 撤回，先锁现状；要修 bug 另开 issue 和 PR |
| 重构 diff 太大 | 小步重构 | 拆小，按单个职责块重新来 |
| workflow 红但 AI 说没问题 | GitHub workflow | 以 workflow 为准，根据日志做最小修复 |
| PR 混了重构、bug、新功能 | PR 审查 | 拆 PR；混合 PR 是质量事故温床 |

## 最小落地清单

```
大项目 AI 重构最小闭环
├── 入口清单
├── 职责地图
├── 状态地图
├── 特征测试
├── 小步 commit
├── PR 模板
├── lint + test workflow
├── required check
├── 人审
└── spec 状态账本
```

| 优先级 | 先做什么 | 原因 |
|---|---|---|
| P0 | 特征测试 + required check | 没有硬门禁，AI 产出不可控 |
| P1 | 入口清单 + 状态地图 | 防止只看局部就重构 |
| P2 | PR 模板 + spec 状态账本 | 让协作和文档跟得上代码 |
| P3 | AI review | 只能增益审查，不能替代门禁 |

## 一句话总结

```
大项目里的 AI 价值
├── 驾驭读不完的代码库
│     └── 架构梳理 + 活文档
├── 敢动没人敢碰的老代码
│     └── 特征测试先锁现状
└── 让 AI 产出进入团队秩序
      └── Git 可追溯 + GitHub workflow 门禁

纪律：
先理解再锁住
  → 先小步再验证
    → 先入 Git 再进主干
      → 先过 workflow 再由人裁决
```
