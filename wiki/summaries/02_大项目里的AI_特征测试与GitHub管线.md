# 总结：大项目里的 AI — 特征测试做安全重构，GitHub 打通协作管线

> 原文：`output/大项目里的 AI：用特征测试做安全重构，用 GitHub 打通协作管线/`
> 模块 7「AI Coding 带来的范式变革」· 第 2 课

## 核心问题

大项目里用 AI，问题不在它会不会写代码，在三件事：

| 风险 | 本质 |
|---|---|
| **看不全** | 上下文窗口有限，AI 只看到局部，改坏的是它没看到的依赖 |
| **改不敢** | 老代码没人说得清，AI 改完行为是否一致没人敢签字 |
| **产出管不住** | 每人一个 agent，PR 像水一样往主干涌，以前的"谁写谁负责"失效 |

换更强的模型没用，窗口还是那么大。能做到的：**每步操作留在 Git、每步改动有测试锁着、出问题能退回去**。

---

## 思维跃迁：人的角色从"实现者"变为"定边界的人"

大项目里，AI 负责在格子里填实现、补测试、做机械重构；人的活是：
- 架构边界
- 模块职责
- 接口契约
- 验收标准
- 最终裁决

**定边界比填实现更难，也更值钱。**

---

## 四步安全流程

### 1. 读懂遗留代码（只读，不改）

让 AI 输出四件东西：
1. public API 和入口清单
2. 职责板块（计价、折扣、税、运费、库存、风控各散落在哪）
3. 状态地图（数据库、全局变量、缓存怎么读写）
4. 可疑行为清单（看着像 bug，但重构前必须先锁住）

> 第一份交付物经常不是代码，是结构理解。

**上下文工程三招**：按需检索（只放当前任务需要的文件）、分层摘要（先读模块摘要再下钻）、活文档（spec 跟代码一起更新）

### 2. 特征测试锁住现状

**特征测试**验证"改完和改前一不一样"，不判断"对不对"。

**铁律：没有特征测试，不许 AI 重构这块。**

| 维度 | 普通测试 | 特征测试 |
|---|---|---|
| 问题 | 行为对不对？ | 改完和改前一不一样？ |
| 现状有 bug | 写正确行为 | 先锁住 bug 的现状，bug 单独修 |

行为面不只是返回值，还包括：数据库副作用、事件/日志、异常路径、public import 路径、性能边界。

重构前先回答三个问题：**为什么重构？哪些行为必须不变？怎么回退？**

### 3. 小步重构，每步跑测试

- 保留 public 入口不变（绞杀者模式：新实现在外围长出、逐步接管）
- 一次只抽一个职责块
- 每抽一块跑一次特征测试
- 重构和修 bug 分开，一个 PR 只做一件事
- 测试红先停：判断是测试写错了、状态没隔离，还是行为真的变了

**另一种难**：代码已拆模块，但靠全局变量、近循环依赖、import 副作用暗中耦合。先画依赖图和状态地图，图画出来之前 spec 和特征测试没有意义。

### 4. GitHub workflow 设为门禁

```
issue → branch → commit → PR → CI(lint + pytest) → 人审 → merge
```

最小可用 workflow（5 行 YAML）：`ruff check` + `pytest -q`，设为 required check，红了合不进去。

**AI review 可以辅助，但不能当门禁**：AI review 的标准会随模型漂移，无法替代确定性检查。

## 让 spec 活起来（状态账本）

```
spec/governance/   ← 长期规则
spec/planned/      ← 已设计、未落地
spec/implemented/  ← 已落地，带实现锚点
spec/archived/     ← 废弃或搁置
```

**核心铁律**：一个改动不算完，直到实现、测试、文档、示例、兼容信息、spec 状态讲的是同一个故事。

issue 分类是治理入口：局部 bug 走最小修复，涉及架构边界先停下来写 spec，别让 AI 把每个 issue 都当成局部补丁。

---

## 质量飞轮

```
AI 生成
  → spec / 特征测试 / lint / pytest / PR 模板验证
  → 失败信息结构化回喂给 AI
  → AI 修正
  → 人做最终裁决
```

AI 的可信度不来自它说"我完成了"，来自把它放进了 **spec + 测试 + Git + workflow + PR + 人审 gate** 组成的系统。

**AI 比人更需要制度**：靠谱工程师有隐性刹车（线上压力、代码审美、对坑的敬畏），AI 没有。规则必须写进 issue 模板、PR 模板、AGENTS.md、lint/pytest、Branch protection。

## 常见翻车及处理

| 翻车 | 处理 |
|---|---|
| AI 架构梳理漏关键入口 | 先追问 public API、调用方、import 面，不要继续生成测试 |
| 特征测试不稳定 | 查数据库、全局状态、时间、随机数——测试不稳定不许重构 |
| AI 顺手修了疑似 bug | 撤回，先锁现状，要修另开 issue 和 PR |
| 重构 diff 太大 | 要求拆小，按单个职责块重新来 |
| workflow 红但 AI 说没问题 | 以 workflow 为准，根据日志做最小修复 |
| PR 里混了重构+bug+新功能 | 拆 PR，混合 PR 是质量事故温床 |

---

## 一句话总结

大项目里 AI 的价值在三件过去很难的事：**驾驭读不完的代码库**（架构梳理 + 活文档）、**敢动没人敢碰的老代码**（特征测试先锁现状）、**让 AI 产出进入团队秩序**（Git 可追溯 + GitHub workflow 门禁）。纪律只有一条：先理解再锁住，先小步再验证，先入 Git 再进主干，先过 workflow 再由人裁决。
