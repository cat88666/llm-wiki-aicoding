# 主题：AI 工程化治理

## 一、核心判断

AI Coding 的组织价值，不是让工程师少负责，而是把人的责任从“填代码”前移到：

```
定边界
定标准
定门禁
审语义
维护长期资产
```

代码可以大量由 AI 生成，但 spec、测试、门禁、回退、最终裁决仍然是工程责任。

## 二、治理对象

| 对象 | 治理问题 | 产物 |
|---|---|---|
| 需求 | 是否明确、可验收 | spec、issue |
| Agent | 是否超范围、自行发挥 | AGENTS、行动卡 |
| 测试 | 是否覆盖关键边界 | 单测、集成、特征测试 |
| 变更 | 是否可审、可回退 | branch、commit、PR |
| 主干 | 是否有硬门禁 | workflow、required check |
| 知识 | 是否随代码更新 | 活文档、状态账本 |

## 三、Spec 状态账本

```
spec/
├── governance/
│   └── 长期规则：命名、目录、禁止项
├── planned/
│   └── 已设计，未实现
├── implemented/
│   └── 已落地，带代码锚点和测试证据
└── archived/
    └── 废弃或搁置
```

一个改动不算完，直到实现、测试、文档、示例、兼容信息讲的是同一个故事。

## 四、Issue 分流

| 类型 | 处理 |
|---|---|
| 局部 bug | 最小修复，先复现再改 |
| 架构边界变化 | 先停下来写 spec |
| 重构 | 先补特征测试 |
| 新功能 | 先定义验收标准 |
| 不确定行为 | 先记录现状，不让 AI 判断对错 |

Issue 是 spec 治理的入口，不只是任务标题。

## 五、团队最小制度

```
最小制度
├── 任务必须有 issue 或 spec 锚点
├── Agent 修改范围必须显式写出
├── 重构必须先有特征测试
├── PR 必须写测试证据和回退方案
├── required check 红灯不能合并
└── 人审负责业务语义和架构边界
```

## 六、一次成功率

一次成功率衡量的是：

```
一次交互能否产出可推进的计划、测试或初版实现
```

它适合用于模型和 Agent 选型。不要只看单次惊艳输出，要用团队代表性任务做基线。

| 低一次成功率的处理 | 含义 |
|---|---|
| 补 spec | 输入不清 |
| 收窄范围 | 任务过大 |
| 增加测试 | 反馈太弱 |
| 换模型或 harness | 工具不适配 |

## 七、一句话

```
AI 工程化治理的目标，是让高产进入团队秩序，而不是让高产绕过团队秩序。
```
