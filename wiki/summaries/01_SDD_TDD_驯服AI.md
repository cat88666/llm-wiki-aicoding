# 总结：用 SDD + TDD 驯服 AI

> 原文：`output/用 SDD + TDD 驯服 AI：让代码既能「写得快」也能「信得过」/`
> 模块 7「AI Coding 带来的范式变革」· 第 1 课

---

## 核心问题

AI 写代码已是日常（腾讯内部 50%、YC 冬季批 25% 公司 95% 代码由 AI 生成），但**写得快 ≠ 写得对**。

Karpathy 从 "vibe coding" 改口为 "agentic engineering"，因为以下负担仍然在人：

```
审查 / 架构 / 调试 / 测试 / 安全
```

---

## 两个核心方法

### 对比总览

| 维度 | SDD（规格驱动开发） | TDD（测试驱动开发） |
|---|---|---|
| 产物 | 规格（spec）是首要产物 | 失败的测试是首要产物 |
| 关注点 | "怎么用"，不急于"怎么实现" | 行为是否正确 |
| 对 AI 的作用 | 给 AI 明确约束，防止自行发挥 | 给 AI 输出提供"裁判" |
| AI 时代优势 | 规格人能几分钟审完 | AI 写测试速度快，恰好是最需要的工具 |

### SDD 规格的四条标准

```
规格必须：
├── 聚焦"怎么用"（不是"怎么实现"）
├── 人能几分钟审完（十几条可逐条对照的约束，不是百页文档）
├── 可验证（"P99 延迟 ≤ 200ms" ✓  "性能不能太差" ✗）
└── 划死不做的事（防止 AI 自行发挥）
```

### TDD 在 AI 时代的复活

```
传统 TDD 困境：人会偷懒，跳过红阶段
AI 时代解法：执行主体换成 Agent，需要明确指令，天然契合 TDD
```

## 配合流程

```
① 写规格（SDD）
    └── 人工审查 spec
         │
② 按规格只写测试（TDD 红，全部 FAIL）
         │
③ AI 补实现，红转绿（TDD 绿）
```

### 验收三层（Python 技术栈）

| 层级 | 工具 | 拦截目标 |
|---|---|---|
| L1 静态 | pyright | 类型错误 |
| L2 自动 | pytest 全量用例 | 行为正确性 |
| L3 人工 | example + 复检 | 易用性 & 业务正确性 |

## 实操要点：待办事项 API 示例

### 规格边界定义

```
数据模型：id / title / done
端点（共 3 个）：
├── POST /todos          新建
├── GET  /todos          列表
└── POST /todos/{id}/done 完成

边界条件：5 条
明确不做：无删除 / 无分页 / 无持久化 / 无鉴权
```

### 关键操作要点

| 要点 | 正确做法 | 错误做法 |
|---|---|---|
| 失败回喂 | 结构化：断言 + 输入 + 期望 + 实际 | 笼统说 "fix the bug" |
| 推进节奏 | 渐进式：一个端点一个红绿循环 | 一次全堆给 AI |
| 审查时机 | 审 spec 和测试结构（前移） | 审上百行实现（后置） |

### 契约传导链

```
后端 spec
  └─→ TDD 红绿循环
        └─→ FastAPI 自动生成 OpenAPI 契约
              └─→ 契约成为前端的 spec
                    └─→ 前端约束（端点 / 字段 / 长度）全部从契约来
```

## 工程固化

### 流水线结构

```
L3 AI Coding
├── 规划 Agent
├── 实现 Agent
└── 审查 Agent
      └── 走同一道 CI/CD 门禁（类型检查 / 测试 / lint，无豁免权）
```

### Rules + Spec + Skills 三位一体

| 沉淀 | 管什么 | 效果 |
|---|---|---|
| Rules | 命名规范、目录约定、禁止项 | 产出贴着团队工程口味 |
| Spec | 模块意图、边界、明确不做 | AI 理解"为什么" |
| Skills | 可复用操作流程 | 不用每次重新拼装步骤 |

> 每踩一次坑就写进 Rules/Spec，知识飞轮越转越快。

### 两道审查关

```
审查关
├── AI 自审（换视角）
│     ├── 边界漏洞
│     ├── 静默吞错
│     ├── spec 偏差
│     └── 缺测试
└── 人工最终决策（不可替代）
      ├── 不可逆操作
      └── 业务语义是否正确
```

## 一句话总结

```
SDD 定义"什么是对"
  → TDD 把"对"变成机器能判的东西
    → API 契约让确定性沿技术栈一层一层往下传导

核心思路：找到"哪一层的产出可以当下一层的 spec"
         就有了一条从源头往下的约束链
```
