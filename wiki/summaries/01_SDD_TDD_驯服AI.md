# 总结：用 SDD + TDD 驯服 AI

> 原文：`output/用 SDD + TDD 驯服 AI：让代码既能「写得快」也能「信得过」/`
> 模块 7「AI Coding 带来的范式变革」· 第 1 课

## 核心问题

AI 写代码已是日常（腾讯内部 AI 生成占比 50%，YC 冬季批次 25% 公司 95% 代码由 AI 写），但写得快不等于写得对。Karpathy 从"vibe coding"改口为"agentic engineering"，原因正在于此：审查、架构、调试、测试、安全的负担都还在。

**本文要解决的问题：在 AI 大量产出代码的前提下，怎么确认它写得对、敢不敢上线。**

---

## 两个核心方法

### SDD（规格驱动开发）
- 规格是首要产物，代码从规格构建出来
- 规格聚焦"怎么用"，不急于讨论"怎么实现"
- 规格必须**人能在几分钟内审完**（十几条可逐条对照的约束，不是百页文档）
- 规格必须可验证——"P99 延迟不超过 200ms"是规格，"性能不能太差"不是
- 规格必须划死不做的事，防止 AI 自行发挥

### TDD（测试驱动开发）
- 先写会失败的测试（红），再补最少实现（绿），再重构
- AI 时代 TDD 的复活原因：**执行主体从会偷懒的人换成了需要明确指令的 Agent**
- AI 写测试速度快，且恰好是 AI 生成代码最需要的"裁判"

### 两者配合
```
先写规格（SDD，人审）
→ 按规格只写测试（TDD 红，全部 FAIL）
→ AI 补实现，红转绿（TDD 绿）
```

**验收三层**（Python 技术栈）：
1. pyright 静态校验 — 拦类型错误
2. pytest 全量用例 — 判行为正确
3. example + 人工复检 — 判易用性和业务正确性

## 实操流程：待办事项 API 示例

规格定义了数据模型（id/title/done）、三个端点（POST /todos、GET /todos、POST /todos/{id}/done）、5 条边界条件，以及明确不做的事（无删除、无分页、无持久化、无鉴权）。

**关键操作要点：**
- 失败信息要结构化回喂（具体断言 + 输入 + 期望 + 实际），不是笼统说"fix the bug"
- 渐进式：一个端点一个红绿循环，不一次堆给 AI
- 审查成本前移：审的是 spec 和测试结构，不是上百行实现

**契约传导链：**
后端 spec → TDD 红绿 → FastAPI 自动生成 OpenAPI 契约 → 契约成为前端的 spec → 前端约束（端点、字段、长度）全部从契约来

## 工程固化

### 流水线
- L3 AI Coding：规划 / 实现 / 审查三个 Agent 分工
- AI 产出走同一道 CI/CD 门禁（类型检查、测试、lint），无豁免权

### Rules + Spec + Skills 三位一体

| 沉淀 | 管什么 | 效果 |
|---|---|---|
| Rules | 命名规范、目录约定、禁止项 | 产出贴着团队工程口味 |
| Spec | 模块意图、边界、明确不做 | AI 理解"为什么" |
| Skills | 可复用操作流程 | 不用每次重新拼装步骤 |

每踩一次坑就写进 Rules/Spec，知识飞轮越转越快。

### 两道审查关
1. **AI 自审**：换视角找边界漏洞、静默吞错、spec 偏差、缺测试
2. **人最终决策**：不可逆操作、业务语义是否正确 — 机器替代不了

---

## 一句话总结

SDD 定义"什么是对"，TDD 把"对"变成机器能判的东西，API 契约让确定性沿技术栈一层一层往下传导。找到"哪一层的产出可以当下一层的 spec"，就有了一条从源头往下的约束链。
