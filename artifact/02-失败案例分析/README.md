# 生产游戏 AI 重构失败案例分析

> 分析对象：[`artifact/01-失败案例/01-生产游戏AI重构失败案例`](../01-失败案例/01-生产游戏AI重构失败案例/)
>
> 分析基线：[`wiki/`](../../wiki/) 中的 SDD、特征测试、TDD、Agent 行为约束、质量门禁与安全重构方法。
>
> 结论边界：本报告基于失败案例中保存的扫描、设计、计划、变更记录和结果审核文档，不代表已经重新验证原始 Java 工程。

## 一、结论

这次重构失败的主要原因不是 AI 不会设计，也不是文档数量不足，而是工程闭环在“实施验证”阶段被主动断开：

```text
扫描和设计很完整
  └── 未关闭业务阻塞项就开始编码
        └── 没有特征测试锁定旧行为
              └── 无编译环境仍继续结构性改造
                    └── 用静态 diff + AI 自审替代编译和测试
                          └── 行为变化、漏提交、混合提交进入待发布状态
```

最终形成了“文档完成度高、真实可发布置信度低”的状态：

| 指标 | 结果 | 说明 |
|---|---:|---|
| 功能扫描 / 设计 / 计划 | 21 / 21 / 21 | 文档产物齐全 |
| 有结果审核的功能 | 12 / 21 | 其余 9 个功能没有结果审核 |
| 审核结论为灰度观察 | 10 / 12 | 绝大多数已审核功能仍有行为或依赖风险 |
| 审核结论为可发布 | 2 / 12 | 仅风控数据上报、好友备注 |
| Git 记录中的未验证项 | 至少 63 条 | 大量变更未编译、未单测、未协议回归 |
| 阻塞型待确认项 | 6 项 | 协议、资金、MQ、幂等和错误语义未拍板 |

按照当前 Wiki 的完成标准，这批改造不能视为“生产级重构完成”，最多只能视为：

```text
高质量问题扫描
+ 部分代码改造草案
+ 静态审核结果
- 可执行测试证据
- 硬质量门禁
- 生产发布准入
```

## 二、核心失败问题

### P0：用静态审核替代编译和测试

失败案例明确记录：

- 执行计划规定“无 java/maven，不能编译/跑测试”，并把 `dev 基线 diff + 编码审核 Prompt` 定义为替代闭环。
- Git 变更记录中至少 63 条验证结果为“未验证”“未编译”或“git 记录未体现编译、单测或协议回归结果”。
- 结果审核大量只给出“回归测试覆盖建议”，没有已执行的测试日志。

这违背了：

- [TDD 红绿循环](../../wiki/concepts/概念-TDD红绿循环.md)：测试必须成为机器可执行判定。
- [AI 质量门禁](../../wiki/concepts/概念-AI质量门禁.md)：没有 lint、test、CI 证据不能进入合并和发布判断。
- [大项目 AI 安全重构](../../wiki/summaries/主题-大项目AI安全重构.md)：每一步验证失败必须停下，不能继续堆改动。

实际后果已经出现：入座链路引用的 `SitDownStateService` 漏提交，造成 P0 编译断裂。该问题直到结果审核后才被发现并补建，证明静态核对不能替代最基础的编译门禁。

**建议：**

1. 立即取消“静态 diff 可替代编译和测试”的规则。
2. 没有可用构建环境时，只允许扫描、写 spec、设计测试，不允许提交结构性代码改造。
3. 建立最低门禁：模块编译、关键单测、协议冒烟任一缺失，状态只能是 `BLOCKED`，不能写“实施 100%”或“可发布”。

### P0：没有特征测试，重构与行为修复混在一起

21 个功能都有扫描、设计和实现计划，但失败案例目录没有保存任何特征测试或测试结果。多个结果审核证明重构同时改变了线上行为：

| 功能 | 行为变化 |
|---|---|
| 入座 | HALL/RACE 从静默成功变为明确拒绝；限流与余额校验顺序变化 |
| 风控 | 规则计算异常从放行变为保守拦截 |
| 进入房间前校验 | 原本失效的私人房禁入校验开始生效 |
| 牌桌状态查询 | HALL 不再执行 Club 校验；解散桌从系统异常变为 `GAME_END` |
| 预站起 / 留座离桌 | 1045 从死功能变为可用；1034 从静默丢包变为明确错误 |
| 用户配置 | 参数校验收紧；错误路径改为空配置响应 |

这些变化可能是合理修复，但不属于“保持行为等价的重构”。在没有特征测试和产品确认时，它们无法证明安全。

**建议：**

1. 重构、Bug 修复、协议行为调整拆成不同 issue 和 PR。
2. 每个遗留功能先建立行为矩阵和特征测试，至少锁定正常、异常、幂等、并发和副作用路径。
3. 疑似 Bug 先按旧行为写特征测试；确认要修后，再用单独红测试驱动修复。

### P0：阻塞型业务决策未关闭，AI 被允许自行选择默认行为

执行计划一方面定义了阻塞型待确认项，另一方面又规定待确认项“一律不停下”，由 AI 选择所谓保守默认并继续。实际已有 6 个阻塞项，涉及：

- 1042、1012、1013 协议是废弃还是适配。
- 审核单幂等依赖 Redis CAS、钱包 eventId 还是新增中间态。
- MQ 是否保证同一审核单有序。
- 建单成功但 MQ 失败时客户端看到成功还是失败。
- 实时账单主块失败时返回半空成功还是错误响应。

这些不是技术实现细节，而是产品语义、资金一致性和分布式状态机契约。AI 无权替业务方拍板。

**建议：**

1. 将待确认项分成 `BLOCKED`、兼容默认、灰度观察三类。
2. 协议废弃、资金口径、幂等策略、MQ 顺序、拦截策略必须由责任人签字确认，未确认不得编码。
3. 在 spec 中记录“决策人、决策日期、最终结论、对应测试”，不要只写安全默认。

### P1：未设计的跨功能重构已经接入主链路

结果审核明确指出，`visitor → CandidateProvider` 重构没有设计文档，却已经接入 `BaseUserLeaveService` 和 `UserLeaveDefault`，并影响站起、退出、离开状态检查、预站起和风控站起。

这说明任务虽然按 21 个功能拆分，但真实共享依赖没有形成独立重构单元。表面上是小功能推进，实际仍存在跨功能大改。

**建议：**

1. 把 CandidateProvider 作为独立共享能力立项，补 public API、调用方、状态和行为等价矩阵。
2. 在共享能力完成特征测试和单独审核前，调用方不得切换。
3. 对 04、06、07、08、18 做一次调用面复扫，确认没有遗漏入口和旧 visitor 行为。

### P1：完成度、审核状态与真实证据不一致

指导文档规定“实施”必须包含代码改造、测试、回归和上线观察，但多个功能在测试、回归、上线观察仍待补充时被标记为“实施编码 100%”。同时：

- 变更记录索引大量状态仍为“未审核”。
- 指导文档却标记 12 个功能已审核。
- 多条记录由非原作者后补，根据 commit message、计划和 diff 推断。
- 结果审核发现指导文档对 3018/3019 是否落地的记录不准确。

状态账本失真后，后续 Agent 和人工评审都会基于错误上下文继续工作。

**建议：**

1. 建立唯一状态表，其他文档只引用，不重复维护百分比。
2. 使用证据驱动状态：`implemented` 必须关联 commit，`verified` 必须关联命令和日志，`approved` 必须关联审核结论。
3. 将“实施编码完成”和“生产完成”拆开，禁止用单一 100% 掩盖未验证状态。
4. 后补记录必须标记 `INFERRED`，原作者或责任人确认后才能转为事实。

### P1：提交边界和回退边界不够独立

失败案例记录了混合提交、笼统提交信息和大量级联回退：

- commit `036cfd439` 混合了 18 预站起与 20 好友备注。
- 多个提交信息仅为“优化重构入座”或“进入房间优化”，无法直接追到 R 项。
- 多条变更记录说明单独回退会导致后续提交编译失败，需要链式回退。

这使“一个功能、一项风险、一个回退点”失效，也降低了审核定位效率。

**建议：**

1. 一个 commit 只对应一个 R 项或一个测试点，提交信息必须包含功能号和 R 编号。
2. 先提交测试，再提交最小实现；共享基础类与调用方切换分开提交。
3. PR 中列出提交依赖图；不能独立回退的提交必须合并为一个明确的原子变更。

### P1：审核发现问题后仍缺少硬阻断

12 个结果审核中，10 个结论为“灰度观察”，只有 2 个“可发布”。但灰度观察项普遍缺少已落地的指标、阈值、灰度比例、自动回滚条件和实际验证结果。

“建议灰度”不是灰度方案。如果没有观测和回滚门禁，它只是把未验证风险推向生产。

**建议：**

1. 灰度观察必须包含指标名、基线值、告警阈值、灰度比例、观察时长和回滚动作。
2. P1 行为变化未获产品确认时，审核结论应是 `BLOCKED`，不是“灰度观察”。
3. “可发布”必须附构建、测试、协议回归和人工语义审核证据。

## 三、失败根因

| 根因 | 表现 | 本质 |
|---|---|---|
| 产出导向替代正确性导向 | 21 套扫描、设计、计划快速完成 | 把文档数量当成进度 |
| Agent 自主权过大 | 待确认项不停下，AI 选择默认行为 | 人没有承担业务裁决责任 |
| 验证环境缺失仍继续编码 | 静态审核替代编译测试 | 没有硬门禁 |
| 任务按文档拆分，未按依赖拆分 | CandidateProvider 跨功能接入 | 真实变更边界失控 |
| 状态重复维护 | 指导、变更记录、结果审核互相不一致 | 缺少唯一事实源 |
| 审核偏重静态等价 | 有建议，无已执行测试证据 | 审核无法阻断错误 |

## 四、建议的恢复顺序

### 立即止损

1. 暂停继续结构性编码和发布。
2. 将 21 个功能统一重置为基于证据的状态：`未验证`、`已验证`、`BLOCKED`。
3. 优先验证已经发现过 P0 的入座链路，并审计未设计的 CandidateProvider 共享改造。
4. 关闭 6 个阻塞型业务决策；未关闭项不再使用 AI 默认值推进。

### 恢复最小工程闭环

```text
每个功能重新准入
├── 一份可审 spec：旧行为、目标行为、明确不做
├── 一组特征测试：锁旧行为和副作用
├── 一个独立 R 项：单职责小改
├── 一次真实验证：编译 + 测试 + 协议冒烟
├── 一个可回退 PR：证据齐全
└── 一次人工裁决：业务语义确认
```

建议先选择低风险的“好友备注”重新跑通完整闭环，再处理用户配置和风控数据上报，最后进入入座、带入、资金、重连和私人房审核链路。

### 建立硬门禁

| 阶段 | 必须满足 |
|---|---|
| 编码准入 | spec 已审、阻塞项关闭、测试设计完成 |
| 重构准入 | 特征测试已运行且通过 |
| PR 准入 | 编译通过、局部测试通过、协议回归通过、diff 单一 |
| 发布准入 | CI 全绿、业务确认完成、灰度与回滚可执行 |
| 完成 | spec、测试、实现、审核、变更记录状态一致 |

## 五、应该保留的有效资产

失败案例并非全部无效。以下资产值得保留并转入新的生产级流程：

- 21 个功能的入口、调用链、状态和协议扫描结果。
- P0/P1 问题证据、待确认项分类和跨负责人契约。
- 结果审核中对行为变化、协议兼容和相邻改动的识别。
- Git 变更记录中的 commit、影响范围和回退依赖。
- 已发现的具体缺陷，例如编译断裂、跨桌风险数据串用、NPE、event key 错配。

保留这些资产，但不要沿用“无编译测试也可继续实施”和“待确认项一律不停下”的执行规则。

## 六、一句话复盘

```text
这次失败不是因为 AI 产出太少，而是因为规格、测试、门禁和人工裁决没有真正限制 AI 的产出。
```
