> 对应分析文档：`docs/Junk-游戏准备重构/03-带入.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame + dx-game-guandan
> 分支：20260615/junk
> 复杂度：H（P0 高）
> 前置依赖：01-进入房间、02-入座（已审，复用 `SitDownStateService` 占座转入座表级锁 / `IOccupyResitHandler`）、10/16 私人房审核（审核单创建）、21 用户配置（自动补码目标值）

## 1. 背景与目标

带入覆盖 1010 `TRANSFER_IN`（`@ThreadModel(TABLE)`）、1011 `TRANSFER_IN_VIEW`（`@ThreadModel(PLAYER)`）、通知 14009 `USER_SCORE_CHANGE_NOTICE`、14022 `OCCUPY_ADD_CHIPS_NOTICE`，含普通钱包 `GUANDAN` + 彩金钱包 `GD_BONUS`、私人审核、自动补至目标值、结算补码、占座转入座。链路：`UserBringInCommand → UserBringInCommandDefault → UserBringInBootstrapService.check/transferInChipScore`，guandan 扩展 `UserBringInCommandBootstrap` / `UserBringInBootstrap`。基于真实代码确认：

- **P0-1：1010 单协议承载过多场景，缺显式业务意图**。`UserBringInBootstrap.check` 同时处理普通带入/彩金/私人审核/自动补至/结算补码；`UserBringInCommandBootstrap.isSettleComplementUser` 又靠 `TableHandStatus`/`playerIsContinue`/`needAddChipUsersMap` 二次识别。新增场景只能继续叠条件分支，互相覆盖风险高；线上无法从请求判断业务场景。
- **P0-2：钱包刷新与任务推进非事务边界**。`UserBringInDefault.transferInChipScore` 先 `saveTransferInChipScore` 再 `transferInAfterRefresh` 最后推占座转入座；占座转入座存在**散落的同步 `new OccupyToSitDownTask(...).run()`**（`UserBringInDefault.java:679`、`UserBringInBootstrap.java:356/865`）与 **异步 `eventManager.publish`**（`UserBringInDefault.java:187`）**并存**，时序不清。钱包上分成功但本地刷新/占座转入座失败时，可能扣款/上分了但桌内仍 OCCUPY。
- **P1-1：私人审核幂等依赖本地 Guava Cache**。`UserBringInDefault.java:89` `LoadingCache<String, AtomicBoolean> CACHE`（1h），guandan `UserBringInBootstrap` 另有 `AUDIT_CACHE`/`AUDIT_PASS_CACHE`。多节点/MQ 回放下本地缓存无法保证审核幂等终态。
- **P1-2：彩金字段沿用鱿鱼命名**。`BringInReviewDto.bringInSquidAmount`（`:110`）存放 `GD_BONUS` 金额，`AmountApplyBootstrap.java:56` 将彩金写入 `bringInSquidAmount`。语义误导 + 跨游戏复用治理成本高。
- **P2：静态取 Bean + 大对象日志**。`UserBringInBootstrap` 多处 `SpringContextUtils`/`SpringUtil`（D9，难测）；多处记录完整 req/reviewDto/钱包（D12）。

目标：**保持 1010/1011/14009/14022 协议与 DTO 100% 兼容**前提下，由服务端归一化业务场景、把"钱包结果→本地快照→用户状态→通知→开局唤醒"封装为有序 applier、占座转入座**统一走 02 已建的 `SitDownStateService` 表级锁**消除散落 `.run()`、审核幂等改 Redis 审核单/事件号、彩金金额内部封装钱包列表视图。**最小安全改造优先**，结构性拆分分级落地，避免与 02/05/17 冲突。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导编排骨架（`BringInFlowService`/`BringInResultApplier` 放 frame），guandan 提供场景/钱包/审核扩展点（彩金、私人审核、结算补码） |
| 协议兼容 | 1010 `addChipScore`/`walletList`、1011、14009、14022 字段不变；内部 `BringInScene` 不下发；14022 仅可新增可选字段 |
| 业务场景识别（P0-1） | 内部 `BringInScene { SIT_DOWN / MANUAL_IN_GAME / AUTO_TO_TARGET / SETTLEMENT_COMPLEMENT / PRIVATE_AUDIT / BONUS_COMPLEMENT }`，由 `BringInSceneResolver` 归一化；命令层只做 DTO 兼容（旧 addChipScore→walletList） |
| 钱包/状态/通知有序化（P0-2） | `BringInResultApplier`：钱包上分结果 → 本地钱包快照 → 用户状态 → 通知 → 开局唤醒，返回 `{walletApplied, stateApplied, noticeSent}`；主动 1010 与审核回调复用同一 applier |
| 占座转入座统一（P0-2） | **散落 `new OccupyToSitDownTask(...).run()` 统一改走 02 已建 `SitDownStateService.occupyToSitDown(tableId,userId)` 表级锁**（`IOccupyResitHandler`）。消除 `UserBringInDefault:679`/`UserBringInBootstrap:356/865` 的裸 run，与 02/`GdSitDownStateService` 一致 |
| 审核幂等（P1-1） | 幂等主键用 `BringInReviewDto.id` + Redis 审核单状态 CAS（或钱包 RPC 事件号）；本地 cache 仅性能优化，不作业务幂等判定 |
| 彩金字段（P1-2） | **外部 DTO `bringInSquidAmount` 字段不动**（协议/存储兼容）；内部建 `ReviewWalletAmount` 列表视图，把 `bringInSquidAmount` 映射为 `GD_BONUS`；新代码禁止直读 squid 承载彩金 |
| 状态转移 | `BringInStateTransition` 只允许显式转移：`OCCUPY→DEFAULT`、`DEFAULT→DEFAULT(nextHand 补码)`、`SETTLEMENT_WAIT→OCCUPY/DEFAULT`；复用 02 `UserItem.status` 语义，不重定义枚举 |
| 静态 Bean / 日志（P2） | 新代码构造器注入；req/reviewDto/钱包 toJson info → debug，info 仅 tableId/userId/scene/钱包类型/金额元数据 |
| 灰度策略 | 保留旧 `UserBringInCommandDefault` 入口，内部双写日志（场景/钱包计划/状态转移）先只读对比，按桌灰度切新 flow；审核回调先启 Redis CAS 再迁钱包执行 |

## 3. 整体架构

### 改造前

```
1010 UserBringInCommand(TABLE) → UserBringInCommandDefault.handleCommand(init/check/handle/after)
  ├─ init 旧 addChipScore→walletList 兼容
  ├─ check 普通/彩金/私审/自动补至/结算补码 全压一处 (P0-1)
  └─ handle → UserBringInBootstrapService.check + transferInChipScore
        └─ UserBringInDefault.transferInChipScore: save→refresh→占座转入座
              ├─ publish(OccupyToSitDownTask) :187   ┐ 时序不清 (P0-2)
              └─ new OccupyToSitDownTask(...).run() :679 ┘
guandan UserBringInCommandBootstrap(结算补码识别) / UserBringInBootstrap(GUANDAN+GD_BONUS/私审/.run():356,865)
  └─ AUDIT_CACHE/AUDIT_PASS_CACHE 本地幂等 (P1-1)；bringInSquidAmount 承载彩金 (P1-2)
```

### 改造后

```
1010 UserBringInCommand(TABLE, 不变)
  └─ UserBringInCommandDefault(入口保留)
       ├─ BringInCommandAdapter   — DTO 兼容(旧 addChipScore→walletList)，不含业务
       └─ BringInFlowService.execute(BringInCommand)
            ├─ BringInSceneResolver   → BringInScene (内部, 不下发)
            ├─ BringInValidatorChain  → 按场景组合校验(复用02入座状态结论)
            ├─ BringInWalletService   → upScore + 账变(钱包RPC事件号)
            ├─ BringInResultApplier   → 本地钱包快照→状态(BringInStateTransition)→通知→开局唤醒
            └─ BringInOccupyBridge    → 占座转入座走 SitDownStateService.occupyToSitDown(表级锁)
1011 UserBringInViewCommand(PLAYER, 不变) → 弹窗金额计算(复用 BringInSceneResolver 判定 needAudit/needAmountApply)
审核回调 TableTransferAuditConsumer/BringInReviewBizService
  └─ BringInAuditCallbackHandler — Redis 审核单 CAS 幂等 → 复用 BringInResultApplier
彩金: ReviewWalletAmount 视图(bringInSquidAmount→GD_BONUS), 外部字段不变
```

**兼容性原则**：
- 1010 `UserTransferInReqDto`(addChipScore/walletList)、`UserTransferinResDto`、1011 `UserTransferInViewRespDto`、14009、14022 字段不变。
- `UserBringInCommand`/`UserBringInCommandDefault`/`UserBringInBootstrapService` public 签名保留（内部委托）。
- `BringInReviewDto.bringInSquidAmount` 字段保留（审核单存储/RPC 兼容）。
- 占座转入座改走 `SitDownStateService.occupyToSitDown`，但 `OccupyToSitDownTask` 本身逻辑不变（02 已封装）。

## 4. 安全改造（R-01 ~ R-05）

### R-01：`BringInSceneResolver` 归一化业务场景（P0-1）
- 新增内部 `BringInScene` 枚举 + `BringInSceneResolver`：输入用户状态/桌状态/钱包列表/用户配置/审核单，输出场景；命令层 `init` 只做协议兼容。
- `UserBringInCommandBootstrap.isSettleComplementUser` 的二次识别收敛进 resolver，单一事实源。

### R-02：`BringInResultApplier` 有序化 + 占座转入座统一表级锁（P0-2）
- 新增 `BringInResultApplier`：固定顺序 钱包结果→本地钱包快照→用户状态(`BringInStateTransition`)→通知(14009/14022)→开局唤醒；返回 `{walletApplied, stateApplied, noticeSent}`，失败点明确。
- **占座转入座**：`UserBringInDefault:679`、`UserBringInBootstrap:356/865` 的 `new OccupyToSitDownTask(...).run()` 统一改为 `sitDownStateService.occupyToSitDown(tableId, userId)`（02 已建表级锁路径），消除散落裸 run 与 publish:187 的时序竞争。
- 主动 1010 与审核回调复用同一 applier。

### R-03：审核幂等改 Redis 审核单 CAS（P1-1）
- 幂等主键 `BringInReviewDto.id`；审核回调先做 Redis 审核单状态 CAS（PASS/REJECT/TIMEOUT 终态），成功才执行钱包上分。
- `UserBringInDefault.CACHE`、`UserBringInBootstrap.AUDIT_CACHE/AUDIT_PASS_CACHE` 降级为**性能优化**（命中即跳过），不再作为唯一业务幂等判定。

### R-04：彩金金额内部钱包列表视图（P1-2）
- 新增内部 `ReviewWalletAmount{walletType, amount}` 列表，`bringInAmount`→`GUANDAN`、`bringInSquidAmount`→`GD_BONUS` 映射在一处完成。
- 外部 `BringInReviewDto.bringInSquidAmount` 字段、审核单存储、RPC 不变；新代码只读 `ReviewWalletAmount`。

### R-05：构造器注入 + 日志脱敏（P2）
- 新增 service 用构造器注入；保留 `UserBringInBootstrap` 既有 `SpringContextUtils` 调用点为兼容（标注，后续迁移）。
- req/reviewDto/钱包 toJson info → debug；info 仅 tableId/userId/scene/walletType/amount 元数据。

| 维度 | 方案 |

| 新旧共存 | 保留 `UserBringInCommandDefault` 入口；新 flow 内部双写日志先只读对比 1010/1011 响应 + 钱包快照 |

| 运行中牌局兼容 | 占座转入座改走 02 表级锁，需回归占座/结算补码/游戏中补码链路 |
| 与其他 Junk 同步 | 占座转入座共用 02 `SitDownStateService`；放弃补码反向路径见 17；带出见 05；账单展示见 09 |

## 9. 验收关键场景

1. 旧客户端只传 `addChipScore` → 适配层转默认钱包，带入成功（兼容）。
2. 新客户端 `walletList` → GUANDAN/GD_BONUS 各按余额校验。
3. 占座用户带入成功 → 走 `SitDownStateService.occupyToSitDown` 表级锁，OCCUPY→DEFAULT，触发开局检查，移除占座超时任务。
4. 游戏中补码 → 只增 `nextBringInChipScore`，当前桌面筹码不立即变。
5. 私人审核通过 → GUANDAN+GD_BONUS 刷新 + 14009；**重复回调不重复上分**（Redis CAS）。
6. 审核 REJECT/TIMEOUT → 补码状态清理/恢复，后续站起或弹窗正确。
7. 1011 余额不足 → `needAmountApply=YES`。

## 11. 本功能不处理的范围

- 不重写 1010/1011/14009/14022 协议；不改钱包 RPC 账变模型。
- 不调整入房/入座/站起/退出公共状态枚举（复用 02）。
- 放弃补码协议缺失归 17；手动带出/退出自动带出归 05/06；审核单创建协议归 10/16（本功能只消费审核结果）。
- `OccupyToSitDownTask` 内部逻辑不改（02 已封装表级锁入口）。
