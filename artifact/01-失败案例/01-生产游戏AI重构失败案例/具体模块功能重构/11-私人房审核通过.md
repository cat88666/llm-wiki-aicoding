## 1. 功能边界

私人房审核通过不是客户端直接请求协议，而是审核系统/MQ/事件回调驱动的带入审核结果处理。它承接 1078 申请审核、占座带入和结算补码链路。

本功能负责:

- 校验审核结果消息。
- 处理 PASS/REJECT/TIMEOUT。
- 审核通过后增加私人房额度。
- 执行普通钱包和彩金钱包带入。
- 审核后衔接占座转入座、结算补码继续状态、失败通知。

本功能不负责:

- 审核单创建，归属 10。
- 固定带入首次入座触发审核，归属 16。
- 普通无审核带入，归属 03。

## 2. 入口与调用链

主要入口:

- `UserBringInDefault.dealBringInReviewDto` 是框架默认审核结果处理。
- `UserBringInBootstrap.dealBringInReviewDto` 是掼蛋覆盖实现，处理主钱包和 `GD_BONUS`。
- `BringInReviewBizService.auditExpire` 将超时审核单更新为 TIMEOUT 并发送审核 MQ。

掼蛋 PASS 链路:

1. `UserBringInBootstrap.dealBringInReviewDto` 读取审核单并做幂等判断。
2. `validateAuditMessage` 校验桌子和解散状态。
3. 清理用户审核字段和占座/审核任务。
4. PASS 时调用 `processGuandanAuditTransferIn`。
5. `processGuandanAuditTransferIn` 增加私人额度、执行 `upScore`、刷新本地钱包、占座转入座或结算继续状态。

REJECT/TIMEOUT:

- 清理审核状态。
- 结算补码场景调用 `handleSettleComplementAuditReject`，可能恢复补码、过级彩金或触发站起。

## 3. 数据流与状态

输入:

- `BringInReviewDto.id/status/tableId/userId/holderId/bringInAmount/bringInSquidAmount/amountApply`。

状态依赖:

- Redis 审核单状态。
- `BaseUserItem.transferAuditStartTime/bringInAuditExpireSecond`。
- `BaseUserItem.status` 是否仍为 OCCUPY。
- `TableHand.needAddChipUsersMap/playerIsContinue`。
- 钱包本地缓存和钱包 RPC 结果。

输出:

- 钱包上分。
- 1010 响应或审核相关通知。
- 14009 分数变化通知。
- 占座转入座任务。
- 结算继续通知。

## 4. 现有设计理解

审核通过链路本质上是“异步补码成功”。它和普通 1010 带入最终都要完成同一件事: 钱包上分、本地钱包刷新、用户状态推进、通知客户端。

掼蛋扩展复杂点在于审核单可能同时包含主筹码和彩金，并且审核回调可能发生在:

- 用户仍占座。
- 用户已不在桌。
- 结算补码等待。
- 审核拒绝/超时后需要重新触发补码或站起。

## 5. 问题扫描

### P0-1 审核幂等仍以本地缓存为主

证据:

- `UserBringInDefault` 使用本地 `LoadingCache<String, AtomicBoolean> CACHE`。
- `UserBringInBootstrap` 又定义 `AUDIT_CACHE` 和 `AUDIT_PASS_CACHE`。
- 审核单业务终态保存在 Redis，但处理幂等没有完全依赖 Redis CAS。

影响:

- 多节点重复消费同一审核通过消息时，仍可能重复处理钱包和状态。
- 服务重启后本地 cache 丢失，历史消息回放风险变高。

建议:

- 以审核单 ID 为幂等键，在 Redis 审核单状态上做 CAS: INIT -> PROCESSING -> PASS_APPLIED。
- 钱包 RPC eventId 与审核单 ID 关联，保障资金幂等。

### P0-2 审核通过的钱包执行和状态推进缺少统一补偿

证据:

- `processGuandanAuditTransferIn` 内执行额度增加、上分、钱包刷新、占座任务、结算唤醒。
- 部分占座转入座通过 `new OccupyToSitDownTask(...).run()` 同步执行。
- `auditPassUserNotExistsTable` 对用户不在桌场景有 TODO 或特殊处理。

影响:

- 审核通过但用户状态变化、钱包刷新或通知失败时，很难判断应该重试哪一步。
- 同步运行任务绕过队列，和占座超时任务并发关系不清晰。

建议:

- 审核通过输出 `AuditApplyPlan`，分步骤落状态: `quotaApplied`、`walletApplied`、`localStateApplied`、`noticeSent`。
- 每一步可幂等重试，失败进入补偿队列。

### P1-1 审核拒绝/超时和结算补码耦合过深

证据:

- `handleSettleComplementAuditReject` 同时移除站起/占座任务、调用 `SettleChipComplementService`、`LevelUpBonusHandler` 和 `UserLeaveBootstrapService.userStand`。
- 该方法位于 `UserBringInBootstrap`，但处理的是结算补码拒绝后的离开策略。

影响:

- 带入服务承担结算、彩金、站起职责。
- 改结算补码策略需要进入带入类修改。

建议:

- 将拒绝/超时交给 `SettleComplementAuditResultHandler`，带入服务只负责转发审核结果。

### P1-2 审核消息校验不足

证据:

- `UserBringInBootstrap.validateAuditMessage` 主要检查桌子和解散。
- 是否仍有申请人、审核单状态是否 INIT、金额是否匹配、holder 是否一致，分散在其他步骤或缺失。

影响:

- 脏消息、过期消息或错桌消息可能进入后续流程后才失败。

建议:

- 建 `AuditMessageValidator`，集中校验审核单状态、金额、桌子、房主、申请人、过期时间。

## 6. 架构审视

D1 本地缓存幂等: 资金相关审核不能依赖单节点 cache。
D6 资金事务: 额度增加、上分、本地钱包、用户状态不是同一事务。
D8 任务时序: 审核回调同步 run 占座转入座，和 EventManager 时序混用。
D11 职责耦合: 带入类同时处理审核、结算、彩金、站起。
D12 日志: 审核通过日志包含金额和用户信息，需要审计脱敏。

## 7. 改造建议

优先级:

1. P0: 审核结果处理改为 Redis CAS 幂等。
2. P0: 资金执行与状态推进拆成可补偿步骤。
3. P1: 拆 `AuditResultHandler`，按 PASS/REJECT/TIMEOUT 分发。
4. P1: 结算补码拒绝处理迁出带入服务。

## 8. 落地设计

建议类:

- `BringInAuditResultConsumer`
- `AuditMessageValidator`
- `AuditIdempotencyService`
- `AuditWalletApplyService`
- `AuditStateApplyService`

`AuditApplyPlan`:

- `auditId`

- `mainAmount`
- `bonusAmount`
- `quotaAmount`
- `userState`
- `applyMode`

## 9. 协议兼容确认

请求兼容:

- 本功能无客户端请求。
- 审核 MQ 消息字段保持。

响应兼容:

- 审核通过后产生的 1010 响应和 14009 通知字段保持。

通知兼容:

- `OCCUPY_ADD_CHIPS_NOTICE`、`USER_SCORE_CHANGE_NOTICE`、掼蛋继续状态通知保持协议号和字段。

## 10. 可测试性设计

单元测试:

- PASS 幂等重复消息。
- REJECT/TIMEOUT 幂等重复消息。
- 主钱包+彩金金额解析。
- 错桌、错 holder、过期审核单拒绝。
- 用户不在桌的审核通过处理。

集成测试:

- 1078 创建审核单后，PASS 回调完成上分和占座转入座。
- PASS 重复回调不重复上分。
- REJECT 后用户收到补码或站起处理。
- TIMEOUT 后审核单状态和 MQ 通知一致。

## 11. 迁移与灰度

迁移步骤:

1. 先新增 Redis CAS 幂等，只记录冲突不阻断。
2. 确认无误后启用 CAS 拦截重复处理。
3. 将 PASS 钱包执行迁到新服务。
4. 最后迁移 REJECT/TIMEOUT 结算补码处理。

回滚:

- CAS 状态保留，但可配置回退到旧 handler。
- 新补偿记录不影响旧协议。

## 12. 验收场景

- 审核通过后主钱包上分成功。
- 审核通过后彩金钱包上分成功。
- 占座用户审核通过后入座成功。
- 审核通过时用户已退出，不重复入座且资金处理有明确策略。
- 审核拒绝后不残留占座转入座任务。
- 审核超时后房主和用户侧状态一致。

## 13. 待确认项

- 审核通过但用户已离桌时，是自动上分再带出，还是作废审核。
- 审核单状态是否允许新增 PROCESSING/PASS_APPLIED。
- 钱包 RPC 是否支持以审核单 ID 做幂等。
- 拒绝/超时时用户是否必须立即站起。

## 14. 本功能不处理的范围

- 不创建审核单。
- 不改 1078 协议。
- 不改房主审核前端。
- 不改钱包 RPC 协议。
