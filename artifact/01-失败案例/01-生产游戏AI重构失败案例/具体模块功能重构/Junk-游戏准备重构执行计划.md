## 0. 目标与自主授权

**目标**：21 个游戏准备功能全部达到「设计 100% / 计划 100% / 实施 100% / 审核完成」，过程可追溯、可回退、协议零破坏、业务对 `origin/20260615/dev` 基线等价。

**自主授权（用户已明确）**：
- 允许在不逐个确认的情况下推进全部任务；执行时间长可接受。
- 待确认项**不阻塞**：按 §5 决策规则取安全默认 + 写入「待与产品确认」，继续推进，不停下等人。
- 本地 commit 自主进行（窄范围 add，见 §5.4）；**绝不 `git push`**（唯一需要用户点头的外向动作）。

**硬约束（红线，任何阶段不可破）**：
1. 协议号、请求 DTO、响应 DTO、通知 DTO 字段 **0 改动**（只允许新增可选字段，且需登记为「协议层观察项」）。
2. 业务逻辑对 `origin/20260615/dev` **等价**：同输入下 成功/失败/错误码/副作用 一致；新增的 `throw`（禁入等）视为业务变化，必须标注「待产品确认」。
3. `dx-game-frame` 只增不删改：可新增类/方法/扩展点、优化方法内部实现；**禁止**删除/重命名/改签名既有公共类/方法/字段。
4. 共享模型（`UserItem.status` / `TableHand` / `TableUserWallet` / 占座状态）不在单功能方案里单方重定义，跨功能影响写清楚。
5. 每次代码改动 → 一条 git 变更记录（§5.4 格式）。

## 5. 自主决策规则（无需用户确认的关键）

### 5.1 待确认项处理（不阻塞）
按 指导.md §5.4 分类，但**一律不停下**：

| 类型 | 自主处理 |
|---|---|
| 产品决策（如禁入是否生效、解散桌返回码变更） | 取**最兼容/零回归默认**，写入文档「待与产品确认」，继续 |
| 协议决策（废弃 vs 适配 vs 完整实现） | 默认「保留协议 + 适配到现有链路」；若明显废弃则占位返回不支持并标注，继续 |
| 技术实现细节 | 取保守实现（空值兜底、最小改动、复用既有扩展点），继续 |
| 测试口径 | 写成验收场景，不留疑问 |
| 跨负责人边界（Ceres/Torsen/Duik） | 只记调用契约，不替其重构 |

### 5.2 设计/编码决策优先级（拿不准时从上往下选）
1. 协议零改动 > 一切。
2. 对 dev 基线业务等价 / 零回归 > 「看起来更合理」。
3. frame 只增不删改；优先复用既有扩展点（`GameHandleFactory` 空间分发、`ValidatorChain`/Contributor、`StateService`/表级锁、`IOccupyResitHandler` 等）。
4. 共享模型不单方重定义。
5. 最小改动；能 additive 不重写。
6. 仍拿不准 → 保守 + 标注「待确认」，继续。

### 5.3 验证策略（无编译环境）
- 无 java/maven，**不能编译/跑测试**。替代闭环 = **dev 基线 diff + 编码审核Prompt**。
- 每个功能编码后**立即自审**：协议 0 改动核对、dev↔junk 业务等价核对、设计 R 项落地核对、并发临界区核对。
- 所有「验证结果」如实写「未编译，静态对比 dev 基线 + 编码审核，待 IDE 编译/回归」，审核状态按 dev 对比结论填。

### 5.4 提交策略
- **窄范围**：`git add <精确文件路径>`，**绝不 `git add -A` / `git add .`**（仓库有用户并行未提交改动，如 指导.md，禁止裹挟）。
- 一功能一阶段一 commit；commit message 关联序号 + R/S 编号；提交后补变更记录（编号 `GIT-YYYYMMDD-序号`，字段见 指导.md §七）。
- **不 push**。push 是唯一保留给用户的动作。

### 5.5 并行编辑共存
- 用户在并行编辑本仓库（`指导.md` 当前有未提交改动）。
- 共享文件（`指导.md`、`变更记录.md`）一律**精准 Edit 追加/改单格**，不整体 Write 重写；编辑前先 re-read 当前内容。
- 每个功能独立文件（`{n}-…`）由本计划负责创建/修改，冲突面小。

### 5.6 失败协议（三次）
1. 诊断根因，针对性修。2. 换方案（不重复同一失败动作）。3. 质疑假设/复扫代码/更新计划。3 次仍不通 → 在 `findings.md` 记录「阻塞」并**跳过该项继续其它任务**，最后统一汇报（不空转、不卡住整体）。
