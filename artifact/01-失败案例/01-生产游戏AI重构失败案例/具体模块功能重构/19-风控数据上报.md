## 1. 功能边界

风控数据上报覆盖 1040 `CommonProto.RISK_DATA_COLLECTION` 和 1041 `CommonProto.RISK_DATA_GRAPH`。

本功能负责:

- 客户端上报 IP、GPS、设备、语音、视频状态。
- 刷新本地风险数据缓存。
- 同步 IP/GPS 数据到大厅座位 Redis。
- 返回当前用户风险数据。
- 生成风控关系图，展示同 IP、GPS 接近、GPS 未开启等关系。

本功能不负责:

- 入座风控拦截、实时风控站起和风险通知，归属 08。
- 用户进入房间时的初始风控数据写入，归属 01。

## 2. 入口与调用链

1040 上报:

1. `RiskDataCollectionCommand` 使用 command 入口，委托 `RiskDataCollectionCommandDefault`。
2. `RiskDataCollectionCommandDefault.handleCommand` 调用 `UserRiskCache.refreshRiskData(req)`。
3. 读取 `UserRiskCache.getUserRiskData(tableId,userId)` 作为响应。
4. `syncRiskDataToRedis` 读取 `BaseTableInfoCache` 和风险数据，构造 `RiskDataDto`，调用 `HallTableUserSeatRedisService.syncRiskData`。
5. 发送 1040 响应。

1041 关系图:

1. `RiskDataGraphCommand` 委托 `RiskDataGraphCommandDefault`。
2. 读取 `UserRiskCache.getTableRiskData(tableId)`。
3. `getUserRiskGraph` 按桌配置计算用户之间的风险关系。
4. 结合 `BaseTableUserCache.PLAYER_USER`、`UserBaseInfoCache`、`UserFriendCache` 组装 `RiskDataGraphRespDto`。
5. 发送 1041 响应。

## 3. 数据流与状态

1040 请求:

- `RiskDataCollectionReqDto.ip`
- `latitude/longitude`
- `deviceType`
- `gpsImposeSwitch`
- `realTimeVoiceSwitch`
- `realTimeVideoSwitch`

缓存:

- `UserRiskCache.USER_RISK_DATA_CACHE`: 以 userId 为 key，3 小时过期。
- 注意 key 不是 tableId+userId，跨桌风险数据会复用同一个用户缓存对象。

1041 响应:

- `RiskDataGraphRespDto.riskPlayerList`
- `UserRiskInfoDto.riskGraphList`
- `UserRiskInfoWrapperDto.typeList/distance/chairId/userId`

## 4. 现有设计理解

1040 是客户端持续上报，1041 是按桌查询关系图。实际风控计算也读取同一份 `UserRiskCache`，因此上报质量直接影响入座拦截和实时风控。

当前实现把数据缓存按 userId 存储，而 `getUserRiskData(tableId,userId)` 在读取时再结合 tableInfo 判断数据是否超时和 GPS 是否置零。

## 5. 问题扫描

### P0-1 风险数据缓存 key 只有 userId，缺少 table 维度

证据:

- `UserRiskCache.USER_RISK_DATA_CACHE` 类型为 `LoadingCache<Long, RiskDataCollectionDto>`。
- `initRiskDataEnterRoom` 和 `refreshRiskData` 都按 `req.getUserId()` 读取缓存。
- `getUserRiskData(tableId,userId)` 再读取 tableInfo 做超时处理。

影响:

- 同一用户跨桌或换桌后，IP/GPS/设备数据可能串用。
- 不同桌子的 GPS 强制开关和超时策略不同，但缓存对象共享。

建议:

- 风控缓存 key 改为 `tableId:userId`，或在 DTO 中记录 tableId 并校验。
- 入房离桌时清理当前桌用户风险缓存。

### P0-2 1040 同步 Redis 缺少空值保护

证据:

- `RiskDataCollectionCommandDefault.syncRiskDataToRedis` 直接读取 `tableInfo.getRoomCode()`、`tableInfo.getTableCode()`。
- 直接读取 `riskData.getIp()`、`getLatitude()`、`getLongitude()`。

影响:

- tableInfo 或 riskData 异常时，1040 整体返回系统异常。
- 客户端上报缺 GPS 时，Redis 同步可能失败，进而影响大厅展示。

建议:

- 将本地缓存刷新和 Redis 同步拆开；Redis 同步失败不影响 1040 主响应，但要有指标和重试。
- tableInfo 缺失返回明确业务错误。

### P1-1 1041 关系图计算对 GPS 空值敏感

证据:

- `RiskDataGraphCommandDefault.getUserRiskGraph` 在 GPS 开启时直接 `riskData.getLatitude().doubleValue()`。
- `isNearby` 也直接读取经纬度。

影响:

- 任何一个用户缺少经纬度都会使关系图计算异常；当前 catch 后返回部分或空关系。
- 关系图异常不一定影响 1041 成功响应。

建议:

- GPS 规则先校验经纬度完整性。
- 关系图响应中标注用户 GPS 数据缺失的风险类型，而不是异常吞掉。

### P1-2 1041 只计算 IP/GPS，和实时风控类型不完全一致

证据:

- `RiskDataGraphCommandDefault.getUserRiskGraph` 只判断 IP 相同、GPS 相近、GPS 未开启。
- `BaseTableRiskService.checkRiskData` 还覆盖实时视频和设备。

影响:

- 风控关系图展示和实时风控结果不一致，用户可能被设备/视频风控站起但关系图没有体现。

建议:

- 关系图复用 08 的 `RiskRuleEngine` 结果，至少在 UI 中补设备/视频风险节点。

## 6. 架构审视

D1 缓存维度: 风险数据按 userId 缓存，和桌维度风控不匹配。
D6 外部同步: 1040 同步 Redis 与主响应耦合。
D7 数据质量: 缺 GPS 数据没有结构化错误。
D12 隐私日志: 1040/1041 日志输出 IP、GPS，必须脱敏或采样。

## 7. 改造建议

优先级:

1. P0: 风险缓存加入 table 维度。
2. P0: 1040 主响应与 Redis 同步解耦。
3. P1: 1041 关系图统一使用风险规则引擎。
4. P1: 风控数据日志脱敏。

通知兼容:

## 10. 可测试性设计

单元测试:

- 同一用户不同 table 风险数据互不污染。
- 缺 tableInfo 的 1040。
- Redis 同步失败不影响主响应。
- GPS 空值关系图不 NPE。
- IP/GPS/设备/视频风险规则一致。

集成测试:

- 1040 上报后入座风控能读取最新数据。
- 1041 返回同 IP 和 GPS 接近关系。
- 换桌后旧桌风险数据不影响新桌。

## 11. 迁移与灰度

迁移步骤:

1. 新旧缓存双写: userId key 和 tableId:userId key。
2. 风控计算影子读取新 key，对比结果。
3. 灰度切换读新 key。
4. 清理旧 key。

回滚:

- 保留旧 userId key 一段时间。
- Redis 同步解耦可独立回滚。

## 12. 验收场景

- 客户端上报 IP/GPS 后 1040 返回更新后的数据。
- 1041 展示同 IP 用户。
- GPS 未开启用户被标记。
- GPS 缺失不导致 1041 系统异常。
- 换桌后风险数据不串桌。
- Redis 同步失败有告警但 1040 可控。

## 13. 待确认项

- 关系图是否要展示设备/视频风险。
- 1040 缺经纬度时客户端期望成功还是错误。
