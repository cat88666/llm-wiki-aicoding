## 1. 功能边界

牌桌状态查询覆盖 1025 `CommonProto.TABLE_STATUS`，用于客户端进入或刷新前查询桌子是否可用、是否需要密码以及轻量配置。

本功能负责:

- 查询牌桌状态。
- 解散桌、缺失桌初始化。
- 排队中用户拦截。
- PC 设备拦截。
- 入房权限轻量检查。
- 返回 `TableStatusRespDto`。

本功能不负责:

- 真正进入房间和数据组装，归属 01。
- 密码校验，归属 12。
- 断线重连数据恢复，归属 14。

## 2. 入口与调用链

入口:

1. `TableStatusCommand` 使用 `@ThreadModel(PLAYER)`，`getType` 返回 `CommonProto.TABLE_STATUS`。
2. `TableStatusCommandDefault.handleCommand` 先拦截 PC 设备。
3. 查询 `RetryLineUpRedisService.getUserLineUp`，排队中返回 `HALL_LINE_UP_ERROR`。
4. `initContext` 若本地无 table，调用 `GameTableBootstrapService.init` 初始化。
5. `check` 按大厅/私人/俱乐部调用不同 `BaseUserCheckAuthService`。
6. `refreshUser` 异步刷新用户基础信息和配置。
7. `after` 返回 `TableStatusRespDto`。

## 3. 数据流与状态

请求:

- `TableStatusReqDto.encryptKey`: 可选，用于 ASD 加密响应。
- 基础请求含 `tableId/userId/deviceType`。

响应:

- `TableStatusRespDto.status`
- `TableStatusRespDto.isPassword`
- `TableStatusRespDto.tableSimpleCfgDto`
- `TableStatusRespDto.asdDto`
- `lastNoticeTimeStamp/appVersionStr` 当前 DTO 有字段但默认链路未明显设置。

状态:

- `BaseTableCache.checkDisbandTable`
- `BaseTableCache.getTable`
- `BaseTableInfoCache.getTableInfo`
- `GameStatusEnum.getByCode(table.getTableStatus())`

## 4. 现有设计理解

1025 是 1000 入房前的轻量状态查询，但它并不是纯读接口。当前实现会在本地 table 缺失时尝试初始化牌桌，并会刷新用户基础信息和配置缓存。

这使 1025 同时承担“查询状态”和“准备进入房间的预热”。

## 5. 问题扫描

### P0-1 查询接口可能触发桌子初始化

证据:

- `TableStatusCommandDefault.initContext` 在 `BaseTableCache.getTable(tableId)` 为空时调用 `GameTableBootstrapService.init(req.getTableId(), req.getUserId())`。

影响:

- 客户端刷新状态可能带来重初始化副作用。
- 初始化失败返回 `TABLE_USER_STATUS_ERROR`，不一定能准确表达桌不存在、解散、初始化失败。

建议:

- 明确 1025 是否允许初始化。如果允许，响应/日志中记录 `initialized=true`。
- 纯查询和初始化预热拆成两个内部步骤，便于监控。

### P1-1 私人房和俱乐部判断存在 else 归属歧义

证据:

- `check` 中先 `if (tableInfo.isHallTable())`，再 `if (tableInfo.isPrivateTable()) ... else ...`。
- 当 hall table 不是 private table 时，会继续进入 else 的俱乐部校验。

影响:

- 大厅桌可能同时执行大厅校验和俱乐部校验，取决于 `isHallTable/isPrivateTable` 的实现。

建议:

- 改成互斥空间枚举 `HALL / PRIVATE / CLUB / RACE`，只执行一种校验链。

### P1-2 响应状态枚举来源不统一

证据:

- `TableStatusEnum` 定义 GAME_WAIT/GAME_STARTING/GAME_END/GAME_DISBAND_MIDDLE。
- `TableStatusCommandDefault` 使用的是 `GameStatusEnum.getByCode(table.getTableStatus())`。
- 掼蛋入房 DTO `TableInfoEnterRoomDto.tableStatus` 取 `tableInfo.getStatus()`。

影响:

- 1025、1000、桌缓存状态可能使用不同枚举或字段来源。
- 客户端刷新状态与进入房间后状态可能不一致。

建议:

- 建 `TableStatusView` 统一转换，对外协议保留原 code。

### P1-3 `refreshUser` 在查询接口中有异步副作用

证据:

- `refreshUser` 在用户不在房间时异步刷新 `UserBaseInfoCache`，同步读取 `UserConfigCache`。

影响:

- 1025 性能受用户配置 RPC 影响。
- 查询接口被频繁调用时可能放大缓存刷新压力。

建议:

- 对刷新动作加限流和指标。
- 将预热职责迁到入房 flow 或独立 prewarm 服务。

## 6. 架构审视

D1 本地缓存: 查询状态依赖本地 table 和 tableInfo，缺失时触发初始化。
D5 状态来源: table status、tableInfo status、DTO status 来源不统一。
D6 外部依赖: 查询接口间接触发用户配置 RPC。
D7 空间策略: 大厅/私人/俱乐部不是互斥结构。

## 7. 改造建议

优先级:

1. P0: 明确并拆分“查询”和“初始化预热”。
2. P1: 建 `GameSpaceResolver`，空间校验互斥。
3. P1: 建 `TableStatusViewAssembler` 统一状态来源。
4. P1: 用户预热增加限流。

## 8. 落地设计

建议类:

- `TableStatusFlowService`
- `TableStatusSnapshotLoader`
- `TableStatusPermissionChecker`
- `TableStatusViewAssembler`
- `TableStatusPrewarmService`

处理顺序:

1. 读取排队状态。
2. 读取或初始化 table snapshot。
3. 根据 game space 执行权限检查。
4. 组装响应。
5. 异步预热用户信息。

## 9. 协议兼容确认

请求兼容:

- 1025 `encryptKey` 保持可选。

响应兼容:

- `status/isPassword/tableSimpleCfgDto/asdDto` 保持。
- 如果新增 `initialized`，必须可选。

错误兼容:

- PC 设备、排队、权限错误继续返回现有错误码。

## 10. 可测试性设计

单元测试:

- table 已存在。
- table 不存在但初始化成功。
- table 不存在且初始化失败。
- 大厅/私人/俱乐部互斥校验。
- 排队中用户拦截。
- encryptKey 生成 ASD。

集成测试:

- 1025 成功后 1000 入房状态一致。
- 解散桌返回终局错误。
- 用户配置预热失败不影响状态响应。

## 11. 迁移与灰度

迁移步骤:

1. 影子生成 `TableStatusView`，对比旧响应。
2. 灰度启用互斥空间校验。
3. 拆分初始化预热日志指标。
4. 最后控制是否保留查询初始化能力。

回滚:

- 关闭新 view assembler，回到旧 default。
- 空间校验保留旧 if 逻辑。

## 12. 验收场景

- 有效桌返回状态和轻量配置。
- 解散桌返回不可进入。
- 排队用户不能直接进入桌。
- PC 设备返回未开放错误。
- 密码房 `isPassword=true`。
- ASD encryptKey 请求返回 asdDto。

## 13. 待确认项

- 1025 是否应当初始化不存在的 table。
- 大厅桌是否可能同时需要俱乐部校验。
- `lastNoticeTimeStamp/appVersionStr` 是否仍需返回。
- `TableStatusEnum` 和 `GameStatusEnum` 是否应统一。

## 14. 本功能不处理的范围

- 不组装入房完整数据。
- 不处理密码校验。
- 不处理重连阶段。
- 不修改客户端状态码。
