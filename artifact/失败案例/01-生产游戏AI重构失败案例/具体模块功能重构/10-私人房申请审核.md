## 1. 功能边界

私人房申请审核覆盖 1078 `CommonProto.AMOUNT_APPLY`，用于私人审核房中玩家申请额度并生成待房主审核的补码/带入审核单。

本功能负责:

- 接收额度申请请求。
- 校验重复申请和审核单是否过期。
- 计算申请额度 `amountApply`。
- 生成 `BringInReviewDto` 并写入 Redis。
- 发送审核 MQ。
- 触发占座补码弹窗通知事件。

本功能不负责:

- 审核通过后的钱包上分和状态推进，归属 11。
- 固定带入首次入座自动触发审核，归属 16。
- 带入金额弹窗计算，归属 03。

## 2. 入口与调用链

入口:

1. `AmountApplyCommand` 使用 `@ThreadModel(TABLE)`，`getType` 返回 `CommonProto.AMOUNT_APPLY`。
2. 命令通过 `GameBaseBootstrapFactory.getDataBoot(AbstractAmountApplyCommand.class)` 获取实现。
3. 框架 `AmountApplyDefault.handleCommand` 执行 `check -> amountApplyStore -> dataHandler -> sendMq -> after -> notice`。
4. 掼蛋 `AmountApplyBootstrap` 继承 `AmountApplyImpl`，重写 `amountApplyStore` 以同时记录主筹码和彩金。

默认处理:

- `AmountApplyImpl.check` 先按桌子配置计算 `amountApply`，非房主校验私有记分钱包余额加申请额度是否覆盖本次带入。
- `AmountApplyDefault.check` 查询 `RetryBringInReviewRedisService.selectTableUserAuditIng`，未过期审核单存在时抛 `COMMON_TRANSFER_AUDIT_DUPLICATE`。
- `AmountApplyDefault.dataHandler` 发布 `BringInReviewEvent`。
- `AmountApplyDefault.sendMq` 发送私人房审核 MQ。
- `AmountApplyDefault.notice` 发布 `AmountApplyUserSitDownNoticeEvent`。

## 3. 数据流与状态

请求:

- `AmountApplyReqVO.amountApply`: 客户端申请额度，部分场景会被服务端按桌配置覆盖。
- `AmountApplyReqVO.walletList`: `WalletAmountReqVO.walletType/addChipScore`。

审核单:

- `BringInReviewDto.id`: 审核单唯一 ID。
- `holderId/tableId/userId`: 房主、桌子、申请人。
- `status`: INIT/PASS/REJECT/TIMEOUT/INVALID。
- `amountApply`: 本次申请额度。
- `bringInAmount`: 主筹码带入金额。
- `bringInSquidAmount`: 掼蛋中复用为 `GD_BONUS` 彩金金额。
- `bringInTime/bringInAuditExpireSecond`: 审核有效期。

存储与消息:

- Redis: `RetryBringInReviewRedisService.addOrUpdate`。
- MQ: `CommonRocketMqProvider.sendPrivateTableAuditMq`。
- Spring event: `BringInReviewEvent`、`AmountApplyUserSitDownNoticeEvent`。

## 4. 现有设计理解

现有申请审核链路是“先建审核单，再让房主/大厅审核系统处理”。协议 1078 本身不执行上分，也不改变玩家最终入座状态，只把用户留在占座/审核流程中。

掼蛋扩展的核心变化是: 同一次审核申请可能包含普通钱包和 `GD_BONUS` 彩金钱包，因此 `AmountApplyBootstrap.amountApplyStore` 用钱包 map 分别取主钱包和彩金金额。

## 5. 问题扫描

### P0-1 申请审核没有显式校验用户仍在可申请状态

证据:

- `AmountApplyDefault.amountApplyStore` 直接 `BaseTableUserCache.getUser(req.getTableId(), req.getUserId())` 后构建审核单。
- `CommonUserBringInService.buildReviewDto` 继续读取用户基础信息、桌信息、`UserTableItemCache` 和钱包。
- `AmountApplyDefault.check` 重点检查重复审核单，并未显式确认申请人仍在房间、仍处于 OCCUPY 或补码流程。

影响:

- 用户退出、站起或缓存短暂缺失时仍可能尝试建审核单，失败表现为 NPE 或系统异常。
- 审核单和桌内状态可能不一致。

建议:

- 增加 `AmountApplyEligibilityValidator`，校验 tableInfo、userItem、user area、user status、wallet list。
- 不满足申请条件时返回明确业务异常，不进入建单。

### P0-2 审核单创建、事件、MQ 非原子

证据:

- `AmountApplyDefault.handleCommand` 依次执行 `amountApplyStore`、`dataHandler`、`sendMq`、`after`、`notice`。
- Redis 写成功后，Spring event 或 MQ 发送失败会进入异常并返回错误，但审核单已经存在。

影响:

- 客户端看到申请失败，但房主侧可能稍后收到审核单，或 Redis 中已有待审核单导致下一次申请重复。
- MQ 成功但通知失败时，前端占座弹窗状态和审核状态不一致。

建议:

- 建 `AmountApplyTransactionRecord`，记录 `stored/eventPublished/mqSent/noticeSent`。
- 失败时按审核单 ID 发送补偿任务或明确返回“申请已受理，通知补发中”。

### P1-1 `amountApply` 服务端覆盖规则不透明

证据:

- `AmountApplyDefault.getAmountApply` 对非房主按固定/非固定带入模式覆盖申请额度。
- 房主则按 `walletList` 金额求和。
- `AmountApplyImpl.check` 再用 `userCashBalance + amountApply < totalBring` 校验。

影响:

- 客户端传入的 `amountApply` 不一定生效，排查申请金额时需理解桌配置。
- 主钱包和彩金一起申请时，总带入和额度申请的关系不直观。

建议:

- 内部新增 `AmountApplyCalcResult`，记录 `clientAmount/serverAmount/calcReason/totalBring/mainAmount/bonusAmount`。
- 响应审核单保持兼容，但日志和测试使用 calc result。

### P1-2 掼蛋彩金仍使用 `bringInSquidAmount`

证据:

- `AmountApplyBootstrap.amountApplyStore` 将 `DxWalletType.GD_BONUS` 写入 `BringInReviewDto.bringInSquidAmount`。

影响:

- 字段命名和掼蛋业务不匹配，审核侧理解成本高。

建议:

- 外部字段保持，内部使用 `reviewWalletAmounts` 视图映射。

## 6. 架构审视

D1 Redis 与本地缓存: 审核单在 Redis，申请资格来自本地桌缓存，两者无统一事务。
D6 事件一致性: Redis、Spring event、MQ、响应、通知分步执行。
D7 错误语义: 建单后通知失败仍可能返回系统异常，用户感知不准确。
D11 DTO 复用: `BringInReviewDto` 同时承载德州、鱿鱼、掼蛋彩金。
D12 日志: 审核单包含用户、商户、金额，必须脱敏和分级。

## 7. 改造建议

优先级:

1. P0: 加申请资格校验器。
2. P0: 为建单后失败引入补偿状态或补发机制。
3. P1: 抽 `AmountApplyCalculator`，记录金额覆盖原因。
4. P1: 用内部钱包金额列表替代直接读写 `bringInSquidAmount`。

## 8. 落地设计

建议类:

- `AmountApplyFlowService`
- `AmountApplyEligibilityValidator`
- `AmountApplyCalculator`
- `BringInReviewFactory`
- `AmountApplyEventPublisher`
- `AmountApplyNoticeCoordinator`

`AmountApplyCommand`:

- 输入 `AmountApplyReqVO`。
- 输出 `AmountApplyResult`，包含审核单和各步骤状态。

## 9. 协议兼容确认

请求兼容:

- 1078 `AmountApplyReqVO.amountApply/walletList` 保持。

响应兼容:

- 1078 仍返回 `BringInReviewDto`。
- 不新增必填字段。

通知兼容:

- 审核申请 MQ 消息结构保持。
- `OCCUPY_ADD_CHIPS_NOTICE` 由后续通知链路保持。

## 10. 可测试性设计

单元测试:

- 重复未过期审核单拒绝。
- 过期审核单允许重新申请。
- 固定带入和非固定带入的 `amountApply` 计算。
- 房主申请额度按钱包金额求和。
- 主钱包+彩金生成审核单。

集成测试:

- 申请成功后 Redis 有 INIT 审核单，MQ 发出。
- Redis 写成功但 MQ 失败时可补偿。
- 用户已退出时申请返回明确业务异常。

## 11. 迁移与灰度

迁移步骤:

1. 新资格校验只记录不拦截，观察线上命中。
2. 开启建单补偿记录。
3. 灰度启用资格拦截。
4. 最后切换金额计算器。

回滚:

- 关闭新校验器。
- 保留旧 `AmountApplyDefault` 步骤顺序。

## 12. 验收场景

- 非房主余额不足时，1078 返回参数/额度异常。
- 已有未过期审核单时，返回重复申请错误。
- 审核单过期后可重新申请。
- 主钱包和彩金金额都进入审核单。
- 申请成功后房主侧收到审核 MQ。
- 申请成功后用户收到占座补码弹窗或额度申请入口。

## 13. 待确认项

- 1078 建单成功但 MQ 失败时客户端应看到成功还是失败。
- 非房主 `amountApply` 覆盖是否需要在响应中展示给前端。
- 彩金审核金额是否需要对外新增独立字段。
- 用户不在 OCCUPY 但仍在房间时是否允许额度申请。

## 14. 本功能不处理的范围

- 不处理审核通过后的上分。
- 不处理固定带入自动触发场景。
- 不改审核 MQ 协议。
- 不改钱包 RPC。
