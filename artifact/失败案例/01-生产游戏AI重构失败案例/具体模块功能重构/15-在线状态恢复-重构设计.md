> 对应分析文档：`docs/Junk-游戏准备重构/15-在线状态恢复.md`
> 设计日期：2026-06-04
> 工程：dx-game-guandan（3016/16019 为掼蛋协议）
> 分支：20260615/junk
> 复杂度：M（P1 中）
> 前置依赖：14-旁观返回重连（复用"重连后补偿"语义）
> 与 14 关系：14=重连数据恢复，15=在线状态恢复；分开

## 1. 背景与目标

在线状态恢复覆盖掼蛋 3016 `GD_USER_ONLINE_STATUS_QUERY`（`@ThreadModel(TABLE)`）、16019 `USER_ONLINE_STATUS_QUERY_NOTICE`，是**补偿协议**：重连后主动拉取并补发桌内在线状态，避免短时间看到旧状态。主链路在线状态由框架 `BaseUserOnlineService` 按心跳计算并发 14023 `USER_ONLINE_STATUS_NOTICE`。基于真实代码确认：

- **P0-1：查询缺空值保护**。`GdUserOnlineStatusQueryCommand`（`:52-54`）`UserTableItem userTableItem = UserTableItemCache.getUserTableItem(tableId,userId)` 后直接 `userTableItem.getUserId()`/`getOnlineStatus()`；遍历桌内用户（`:57-58`）同样直接 `currentUserTableItem.getUserId()`。重连用户刚退出 / 缓存缺失时 3016 系统异常；**一个用户缓存缺失会中断本次所有在线状态补发**（虽有外层 catch，但整批补发失败）。
- **P1-1：16019 单人通知 vs 整桌快照语义不清**。命令先 `SendUtil.send` 返回当前用户状态，再对 `tableUserIds` 循环、对每个 `currentUserId` 发送**其自己**的 `UserOnlineStatusMsgDto`（`:57-60`）。每个玩家只收到自己的状态，不是整桌快照；若前端期望"请求后恢复所有人状态"，当前语义不足。
- **P1-2：只覆盖 PLAYER_USER 不覆盖旁观**。`fetchTableUserIds(tableId, PLAYER_USER)`（`:48`）；旁观者重连后在线状态恢复不在 3016 范围。

目标：**3016 请求、`UserOnlineStatusMsgDto`、16019、14023 字段 100% 兼容**前提下，加 `UserTableItem` 缺失兜底（修 P0-1：当前用户缺失返回明确业务错误或默认状态，遍历跳过缺失项 + 指标）、明确 16019 补偿范围（单人 vs 整桌，待产品）、在线状态组装抽 `OnlineStatusSnapshotService`。**最小安全改造优先**。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Guandan 主导（3016/16019 掼蛋协议） |
| 协议兼容 | 3016 请求、`UserOnlineStatusMsgDto`、16019、14023 字段不变 |
| 空值兜底（P0-1） | 当前用户 `UserTableItem` 缺失 → 返回明确业务错误或默认在线状态（待产品定，§10）；遍历其他用户跳过缺失项 + 记指标，不中断整批补发 |
| 16019 补偿范围（P1-1） | **本期保持现状语义**（每玩家收到自己状态）；是否需整桌快照待产品确认；如需，新增**可选**列表字段或对请求用户循环发各玩家状态，保持旧字段兼容 |
| 覆盖范围（P1-2） | 本期保持只恢复坐下玩家（PLAYER_USER）；旁观者是否纳入待产品；如需扩 `PLAYER_USER+WATCH_USER` 保旧字段兼容 |
| 组装抽取 | `OnlineStatusSnapshotService` 输出 `OnlineStatusSnapshot{requestUserStatus,playerStatuses,watchStatuses,missingUserIds}`；`OnlineStatusNoticeService` 统一发送 |
| 灰度策略 | 先加空值兜底 + 日志指标 → 灰度整桌快照（若产品要） → 评估旁观范围 |

## 3. 整体架构

### 改造前

```
3016 GdUserOnlineStatusQueryCommand(TABLE)
  ├─ tableUserIds = fetchTableUserIds(PLAYER_USER) :48 (不含旁观 P1-2)
  ├─ userTableItem = getUserTableItem(tableId,userId) :52 → getUserId/getOnlineStatus :54 (无空保护 P0-1)
  ├─ SendUtil.send(当前用户状态) :55
  └─ for currentUserId in tableUserIds:
        currentUserTableItem.getUserId/getOnlineStatus :58 (无空保护 P0-1)
        sendNotice(currentUserId, 16019 其自己状态) :60 (单人语义 P1-1)
主链路: BaseUserOnlineService.handle → checkOnlineStatus(心跳) → UserTableItem.onlineStatus → 14023
```

### 改造后

```
3016 GdUserOnlineStatusQueryCommand(TABLE, 不变)
  └─ GdOnlineStatusQueryService.query(tableId, userId)
       ├─ OnlineStatusSnapshotService → OnlineStatusSnapshot(requestUserStatus/playerStatuses/missingUserIds)
       │     当前用户缺失 → 业务错误或默认状态(R-01); 遍历缺失跳过+指标
       ├─ SendUtil.send(3016 当前用户状态)
       └─ OnlineStatusNoticeService → 16019(现状单人语义保持; 整桌快照可选 R-02)
14023 框架在线状态通知不变
```

**兼容性原则**：
- 3016 请求、`UserOnlineStatusMsgDto`（userId/onlineStatus 0在线/1网络不佳/2掉线）、16019、14023 字段不变。
- `GdUserOnlineStatusQueryCommand` 入口保留（内部委托 service）。
- 不改心跳阈值 / `BaseUserOnlineService` / 14023。

## 4. 安全改造（R-01 ~ R-02）

### R-01：`UserTableItem` 缺失兜底（P0-1）
- 当前用户 `UserTableItem==null` → 返回明确业务错误或默认在线状态（待产品定）；遍历其他用户跳过缺失项 + warn + 指标，**单个缺失不中断整批补发**。
- 抽 `OnlineStatusSnapshotService`，缺失项记入 `OnlineStatusSnapshot.missingUserIds`。

### R-02：16019 补偿范围明确（P1-1，待产品）
- 本期保持现状（每玩家收到自己状态）；文档明确 16019 语义。
- 若产品要求"请求后整桌恢复"：新增**可选**列表字段或对请求用户循环发各玩家状态，**保旧字段兼容**。

> P1-2（旁观范围）：本期文档明确 3016 只恢复坐下玩家；旁观纳入待产品，扩 WATCH_USER 时保旧字段兼容。

## 7. 可测试性设计

- `OnlineStatusSnapshotServiceTest`：当前用户在线/网络不佳/掉线；当前用户 UserTableItem 缺失；桌内某玩家缺失（跳过+missingUserIds）；空玩家列表。
- 集成：重连后 3016 当前用户收到响应；桌内玩家收到 16019；心跳变化后读最新 onlineStatus；3016 与 14023 口径一致。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | 先加空值兜底 + 指标 → 灰度整桌快照(若产品要) → 评估旁观范围 |
| 回滚 | 保留逐玩家通知行为；snapshot 内部结构不进协议 |
| 与其他 Junk 同步 | 复用 14 重连后补偿语义；不改 14023/心跳 |

## 9. 验收关键场景

1. 在线玩家 onlineStatus=0；网络不佳=1；掉线=2。
2. 重连后 3016 主动刷新当前用户状态。
3. **当前用户 UserTableItem 缺失 → 明确业务错误/默认状态，不系统异常（P0-1）**。
4. **桌内某玩家缺失 → 跳过该项继续补发，不中断整批（P0-1）**。
5. 3016 与 14023 状态口径一致。
