# 旁观 / 返回 / 重连重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/14-旁观返回重连.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame + dx-game-guandan（重连数据组装在 guandan 为主）
> 分支：20260615/junk
> 复杂度：H（P0 高）
> 前置依赖：01-进入房间（已建 `GdReconnectStateService.recover→restartLoadHand`，重连作为 1000 入房 alreadyInRoom 分支）、02-入座、12-进入房间前校验
> 与 15 关系：14=重连**数据**恢复（手牌/结算/占座），15=在线**状态**恢复（心跳/onlineStatus）；**分开设计**，14 提供阶段/快照基础，15 复用其"重连后补偿"语义

## 1. 背景与目标

旁观/返回/重连覆盖普通入房成旁观、用户返回、断线重连、准备/游戏中/结算阶段数据恢复。基于真实代码确认：

- **P0-1：1012/1013 协议号存在但无命令实现**。`CommonProto.USER_BACK=1012`、`USER_RECONNECT=1013` 定义，但两工程无 `UserBackCommand`/`ReconnectCommand`；掼蛋重连数据实际通过 **1000 入房响应** 的 `TableHandEnterRoomDto`/`TableUserHandEnterRoomDto` 承载（`TableUserHandBootstrap` 组装，01 的 `GdReconnectStateService.recover` 在 alreadyInRoom 分支调 `restartLoadHand`）。客户端若发 1012/1013 可能无处理，协议与链路不一致。
- **P0-2：重连判断依赖 `tableUserHandMap` 非空，可能误判**。`determineIsReconnect` 中 `tableUserHandMap` 非空直接返回 true；WAIT/SETTLE_END 仅在 map 为空时才查当前用户。**桌内其他玩家有手牌时，新进旁观用户也可能被判为重连**；游戏中历史手牌缓存残留扩大误判。
- **P1-1：`buildTableHandInfo` 对 `tableHand` 缺入口空值保护**。读 `TableHandCache.getTableHand(tableId)` 后直接 `BeanCopyUtil.copyProperties(tableHand,...)`/`tableHand.getHandNumNoException()` → 准备阶段/缓存缺失 NPE。
- **P1-2：重连恢复字段散在 `TableHand` transient**。上贡/还贡/随机牌/换座/结算 teams 等多个 `transient ...ForReconnect` 字段，不持久化（节点重启丢失），清理依赖 `resetForNewHand`，遗漏字段污染下一手。

目标：**1000 入房请求/响应、`TableHandEnterRoomDto`/`TableUserHandEnterRoomDto` 字段、`WATCH_USER_CONTINUOUS_BILLING_NOTICE`、取消托管通知 100% 兼容**前提下，**修正重连判断按当前 userId 建模（修 P0-2 误判）**、空 tableHand 安全组装、阶段恢复字段集中到 `ReconnectSnapshot`、明确 1012/1013 协议策略。**最小安全改造优先**，复用 01 已建 `GdReconnectStateService`。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | 重连作为 1000 入房分支保留（复用 01 `GdReconnectStateService`）；数据组装在 guandan `TableUserHandBootstrap` 收口 |
| 协议兼容 | 1000 请求/响应、`TableHandEnterRoomDto.isReconnect/reconnectStage`、`TableUserHandEnterRoomDto.userStatus/occupySeatStartTime/occupySeatCountdown/userTransferInViewRespDto`、`WATCH_USER_CONTINUOUS_BILLING_NOTICE` 字段不变 |
| **重连判断修正（P0-2 核心）** | `ReconnectIntentResolver`：以**当前 userId 是否有有效 `TableUserHand` 或仍在玩家态(DEFAULT/OCCUPY)** 为核心；`tableUserHandMap` 非空仅作"桌进行中"辅助证据，不再"非空即重连"。修复新旁观用户误判 |
| 空 tableHand 安全组装（P1-1） | `TableHandSnapshotLoader` 对空 `tableHand` 返回 `NoHandSnapshot`；DTO 按 snapshot 类型输出空手牌结构，不直接访问 |
| 阶段恢复字段集中（P1-2） | `ReconnectSnapshot{isReconnect,stage,tableHandStatus,currentUserArea,currentUserStatus,settlementState,occupyState,cachedNotices}` 按阶段集中生成/清理；评估关键结算/转蛋恢复数据是否需 Redis/事件重放（标注，本期不强迁） |
| **1012/1013 协议策略（阻塞型）** | **安全默认：标注掼蛋重连以 1000 为准**；1012/1013 若客户端仍发，实现薄适配命令转发 1000 重连 flow，否则 NoSupport。废弃 vs 适配 待产品/客户端确认（§10） |
| 阶段枚举 | `ReconnectStageResolver` 统一 PREPARE/PLAYING/SETTLEMENT 与 `TableHandStatusEnum` 映射，消除散落字符串 |
| 灰度策略 | 新旧重连判断并行打日志 → 先修旁观误判 → 收口阶段 resolver → 引入 snapshot 替换 DTO 组装 → 最后定 1012/1013 |

## 3. 整体架构

### 改造前

```
1000 入房完成 → EnterRoomNotice.sendNotice(非重连发 WATCH_USER_CONTINUOUS_BILLING_NOTICE)
重连数据(alreadyInRoom 分支, 01 GdReconnectStateService.recover → restartLoadHand):
  TableUserHandBootstrap.buildTableHandInfo (tableHand 无空保护 P1-1)
    ├─ fillReconnectStatus → determineIsReconnect(tableUserHandMap 非空即 true, P0-2 误判) / determineReconnectStage
    ├─ buildSettleEndReconnectData / fillTurnedEggReconnectNotices / fillWaitBetAndUserBetActions / fillDistributionInfos
    └─ TableUserHandEnterRoomDto(手牌/补码弹窗/状态/占座倒计时)
TableHand transient ...ForReconnect 字段(节点重启丢失 P1-2)
1012/1013 协议号存在 ── 无 command 实现 (P0-1)
```

### 改造后

```
1000 入房(不变) → 重连分支
  └─ ReconnectIntentResolver → ReconnectSnapshot
       ├─ isReconnect = 当前 userId 有有效 TableUserHand || 仍在玩家态(DEFAULT/OCCUPY)  (R-01 修 P0-2)
       │   (tableUserHandMap 非空仅辅助"桌进行中")
       ├─ ReconnectStageResolver → stage(PREPARE/PLAYING/SETTLEMENT)
       ├─ TableHandSnapshotLoader → NoHandSnapshot|HandSnapshot (空 tableHand 安全 R-02)
       ├─ ReconnectTableHandAssembler / ReconnectUserHandAssembler(按 snapshot 组装)
       └─ ReconnectNoticeCache(阶段恢复字段集中生成/清理 R-03)
1012/1013: 薄适配转发 1000 重连 flow 或 NoSupport(R-04, 待产品)
```

**兼容性原则**：
- 1000 请求/响应、`TableHandEnterRoomDto`/`TableUserHandEnterRoomDto` 字段、`WATCH_USER_CONTINUOUS_BILLING_NOTICE`、取消托管通知、手牌加密不变。
- `TableUserHandBootstrap.buildTableHandInfo` public 签名保留（内部委托 assembler）。
- 复用 01 `GdReconnectStateService`；不改 1000 入房权限校验。

## 4. 安全改造（R-01 ~ R-04）

### R-01：重连判断按当前 userId 建模（P0-2 核心修复）
- `ReconnectIntentResolver`：`isReconnect = 当前 userId 有有效 TableUserHand || 当前用户仍在玩家态(DEFAULT/OCCUPY)`；`tableUserHandMap` 非空仅作桌进行中辅助。修复新旁观用户在游戏中进入被误判重连。

### R-02：空 tableHand 安全组装（P1-1）
- `TableHandSnapshotLoader` 对 `getTableHand==null` 返回 `NoHandSnapshot`；`buildTableHandInfo`/assembler 按 snapshot 类型输出空手牌结构，不再直接 `copyProperties`/`getHandNumNoException` NPE。

### R-03：阶段恢复字段集中 `ReconnectSnapshot`（P1-2）
- 新增 `ReconnectSnapshot` + `ReconnectNoticeCache`：上贡/还贡/随机牌/换座/结算 teams 等 transient reconnect 字段按阶段集中生成/清理，与 `resetForNewHand` 对齐避免遗漏污染下一手。
- 关键结算/转蛋恢复数据是否需 Redis/事件重放：标注评估，本期不强迁（D1 单点风险记录）。

### R-04：1012/1013 协议策略（P0-1，阻塞型，待产品）
- **安全默认**：文档标注掼蛋重连以 1000 为准；若客户端仍发 1012/1013，实现薄适配命令转发 1000 重连 flow（不新增必填字段），否则 NoSupport（对齐 17/18）。废弃 vs 适配待产品确认。

## 6. 协议兼容确认

| 协议号 | 类型 | DTO | 字段是否变更 | 兼容性 |
|--------|------|-----|--------------|--------|

## 7. 可测试性设计

- `ReconnectIntentResolverTest`：当前用户有 TableUserHand 判重连；**其他用户有手牌但当前用户新旁观 → 不判重连（P0-2）**；WAIT 准备/伪结算态/SETTLEMENT 重连。
- `TableHandSnapshotLoaderTest`：空 tableHand → NoHandSnapshot 安全组装。
- 集成：游戏中重连恢复手牌+等待操作；结算重连恢复 playerIsContinue/needAddChipUsersMap；转蛋重连恢复随机牌/换座；新旁观不触发重连。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | 新旧重连判断并行打日志对比 → 先修旁观误判 → 收口阶段 resolver → snapshot 替换 DTO 组装 → 定 1012/1013 |
| 回滚 | 保留旧 `determineIsReconnect` 开关；snapshot 内部结构不进协议 |
| 运行中牌局兼容 | 复用 01 GdReconnectStateService；回归各阶段重连数据恢复 |
| 与其他 Junk 同步 | 入房见 01；在线状态见 15（复用重连后补偿语义）；账单旁观列表见 09 |

## 9. 验收关键场景

1. 新用户旁观进入 → 不标记重连（P0-2 修复）。
2. 已坐下玩家杀进程后进入 → isReconnect=true。
3. 准备阶段 OCCUPY 用户重连 → 恢复补码倒计时。
4. 游戏中重连 → 恢复手牌/最近操作/等待操作。
5. 结算阶段重连 → 恢复 playerIsContinue/needAddChipUsersMap。
6. 空 tableHand → 安全组装空手牌，不 NPE。
7. 1012/1013 策略有明确文档。
