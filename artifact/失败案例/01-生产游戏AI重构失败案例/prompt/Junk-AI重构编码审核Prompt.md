# Junk 游戏准备重构 - 代码审核 Prompt

> 用途：对 Junk 负责的"游戏准备"模块**已落地的代码改造**做审核。每个功能独立产出结果审核报告：`docs/Junk-游戏准备重构/{序号}-{功能名称}-结果审核.md`。
>
> 核心问题（按顺序回答）：
>
> 1. **是否达到审核条件**（指导文档中该功能完成度是否已含"实施"）
> 2. **与 dev 基线对比**：协议是否变化、业务是否等价
> 3. **是否符合重构文档预期**：设计文档每条 R-XX 是否落地
> 4. **是否有严重问题**：P0/P1/P2 分级风险
> 5. **结果写入**：`docs/Junk-游戏准备重构/{序号}-{功能名称}-结果审核.md`

---

## 步骤一：审核准入检查

> 这一步**必须先做**。不满足准入条件就停止审核，直接输出"功能 {序号}-{功能} 未达到审核条件：xxx"。

1. 读 `docs/Junk-游戏准备重构/Junk-游戏准备重构指导.md` 的"任务清单"。
2. 找到当前 `{功能序号}-{功能名称}` 行，检查"任务完成度"列：
   - 含 "实施" 且非 0% → ✓ 可审核
   - 仅有"扫描 / 设计 / 计划" → ✗ 未进入实施阶段，停止
3. 读 `Junk-游戏准备重构git变更记录.md`，确认本功能有至少 1 条 commit 已落库；如全部为"待提交"，停止。
4. 确认对应设计文档 `{序号}-{功能}-重构设计.md` 与实现计划 `{序号}-{功能}-实现计划.md` 均存在；缺失任一，停止。

满足以上 4 条后才进入步骤二。

## 步骤六：审核维度

| 维度 | 检查点 | 严重度 |
|---|---|---|
| **协议层** | 1. 协议号是否变化<br>2. ReqDto/RespDto 字段是否变化（删/改/改名/必填变可选）<br>3. 通知 DTO 是否变化 | **任一变化即 P0** |
| **业务逻辑等价性** | 1. 校验项是否齐全（dev 的每一条 check 在 junk 是否都有）<br>2. 校验顺序是否一致（错误码先后影响客户端感知）<br>3. 状态写入路径是否等价（WATCH/PLAYER 集合、AtomicCounter）<br>4. 重连判定条件是否一致<br>5. 快照组装字段是否齐全<br>6. 后置事件/通知/MQ 是否一致 | P0/P1 |
| **新增校验** | dev 没有、junk 新增的 `throw` 是否破坏既有玩家行为 | **P0** |
| **并发与原子性** | 原子计数器替换为集合 size 时，check 与 write 是否在同一临界区 | P1 |
| **健壮性退化** | dev 有的空值校验/业务异常 junk 是否被移除（变 NPE 走系统错误） | P1 |
| **路由/装配** | Handler 注册机制变化、cast 兼容性、Spring 注入顺序 | P1 |
| **可观测性** | 日志标签丢失、info→debug 收敛后线上排错信息密度 | P2 |
| **时序差异** | 同步→事件、事件失败 catch 吞 | P2 |
| **设计落地** | 设计文档 R-01 ~ R-NN 每条是否真的落地，命名/位置是否一致 | P0/P1（取决于偏离程度） |

## 步骤七：审核约束（红线）

1. **业务逻辑不能改变**：玩家在同等输入下得到的结果（成功/失败/错误码/副作用）必须等价。
2. **前端接口协议不能改变**：协议号、请求字段、响应字段、通知字段 0 改动；新增可选字段需列入"协议层观察项"。

## 步骤八：审核流程

```text
for 每个范围内文件:
    git show origin/20260615/dev:<path>   → 取 baseline 内容
    cat <path>                            → 取 junk 当前内容
    逐项对照"审核维度"打 ✓ / ⚠ / ✗
    把不等价的位置抓证据：文件 + 行号 + 旧代码片段 + 新代码片段 + 影响一句话

并发专项：
    凡见到 "check 用集合 size + 后续写入" 组合 → 必检查临界区是否覆盖
```

不要凭印象、不要遗漏证据。

## 步骤十：审核纪律

1. **不许臆断**：每个结论都要给"文件路径 + 行号 + 代码片段"证据
2. **不许改代码**：审核阶段只读，发现问题写报告、提建议，不直接 fix
3. **不许跳级**：P0 风险必须显式标出，不能藏在 P1 列表里
4. **dev 是基线**：判断"行为变化"以 `origin/20260615/dev` 为准，不以"看起来更合理"为准
5. **新增的禁入校验 = 业务变化**：即使产品视角"应当存在"，dev 上没有就是变化，需产品确认
6. **范围外要标注**：差异文件中不属于本功能的（如 1041/1029 等其他协议），列在"相邻改动未审核"附录
7. **并发场景必查**：只要看到"check 用集合 size + 后续写入"组合，必检查临界区是否覆盖
8. **设计 ≠ 落地**：不允许把"设计文档说应该这样做"当作"代码已经这样做"；必须以 junk 实际代码为准

## 附：审核结论三档

| 结论 | 标准 |
|---|---|
| ✓ **可发布** | 无 P0；P1/P2 可接受或已在灰度计划 |
| ⚠ **灰度观察** | 存在 P1，但有降级方案或开关；可灰度上线观察 |
| ✗ **不可发布** | 存在 P0（协议变化 / 新增禁入未确认 / 并发可超限 / 设计严重偏离），必须先修复再审 |

报告头部 `审核结论` 字段必须从此三档中明确选一档，不能模糊。
