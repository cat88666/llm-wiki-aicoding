# 05-一次启动自主编排

> 目标：让 AI 在一次启动后持续推进所有可执行任务，直到全部完成或只剩必须由人工解决的阻塞项。

## 1. 自主运行边界

```text
一次启动
  ≠ 一次提交全部重构
  ≠ 跳过业务确认
  ≠ 测试失败仍继续

一次启动
  = AI 自动维护状态
  + 自动选择下一原子任务
  + 自动验证和修复
  + 自动跳过独立阻塞链
  + 最终统一汇报
```

## 2. 启动前唯一人工输入

| 输入 | 必填内容 |
|---|---|
| 工程路径 | frame、guandan 及相关工程绝对路径 |
| 工作分支 | AI 可以提交的目标分支 |
| 基线分支 | 用于行为和 diff 对比的稳定分支 |
| 构建命令 | 每个工程可执行的编译命令 |
| 测试命令 | 局部测试、全量测试、协议冒烟命令 |
| 权限边界 | 是否允许建分支、commit、push、建 PR |
| 人工决策 | 已确认的产品、协议、资金和风控结论 |

缺少工程路径、分支或构建测试命令时，不得启动代码重构。

## 3. 任务图

AI 先把任务拆成有依赖的原子节点：

```text
环境基线
  ├── 低风险校准：20 → 21 → 19
  ├── 核心入口：12 → 01 → 02
  ├── 离开共享能力：07 → 04 / 06 / 18
  ├── 资金链：09 → 03 → 05 → 17
  ├── 私人房链：10 → 11 → 16
  └── 重连链：14 → 15
```

每个功能内部继续拆成：

```text
SCAN → SPEC → CHARACTERIZATION_TEST → R-01 → VERIFY → REVIEW → DONE
```

共享能力必须先独立立项和验证，再允许调用方切换。禁止把跨功能共享重构藏进某个功能提交。

## 4. 调度规则

AI 每轮只选择一个原子任务执行，选择顺序：

1. 修复导致构建或测试失败的 `FAILED` 任务。
2. 完成会解锁最多下游任务的共享前置任务。
3. 推进低风险、依赖少的 `READY` 任务。
4. 高风险资金、并发、重连任务最后执行。

任务之间完全独立时可以并行分析或运行测试，但同一文件、同一状态机、同一共享服务不得并行修改。

## 5. 单任务自动循环

```text
读取任务卡和相关代码
  → 核实真实入口、调用方和当前 diff
  → 补或运行特征测试
  → 测试必须先证明旧行为
  → 实现一个最小 R 项
  → 编译 + 局部测试 + 相关回归
  → 检查 diff 范围和协议兼容
  → 原子 commit
  → 更新状态表、证据、回滚方式
```

验证失败时，AI最多做三轮基于新证据的最小修复。三轮后仍失败则标记 `BLOCKED`，保留日志并继续其他独立任务。

## 6. 硬停止条件

| 条件 | 状态 | 后续 |
|---|---|---|
| 缺编译或测试环境 | `BLOCKED_ENV` | 只准备环境修复和测试草案 |
| 产品或协议语义未确认 | `BLOCKED_DECISION` | 生成阻塞决策单 |
| 资金、MQ、幂等契约不明确 | `BLOCKED_DECISION` | 不采用 AI 默认值 |
| 测试不稳定 | `BLOCKED_TEST` | 先稳定夹具 |
| diff 超出当前 R 项 | `FAILED_SCOPE` | 撤回多余改动并重拆 |
| 发现未设计共享重构 | `BLOCKED_DESIGN` | 独立立项 |
| 需要合并、发布或改生产数据 | `BLOCKED_APPROVAL` | 等待人工批准 |

## 7. 自动恢复规则

- 命令失败：读取完整错误，定位最小根因，不重复盲试。
- 测试失败：只修复与当前 R 项直接相关的问题；发现旧缺陷则另建任务。
- 工作区出现用户改动：保留用户改动，收窄写入范围。
- 基线漂移：重新计算 diff 和相关测试，不沿用旧审核结论。
- 状态账本冲突：以代码、测试日志和 Git 证据为准，修正状态表。

## 8. 运行结束条件

只有以下两种合法结束：

| 结论 | 条件 |
|---|---|
| `COMPLETE` | 所有目标任务为 `DONE`，全量门禁通过，最终验收报告完整 |
| `BLOCKED` | 无可执行任务，只剩需要人工决策、权限或外部环境的阻塞项 |

不得因为 token、时间、文档写完、静态审核完成或部分功能完成而宣称重构完成。
