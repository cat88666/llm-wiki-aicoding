# 01-SDD 规格驱动流程

> SDD 的目标：先让人定义“什么是对”，再让 AI 在边界内实现。

---

## 1. SDD 在 Junk 重构中的位置

```
旧资料 / 代码扫描
  └── 提炼 spec
        ├── 功能边界
        ├── 协议契约
        ├── 状态契约
        ├── 行为矩阵
        ├── 非目标
        └── 验收标准
              └── 通过后才允许写测试和编码
```

### spec 必须回答

| 问题 | 示例 |
|---|---|
| 这个功能管什么 | 入座只管占座、入座校验、状态切换、自动带入触发 |
| 不管什么 | 不处理游戏开始、出牌、结算派彩 |
| 外部契约是什么 | 协议号、DTO 字段、通知字段、RPC/MQ 参数 |
| 状态怎么变 | WATCH_USER → OCCUPY → PLAYER_USER |
| 哪些行为不能变 | 错误码、通知时机、钱包副作用、缓存写入 |
| 怎么验收 | 可执行测试、协议冒烟、构建与 CI 结果 |

## 2. 每个功能的 spec 结构

```text
功能 spec
├── 1. 功能边界
│     ├── 目标
│     ├── 非目标
│     └── 依赖功能
├── 2. 协议契约
│     ├── 协议号
│     ├── 请求 DTO
│     ├── 响应 DTO
│     └── 通知 DTO
├── 3. 状态契约
│     ├── 本地缓存
│     ├── Redis
│     ├── Table / Hand / UserItem
│     └── 钱包 / 账单 / 审核单
├── 4. 行为矩阵
│     ├── 正常路径
│     ├── 异常路径
│     ├── 重复请求
│     ├── 并发路径
│     └── 重连路径
├── 5. 兼容性约束
├── 6. 明确不做
├── 7. 验收标准
└── 8. 回滚方式
```

## 3. 行为矩阵模板

| 场景 | 输入 / 前置状态 | 当前旧行为 | 重构后要求 | 验收方式 |
|---|---|---|---|---|
| 正常路径 | 待补 | 待补 | 与旧行为一致 | 特征测试 |
| 重复请求 | 待补 | 待补 | 不重复写状态 / 不重复扣款 | 幂等测试 |
| 并发请求 | 待补 | 待补 | 状态线性化 | 并发测试 / 静态审核 |
| 缓存缺失 | 待补 | 待补 | 明确兜底，不系统异常 | 异常路径测试 |
| RPC 失败 | 待补 | 待补 | 错误码和补偿一致 | mock 测试 |
| MQ 失败 | 待补 | 待补 | 成功/失败语义明确 | 注入测试 |
| 重连恢复 | 待补 | 待补 | 快照字段一致 | 协议回归 |

## 4. Junk 关键状态契约

| 状态 | 影响功能 | spec 必须写清 |
|---|---|---|
| `UserItem.status` | 入座、站起、退出、重连、风控 | 状态枚举、读写点、非法流转 |
| `TableHand` | 带入、带出、补码、站起、结算衔接 | 当前手、结算中、等待中、补码状态 |
| `TableUserWallet` | 带入、带出、实时账单、退出 | 普通钱包、彩金、下一手带入带出 |
| 占座状态 | 入座、站起、退出、超时任务 | OCCUPY 创建、转入座、释放 |
| 风控缓存 | 风控、入座、风控数据上报 | user/table 维度、跨桌隔离、过期策略 |
| 私人房审核单 | 申请审核、审核通过、固定带入 | 状态机、幂等 key、失败补偿 |

---

## 5. spec 审核清单

| 检查项 | 通过标准 |
|---|---|
| 目标清楚 | 能一句话说出本功能要解决什么 |
| 非目标清楚 | 不会把其他负责人模块混进来 |
| 协议清楚 | 协议号、DTO、通知字段列全 |
| 状态清楚 | 所有缓存、模型、钱包、MQ 副作用列全 |
| 行为可测 | 每条关键行为能对应测试或审核项 |
| 风险分级 | P0/P1/P2 明确 |
| 回滚明确 | 出问题能 revert 或切旧 flow |

**不通过 spec 审核，不允许进入编码。**
