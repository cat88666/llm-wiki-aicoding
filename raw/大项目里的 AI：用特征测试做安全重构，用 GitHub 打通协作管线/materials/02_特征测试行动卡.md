# 特征测试行动卡

> 用在：已经完成架构梳理，准备重构遗留代码之前。目标是把当前行为锁住，证明后续改结构时行为没变。

## 输入

- 架构梳理输出。
- 待重构入口代码。
- 可疑行为清单。
- 已有测试或线上样例。

## AI 动作

1. 先写当前行为 spec。
2. 根据 spec 生成特征测试。
3. 覆盖关键路径、边界输入、可疑行为、老入口、隐式状态。
4. 为数据库、全局状态、缓存准备 reset / 临时隔离。
5. 跑测试到全绿。

## 人审动作

- 检查测试锁的是「现状」，不是「理想」。
- 检查可疑行为有没有被原样锁住。
- 检查测试是否稳定，是否隔离数据库和全局状态。
- 检查 public API、返回结构、事件、日志、warning、import 路径是否属于行为面。

## 完成标准

- 特征测试全绿。
- 每条可疑行为至少有一条测试。
- 测试可以重复跑，结果稳定。
- spec 和测试说的是同一组现状。

## 禁止事项

- 禁止把 bug 修复混进特征测试。
- 禁止用「应该是」写断言。
- 禁止在测试不稳定时开始重构。
- 禁止只测 happy path。

## 可复制 Prompt

```text
基于刚才的架构梳理，请先写一份当前行为 spec，然后生成特征测试。

要求：
1. 特征测试锁住当前行为，不判断当前行为是否合理；
2. 可疑行为必须原样锁住，并在测试名或注释里标明「现状」；
3. 如果代码读写数据库、全局变量、缓存或配置，测试前必须 reset 并使用临时隔离；
4. 覆盖 public API、返回结构、错误/拒绝路径、事件/日志/审计记录等行为面；
5. 生成后说明每条测试锁住了哪条现状。

不要修改业务代码。先只生成 spec 和测试。
```
