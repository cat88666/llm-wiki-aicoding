# 03-小步实施流水线

> 目标：把每个 Junk 功能拆成可以独立验证、独立提交、独立回滚的小步骤。

---

## 1. 单功能流水线

```text
选择一个功能
  └── 建功能重构包
        ├── scan：真实代码证据
        ├── spec：SDD 规格
        ├── tests：特征测试 / TDD 场景
        ├── plan：R 项拆解
        ├── commits：小步实现
        ├── review：结果审核
        └── release：灰度 / 回滚 / 观测
```

---

## 2. R 项拆解方式

| R 项类型 | 示例 | commit 粒度 |
|---|---|---|
| R-01 P0 修复 | 空指针、重复扣款、错误计数 | 一个 bug 一个 commit |
| R-02 兼容兜底 | 缓存缺失、RPC 失败默认值 | 一个兜底点一个 commit |
| R-03 特征测试 | 锁旧行为 | 测试单独 commit |
| R-04 结构抽取 | validator / assembler / state service | 一个职责块一个 commit |
| R-05 日志观测 | 日志收敛、指标补充 | 和业务改动分开 |

---

## 3. 推荐实施顺序

```
先测试
  └── 再 P0/P1 最小修复
        └── 再结构抽取
              └── 再清理重复
                    └── 最后优化日志和观测
```

### 禁止顺序

| 禁止 | 原因 |
|---|---|
| 先大规模抽象 | 旧行为没锁住，无法判断是否改坏 |
| 先全量重命名 | diff 噪声大，审核困难 |
| 重构和 bugfix 混合 | 线上回归时无法定位 |
| 一个 PR 改多个功能 | 影响面不可控 |
| 修改协议后再补说明 | 客户端兼容风险不可接受 |

---

## 4. 单步执行模板

```text
R-xx 执行
├── 目标
│     └── 本步只解决什么
├── 禁止
│     └── 本步不能改什么
├── 代码范围
│     ├── 必改文件
│     ├── 可新增文件
│     └── 禁止触碰文件
├── 测试
│     ├── 先写/补哪条测试
│     ├── 必跑命令
│     └── 无法运行时的静态审核项
├── 验收
│     ├── 响应
│     ├── 状态
│     ├── 副作用
│     └── 日志/指标
└── 回滚
      └── revert / 开关 / 旧 flow
```

---

## 5. AI 执行指令规则

给 AI 的任务必须是小而硬的：

| 好指令 | 差指令 |
|---|---|
| 只实现 `R-01 私人房人数计数等价修复`，不得修改 DTO | 优化进入房间 |
| 先补 `重复入座不重复占座` 特征测试，再改实现 | 重构入座模块 |
| 抽出 `BringInAmountPolicy`，保持所有调用结果不变 | 整理带入代码 |
| 对比 dev 基线，审核 1006 协议字段是否变化 | 看看有没有问题 |

### AI 输出必须包含

| 输出 | 要求 |
|---|---|
| 改动摘要 | 只说当前 R 项 |
| 文件清单 | 新增、修改、未触碰 |
| 验证结果 | 命令、结果、无法运行原因 |
| 行为等价证据 | 测试或静态对比 |
| 风险 | 未验证项、待确认项 |
| 回滚 | 对应 commit 或开关 |

---

## 6. 变更记录字段

| 字段 | 内容 |
|---|---|
| 编号 | `JUNK-YYYYMMDD-序号` |
| 功能 | 例如 `02 入座` |
| R 项 | 例如 `R-03 SitDownValidatorChain` |
| commit | 本地 commit hash |
| 变更文件 | 精确文件路径 |
| 协议影响 | 无 / 新增可选字段 / 待确认 |
| 行为影响 | 等价 / 有变化待确认 |
| 测试证据 | 命令和结果 |
| 审核结论 | 可发布 / 灰度观察 / 驳回 |
| 回滚方式 | revert / 开关 / 旧 flow |
