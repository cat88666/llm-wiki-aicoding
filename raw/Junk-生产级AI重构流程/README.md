# Junk 生产级 AI 重构流程

> 目标：以真实生产项目重构为目标，把 Junk 负责的游戏准备模块，从“资料很多但无法闭环”整理成一条可执行、可验证、可回退的工程流水线。
>
> 方法来源：
> - `wiki/summaries/01_SDD_TDD_驯服AI.md`
> - `wiki/summaries/02_大项目里的AI_特征测试与GitHub管线.md`
>
> 约束：本目录是全新的生产重构流程入口，不修改、不搬迁、不覆盖旧资料。

---

## 1. 核心原则

```
生产重构不是让 AI 直接改代码
  └── 而是先建立可验证的工程约束
        ├── SDD：先定义什么是对
        ├── 特征测试：先锁住旧行为
        ├── TDD：每个新增/修复行为先有裁判
        ├── 小步重构：一次只改一个职责块
        └── GitHub 门禁：所有变更必须可追溯、可审核、可回退
```

### 从旧模式切换到生产模式

| 旧模式 | 生产模式 |
|---|---|
| 先让 AI 扫描并给建议 | 先定义功能 spec 和验收标准 |
| 扫描、设计、计划、审核散落 | 每个功能只有一个“重构包”作为准入口 |
| AI 直接结构性改造 | 没有特征测试，不允许结构性重构 |
| 一个功能一次性大改 | 按 R 项小步提交，小 diff 审核 |
| 文档写完就算推进 | spec、测试、实现、审核、变更记录必须一致 |
| 待确认项全部堆在文末 | 分成阻塞项、保守默认项、观察项 |

## 3. 执行总图

```
Phase 0：建立生产基线
  ├── 确认分支 / 对比分支 / 代码版本
  ├── 建唯一状态表
  ├── 建功能依赖图
  └── 明确 Junk / 非 Junk 边界

Phase 1：SDD 规格化
  ├── 21 个功能逐个建立 spec
  ├── 协议契约、状态契约、行为矩阵写清楚
  ├── 明确目标 / 非目标 / 禁止项
  └── spec 审核通过后才能进入测试设计

Phase 2：特征测试锁现状
  ├── 正常路径
  ├── 异常路径
  ├── 幂等路径
  ├── 并发路径
  └── RPC / MQ / 缓存 / 通知 / 钱包副作用

Phase 3：TDD + 小步重构
  ├── 每个 R 项先写测试或回归场景
  ├── AI 只实现当前 R 项
  ├── 每步跑测试 / 静态等价检查
  └── 每步独立 commit + 变更记录

Phase 4：PR 门禁
  ├── 协议 0 破坏检查
  ├── 行为等价检查
  ├── 测试证据检查
  ├── 结果审核
  └── 人工最终裁决

Phase 5：灰度与复盘
  ├── 回滚方案
  ├── 观测指标
  ├── 待确认项关闭
  └── 规则沉淀进模板 / Prompt / AGENTS.md
```

## 4. Junk 功能推进顺序

### 先审低风险样本

| 顺序 | 功能 | 目的 |
|---:|---|---|
| 1 | 好友备注 | 校准 spec、测试、PR 模板 |
| 2 | 用户配置查询 / 修改 | 校准协议兼容和缓存更新 |
| 3 | 风控数据上报 | 校准风控缓存、上报副作用 |

### 再锁核心入口链

```text
进入房间前校验
  → 进入房间
    → 入座
      → 带入
        → 站起
          → 带出
            → 退出
```

### 最后处理高风险依赖链

| 链路 | 顺序 |
|---|---|
| 资金链 | 实时账单 → 带入 → 带出 → 放弃补码 |
| 私人房链 | 私人房申请审核 → 私人房审核通过 → 私人房固定带入审核 |
| 重连链 | 旁观 / 返回 / 重连 → 在线状态恢复 |
| 离开链 | 离开状态检查 → 站起 / 退出 / 预站起留座离桌 |

## 5. 完成定义

一个功能只有同时满足以下条件，才算“生产级重构完成”：

| 维度 | 完成标准 |
|---|---|
| spec | 目标、非目标、协议、状态、行为矩阵、验收标准齐全 |
| 特征测试 | 旧行为已被测试或等价审核锁住 |
| 实现 | 每个 R 项小步落地，无混合提交 |
| 验证 | 测试、静态 diff、冒烟记录齐全 |
| 审核 | 结果审核明确：可发布 / 灰度观察 / 驳回 |
| Git | commit、PR、变更记录、回滚方式完整 |
| 生产 | 灰度方案、观测指标、待确认项处理清楚 |
