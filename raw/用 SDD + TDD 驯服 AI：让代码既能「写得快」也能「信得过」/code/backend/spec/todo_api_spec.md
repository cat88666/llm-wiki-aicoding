# 待办事项 API · 规格（SDD 产物）

> 这份 spec 是课堂演示里"和 AI 探讨 → 人确认"的产物。
> 它是**人能快速 review 的东西**——审这十几条，比审上百行实现快。
> 它也是 TDD 的源头：下面每条"校验规则/错误"都会变成一条测试。

## 数据模型 Todo

| 字段 | 类型 | 约束 |
|---|---|---|
| `id` | int | 服务端自增，从 1 开始，只读 |
| `title` | str | **必填，去除首尾空白后非空，长度 1–100** |
| `done` | bool | 默认 `false` |

## 端点

### 1. 新建待办 `POST /todos`
- 请求体：`{ "title": str }`
- 成功：`201`，返回完整 Todo（含 `id`、`done=false`）
- 失败：
  - `title` 空 / 仅空白 / 超过 100 字 → `422`（校验错误）

### 2. 列出待办 `GET /todos`
- 成功：`200`，返回 `Todo[]`（按 `id` 升序）

### 3. 标记完成 `POST /todos/{id}/done`
- 成功：`200`，返回更新后的 Todo（`done=true`）
- 失败：`id` 不存在 → `404`

## 边界（会逐条变成测试）

- 空标题 `""` → 422
- 仅空白标题 `"   "` → 422（去空白后为空）
- 超长标题（101 字）→ 422
- 标记不存在的 id（如 999）→ 404
- 重复标记完成：幂等，仍返回 `done=true`、200

## 不在本次范围（明确划出去，避免 AI 自由发挥）

- 不做删除、不做编辑标题、不做分页、不做持久化（内存存储即可）
- 不做鉴权
