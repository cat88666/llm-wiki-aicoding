# 生成-验证-修正飞轮行动卡

> 用在：把单次 AI 编码升级成可持续的工程流程。目标是让 AI 产出每次都经过可复现验证，再由人裁决。

## 输入

- 任务或 issue。
- spec / 规则文件。
- 测试和 CI。
- PR 审查流程。

## AI 动作

1. 生成：按 spec 或 issue 做最小实现。
2. 验证：运行测试、lint、workflow。
3. 修正：根据结构化失败信息做最小修改。
4. 记录：在 PR 或变更说明里写清验证结果。

## 人审动作

- 先定义验证标准。
- 审失败信息，不只审 AI 的解释。
- 裁决机器无法判断的语义问题。
- 决定是否 merge 或另开 issue。

## 完成标准

- 每个产出都有对应验证。
- 失败能回喂并复跑。
- 人只在标准、语义、合并上做最终裁决。
- 流程能被团队重复执行。

## 禁止事项

- 禁止用「AI 说已经完成」当完成标准。
- 禁止没有验证就进入下一步。
- 禁止失败信息不结构化，只说「有 bug」。
- 禁止把一次成功当成长期可信。

## 可复制 Prompt

```text
请按生成-验证-修正流程执行本任务：

1. 先列出本任务的验证标准；
2. 做最小实现；
3. 运行验证命令，并贴出关键结果；
4. 如果失败，请按「失败项 / 输入 / 期望 / 实际 / 最小修复」格式分析；
5. 修复后重新运行验证；
6. 最后输出：改动范围、验证结果、剩余风险、需要人裁决的问题。

不要跳过验证，不要把未验证内容说成已完成。
```
