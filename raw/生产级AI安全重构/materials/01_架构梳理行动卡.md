# 架构梳理行动卡

> 用在：准备让 AI 改一段遗留代码之前。目标不是马上改代码，而是先让 AI 只读，交付一张「我理解了什么」的结构图。

## 输入

- 一个入口函数、类、模块或目录。
- 已知调用方式或业务入口。
- 可选：历史 issue、README、已有测试、线上报错。

## AI 动作

1. 只读代码，不改文件。
2. 列出 public API 和下游可能依赖的入口。
3. 拆出职责板块：计价、状态、持久化、通知、报表等。
4. 画状态地图：数据库、全局变量、缓存、配置、事件、日志、外部服务。
5. 找出「几套真相」：重复实现、旧入口、新入口、并行规则。
6. 列出可疑行为：像 bug，但还不能顺手修。

## 人审动作

- 确认入口有没有漏。
- 确认 public API / import 路径有没有漏。
- 确认隐式状态有没有漏。
- 追问 AI：如果改这里，可能破坏哪里？
- 把「先锁住」和「后续可能修」分开标注。

## 完成标准

- 有入口清单。
- 有职责板块。
- 有状态地图。
- 有可疑行为清单。
- 能说清哪些行为必须先保持不变。

## 禁止事项

- 禁止边读边改。
- 禁止把「看起来不合理」直接当 bug 修。
- 禁止只看返回值，忽略数据库、日志、事件、import 路径。

## 可复制 Prompt

```text
请只读下面这段遗留代码，不要修改任何文件。

请输出：
1. public API / 入口清单，标出下游可能依赖的调用面；
2. 职责板块拆分，说明每块现在散落在哪些函数/类/文件里；
3. 状态地图，列出数据库、全局变量、缓存、配置、事件、日志等隐式输入输出；
4. 现有系统里可能并存的多套实现或多套规则；
5. 可疑行为清单：看起来像 bug，但重构前必须先原样锁住的行为；
6. 如果要重构，第一批应该补哪些特征测试。

要求：先描述现状，不要给重构方案，不要顺手修 bug。
```
