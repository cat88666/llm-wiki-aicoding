# spec ↔ issue 治理行动卡

> 用在：一次性 spec 和测试已经产生，想让它们跟着项目长期维护。目标是把 spec 变成状态账本，不让它退化成过期文档。

## 输入

- 当前行为 spec。
- 已有测试。
- issue 列表。
- 项目规则、兼容要求、团队约定。

## AI 动作

1. 建规则文件：`AGENTS.md` 或 `CLAUDE.md`。
2. 建 `spec/` 状态账本：
   - `governance/`：长期规则。
   - `planned/`：已设计、未落地。
   - `implemented/`：已落地，带实现锚点。
   - `archived/`：废弃或搁置。
3. 把一次性 spec 改成单条 spec 文件。
4. 给 issue 分类：局部 bug、设计变更、兼容变更、同根因问题。
5. 在 PR 里对账：实现、测试、文档、spec 状态是否一致。

## 人审动作

- 确认哪些规则必须写进 agent 规则文件。
- 判断 issue 是否会影响架构、公共接口或兼容线。
- 确认 spec 的状态迁移是否正确。
- 确认「完成」是否包含实现、测试、文档、示例、兼容信息、spec 状态。

## 完成标准

- agent 开工前有规则文件可读。
- spec 目录有状态，不是一堆散文档。
- issue 能映射到 spec 缺口或已有契约。
- PR 合并时 spec 状态同步更新。

## 禁止事项

- 禁止把重要规则只留在聊天记录里。
- 禁止实现改了但 spec 不改。
- 禁止多个同根因 issue 被逐个打补丁。
- 禁止分不清影响面时直接开工。

## 可复制 Prompt

```text
请基于当前项目，帮我建立一个最小 spec 状态账本。

要求：
1. 生成 AGENTS.md，写清 issue 分类、分支命名、测试要求、spec 对账规则和完成定义；
2. 建 spec/README.md，列出 governance/planned/implemented/archived 四类状态；
3. 把当前这份行为 spec 按模板整理成一条 spec，包含状态、契约、验收标准、实现锚点、兼容影响；
4. 给现有 issue 分类：局部 bug / 设计变更 / 公共接口或兼容影响 / 多 issue 同根因；
5. 输出 PR 合并前的 spec 对账清单。

不要写业务实现，先建立治理结构。
```
