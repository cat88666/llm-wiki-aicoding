# Junk 游戏准备重构 - AI 深度扫描 Prompt

> 用途：用于驱动 AI 对 Junk 负责的"游戏准备"模块做深度代码扫描，产出每个大功能的独立问题清单、改造建议、核心理由、落地顺序和验收场景。
>
> 核心原则：**允许内部彻底重构，但入口协议请求参数、出口协议响应参数、通知 DTO 字段必须保持向后兼容。**

---

## 1. 使用方式

### 1.1 基础环境确认

项目开始之前，必须先和用户确认以下信息，确认后再开始扫描、写文档或修改代码：

```text
项目目录：dx-b-server-jk/guandan/dx-game-guandan
扫描项目：dx-game-frame dx-game-guandan
默认分支：20260615/junk
```

确认口径：

- 项目目录是否为 `dx-b-server-jk/guandan/dx-game-guandan`。
- 扫描项目是否只包含 `dx-game-frame` 和 `dx-game-guandan`。
- 当前工作分支是否为 `20260615/junk`。
- 如果实际路径或分支与默认值不一致，必须先让用户确认以哪个路径和分支为准。

### 1.2 执行步骤

1. 从 `docs/重构功能拆解.md` 的 `2. 游戏准备(Junk)` 选择一个大功能。
2. 将本 Prompt 中的 `{功能名称}` 和 `{子功能列表}` 替换为实际内容。
3. 在 `dx-b-server-jk/guandan` 目录下同时扫描：
   - `dx-game-guandan`
   - `dx-game-frame`
4. 将输出写入 `dx-b-server-jk/guandan/dx-game-guandan/docs/Junk-游戏准备重构/{序号}-{功能名称}.md`。
5. 每个功能独立输出，不把多个功能混成一个大报告。
6. 扫描前先记录当前代码版本：`git rev-parse HEAD`。

## 4. 已知架构缺陷与标杆

### 4.1 已确认缺陷（扫描时验证当前功能是否存在）

| 编号 | 缺陷 | 关键证据 |
|---:|---|---|
| D1 | 单节点内存状态，无持久化/容灾 | `GuandanMasterManager.masterMap = new HashMap<>()` |
| D2 | 并发安全模型混乱 | TableHand 内 HashMap/ConcurrentHashMap/AtomicBoolean/synchronizedList 四种混用 |
| D3 | Thread.sleep 阻塞业务线程 | `GameStartService:375 Thread.sleep(2000L)` |
| D4 | 无对象池，高频创建 108 Card | `CardUtils.getCardFromId()` 每次 `getNewDecks().stream()` |
| D5 | TableHand 70+ 字段上帝对象 | 12+ 领域职责塞在一个类 |
| D6 | 无显式状态机 | `GameStatusEnum gameStatus` 被注释掉 |
| D7 | 零测试覆盖 | 无 `src/test` 目录 |
| D8 | 结算无事务/幂等 | 无幂等 key，`waitBetMap.remove` 被注释 |
| D9 | 静态方法 + SpringContextUtils.getBean 反模式 | `PlayNoticeService` |
| D10 | 魔法数字散布 | 53/54 大小鬼, 1/2 玩法类型, 4000/2000 动画延迟 |
| D11 | 断线重连靠 7 个 transient 字段 | 节点重启全丢 |
| D12 | 日志滥用 | GsonUtils.toJson(tableHand) 序列化 70+ 字段到 info 日志 |

扫描时：如果发现 D1-D12 在当前功能中存在，问题表"对应缺陷"列填写编号；独立发现的新问题填"新发现"。

### 4.2 德州标杆架构（改造方向参考）

| 模式 | 德州实现 | 掼蛋应对齐 |
|---|---|---|
| Command-Context-Handler 三层 | 协议→CommandContext→TableSpaceService 策略接口 | 替换当前 Command 直调 Service 的扁平结构 |
| 空间类型多态路由 | `DxGameServerFactory.getHandle(spaceType, gameModel)` | 支持 CLUB/PRIVATE/HALL/RACE 差异化逻辑 |
| 事件驱动状态流转 | Spring ApplicationEvent（9 种领域事件） | 替换 EventManager 仅做定时调度的用法 |
| TABLE 线程单线程绑定 | 所有牌桌状态变更严格在 tableId 线程 | 确保不被 EventManager 回调/异步任务绕过 |
| Redis 全量状态缓存 | `dx::game::texas::table:{tableId}` | 替换纯本地 HashMap 的单点风险 |
| 延迟任务体系 | OnceEventTypeEnum + LoopEventTypeEnum | 替换 Thread.sleep 和散落的 Timer |

## 5. 重构铁律

### 5.1 协议兼容

必须保持不变：

- 请求协议号
- 请求 DTO 字段
- 响应 DTO 字段
- 通知协议号
- 通知 DTO 字段
- 客户端已依赖的字段语义

允许改变：

- 内部类结构
- Service / Bootstrap / Handler 调用链
- 缓存封装方式
- 校验链拆分
- 任务调度封装
- frame 新增扩展点

如果发现字段当前赋值疑似错误，但可能已被客户端依赖：

```text
标记为：兼容性待确认
不要直接建议删除或改字段语义
```

### 5.2 frame 修改边界

`dx-game-frame` 是公共框架，可以做向后兼容演进。

可以做：

- 新增接口、抽象类、工具类
- 新增方法
- 新增扩展点
- 优化已有方法内部实现
- 在 guandan 中覆盖或组合 frame 默认实现

不能做：

- 删除已有公共类
- 删除已有公共方法
- 重命名已有公共类 / 方法 / 字段
- 修改已有公共方法签名
- 破坏麻将侧已有调用预期

替换旧实现时，推荐：

```text
新增新方法 / 新类 -> 旧方法 @Deprecated -> guandan 迁移到新链路 -> 后续再评估清理
```

### 5.3 共享模型慎改

如果方案涉及 UserItem.status / TableHand / TableUserWallet 等多功能共享的模型变更，必须在"迁移与灰度-与其他 Junk 功能同步要求"中说明影响范围，不要在单个功能的方案中独自重新定义共享模型。

## 6. 扫描方法

### Step 0：复杂度分级

先判断当前功能复杂度，并在输出头部标注。

| 等级 | 判断标准 | 输出深度 |
| --- | --- | --- |
| L | 查询、展示、轻量校验，无核心状态变更（如"牌桌状态查询"、"在线状态恢复"） | 简化输出，但仍需覆盖架构审视核心问题；可以用一段结论或精简表格呈现，必须有证据 |
| M | 有状态变更，但流程较线性（如"放弃补码"、"进入房间前校验"） | 完整扫描 |
| H | 涉及钱包、风控、占座、重连、结算衔接、异步任务、多阶段状态（如"进入房间"、"入座"、"带入"、"站起"、"退出"） | 完整扫描，问题和方案必须展开 |

### Step 0.5：跨功能决策继承（非首个功能时执行）

如果当前不是第一个扫描的功能，先查阅已完成的扫描文档，提取与当前功能相关的已有决策：

1. 已完成文档中涉及当前功能的"跨功能同步要求"或"依赖接口"。
2. 已采用的设计模式和架构决策（如：前序功能已决定用 FSM 管理 UserItem.status，当前功能应沿用而非另起炉灶）。
3. 已对共享模型（UserItem.status / TableHand / TableUserWallet）做出的拆分/扩展决策。

在输出文档头部标注：`前置依赖文档：01-进入房间.md, 02-入座.md`（列出已参考的文档）。如果是第一个扫描的功能，跳过此步。

### Step 4：先理解现有设计

在扫描问题前，先用自然语言说明当前代码是怎么工作的。

必须输出：

1. **主流程叙事**：一个正常请求从进入到完成，经过哪些步骤？每步做了什么业务动作？（1-2-3 编号描述，不贴代码）
2. **条件分支**：有哪些条件分支？每个分支的触发条件和处理方式？
3. **异常路径**：RPC 失败、超时、资源不足等场景的处理。
4. **当前设计中应保留的合理点**：哪些设计决策是合理的、应该在重构中保留的？（如：某个校验顺序是经过线上验证的、某个通知时机是配合前端动画的、某个缓存策略是有性能考量的）
5. **业务规则隐含知识**：代码中不明显但重要的业务规则。（如：带入金额计算涉及彩金和普通筹码的优先级、风控站起需要区分"本手结束后"和"立即"）

如果无法解释某段逻辑：

```text
待理解：{类名:行号} {逻辑描述}
```

不能把"待理解"直接当成 bug。这一步的输出应该让一个不看代码的人也能理解这个功能当前是怎么工作的。

| 优先级 | 标准 |
| --- | --- |
| P0 | 可能导致资金错误、牌桌状态损坏、玩家状态错误、严重线上故障 |
| P1 | 可能导致卡流程、重连错误、通知不一致、权限 / 风控绕过 |
| P2 | 维护困难、重复逻辑、扩展成本高、性能浪费 |

维度：

- 正确性
- 并发安全
- 幂等防重
- 异常补偿
- 阶段覆盖
- 性能
- 代码质量
- 协议兼容
- 可测试性

必须检查：

```text
1. 请求重发是否幂等
2. 同一用户并发操作是否安全
3. 同一桌多用户并发操作是否安全
4. 状态变更是否都在线程模型约束内
5. RPC 失败是否有补偿
6. 通知失败是否影响主流程
7. 缓存与 RPC / DB 是否可能不一致
8. 游戏准备中 / 游戏中 / 结算中 / 重连中是否都有定义
9. 是否有大对象日志、重复序列化、N+1 调用
10. 是否有 SpringContextUtils.getBean / static 反模式
```

模式结论只能是：

```text

```

每个结论必须附理由。"不适合"同样重要——它证明你思考过而非遗漏。

## 8. 输出质量门禁

## 10. Prompt 设计理由

| 设计点 | 理由 |
| --- | --- |
| 先定位入口再分析 | 防止按类名猜测，必须从协议和真实调用链出发 |
| 嵌入完整包路径和协议号范围 | 消除 AI 按类名猜测的风险，确保扫描基于真实代码结构 |
| 先理解再评判 | 防止把看不懂的历史逻辑误判成 bug；保留精华设计 |
| 问题必须有证据 | 避免输出空泛架构批评 |
| 建议必须关联问题 | 避免"为了重构而重构" |
| 嵌入 D1-D12 已知缺陷 | 避免重复发现已知问题，同时验证每个功能是否受影响 |
| 嵌入德州标杆架构 | 确保改造方向对齐已验证的架构模式，而非自由发挥 |
| 嵌入功能间依赖全景图 | 19 个功能共享核心模型，提供已知依赖避免每次从零发现，也防止单功能方案破坏全局一致性 |
| 跨功能决策继承（Step 0.5） | 后续扫描引用前序文档决策，避免同一共享模型被多个方案各自重新定义 |
| 8 种模式逐一评估 | 防止只挑熟悉的模式硬套；"不适合"的结论同样有价值 |
| 允许结论为"不需要模式" | 避免过度设计；简单功能用直白 Service 实现反而是最优解 |
| 改造建议三级分档 | 最小安全改造/结构性重构/彻底重写，匹配不同风险偏好和排期约束 |
| 可测试性设计独立步骤 | D7 零测试覆盖不能只补测试用例，必须在架构层面让代码可测试 |
| 非 Junk 边界分析接口契约 | 只标记"非 Junk 范围"会丢失接口依赖信息，导致 Junk 侧重构后对接出错 |
| 共享模型慎改 | UserItem.status / TableHand 等被 6+ 功能共读写，单功能方案不能独自重定义 |
| 每功能独立文档 | 19 个功能共享状态但职责不同，必须逐个闭环 |
| 强化协议兼容 | 当前目标是内部重构，不是破坏客户端协议 |
| 加入迁移灰度 | 线上游戏服务不能停服重写 |
| 加入质量门禁 | 保证所有功能扫描报告格式和深度稳定 |
