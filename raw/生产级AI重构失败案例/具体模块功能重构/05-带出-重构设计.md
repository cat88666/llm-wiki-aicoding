## 1. 背景与目标

带出覆盖 1023 `USER_BRING_OUT_VIEW`（`@ThreadModel(PLAYER)`）、1024 `USER_BRING_OUT`（`@ThreadModel(TABLE)`）、通知 14009，以及站起/退出链路的自动带出。链路：`UserBringOutViewCommand`/`UserBringOutCommand → *CommandDefault → UserBringOutBootstrapService`，guandan 扩展 `UserBringOutBootstrap.autoBringOut`。基于真实代码确认：

- **P0-1：自动带出分支多，规则隐含在实现顺序**。`UserBringOutBootstrap.autoBringOut`（`:137`）依次判断 `bonusFlag`（`:141`）、`shouldCurrentBringOutForSettleStand`（`:142`）、私人审核结算离场、房主强制、轮服务费等；分支顺序即业务优先级，无显式规则表。由 06 退出 `GdUserLeaveBootstrapService.playerQuit`（`:162/171`）、过级彩金 `LevelUpBonusHandler`（`:317`）等多入口触发。新增离场原因易把"下一手预约"与"立即下分"放错位置。
- **P0-2：带出命令入口 NPE**。`UserBringOutDefault.java:52-53` 先 `tableHand = BaseTableHandCache.getTableHand(...)` 再 `tableHand.getHandCode()`（无空保护）；`:83` `getTableHand(tableId).getTableHandStatus()` 链式 NPE。桌已解散/缓存未命中时本应业务异常，却 NPE；若发生在 try 前，错误响应可能发不出。
- **P1-1：掼蛋覆盖未复用框架 `checkBringOutTableLimit`**。`UserBringOutCommandExtend` 重写校验只处理私人固定房保留筹码与金额，未续调表级限制；框架后续扩展表级带出限制会被绕过，私人固定房与普通房限制来源不一致。
- **P1-2：成功响应用 error builder**。`UserBringOutDefault.sendBringOutResp` `:118/120` 用 `NetResponseBuilder.buildErrorResponse(..., createMessageException(SUCCESS, ...))` 构造**成功**响应。若 builder 语义/客户端埋点按 error 分类，成功带出被错误统计；未来统一网关可能误拦。
- **P2：资金敏感日志 + 状态集中**。带出金额/桌面筹码/钱包列表 toJson（D12）；结算/彩金/强制离场/轮费状态全集中在 `UserBringOutBootstrap`（D11）；`downScore` 成功→本地同步→通知 非显式事务（D6）。

目标：**保持 1023/1024/14009 协议字段 100% 兼容**前提下，把自动带出决策抽成可测试的 `BringOutPolicyResolver`、修复入口 NPE（tableHand 移入 initContext 空值兜底）、校验规则链化（覆盖类只追加不替换）、统一带出钱包/通知封装；`buildErrorResponse` 成功响应**先锁兼容测试再评估**改正常 builder。**最小安全改造优先**，对齐 07 `LeaveIntent`。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导骨架（policy/钱包/通知放 frame），guandan 提供彩金/结算/私审/轮费 扩展决策 |
| 协议兼容 | 1023/1024 `bringOutScore`/`walletList`、`UserBringOutViewRespDto`/`UserBringOutRespDto`（含 `bringOutStatus` 枚举语义）、14009 字段不变 |

| 校验规则链（P1-1） | 拆 `CommonBringOutAmountRule`/`PrivateFixedKeepRule`/`TableLimitRule`；掼蛋覆盖类**只追加规则不替换整段**，保证 `checkBringOutTableLimit` 不被绕过 |

| 钱包/通知封装 | `BringOutWalletService`（`DownScoreReq` + 主钱包/彩金金额列表 + 本地缓存同步可重试）、`BringOutNoticeService`（统一 1024 响应 + 14009） |
| 灰度策略 | 先引入 policy resolver 只做影子决策日志，对比旧分支输出；先灰度手动带出再自动带出，最后收口彩金/私审特殊分支 |

## 3. 整体架构

### 改造前

```
1023 UserBringOutViewCommand(PLAYER) → ViewCommandDefault(check/handle/after) → getCanBringOutChipScore + checkCanLeaveResult
1024 UserBringOutCommand(TABLE) → CommandDefault.check(玩家/钱包/金额/最小保留) → UserBringOutBootstrap.handleManual
     UserBringOutDefault: tableHand=getTableHand(); tableHand.getHandCode() :52-53 (P0-2 NPE)
                          sendBringOutResp → buildErrorResponse(SUCCESS) :118/120 (P1-2)
自动带出: GdUserLeaveBootstrapService.playerQuit:162/171 → UserBringOutBootstrap.autoBringOut:137
          (bonusFlag/settleStand/private/forced/roundFee 分支隐含优先级 P0-1)
```

### 改造后

```
1023 UserBringOutViewCommand(PLAYER, 不变) → BringOutViewService.queryCanBringOut (只读, 复用 PolicyResolver 判定 timing/retain)
1024 UserBringOutCommand(TABLE, 不变)
  └─ UserBringOutCommandDefault(入口保留)
       ├─ initContext: tableHand 空值兜底 handCode="-" (R-02), 先 checkDisband/InfoCache
       └─ BringOutCommandService.execute
            ├─ BringOutValidatorChain (CommonAmount/PrivateFixedKeep/TableLimit, 覆盖类只追加 R-03)
            ├─ BringOutPolicyResolver → BringOutPolicy (R-01)
            ├─ BringOutWalletService → downScore + 本地缓存同步(可重试)
            └─ BringOutNoticeService → 1024 响应(R-04 builder待确认) + 14009
自动带出: AutoBringOutService.auto(tableId,userId,LeaveIntent) ← 06退出/04站起/彩金/强制 复用 PolicyResolver
```

**兼容性原则**：
- 1023/1024 请求响应、`bringOutStatus` 枚举（含 `NEXT_HAND_SUCCESS`/`NEXT_HAND_SUCCESS_ROUND_FEE` 等）、14009 字段不变。
- `UserBringOutCommand`/`*CommandDefault`/`UserBringOutBootstrapService` public 签名保留。
- 站起/退出是否允许的离开校验不动（归 04/06/07）；钱包 RPC 账变模型不动。

## 4. 安全改造（R-01 ~ R-05）

### R-01：`BringOutPolicyResolver` 自动带出决策（P0-1）
- 新增 `BringOutPolicy` + `BringOutPolicyResolver`：把 `autoBringOut` 内 `shouldCurrentBringOutForSettleStand`/`...ForOwnerForcedLeave`/`...ForPrivateAuditSettleLeave`/`...ForRoundFeeActiveLeave` 收敛为按优先级的规则表，输出 CURRENT/NEXT_HAND/RETAIN/SKIP + reason。
- `reason` 对齐 07 `LeaveIntent`（QUIT/STAND/FORCED/...），自动带出入口（06/04/彩金/强制）传 intent。

### R-02：入口 NPE 修复（P0-2）
- `tableHand` 读取移入 `initContext`，空值兜底 `handCode="-"`；`getTableHandStatus()` 链式调用前判空。
- 先 `BaseTableCache.checkDisbandTable` / `BaseTableInfoCache` 再组装日志；保证缓存缺失返回业务异常而非 NPE。

### R-03：校验规则链化（P1-1）
- 拆 `CommonBringOutAmountRule`（金额/精度/最小保留/最大可带出）、`PrivateFixedKeepRule`（私人固定房保留）、`TableLimitRule`（`checkBringOutTableLimit`）。
- 掼蛋覆盖类（原 `UserBringOutCommandExtend`）改为**追加** `PrivateFixedKeepRule`，不替换整段，确保表级限制不被绕过。

### R-04：成功响应 builder 锁兼容（P1-2，谨慎）
- **本期不改 builder**：补测试锁定 `buildErrorResponse(SUCCESS, bringOutStatus.desc)` 的实际报文结构 + 客户端解析路径；`BringOutNoticeService` 封装当前行为。
- 中期（待确认后）改正常 `buildResponse` 并灰度对比。列入待确认。

### R-05：钱包/通知封装 + 日志脱敏（P2/D6）
- `BringOutWalletService`：封装 `DownScoreReq`（主钱包 + 彩金金额列表）+ 下分成功后本地缓存同步（可重试）；明确 downScore→本地同步→通知 顺序。
- 资金敏感 toJson info → debug；info 仅 tableId/userId/reason/applyTiming/金额元数据。

## 7. 可测试性设计

| 方面 | 当前 | 改造后 |
|------|------|--------|
| 决策 | 长 if 链 + 方法名推断 | `BringOutPolicyResolver` 纯函数可单测（7 种 reason × timing） |

| 校验 | 覆盖类整段替换 | 规则链可单独测，覆盖只追加 |

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | 先引入 PolicyResolver 只做影子决策日志，对比旧分支输出不一致即打印完整原因 |
| 灰度维度 | 先手动带出，再自动带出，最后收口彩金/私审特殊分支 |
| 回滚 | 配置恢复旧 `UserBringOutBootstrap` 分支；policy 不进协议 |
| 运行中牌局兼容 | 下分成功后本地同步可重试；回归退出/站起/结算离场自动带出 |
| 与其他 Junk 同步 | reason 对齐 07 `LeaveIntent`；占座/带入见 02/03；账单展示见 09 |

## 9. 验收关键场景

1. 1023 查询返回最小保留 + 最大可带出。
2. 金额 ≤0 或超最大可带出 → 业务异常。
3. 参与本手手动带出 → 1024 返回下一手成功状态。
4. 未参与本手/结算特殊离场 → 立即下分。
5. 退出自动带出 → 主钱包 + 彩金都下分。
6. 下分成功 → 14009，客户端桌面筹码刷新。
7. 无 tableHand（解散/未命中） → 业务异常，不 NPE。
