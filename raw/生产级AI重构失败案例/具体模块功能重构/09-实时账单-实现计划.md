## R-03：`BillingStatsClient` 统一统计 RPC 失败口径

- **目标文件（新增）**：`dx-game-guandan-service/.../service/biz/billing/BillingStatsClient.java`、`BillingStatItem.java`（`com.dx.game.guandan.model.dto.billing` 或 billing 包）
- **步骤**：
  1. 定义 `enum SourceStatus { OK, EMPTY, FAILED, FALLBACK }`；`BillingStatItem<T>{ T data; SourceStatus status; }`。
  2. `BillingStatsClient` 封装现有 5 个调用：`getPlayerBonusProfitLossMap` / `getPlayerWinLoseScoreMap` / `getPlayerRoundCountMap` / `selectListByTableId`(手记录) / `getWalletSummaryForGuandanByTableId`，各返回 `BillingStatItem`。
  3. 失败/空判定沿用现有 `CodeMsgEnum.SUCCESS` + null 判断；局数失败标 `FAILED`，由调用方决定回退手数（标 `FALLBACK`）。
  4. 日志降级（见 R-05）。
- **测试**：每个统计源 mock 成功/空/异常 → status 正确；局数失败回退手数。
- **验收**：5 个统计源失败均不抛到主流程；status 进日志/指标。
- **回滚**：删除 `BillingStatsClient`，调用点恢复直接调 `baseDubboRpcManager`。

## R-02：`BillingSnapshot` 一次性快照

- **目标文件（新增）**：`BillingSnapshot.java`（billing 包，内部 DTO，不进协议）
- **步骤**：
  1. 字段：`tableId/table/tableInfo/tableHand/allUserItems(Map<Long,UserTableItem>)/activeDisplayUserIds/quitUserIds/friendRemarks(Map<Long,UserFriendRemarkDto>)/capturedAt`。
  2. 静态 `capture(tableId, requestUserId)`：一次性读 `TableCache`/`TableInfoCache`/`TableHandCache`/`UserTableItemCache.getAllUserTableItem`/`BaseTableUserCache.fetchTablePlayerUserIds(DEFAULT+SIT_DOWN_WAIT_*)`/`fetchTableUserIds(QUIT_USER)`/`UserFriendCache.getFriendRemarkCache`。
  3. `setRealTimeData`/`setWatchUsers`/`setKickerList` 改为只读 snapshot，不再各自重读 cache。
- **测试**：同一 snapshot 下 activeDisplayUserIds 与金额来源时间点一致；并发站起/退出场景展示状态与金额不错位。
- **验收**：一次 3005 只读一次各 cache；日志带同一 `capturedAt`。
- **回滚**：删除 snapshot，恢复各方法内独立读取。

## R-01：主块失败抛出 + 关键空值兜底（P0-1）

- **目标文件**：`ContinuousBillingHandler.java`、`ContinuousBillingCommand.java`
- **步骤**：
  1. `handle`（或新 `BillingFlowService.build`）：`realTimeData` 主块 + `playAttributes` 构建失败 **不再 catch 吞掉**，向上抛出（业务/系统异常）。
  2. `ContinuousBillingCommand.handleCommand`：**移除"无条件 after"**——`build` 成功才 `after` 发响应；异常走既有 catch（L53-63）发错误响应。
  3. `playAttributes`：`snapshot.tableHand == null` → 兜底（`context.getTable()` 玩法属性或空对象），不再 `tableHand.getPlayAttributes()` 直接 NPE（修 L73）。
  4. `watchUsers`/`kickerList` 包在各自 try/catch（见 R-04），失败返回空列表不拖垮主块。
- **测试**：mock 主块组装抛异常 → 3005 错误响应（非半空成功）；tableHand=null → playAttributes 兜底不 NPE。
- **验收**：组装失败时客户端收到错误响应且监控可见；正常路径响应字段与 dev 一致。
- **回滚**：恢复 handle 整体 try/catch + 命令层无条件 after。
- **⚠ 兼容性待确认**：主块失败由"空成功"改"错误响应"是行为变化，需客户端确认（设计 §10）。

## R-04：踢人/旁观独立 + 空保护（P1-2）

- **目标文件**：`ContinuousBillingHandler.java`（`setKickerList`/`setWatchUsers`）
- **步骤**：
  1. `setKickerList`：`userBaseInfo`/`tableInfo`/`table` 任一为 null → warn + 返回空列表；权限判断在空值兜底后做。
  2. `setKickerList`/`setWatchUsers` 各自 try/catch，异常返回空列表 + warn + 指标。
- **测试**：userBaseInfo/tableInfo/table 缺失 → 踢人列表空、主账单正常；非房主非总代 → 踢人列表空。
- **验收**：附加块失败不影响主账单。
- **回滚**：移除独立 try/catch 与空保护。
