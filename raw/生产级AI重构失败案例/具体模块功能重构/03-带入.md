## 1. 功能边界

带入覆盖 1010 `CommonProto.TRANSFER_IN`、1011 `CommonProto.TRANSFER_IN_VIEW`、14009 `USER_SCORE_CHANGE_NOTICE`、14022 `OCCUPY_ADD_CHIPS_NOTICE`，以及私人审核房和掼蛋彩金钱包的扩展链路。

本功能负责:

- 带入弹窗金额计算: 最小带入、最大带入、可带入、是否需要额度申请、普通钱包和 `GD_BONUS` 彩金钱包。
- 普通钱包带入: `DxWalletType.GUANDAN` 上分、桌面筹码、下一手补码、账变明细。
- 彩金钱包带入: `DxWalletType.GD_BONUS` 的审核、上分、桌面彩金筹码和过级彩金补码。
- 占座转入座衔接: 带入完成后发布或同步运行 `OccupyToSitDownTask`。
- 结算补码衔接: 结算阶段补码、补码后继续状态、`IS_CONTINUE_SERVER_NOTICE` 刷新。

本功能不负责:

- 入房权限、桌子权限、用户列表初始化，归属 01。
- 坐下、占座状态创建和座位合法性，归属 02。
- 手动带出、退出后自动带出，归属 05/06。
- 审核单创建协议本身，归属 10/16；审核结果处理在本功能中作为回调消费。

## 2. 入口与调用链

主协议入口:

- `UserBringInCommand` 使用 `@ThreadModel(TABLE)`，`handleCommand` 委托 `UserBringInCommandDefault`，`getType` 返回 `CommonProto.TRANSFER_IN`。
- `UserBringInViewCommand` 使用 `@ThreadModel(PLAYER)`，委托 `UserBringInViewCommandDefault`，`getType` 返回 `CommonProto.TRANSFER_IN_VIEW`。

框架默认链路:

1. `UserBringInCommandDefault.handleCommand` 初始化 `UserTransferInContext`，执行 `check -> handle -> after`。
2. `init` 兼容旧字段 `addChipScore`，如果 `walletList` 为空则把旧金额转成默认钱包列表。
3. `check` 要求用户在 `PLAYER_USER`，固定带入模式校验固定金额，非固定带入校验钱包类型、金额正数、最多两位小数。
4. `handle` 调用 `UserBringInBootstrapService.check` 和 `transferInChipScore`。
5. `UserBringInDefault.transferInChipScore` 保存上分、刷新桌面筹码、响应客户端，并在占座场景发布 `OccupyToSitDownTask`。

掼蛋扩展链路:

- `UserBringInCommandBootstrap.handleCommand` 在普通框架链路前后增加结算补码用户识别、异常刷新补码弹窗、继续状态回写。
- `UserBringInBootstrap.check` 允许 `GUANDAN` 和 `GD_BONUS` 钱包，限制游戏中不能补彩金，并将私人审核、自动补至目标值、结算强制补码合并进金额校验。
- `UserBringInBootstrap.transferInAfterRefresh` 同时刷新普通钱包和彩金钱包，并在占座成功后同步运行 `OccupyToSitDownTask`、唤醒开局检查。
- `UserBringInBootstrap.dealBringInReviewDto` 消费审核结果，PASS 时调用 `processGuandanAuditTransferIn`，REJECT/TIMEOUT 时清理或恢复补码状态。

## 3. 数据流与状态

请求数据:

- `UserTransferInReqDto.addChipScore`: 旧客户端单金额字段。
- `UserTransferInReqDto.walletList`: 新钱包列表，元素包含钱包类型和带入金额。

弹窗响应:

- `UserTransferInViewRespDto` 输出 `balance`、`creditAvailableBalance`、`privateFirstDown`、`autoComplement`、`autoBringChipScore`、`needAmountApply`、`tableConfigMinBringChipScore`、`tableConfigMaxBringChipScore` 等字段。

带入响应:

- `UserTransferinResDto` 输出 `chipScore`、`nextBringInChipScore`、`balance`、`scoreChangeMsg`。

核心缓存状态:

- `BaseTableUserCache` 保存 `PLAYER_USER / WATCH_USER / QUIT_USER`。
- `BaseUserItem.status` 在 02 中已确认为准备阶段关键状态，带入链路读取 `OCCUPY` 并转为入座。
- `TableUserWalletCache` 保存桌面钱包、累计带入、下一手补码和账变明细。
- `TableHand.needAddChipUsersMap`、`playerIsContinue` 保存结算补码和继续状态。
- `RetryBringInReviewRedisService` 保存私人房审核单，`BringInReviewDto.bringInAmount` 与 `bringInSquidAmount` 分别记录主筹码和彩金。

## 4. 现有设计理解

现有实现把三类业务压在一个带入链路里:

- 普通坐下带入: 用户占座后补主筹码，成功后占座转入座。
- 游戏中补码: 已参与本手时写入 `nextBringInChipScore`，当前手结束后再入桌面。
- 私人审核 / 彩金补码: 审核单通过后再执行钱包上分，掼蛋扩展同时处理 `GUANDAN` 与 `GD_BONUS`。

这套设计的核心意图是保持客户端 1010/1011 协议不变，通过后端判定当前是普通补码、自动补至、彩金补码、固定带入审核还是结算补码。

## 5. 问题扫描

### P0-1 带入协议承载过多场景，缺少显式业务意图

证据:

- `UserBringInBootstrap` 注释明确说明 1010 没有标记能区分“补入100”和“自动补至100”，只能靠条件拦截。
- `UserBringInBootstrap.check` 同时处理普通带入、彩金、私人审核、自动补至目标值、结算补码。
- `UserBringInCommandBootstrap.isSettleComplementUser` 又基于 `TableHandStatus`、`playerIsContinue`、`needAddChipUsersMap` 二次识别结算补码。

影响:

- 新增补码场景时只能继续添加条件分支，容易和已有自动补至、固定带入、结算补码互相覆盖。
- 客户端只看到同一协议和相近金额字段，排查线上问题时很难从请求本身判断业务场景。

建议:

- 内部新增 `BringInScene`，在不改 1010 DTO 的前提下由服务端归一化出 `SIT_DOWN / MANUAL_IN_GAME / AUTO_TO_TARGET / SETTLEMENT_COMPLEMENT / PRIVATE_AUDIT / BONUS_COMPLEMENT`。
- `UserBringInCommandDefault.init` 只做协议兼容，场景识别交给独立 `BringInSceneResolver`。

### P0-2 钱包刷新与任务推进存在非事务边界

证据:

- `UserBringInDefault.transferInChipScore` 先 `saveTransferInChipScore`，再 `transferInAfterRefresh`，最后发布 `OccupyToSitDownTask`。
- `UserBringInBootstrap.transferInAfterRefresh` 在占座场景会直接 `new OccupyToSitDownTask(...).run()`，并唤醒 `ImmediateGameStartCheckService`。
- `TableUserWalletCache.addChipScoreAndRecordDetail`、`BaseTableUserHandCache.addTableUserHandSupply` 和用户状态变更分散在不同服务。

影响:

- 钱包上分成功但本地缓存刷新或占座转入座失败时，用户可能已经扣款/上分，但桌内仍停留 OCCUPY。
- 同步 `run()` 绕过事件队列时序，和 EventManager 中已存在的占座/超时任务关系不清晰。

建议:

- 引入 `BringInResultApplier`，以“钱包结果 -> 本地钱包快照 -> 用户状态 -> 通知 -> 开局唤醒”的顺序封装。
- 对审核回调和主动 1010 复用同一个 applier，返回结构中明确 `walletApplied`、`stateApplied`、`noticeSent`。

### P1-1 私人审核幂等依赖本地 Guava Cache

证据:

- `UserBringInDefault` 使用 `LoadingCache<String, AtomicBoolean> CACHE` 做审核处理幂等。
- `UserBringInBootstrap` 又定义 `AUDIT_CACHE` 与 `AUDIT_PASS_CACHE`。

影响:

- 多节点部署时，本地缓存只能保证单节点幂等；审核 MQ 或回调被不同节点消费仍可能重复处理。
- 本地缓存过期策略无法表达审核单终态，回放消息时行为不可预测。

建议:

- 幂等主键使用 `BringInReviewDto.id`，落到 Redis 审核单状态 CAS 或钱包 RPC 事件号。
- 本地 cache 只能作为性能优化，不能作为业务幂等判定。

### P1-2 彩金字段命名继承鱿鱼语义

证据:

- `BringInReviewDto.bringInSquidAmount` 在掼蛋中存放 `GD_BONUS` 带入金额。
- `AmountApplyBootstrap.amountApplyStore` 将 `DxWalletType.GD_BONUS` 写入 `bringInSquidAmount`。

影响:

- 掼蛋文档和代码阅读时会把彩金误读成鱿鱼池。
- 后续跨游戏复用审核单时，字段语义继续扩大，增加数据治理成本。

建议:

- 保持外部 DTO 字段不变，内部建立 `ReviewWalletAmount` 列表视图，将 `bringInSquidAmount` 映射为 `GD_BONUS`。
- 新代码禁止直接用 `squid` 命名承载掼蛋彩金。

## 6. 架构审视

当前链路的主要问题不是单个 if 复杂，而是“协议兼容、钱包核算、占座状态、审核状态、结算继续状态”全部互相直接读取缓存。

D1 单节点缓存: `BaseTableCache`、`BaseTableUserCache`、`TableUserWalletCache` 均为本地缓存，审核和 MQ 跨节点回放时必须有 Redis/RPC 终态兜底。
D2 并发容器: `TableHand` 同时有 `HashMap`、`ConcurrentHashMap`、`AtomicBoolean`，带入链路读写 `needAddChipUsersMap` 与 `playerIsContinue` 时缺少统一锁边界。
D5 状态机: `BaseUserItem.status` 被带入、入座、站起、退出、风控共用，必须沉淀状态转移表。
D9 Spring 静态取 Bean: `UserBringInBootstrap` 多处使用 `SpringContextUtils` / `SpringUtil`，测试隔离差。
D12 日志: 多处记录完整 `req`、`reviewDto`、钱包信息，需继续确认脱敏边界。

## 7. 改造建议

优先级:

1. P0: 建 `BringInFlowService`，对外保留 1010/1011，对内拆出场景识别、校验、钱包执行、状态推进、通知。
2. P0: 建 `BringInStateTransition`，只允许 `OCCUPY -> DEFAULT`、`DEFAULT -> DEFAULT(nextHand)`、`SETTLEMENT_WAIT -> OCCUPY/DEFAULT` 这些显式转移。
3. P1: 审核回调统一走 Redis 审核单 CAS，替换本地 cache 幂等判定。
4. P1: 将 `GD_BONUS` 金额封装成钱包列表，减少 `bringInSquidAmount` 直读。

## 8. 落地设计

建议类:

- `BringInCommandAdapter`: 只负责 DTO 兼容、旧字段转钱包列表、构造 command。
- `BringInSceneResolver`: 输入用户状态、桌子状态、钱包列表、用户配置，输出 `BringInScene`。
- `BringInValidatorChain`: 按场景组合校验器，复用 02 的入座状态结论。
- `BringInWalletService`: 统一封装 `upScore`、本地钱包刷新、账变记录。
- `BringInOccupyBridge`: 只处理带入完成后的占座转入座、占座超时任务移除和开局唤醒。
- `BringInAuditCallbackHandler`: 只消费审核结果，不直接拼钱包逻辑。

数据对象:

- `BringInCommand`: `tableId/userId/scene/walletAmounts/requestId/sourceProto`.
- `BringInPlan`: `immediateAmounts/nextHandAmounts/needAudit/needAmountApply`.
- `BringInApplyResult`: `walletSnapshots/stateChange/notices`.

## 9. 协议兼容确认

请求兼容:

- 1010 `UserTransferInReqDto.addChipScore` 保留，服务端继续在适配层转为默认钱包。
- 1010 `walletList` 保留，不新增必填字段；内部 `scene` 不下发。
- 1011 请求不变。

响应兼容:

- 1010 `UserTransferinResDto` 字段保留。
- 1011 `UserTransferInViewRespDto` 字段保留，`needAmountApply`、`autoComplement`、`autoBringChipScore` 语义不变。

通知兼容:

- 14009 `USER_SCORE_CHANGE_NOTICE` 保持原 DTO。
- 14022 `OCCUPY_ADD_CHIPS_NOTICE` 保持现有 map 字段，新增字段只能可选。
- 掼蛋继续状态通知 `IS_CONTINUE_SERVER_NOTICE` 不由本次改造改协议号。

## 10. 可测试性设计

单元测试:

- `BringInSceneResolverTest`: 覆盖普通坐下、游戏中补码、自动补至、结算补码、私人审核、彩金补码。
- `BringInValidatorChainTest`: 覆盖钱包类型、金额精度、固定带入、余额不足、彩金桌非彩金桌。
- `BringInAuditCallbackHandlerTest`: 覆盖 PASS/REJECT/TIMEOUT、重复回调、多钱包金额。

集成测试:

- 占座后带入成功转入座，并移除占座超时任务。
- 游戏中带入只增加 `nextBringInChipScore`，不立即改变当前桌面筹码。
- 私人审核通过后普通钱包和彩金钱包都刷新，并发送 14009。

## 11. 迁移与灰度

迁移步骤:

1. 保留现有 `UserBringInCommandDefault` 入口，在内部调用新 flow。
2. 新旧 flow 双写日志: 记录场景识别、钱包计划、状态转移，但只由旧 flow 生效。
3. 对指定桌台灰度启用新 flow，比较 1010/1011 响应和钱包快照。
4. 审核回调先启用 Redis CAS 幂等，再迁移钱包执行逻辑。

回滚:

- 保留原 `GameBaseBootstrapFactory` dataBoot 绑定，配置开关可切回旧实现。
- 新增内部字段不进入协议，回滚不影响客户端。

## 12. 验收场景

- 旧客户端只传 `addChipScore`，服务端仍成功带入默认钱包。
- 新客户端传 `walletList`，主钱包和彩金钱包按各自余额校验。
- 占座用户带入成功后，状态从 OCCUPY 进入 DEFAULT，并触发开局检查。
- 已参与本手的玩家补码，返回 `nextBringInChipScore`，本手桌面筹码不立即增加。
- 私人审核房非房主余额不足时，1011 返回 `needAmountApply=YES`。
- 审核通过重复回调不会重复上分。
- 审核拒绝或超时后，用户仍能按补码规则收到后续站起或弹窗处理。

## 13. 待确认项

- 自动补至目标值是否需要客户端显式展示“补至”与“补入”的不同文案。
- `GD_BONUS` 在审核单中的外部字段是否允许新增更准确的扩展字段。
- 审核回调 MQ 是否保证同一 `auditId` 有序；如果无序，必须先做 Redis CAS。
- 结算补码阶段 `playerIsContinue` 的 0/1 语义是否已和前端完全一致。

## 14. 本功能不处理的范围

- 不重写 1010/1011/14009/14022 协议。
- 不修改钱包 RPC 的账变模型。
- 不调整入房、入座、站起、退出的公共状态枚举。
- 不处理放弃补码协议缺失问题，该问题归属 17。
