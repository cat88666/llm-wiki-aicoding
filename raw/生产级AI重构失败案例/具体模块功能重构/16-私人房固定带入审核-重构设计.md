# 私人房固定带入审核重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/16-私人房固定带入审核.md`
> 设计日期：2026-06-04
> 工程：dx-game-guandan（识别/通知策略，掼蛋侧为主）
> 分支：20260615/junk
> 复杂度：H（P0 高，多链组合）
> 前置依赖：02-入座、03-带入（needAmountApply 弹窗）、10-私人房申请审核（建单）、11-私人房审核通过（审核回调）

## 1. 背景与目标

私人房固定带入审核**非独立协议**，是"私人审核房 + 非房主 + 首次入座 + 固定带入模式 + 需要额度申请"时，入座/带入弹窗/1078 额度申请/审核回调四条链路的组合。基于真实代码确认（scan 已定位）：

- **入座触发**：`UserSitDownBootstrap` 调 `SettleComplementAuditService.shouldKeepPrivateAuditUserForAmountApply`，命中→`SettleComplementNoticeService.sendSitDownAmountApplyNoticeWithoutToast`（保留额度申请弹窗 + 占座倒计时）。
- **坐下完成监听**：`UserSitDownFinishListener.resendOccupyAddChipNotice` 重新坐下/兜底转入座后再查补码结果，命中私人审核同样调 `sendSitDownAmountApplyNoticeWithoutToast`。
- **带入弹窗**：`UserBringInBootstrap.getCanBringChipScore` 对私人审核房设 `needAmountApply`。

问题：
- **P0-1：固定带入审核场景识别分散**。`getCanBringChipScore`（算 needAmountApply）、`shouldKeepPrivateAuditUserForAmountApply`（是否保留额度申请流程）、`UserSitDownBootstrap` 与 `UserSitDownFinishListener.resendOccupyAddChipNotice` **同一判断分支重复**。新增条件需改多入口；重新坐下/结算补码/首次坐下的前端弹窗类型易不一致。
- **P0-2：「已有待审核单」未纳入当前 keep 判断**。`shouldKeepPrivateAuditUserForAmountApply` 当前只判 `needAmountApply=YES`；同处有注释掉的 `hasPendingTransferAudit(tableId,userId)`（方法仍在，可判 Redis 未过期审核单）。用户已提交 1078 后若后续 viewResp 临时不返回 `needAmountApply=YES`，**可能不再保留审核等待态 → 被补码超时任务强制站起**（资金/体验风险）。
- **P1-1：`addChipType` 语义需更强约束**。`sendSitDownAmountApplyNoticeWithoutToast` 明确重新坐下额度申请必须 `AddChipTypeEnum.SIT_DOWN`，但普通补码 `resolveAddChipType` 可能返回 DEFAULT/FORCE。前端按 addChipType 决定文案，错误类型导致"额度申请"与"剩余筹码过少"叠加。

目标：**不新增入座/带入/1078 请求字段，`OCCUPY_ADD_CHIPS_NOTICE`/`needAmountApply` 字段与 addChipType 语义保持**前提下，把场景识别收口到 `PrivateFixedBringInAuditPolicy`、**恢复并测试 `needAmountApply || hasPendingTransferAudit` 的 keep 判断（修 P0-2）**、addChipType 由策略统一输出。**最小安全改造优先**。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Guandan 主导（识别/通知策略在 guandan `settle`/`bootstrap`） |
| 协议兼容 | 入座/带入/1078 请求不新增字段；`UserTransferInViewRespDto.needAmountApply`、`OCCUPY_ADD_CHIPS_NOTICE`、`addChipType` 语义保持 |
| 场景识别收口（P0-1） | 新增 `PrivateFixedBringInAuditPolicy.decide(...) → PrivateAuditDecision{needAudit, hasPendingAudit, keepOccupy, noticeAddChipType, suppressToast, reason}`；入座/监听/带入弹窗只调 policy，不复制分支 |
| **待审核单纳入 keep（P0-2，核心修复）** | 恢复 `shouldKeepPrivateAuditUserForAmountApply` = `needAmountApply=YES \|\| hasPendingTransferAudit(tableId,userId)`；对重复审核单与等待审核单分别打日志。**修复待审核用户被补码超时任务强制站起** |
| addChipType 统一（P1-1） | `PrivateAuditDecision.noticeAddChipType` 由 policy 输出，重新坐下额度申请固定 `AddChipTypeEnum.SIT_DOWN`；测试覆盖重新坐下/结算补码分支 |

| 灰度策略 | 新 policy 只读对比旧分支 → 启用 pending audit keep 判断 → 收口入座/监听调用 → 最后迁带入弹窗判断 |

## 3. 整体架构

### 改造前

```
入座 UserSitDownBootstrap → shouldKeepPrivateAuditUserForAmountApply(仅 needAmountApply=YES, P0-2)
                          → sendSitDownAmountApplyNoticeWithoutToast(保留弹窗+占座倒计时)
坐下完成 UserSitDownFinishListener.resendOccupyAddChipNotice → 同一判断分支(重复 P0-1)
带入弹窗 UserBringInBootstrap.getCanBringChipScore → 设 needAmountApply
（hasPendingTransferAudit 已注释掉, 未纳入 keep 判断 P0-2）
addChipType: sendSitDownAmountApplyNoticeWithoutToast=SIT_DOWN, 但 resolveAddChipType 可能 DEFAULT/FORCE (P1-1)
```

### 改造后

```
PrivateFixedBringInAuditPolicy.decide(tableInfo, userItem, viewResp) → PrivateAuditDecision
  ├─ needAudit = isPrivateTransferAudit && holderId!=userId && (needAmountApply=YES || hasPendingTransferAudit)  (R-02)
  ├─ keepOccupy / suppressToast
  └─ noticeAddChipType = 重新坐下额度申请→SIT_DOWN (R-03)
入座 UserSitDownBootstrap            → 调 policy（不复制分支 R-01）
坐下完成 resendOccupyAddChipNotice   → 调 policy（不复制分支 R-01）
带入弹窗 getCanBringChipScore        → 调 policy 判定 needAmountApply（统一来源）
PrivateAuditNoticeService            → 按 decision 发 OCCUPY_ADD_CHIPS_NOTICE(addChipType=SIT_DOWN, suppressToast)
PrivateAuditPendingService           → hasPendingTransferAudit 封装 + 重复/等待审核单日志
```

**兼容性原则**：
- 入座/带入/1078 请求、`needAmountApply`、`OCCUPY_ADD_CHIPS_NOTICE`、`addChipType` 枚举值/语义不变。
- `SettleComplementAuditService`/`SettleComplementNoticeService` public 签名保留（内部委托 policy）。
- 用户 `status=OCCUPY` 不变（OccupyReason 仅内部）。
- 建单（10）/审核回调（11）/普通补码（03）链路不动。

## 4. 安全改造（R-01 ~ R-03）

### R-01：场景识别收口 `PrivateFixedBringInAuditPolicy`（P0-1）
- 新增 `PrivateFixedBringInAuditPolicy` 输出 `PrivateAuditDecision`；`UserSitDownBootstrap`/`UserSitDownFinishListener.resendOccupyAddChipNotice`/`UserBringInBootstrap.getCanBringChipScore` 改为调 policy，删除各自重复分支。

### R-02：待审核单纳入 keep 判断（P0-2，核心修复）
- `shouldKeepPrivateAuditUserForAmountApply` 恢复为 `needAmountApply=YES || hasPendingTransferAudit(tableId,userId)`（封装到 `PrivateAuditPendingService`）。
- 修复：用户已提交 1078 待审核时，即使 viewResp 临时不返回 `needAmountApply`，仍保留占座/审核等待态，不被补码超时任务强制站起。
- 重复审核单 / 等待审核单 分别日志。

### R-03：addChipType 由策略统一输出（P1-1）
- `PrivateAuditDecision.noticeAddChipType`：重新坐下额度申请固定 `AddChipTypeEnum.SIT_DOWN`；`PrivateAuditNoticeService` 按 decision 发 `OCCUPY_ADD_CHIPS_NOTICE`，避免"额度申请"与"剩余筹码过少" toast 叠加。

## 7. 可测试性设计

- `PrivateFixedBringInAuditPolicyTest`：私人审核房非房主固定带入 needAudit；房主不需要；非私人审核房不需要；needAmountApply=YES 保留占座；**已有未过期审核单即使 viewResp 变化也保留占座（P0-2）**；addChipType=SIT_DOWN。
- 集成：首次固定带入余额不足→额度申请弹窗；提交 1078 后不被超时任务站起；审核通过占座转入座；审核拒绝按补码/站起。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | policy 只读对比旧分支 → 启用 pending audit keep 判断 → 收口入座/监听 → 迁带入弹窗判断 |
| 回滚 | 关闭 policy 开关恢复旧分支；不改外部协议 |
| 与其他 Junk 同步 | 占座转入座/补码见 02/03；建单见 10；审核回调见 11；OCCUPY status 语义不动 |

## 9. 验收关键场景

1. 固定带入非房主额度不足 → 触发审核（额度申请弹窗）。
2. 房主固定带入 → 不触发审核。
3. 彩金固定带入金额进审核单。
4. **提交 1078 后即使 viewResp 临时变化，仍保留占座/审核等待，不被超时站起（P0-2）**。
5. 重新坐下补发额度申请弹窗 addChipType=SIT_DOWN，不出现普通补码 toast。
6. 审核通过后继续开局。
