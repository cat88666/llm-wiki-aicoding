## 1. 背景与目标

当前站起协议 1007 由 `UserStandCommand → UserStandCommandDefault → UserStandTask → UserLeaveBootstrapService.playerStand` 串联，存在：

- **P0**：`UserStandCommandDefault.handle` 发布站起任务时使用 **退出任务** 的 event key —— `EventUtils.getKey(OnceEventTypeEnum.USER_QUIT_TASK, tableId, userId)`。`OnceEventTypeEnum` 里 **已有** `USER_STAND_TASK` 定义但没被使用，结果同一用户并发触发 1007/1006 时 EventManager 中两个任务会互相覆盖，且站起任务在 event queue 里以 "玩家退出" 显示，给排障增加噪音。
- **P1**：`UserStandNotice.notice` 组装通知时直接读 `TableUserWalletCache.getChipScore(tableId, userId)`，钱包缓存缺失时会抛异常 —— 站起主流程已经完成（PLAYER → WATCH 转移），但通知组装失败，客户端看不到状态更新；同时桌内其他人也收不到 14007。
- **P2**：`UserStandCommandDefault.handleCommand` / `UserStandNotice.notice` / `UserStandTask.run` 大量 `info` 级 `GsonUtils.toJson(req/context/userStandNoticeDto)`，含设备 / 用户上下文，高峰 CPU 与日志量放大。
- **P0（不在本期）**：站起、退出共用 `commonCheckUserLeave` 但业务语义不同（站起保留筹码，退出自动带出），缺显式 `LeaveIntent` 模型。这是跨 04/05/06/07 四个功能的结构性改造，留给 06 退出 一起做（届时引入 `LeaveIntent` + `LeavePermissionService`）。

目标：消除任务 event key 冲突的 P0，加固通知组装的 NPE 兜底，收敛 1007 链路含敏感 payload 的 info 日志，保持 1007/14007 协议字段 100% 兼容。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导。改造完全位于 `dx-game-frame`，掼蛋无新增/无修改 |
| `LeaveIntent` 大改 | **不在本期**。留给 06 退出 一起做。本期不引入新枚举，不改 `commonCheckUserLeave` 签名 |
| 任务 event key 修复方式 | 把 `UserStandCommandDefault.handle` 中 `USER_QUIT_TASK` 改为 `USER_STAND_TASK`（枚举常量已存在，零迁移成本） |
| 旧 `USER_QUIT_TASK` 残留 key 处理 | **不主动清理**。新旧 key 字符串完全不同（`TABLE_USER_QUIT_*` vs `TABLE_USER_STAND_*`），不存在脏数据冲突；EventManager 内现有残留任务跑完后自然消失 |
| `UserStandNotice` 钱包兜底策略 | `TableUserWalletCache.getChipScore` 抛任何异常 → 设 chipScore = 0 并 warn 日志，**不**中断通知发送 |
| `UserStandRespDto` 字段是否新增 | 不新增。`status` 字段未改；金额字段语义不变，仅在异常时返回 0 |
| 日志收敛策略 | 整段 `GsonUtils.toJson(context/req/userStandNoticeDto)` 全部降 `debug`；info 仅保留 tableId/userId/standType/kickerReason 元数据 |
| 灰度开关 | 不做。event key 修复直接生效；通知兜底是纯防御代码 |

## 3. 整体架构

### 改造前

```
UserStandCommand (TABLE 线程, 1007)
  └─ UserStandCommandDefault.handleCommand
       ├─ init() — standType=DEFAULT
       ├─ check() — commonCheckUserLeave
       └─ handle()
            └─ EventUtils.getKey(USER_QUIT_TASK, tableId, userId)  ← P0 用错枚举
            └─ eventManager.publish(new UserStandTask(...))

UserStandTask.run() (frame)
  └─ UserLeaveBootstrapService.playerStand(context)
       └─ GdUserLeaveBootstrapService.playerStand
            └─ ...
                 └─ UserStandNotice.notice(context, bringOutResult, userItem)
                       └─ TableUserWalletCache.getChipScore(...)  ← P1 NPE 风险
                       └─ SendUtil.sendNotice(...)               (14007)
```

### 改造后

```
UserStandCommand (TABLE 线程, 1007)
  └─ UserStandCommandDefault.handleCommand
       ├─ init() — standType=DEFAULT
       ├─ check() — commonCheckUserLeave
       └─ handle()
            └─ EventUtils.getKey(USER_STAND_TASK, tableId, userId) ← R-01 修复
            └─ eventManager.publish(new UserStandTask(...))

UserStandTask.run() (frame, 日志降级 debug)
  └─ UserLeaveBootstrapService.playerStand(context)
       └─ ...
            └─ UserStandNotice.notice(...)
                  └─ safeFetchChipScore(tableId, userId) ← R-02 try/catch + warn
                  └─ SendUtil.sendNotice(...)            (14007 永不被钱包阻塞)
```

**兼容性原则**：
- 协议号 1007 请求/响应 / 14007 通知字段不变
- `UserStandCommand`、`UserStandCommandDefault`、`UserStandTask` public 方法签名不变
- `UserStandContext` 字段不变；`UserStandRespDto` / `UserStandNoticeDto` 不动
- `UserLeaveBootstrapService.playerStand` 调用契约不变
- 调用方 `UserStandCommand` / `UserSpecifyCommandDefault`（强制站起）/ `BaseTableRiskService.realtimeCheckRisk`（08 已重构）/ `RiskStandLeaveUserVisitor` 全部无感

## 4. 安全改造

### R-01：修正站起任务 event key（P0）

**文件**：`UserStandCommandDefault.java` L87

```java
// 改前
String eventIdKey = EventUtils.getKey(OnceEventTypeEnum.USER_QUIT_TASK, tableId, userId);
// 改后
String eventIdKey = EventUtils.getKey(OnceEventTypeEnum.USER_STAND_TASK, tableId, userId);
```

`OnceEventTypeEnum.USER_STAND_TASK("TABLE_USER_STAND_%s_UID_%s", "玩家站起")` 已存在（见 `OnceEventTypeEnum.java:26`），无需新增枚举值。

### R-02：`UserStandNotice` 钱包兜底（P1）

**文件**：`UserStandNotice.java`

新增 private helper：

```java
private long safeFetchChipScore(Long tableId, Long userId) {
    try {
        Long chipScore = TableUserWalletCache.getChipScore(tableId, userId);
        return chipScore == null ? 0L : chipScore;
    } catch (Exception e) {
        log.warn("用户站起通知-读取桌面筹码失败,使用0兜底 tableId:{} userId:{}", tableId, userId, e);
        return 0L;
    }
}
```

`notice(...)` 内：
```java
userStandNoticeDto.setChipScore(safeFetchChipScore(tableId, userId));
```

> 注：`UserStandNoticeDto.chipScore` 字段类型与原 `TableUserWalletCache.getChipScore` 返回保持一致（实际为 `Long`）。兜底返回 `0L`，客户端约定"未知/异常 → 显示 0"。

### R-03：日志收敛（P2）

| 位置 | 改前 | 改后 |
|------|------|------|
| `UserStandCommandDefault.handleCommand` | `log.info("用户站起-请求协议 req {}", GsonUtils.toJson(req))` | `log.info("用户站起-请求协议 tableId:{} userId:{}", req.getTableId(), req.getUserId())`；toJson 改 debug |
| `UserStandCommandDefault.handle` 用户不在 PLAYER 列表分支 | `log.info("用户站起-里面不存在该用户,userId:{},context:{},uidList:{}", ...)` | `log.info("用户站起-用户不在 PLAYER_USER tableId:{} userId:{} standType:{}", ...)`；context/uidList toJson 改 debug |
| `UserStandTask.run` | `log.info("用户站起任务,context:{}", GsonUtils.toJson(context))` | `log.info("用户站起任务 tableId:{} userId:{} standType:{}", ...)`；完整 context 改 debug |
| `UserStandNotice.notice` 开始 | `log.info("用户站起通知开始,context:{}", GsonUtils.toJson(context))` | `log.info("用户站起通知开始 tableId:{} userId:{} standType:{}", ...)`；context 改 debug |
| `UserStandNotice.notice` 结束 | `log.info("用户站起通知完成,userStandNoticeDto:{}", GsonUtils.toJson(userStandNoticeDto))` | `log.info("用户站起通知完成 tableId:{} userId:{} chairId:{}", ...)`；DTO 改 debug |
| 异常路径 | `log.warn/error("...,req:{}", GsonUtils.toJson(req), e)` | 保留 warn/error，但 req JSON 替换为 `tableId+userId` 元数据 |

## 10. 验收关键场景

1. **event key 冲突消除**：同一用户先 1007 后 1006，EventManager 中能看到两个独立 key 并存（`TABLE_USER_STAND_*` 和 `TABLE_USER_QUIT_*`），不再互相覆盖。
2. **钱包缺失兜底**：mock `TableUserWalletCache.getChipScore` 抛 NPE，1007 链路仍然发出 14007 通知，`chipScore=0`，warn 日志能定位失败 userId。
3. **日志体积**：1 次正常 1007 产生 ≤ 4 条 info（请求 / 任务开始 / 通知开始 / 通知完成），全部只含 tableId+userId+standType 元数据，无 toJson 大对象。
4. **强制站起链路兼容**：08 重构后 `realtimeCheckRisk` 调 `UserLeaveBootstrapService.userStand(...)` → `playerStand(...)` → `UserStandNotice.notice(...)`，全链路行为不变；新钱包兜底在风控站起场景同样生效。
5. **占座转 TRANSFER**：占座用户关闭补码弹窗触发的站起类型修正逻辑不变（在 `GdUserLeaveBootstrapService.playerStand` 中，不在本期改动范围）。
