# Junk-游戏准备重构指导

> 本文档是 `Junk-游戏准备重构` 的总入口，用于统一工作流程、任务清单、任务完成度、核心产物和未决问题。

## 一、文档定位

本目录用于沉淀 Junk 负责的游戏准备模块重构分析。每个大功能一份扫描文档，必要时继续拆出重构设计文档和实现计划文档。

核心目标：

- 统一重构工作的入口文档，避免任务散落。
- 明确每个功能的扫描文档、设计产物、实现计划和完成度。
- 保留每个任务仍然未知的问题，方便后续评审、排期和验证。
- 控制 Junk 范围，不把其他负责人模块混入当前任务清单。
- 提高文档作为 AI 开发输入材料的可执行性、可验证性和可回退性。

## 二、工作流程

```text
Junk-游戏准备重构
├── 1. 选择任务
│   ├── 从任务清单选择一个功能
│   ├── 确认前置依赖文档是否已完成
│   └── 确认任务是否属于 Junk 负责范围
├── 2. 深度扫描
│   ├── 使用 ../prompt/Junk-AI重构功能扫描Prompt.md
│   ├── 扫描 dx-game-guandan 与 dx-game-frame
│   ├── 填写对应功能扫描文档
│   └── 输出问题、风险、改造建议和核心理由
├── 3. 重构设计
│   ├── 明确重构目标和非目标
│   ├── 明确 Frame 与 Guandan 的职责边界
│   ├── 明确模型、服务、接口、状态和异常处理
│   └── 输出 xx-重构设计.md
├── 4. 实现计划
│   ├── 将设计拆成可执行步骤
│   ├── 标记任务依赖和验证点
│   ├── 保留实现过程中的未知问题
│   └── 输出 xx-实现计划.md
├── 5. 实施验证
│   ├── 按实现计划推进代码改造
│   ├── 补充必要测试和回归点
│   ├── 更新任务完成度
│   └── 记录新增风险和遗留问题
└── 6. 收尾复盘
    ├── 汇总最终核心产物
    ├── 汇总仍未解决的问题
    ├── 确认上线、回滚、观测关注点
    ├── 更新本指导文档
    └── 更新git变更记录
```

## 四、完成度口径

| 完成度项 | 口径 |
|---|---|
| 扫描 | 对应 `xx-功能.md` 已完成问题扫描、风险归纳、改造建议和核心理由 |
| 设计 | 对应 `xx-功能-重构设计.md` 已完成目标方案、边界、模型、接口、异常和验证设计 |
| 计划 | 对应 `xx-功能-实现计划.md` 已完成可执行步骤、依赖、验证点和任务拆分 |
| 实施 | 代码改造、测试、回归、上线观察完成后再更新，不用仅凭文档完成推断 |

## 五、AI 开发输入质量门禁

本节用于提升三个置信度：

| 指标 | 当前参考值 | 目标值 | 提升关键 |
|---|---:|---:|---|
| 作为 AI 开发输入材料 | 78% | 88% ~ 92% | 补代码证据、补 AI 开发卡、补测试命令 |
| 直接按文档全量自动实现 | 55% | 68% ~ 75% | 待确认项闭环，代码级步骤明确 |
| 综合可投入置信度 | 68% | 80% ~ 85% | 增加质量门禁和试开发验证 |

`直接按文档全量自动实现` 不追求 90% 以上。Junk 游戏准备涉及协议、资金、状态机、多人并发和跨模块补偿，必须保留人工评审。

### 5.1 准入标准

一个功能进入 AI 开发前，必须同时满足：

- 扫描文档完成，且 14 个主章节齐全。
- P0/P1 问题均有代码证据、影响说明、改造建议和验收场景。
- 设计文档明确目标、非目标、协议兼容、状态变化、异常处理和回滚方式。
- 实现计划拆成可执行小任务，每个任务都有目标文件、修改步骤、测试要求和验收标准。
- 阻塞型待确认项已关闭，或明确标记为 `BLOCKED`，不能交给 AI 自行决策。
- 文档记录的代码版本与当前开发基线一致；不一致时必须先复扫相关代码。

### 5.2 质量门禁评分

每篇功能文档按 100 分评分，低于 85 分不得进入 AI 开发。

| 维度 | 分值 | 要求 |
|---|---:|---|
| 章节完整，边界清楚 | 10 | 功能范围、非目标、依赖关系明确 |
| 入口 / 调用链有代码证据 | 15 | 有文件路径、类名、方法名，关键问题必须有行号或 grep 命令 |
| 状态 / 缓存 / RPC / MQ / 通知清楚 | 15 | 说明读写点、状态变化和副作用 |
| P0/P1 有证据和影响 | 15 | 不能只有结论，必须说明线上风险 |
| 改造方案可拆任务 | 15 | 能拆成 1 天内可完成的小任务 |
| 协议兼容明确 | 10 | 请求、响应、通知 DTO 字段不变或只新增可选字段 |
| 测试和验收可执行 | 10 | 有单测、集成、回归或冒烟命令 |
| 待确认项已分类 | 5 | 区分产品决策、协议决策、技术假设、测试口径 |
| 回滚 / 灰度 / 观测明确 | 5 | 有配置开关、旧 flow 保留、revert 或观测指标 |

### 5.4 待确认项分类

`待确认项` 不能全部留成普通问题。进入 AI 开发前必须分类处理。

| 类型 | 处理方式 |
|---|---|
| 产品决策 | 必须人工确认；未确认则标记 `BLOCKED` |
| 协议决策 | 必须确认废弃、适配或完整实现 |
| 技术实现细节 | 给出默认假设和保守实现方案 |
| 测试口径 | 写成验收标准，不允许只保留疑问 |
| 跨负责人边界 | 写清调用契约，不替其他负责人重构 |

阻塞型示例：

- 14 旁观 / 返回 / 重连：1012/1013 是废弃，还是适配到 1000。
- 17 放弃补码：1042 是否需要实现，还是明确返回不支持。
- 18 预站起 / 留座离桌：1045 是否仍使用，1034 是否属于掼蛋范围。

### 5.5 试开发校准

批量 AI 开发前，必须先选择 3 个样本做试开发：

| 样本类型 | 推荐功能 | 目的 |
|---|---|---|
| 小修复 | 20 好友备注 | 验证文档是否能支撑低风险代码修复 |
| 中等改造 | 21 用户配置查询 / 修改 | 验证协议兼容和缓存更新口径 |
| 高风险链路 | 03 带入 或 14 旁观 / 返回 / 重连 | 验证状态、资金、重连类复杂任务 |

每个样本必须记录：

- AI 是否需要额外读代码才能理解任务。
- 是否出现文档误导或证据缺失。
- 是否能按文档写出测试。
- review 发现的问题数量和严重程度。
- 最终修改是否保持协议兼容。

试开发通过后，再批量推进同类任务。试开发发现文档缺口时，先修文档，再继续开发。

### 5.6 推荐提升路径

按以下顺序提升置信度：

1. 统一 01 ~ 21 的代码版本基线。
2. 给 03 ~ 21 补文件路径、方法、行号或搜索命令级证据。
3. 给每个 P0/P1 增加 `AI 开发卡`。
4. 关闭 14/17/18 等阻塞型协议策略问题。
5. 给每篇文档按 100 分门禁评分。
6. 选择 3 个样本做 AI 试开发。
7. 根据试开发结果调整模板，再批量推进。

## 六、维护规则

- 新增任务时，先补充本清单，再创建对应任务文档。
- `任务文档` 指功能扫描文档。
- `任务核心产物(md)` 只记录重构设计、实现计划、复盘结论等核心产物，不记录临时草稿。
- `任务完成度` 必须按扫描、设计、计划、实施分阶段维护，不能只写一个模糊百分比。
- `任务存在的未知的问题` 必须保留未确认事项，不要提前写成结论。
- 如果某个功能被合并到其他功能推进，应在任务清单中保留原序号，并明确合并去向。

## 七、git 变更记录

所有进入 `Junk-游戏准备重构` 范围的代码变更，都必须在提交前或提交后补充 git 变更记录，保证每次变更可追溯、可审核、可回退。

详细记录统一维护在：[Junk-游戏准备重构git变更记录.md](Junk-游戏准备重构git变更记录.md)。

### 记录要求

- 每次变更必须对应一个明确任务，优先关联本指导文档任务清单中的序号和功能。
- 每次变更必须记录 git 分支、提交号、变更文件、核心改动、验证结果和回退方式。
- 每次变更必须说明对应的任务文档、重构设计文档或实现计划文档。
- 每次变更必须记录审核状态，未审核、已审核、审核驳回要明确区分。
- 每次变更必须记录未知问题，不能把未验证事项写成已完成结论。
- 多个功能混合提交时，必须在记录中拆分影响范围，不允许只写一个笼统说明。

### 记录格式

每条记录必须按以下字段维护：

| 字段 | 说明 |
|---|---|
| 记录编号 | 使用 `GIT-YYYYMMDD-序号`，例如 `GIT-20260604-001` |
| 任务序号 / 功能 | 对应任务清单中的序号和功能 |
| 关联文档 | 任务文档、重构设计文档、实现计划文档 |
| git 分支 | 本次变更所在分支 |
| commit / MR | commit hash、MR 地址或待提交状态 |
| 变更范围 | 涉及模块、类、接口、配置、测试 |
| 核心变更 | 用 3 到 5 条说明实际改了什么 |
| 验证结果 | 编译、单测、回归、日志、人工验证结果 |
| 审核状态 | 未审核 / 已审核 / 审核驳回 |
| 回退方式 | revert commit、回滚文件、关闭开关或恢复旧实现 |
| 未知问题 | 尚未确认、尚未验证、依赖外部条件的问题 |
