## 1. 功能边界

站起覆盖 1007 `CommonProto.USER_STAND`、14007 `USER_STAND_NOTICE`，以及房主/中控/风控/结算/补码失败等系统站起场景。

本功能负责:

- 用户主动站起请求。
- 占座用户关闭补码弹窗后的站起类型修正。
- 过级、结算、风控、现金不足、强制站起等离桌前校验。
- 站起后从 `PLAYER_USER` 转为 `WATCH_USER`。
- 站起后保留筹码、过级彩金结算、轮服务费扣除和通知。

本功能不负责:

- 退出房间并清理桌内用户，归属 06。
- 手动带出查询与下分，归属 05。
- 预站起标记设置，归属 18。
- 风控数据计算，归属 08/19。

## 2. 入口与调用链

主动站起入口:

1. `UserStandCommand` 使用 `@ThreadModel(TABLE)`，`getType` 返回 `CommonProto.USER_STAND`。
2. `UserStandCommandDefault.handleCommand` 执行 `init -> check -> handle`。
3. `init` 默认设置 `isSystem=false`、`standType=DEFAULT`。
4. `check` 调用 `UserLeaveBootstrapService.commonCheckUserLeave`，只允许 `BaseUserLeaveService.OK_LEAVE_LIST`。
5. `handle` 如果用户不在 `PLAYER_USER`，直接返回成功；否则发布 `UserStandTask`。
6. `UserStandTask.run` 调用 `UserLeaveBootstrapService.playerStand(context)`。

掼蛋实际执行:

- `GdUserLeaveBootstrapService.playerStand` 执行共用离开检查、旁观上限、占座转 `TRANSFER`、保存 `WATCH_USER`、过级彩金结算、轮服务费、筹码保留、站起事件和通知。
- `UserStandNotice.notice` 响应 1007 并广播 14007，通知中读取桌面筹码。

## 3. 数据流与状态

输入:

- `UserStandReqDto` 继承基础请求，业务参数主要来自服务端上下文。
- `UserStandContext.standType` 区分默认站起、风控站起、房主强制、补码失败、预站起等。
- `UserStandContext.kickerReason` 进入进出记录与前端提示。

状态:

- `BaseTableUserCache.PLAYER_USER -> WATCH_USER` 是主动站起的核心转移。
- 占座用户站起时，`UserLeaveDefault.playerStand` 和 `GdUserLeaveBootstrapService.playerStand` 会把默认站起修正为 `UserStandTypeEnum.TRANSFER`。
- `BaseUserItem.prepStandStatus` 在 `BaseUserStandFinishedListener` 中清零。
- 掼蛋站起不自动全额带出，`leaveChipRetainService.retainTableChipsAfterLeave` 会保留桌面筹码。

通知:

- 1007 响应给本人。
- 14007 广播站起通知。
- 站起后还可能触发 14009 分数变化、结算补码弹窗或继续状态通知。

## 4. 现有设计理解

站起和退出共用离开检查，但业务语义不同:

- 站起: 人离开座位但仍在房间内，转为旁观，掼蛋场会保留桌面筹码。
- 退出: 人离开房间，进入 `QUIT_USER`，通常需要自动带出。

掼蛋扩展把站起放在结算和过级彩金链路中处理，核心原因是“站起后仍可能保留桌面筹码”，不能套用框架德州的离桌自动带出逻辑。

## 5. 问题扫描

### P0-1 站起、退出共用检查但缺少显式离开意图模型

证据:

- `UserStandCommandDefault.check` 和 `UserQuitCommandDefault.check` 都调用 `commonCheckUserLeave`。
- `GdUserLeaveBootstrapService.checkSitDownUserRealCanLeave` 内同时判断结算、过级、风控、现金不足、房主强制、补码。
- `GdUserLeaveBootstrapService.playerStand` 与 `playerQuit` 大量重复，但一个保留筹码，一个自动带出。

影响:

- 新增“结算中允许站起但不允许退出”这类规则时，容易误改共用校验导致两条链路一起变化。
- 站起保留筹码、退出自动带出的差异藏在执行阶段，前置校验不能表达差异。

建议:

- 引入 `LeaveIntent.STAND / QUIT / FORCE_STAND / FORCE_QUIT / RISK_STAND / TRANSFER_FAILED`。
- `LeavePermissionService` 输出 `LeaveDecision`，包含 `allowed`、`delayType`、`bringOutPolicy`、`retainChipPolicy`。

### P0-2 站起任务事件 key 使用退出任务枚举

证据:

- `UserStandCommandDefault.handle` 发布 `UserStandTask` 时使用 `EventUtils.getKey(OnceEventTypeEnum.USER_QUIT_TASK, tableId, userId)`。
- `UserQuitCommandDefault.handle` 也使用 `OnceEventTypeEnum.USER_QUIT_TASK`。

影响:

- 同一用户同时触发站起与退出时，事件 key 可能互相覆盖或删除。
- 排查 EventManager 队列时，站起任务显示为 quit 语义，增加误判概率。

建议:

- 新增或使用独立 `USER_STAND_TASK` 事件类型。
- 迁移时同时保留旧 key 清理，避免旧任务残留。

### P1-1 `checkSitDownUserRealCanLeave` 存在空指针和职责过载

证据:

- `GdUserLeaveBootstrapService.checkSitDownUserRealCanLeave` 获取 `TableInfo` 后直接读取 `raceGlobalConfig`。
- 同方法读取 `TableHand`、结算补码、风控、强制离开、过级配置等多个上下文。

影响:

- 表缓存短暂缺失时，主动站起会被系统异常吞并为通用错误。
- 每个新增限制都继续塞入该方法，难以构造单元测试。

建议:

- 拆为 `LeaveRule` 链: `TableExistsRule`、`SettlementComplementRule`、`RiskRule`、`TurnedEggRule`、`ForcedLeaveRule`。
- 每条规则只输出明确拒绝原因或通过。

### P1-2 站起通知读取钱包缓存缺少失败兜底

证据:

- `UserStandNotice.notice` 组装通知时读取 `TableUserWalletCache.getChipScore`。
- `TableUserWalletCache.getWallet` 依赖 `UserTableItemCache.getUserTableItem`，缺失时会抛异常。

影响:

- 站起主流程已完成但通知组装失败时，客户端可能看不到最终状态。
- 钱包缓存问题会扩大为离桌状态感知问题。

建议:

- 通知 DTO 组装增加钱包快照兜底，失败时仍发送站起状态，金额字段记录为 null 或 0 并打指标。

## 6. 架构审视

D1 单节点缓存: 站起依赖 `BaseTableUserCache`、`TableHandCache`、`TableUserWalletCache`，跨节点任务回放需要按 table 线程或 Redis 状态确认。
D5 状态机: `PLAYER -> WATCH`、`OCCUPY -> WATCH`、`PLAYER -> QUIT` 的边界分散在 stand/quit/bringout。
D8 事件语义: 站起任务使用退出事件 key，破坏任务可观测性。
D11 数据结构: `TableHand` 的补码、继续、过级状态是多张 map，站起规则直接读取。
D12 日志: 离开检查日志包含完整用户/桌子快照，需要继续脱敏。

## 7. 改造建议

优先级:

1. P0: 独立 `StandFlowService`，只负责 `STAND` 意图。
2. P0: 修正任务事件类型，建立站起、退出、带出任务的统一命名。
3. P1: 抽离 `LeaveRuleChain`，站起和退出按不同意图复用规则。
4. P1: 通知组装与状态变更解耦，站起成功不被通知金额读取失败阻断。

## 8. 落地设计

建议类:

- `StandCommandAdapter`: 适配 1007 请求。
- `LeaveIntentResolver`: 识别主动、系统、房主、风控、补码失败。
- `StandRuleChain`: 只执行站起可用规则。
- `StandStateService`: 统一执行 `PLAYER/OCCUPY -> WATCH`。
- `StandAfterService`: 执行筹码保留、过级彩金、轮服务费、事件通知。
- `StandNoticeAssembler`: 独立组装 1007/14007。

关键设计:

- `StandStateChange` 必须记录 before/after user area、status、standType、kickerReason。
- 所有站起后动作读取同一份 `StandExecutionSnapshot`，避免各自重新从缓存取。

## 9. 协议兼容确认

请求兼容:

- 1007 请求不新增必填字段。
- 强制站起仍通过服务端上下文设置 `standType` 和 `kickerReason`。

响应兼容:

- 1007 响应 DTO 保持。
- 14007 通知字段保持，金额字段语义不变。

通知兼容:

- 站起后可能联动 14009，不变更协议号。
- 房主/风控强制站起前端提示仍由既有 `kickerReason` 映射。

## 10. 可测试性设计

单元测试:

- `StandRuleChainTest`: 覆盖普通可站起、过级限制、结算补码、风控、现金不足。
- `StandStateServiceTest`: 覆盖 `PLAYER -> WATCH`、`OCCUPY -> WATCH`、已不在玩家列表的幂等成功。
- `StandTaskKeyTest`: 确认站起和退出任务 key 不互相覆盖。

集成测试:

- 占座用户关闭补码弹窗，站起类型为 `TRANSFER`。
- 掼蛋主动站起后用户仍在房间旁观，桌面筹码保留。
- 结算中允许站起场景，继续状态和补码弹窗不丢。
- 风控站起触发后通知本人和桌内其他用户。

## 11. 迁移与灰度

迁移步骤:

1. 先引入 `LeaveIntent` 和站起专用任务 key，旧任务 key 做兼容清理。
2. 将 `GdUserLeaveBootstrapService.playerStand` 内重复逻辑迁入 `StandFlowService`。
3. 打开只读影子日志，比较旧链路和新链路的 `StandDecision`。
4. 单桌灰度启用，观察 1007/14007、用户区域、钱包快照。

回滚:

- 保留原 `UserStandCommandDefault` 调用路径。
- 新增事件 key 回滚时同时清理新旧 key。

## 12. 验收场景

- 玩家主动站起成功后进入 `WATCH_USER`，14007 广播。
- 玩家已不在 `PLAYER_USER` 时，1007 幂等返回成功。
- 占座补码失败站起，站起类型为 `TRANSFER`。
- 过级中途离开限制命中时，1007 返回明确业务异常。
- 结算中站起保留筹码，退出链路不被误触发。
- 风控站起和房主强制站起的原因码正确落进通知和进出记录。

## 13. 待确认项

- 掼蛋站起保留筹码是否所有场次都成立，私人房固定带入是否有例外。
- 站起任务是否允许和退出任务并发，产品上哪个动作优先。
- `UserStandNotice` 金额字段失败时前端是否接受 null/0 兜底。
- 过级彩金结算中站起是否必须等待彩金结算结果再发 14007。

## 14. 本功能不处理的范围

- 不改退出后的自动带出策略。
- 不改风控算法。
- 不改预站起协议 1045。
- 不重写钱包 RPC 或账变记录。
