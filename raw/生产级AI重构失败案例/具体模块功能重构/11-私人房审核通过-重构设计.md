> 对应分析文档：`docs/Junk-游戏准备重构/11-私人房审核通过.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame + dx-game-guandan
> 分支：20260615/junk
> 复杂度：H（P0 高，资金 + 异步回调）
> 前置依赖：10-私人房申请审核（建单）、03-带入（复用 Redis CAS 幂等 / ResultApplier / 占座转入座 02 表级锁 / ReviewWalletAmount）、16-私人房固定带入审核（共享审核单）
> 跨负责人契约：房主审核动作归 Ceres——本功能消费审核结果（PASS/REJECT/TIMEOUT），不重构房主审核前端/动作

## 1. 背景与目标

私人房审核通过**非客户端请求协议**，由审核系统 / MQ / 事件回调驱动，承接 1078 申请审核、占座带入、结算补码。入口：`UserBringInDefault.dealBringInReviewDto`（frame 默认）、`UserBringInBootstrap.dealBringInReviewDto`（guandan 覆盖，处理主钱包 + `GD_BONUS`）、`BringInReviewBizService.auditExpire`（超时→TIMEOUT + MQ）。PASS → `processGuandanAuditTransferIn`（增额度/upScore/钱包刷新/占座转入座/结算继续）；REJECT/TIMEOUT → `handleSettleComplementAuditReject`。基于真实代码确认（部分与 03 共享，已在 03 核实）：

- **P0-1：审核幂等以本地缓存为主**。`UserBringInDefault.LoadingCache<String,AtomicBoolean> CACHE`（`:89`）+ guandan `AUDIT_CACHE`/`AUDIT_PASS_CACHE`；审核单业务终态在 Redis，但处理幂等未完全依赖 Redis CAS。多节点重复消费 PASS / 服务重启回放 → 可能重复处理钱包和状态（**资金重复风险**）。
- **P0-2：钱包执行 + 状态推进缺统一补偿**。`processGuandanAuditTransferIn` 串联 增额度→upScore→钱包刷新→占座任务→结算唤醒；占座转入座部分通过 `new OccupyToSitDownTask(...).run()` 同步（与 03 同问题，散落 `UserBringInBootstrap:356/865`）；`auditPassUserNotExistsTable`（用户不在桌）有特殊/TODO 处理。任一步失败难判断重试点；同步 run 绕过队列、与占座超时任务时序不清。
- **P1-1：REJECT/TIMEOUT 与结算补码耦合过深**。`handleSettleComplementAuditReject`（在 `UserBringInBootstrap`）同时移除站起/占座任务、调 `SettleChipComplementService`/`LevelUpBonusHandler`/`UserLeaveBootstrapService.userStand`。带入服务承担结算/彩金/站起职责，改结算补码策略需进带入类。
- **P1-2：审核消息校验不足**。`validateAuditMessage` 主要查桌子/解散；审核单状态是否 INIT、金额是否匹配、holder/申请人是否一致、是否过期 分散或缺失 → 脏/过期/错桌消息进入后续才失败。

目标：**审核 MQ 消息、1010/14009/OCCUPY_ADD_CHIPS_NOTICE 协议字段 100% 兼容**前提下，审核结果处理改 **Redis 审核单 CAS 幂等**（INIT→PROCESSING→PASS_APPLIED）、资金执行与状态推进拆为可补偿步骤（`AuditApplyPlan`）、PASS/REJECT/TIMEOUT 分发、结算补码拒绝迁出带入服务、集中审核消息校验。**复用 03 已建的 Redis CAS / ResultApplier / 占座转入座 02 表级锁 / ReviewWalletAmount**，避免重复造。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 提供审核结果处理骨架，guandan 提供彩金/结算补码扩展 |
| 协议兼容 | 无客户端请求；审核 MQ 字段、PASS 后产生的 1010 响应/14009/`OCCUPY_ADD_CHIPS_NOTICE`/继续状态通知 不变 |
| 幂等（P0-1） | **审核单 ID 为幂等键 + Redis CAS**：INIT→PROCESSING→PASS_APPLIED（新增中间态待确认，§10）；钱包 RPC eventId 关联审核单 ID 保障资金幂等。本地 cache 降级为性能优化。**与 03 R-03 同一 Redis CAS 机制** |
| 资金/状态补偿（P0-2） | `AuditApplyPlan{auditId,status,mainAmount,bonusAmount,quotaAmount,userState,applyMode}` + 分步状态 `quotaApplied/walletApplied/localStateApplied/noticeSent`；每步可幂等重试，失败进补偿队列。占座转入座**走 02 `SitDownStateService.occupyToSitDown` 表级锁**（消除 `.run()`，与 03 R-02 一致） |
| PASS/REJECT/TIMEOUT 分发（P1-1） | `BringInAuditResultConsumer` 按 status 分发；REJECT/TIMEOUT 结算补码处理迁到 `SettlementAuditRejectHandler`，带入服务只转发审核结果 |
| 消息校验（P1-2） | `AuditMessageValidator` 集中校验 审核单状态/金额/桌子/房主/申请人/过期时间 |
| 彩金（与 03/10 一致） | 内部 `ReviewWalletAmount` 视图（`bringInSquidAmount→GD_BONUS`），外部字段不变 |
| 用户已离桌 PASS | `auditPassUserNotExistsTable` 策略显式化（自动上分再带出 / 作废审核）——待产品确认（§10） |
| 灰度策略 | Redis CAS 先只记冲突不阻断 → 启用 CAS 拦截 → PASS 钱包执行迁新服务 → 最后迁 REJECT/TIMEOUT 结算补码 |

## 3. 整体架构

### 改造前

```
审核 MQ/事件回调 → UserBringInDefault.dealBringInReviewDto / UserBringInBootstrap.dealBringInReviewDto(覆盖)
  ├─ 幂等: 本地 LoadingCache CACHE:89 + AUDIT_CACHE/AUDIT_PASS_CACHE (P0-1)
  ├─ validateAuditMessage (仅桌子/解散 P1-2)
  ├─ PASS → processGuandanAuditTransferIn: 增额度→upScore→钱包刷新→占座转入座(.run 同步)→结算唤醒 (P0-2 无补偿)
  └─ REJECT/TIMEOUT → handleSettleComplementAuditReject(结算/彩金/站起 耦合 P1-1)
BringInReviewBizService.auditExpire → TIMEOUT + MQ
```

### 改造后

```
审核 MQ/事件 → BringInAuditResultConsumer
  ├─ AuditMessageValidator      — 状态/金额/桌子/房主/申请人/过期 集中校验(R-04)
  ├─ AuditIdempotencyService    — Redis 审核单 CAS INIT→PROCESSING→PASS_APPLIED(R-01, 复用03 CAS)
  └─ 按 status 分发:
       ├─ PASS  → AuditWalletApplyService(upScore, eventId=auditId) → AuditStateApplyService
       │           (AuditApplyPlan 分步: quotaApplied/walletApplied/localStateApplied/noticeSent, 可补偿 R-02)
       │           占座转入座 → SitDownStateService.occupyToSitDown(02 表级锁)
       └─ REJECT/TIMEOUT → SettlementAuditRejectHandler(结算补码拒绝迁出带入服务 R-03)
ReviewWalletAmount 视图(bringInSquidAmount→GD_BONUS) 复用 03/10
```

**兼容性原则**：
- 审核 MQ 消息、`BringInReviewDto`、1010 响应/14009/`OCCUPY_ADD_CHIPS_NOTICE`/继续状态通知 字段不变。
- `dealBringInReviewDto` public 签名保留（内部委托新 consumer/service）。
- 占座转入座复用 02 表级锁；房主审核（Ceres）链路不动。

## 4. 安全改造（R-01 ~ R-04）

### R-01：Redis CAS 审核幂等（P0-1，复用 03 R-03）
- 审核单 ID 幂等键；`AuditIdempotencyService` 对 Redis 审核单状态 CAS：INIT→PROCESSING→PASS_APPLIED（REJECT/TIMEOUT 终态同理）。钱包 RPC eventId 关联审核单 ID。
- 本地 `CACHE`/`AUDIT_CACHE`/`AUDIT_PASS_CACHE` 降为性能优化（命中跳过），不作唯一幂等判定。

### R-02：资金/状态可补偿（P0-2，复用 03 R-02）
- `AuditApplyPlan` + 分步状态；PASS 钱包执行 → 本地钱包刷新 → 状态推进 → 通知，每步幂等可重试；失败进补偿队列。
- 占座转入座统一走 02 `SitDownStateService.occupyToSitDown` 表级锁（消除散落 `.run()`）。
- `auditPassUserNotExistsTable` 策略显式化（见 §10 待确认）。

### R-03：REJECT/TIMEOUT 结算补码迁出（P1-1）
- 新增 `SettlementAuditRejectHandler`：把 `handleSettleComplementAuditReject` 的结算/彩金/站起处理迁出带入服务；带入服务只转发审核结果给 handler。

### R-04：审核消息集中校验（P1-2）
- `AuditMessageValidator`：审核单状态（应 INIT/PROCESSING）、金额匹配、桌子存在/未解散、holder 一致、申请人仍存在、未过期；脏/过期/错桌消息早失败。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | Redis CAS 先只记冲突不阻断 → 启用拦截 → PASS 钱包执行迁新服务 → 迁 REJECT/TIMEOUT 结算补码 |
| 回滚 | CAS 状态保留，可配置回退旧 handler；补偿记录不影响旧协议 |
| 运行中牌局兼容 | 占座转入座走 02 表级锁；回归审核 PASS/REJECT/TIMEOUT + 结算补码 |
| 与其他 Junk/Ceres 同步 | 复用 03 CAS/ResultApplier/表级锁/视图；建单见 10；固定带入见 16；房主审核 MQ 契约对齐 Ceres |

## 9. 验收关键场景

1. PASS → 主钱包 + 彩金上分成功；占座用户入座成功（02 表级锁）。
2. PASS 重复回调 → CAS 拦截，不重复上分。
3. PASS 时用户已退出 → 按显式策略处理（不重复入座，资金有明确去向）。
4. REJECT → 不残留占座转入座任务，结算补码恢复/站起由 `SettlementAuditRejectHandler` 处理。
5. TIMEOUT → 审核单状态与房主/用户侧一致。
6. 错桌/错 holder/过期消息 → `AuditMessageValidator` 早拒绝。

## 10. 待确认项

| 待确认项 | 类型 | 处理 |
|----------|------|------|
| 审核通过但用户已离桌：自动上分再带出 vs 作废审核 | 产品/财务 | 本期 `auditPassUserNotExistsTable` 策略显式化，默认沿用现状，待确认 |
| 审核单状态是否允许新增 PROCESSING/PASS_APPLIED | 协议/存储 | CAS 需要中间态；若不允许，用钱包 eventId 幂等替代 |
| 钱包 RPC 是否支持以审核单 ID 做幂等 | 技术 | 决定资金幂等实现（CAS + eventId） |
| 拒绝/超时时用户是否必须立即站起 | 产品 | 本期沿用现状，迁 handler 后行为不变 |

## 11. 本功能不处理的范围

- 不创建审核单（归 10）；不处理固定带入首次入座触发（归 16）。
- 不重构房主审核动作/前端（归 Ceres）。
- 不改 1078/审核 MQ/钱包 RPC 协议。
