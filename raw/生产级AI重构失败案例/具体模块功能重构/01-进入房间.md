## 1. 功能边界

本功能覆盖 `CommonProto.ENTER_ROOM` 进入牌桌协议，协议号为 1000。当前功能由 frame 通用协议入口承接，掼蛋侧通过 bootstrap 子类补充用户状态、重连、手牌、桌面配置、加密手牌等游戏专有数据。

子功能范围：

| 子功能 | 当前覆盖情况 |
| --- | --- |
| 主流程 | `EnterTableCommand -> GameHandleFactory -> 空间类型 Handler -> EnterBaseTableHandlerDefault` |
| 账号锁定校验 | `UserCheckAuthBootstrap.checkUserLockClub/checkUserLockPrivate/checkTableAuth` |
| 代理风控锁定校验 | `userBaseInfo.getTopProxyWindCtlStatus()` |
| 桌子权限校验 | 俱乐部、私人房、密码、直属会员、一级代理可见性、游客开关 |
| 设备限制校验 | PC 禁入、俱乐部 H5 禁入、私人房 iOS 限制 |
| 房间数据组装 | `EnterRoomRespDtoV2`，包含桌信息、手牌信息、玩家列表、用户配置、加密信息 |
| 重连 / 已在房间处理 | `GdGameUserBootstrap.restartLoadHand` 和 `TableUserHandBootstrap` 重连快照 |

不处理范围：房主操作、打赏、好友备注、用户配置查询/修改、投屏、大厅换桌。若进房链路读取这些数据，只记录接口契约，不纳入 Junk 改造方案。

## 4. 现有设计理解

正常流程：

1. 客户端发送 `ENTER_ROOM(1000)`，`EnterTableCommand` 在 PLAYER 线程处理，先拦 PC，再初始化桌用户基础数据。
2. `GameHandleFactory` 根据牌桌空间类型选择 HALL/CLUB/PRIVATE/RACE 处理器。
3. 处理器初始化 `EnterRoomContextV2`，判断用户是否已在房间，读取 `UserBaseInfo`。
4. 校验设备、牌桌状态、币种、旁观上限、密码、踢出、账号锁定、俱乐部/私人房权限、游客和人数上限。
5. 新用户进入时创建 `UserItem` 并加入 WATCH_USER；已在房间时走重连，掼蛋侧取消托管并清理预退出/预站起标记。
6. 组装进房快照：用户配置、桌内玩家、TableHand、TableUserHand、TableInfo、用户配置、风控基础数据。
7. 发布进房完成事件，发送 `ENTER_ROOM` 响应，清理离线时间，私人房加入 Redis 缓存，并广播新观察者通知。

条件分支：

| 分支 | 触发条件 | 当前处理 |
| --- | --- | --- |
| 已在房间/重连 | `BaseUserCheckService.checkUserIsInRoom` 为 true | 不走新用户加入，掼蛋侧取消托管、清理 `prepKickStatus` |
| 新用户 | 不在 PLAYER/WATCH_USER | 创建 `UserItem`，加入 WATCH_USER，写入设备/IP，发送阶段 MQ |
| 私人房 | `tableInfo.isPrivateTable()` | 专用校验：私人房锁定、代理开关、游客、人数上限、固定入房缓存 |
| 大厅/俱乐部/比赛 | 非私人房空间 | 走通用校验：密码、俱乐部、代理风控、币种、旁观上限 |
| 手级重连 | `TableUserHandCache` 有数据或 `TableHandStatus` 为 WAIT/SETTLE_END | `TableUserHandBootstrap` 返回出牌、贡还贡、转蛋、结算、补码倒计时等数据 |
| 加密手牌 | 请求带 `encryptKey` | 写 `UserSecureCache` 并返回 `AsdRespDto`，手牌快照加密 |

合理点：

- 用空间类型 Handler 区分 HALL/CLUB/PRIVATE/RACE，方向是正确的，保留该路由方式。
- `EnterRoomRespDtoV2` 包含桌、手、玩家、用户配置，满足客户端断线重连一次恢复所需。
- 已在房间用户免密码、退出用户免密码，符合重连/回桌体验。
- 掼蛋侧 `TableUserHandBootstrap` 已补齐转蛋、贡还贡、结算后补码、延时状态等重连场景，应保留这些业务语义。

待理解：

- `EnterRoomContextV2.isNewRoomUser` 实际表示“是否已在房间”，命名与含义相反，证据：`EnterBaseTableHandlerDefault.java:78`。
- `BaseUserEnterRoomFinishListener` 中 `!context.getIsNewRoomUser()` 才写进出记录和异常带出，结合命名需要确认是否符合预期，证据：`BaseUserEnterRoomFinishListener.java:52-60`。

## 6. 架构审视

| 问题 | 结论 |
| --- | --- |
| 本功能的本质职责是什么 | 用户进入/恢复牌桌上下文，并返回可恢复客户端状态的完整快照 |
| 变化维度是什么 | 空间类型、用户是否已在房间、私人房/俱乐部权限、设备类型、手牌阶段、是否加密 |
| 核心状态流转是什么 | 不在桌 -> WATCH_USER；已在桌 -> 重连恢复；退出用户 -> 复用 QUIT_USER 的 `UserItem` 回到 WATCH_USER |
| 外部交互模式是什么 | 同步响应为主，附带 Spring Event、MQ 阶段上报、Redis 私人房缓存、客户端广播 |
| 是否适合状态机 | 部分适合。进房用户状态适合抽成 `ENTERING/IN_ROOM/RECONNECTING/FAILED`，但快照组装不需要状态机 |
| 是否适合策略模式 | 适合。HALL/CLUB/PRIVATE/RACE 已是策略雏形，应把校验和后置动作继续空间类型化 |
| 是否适合模板方法 | 适合。`init -> check -> mutate -> assemble -> after` 已是模板方法结构，但现在分散在 handler 和 base default |
| 是否适合责任链 | 适合。设备、账号锁定、代理风控、桌权限、密码、游客、人数上限应拆成可组合校验链 |
| 是否适合事件驱动 | 部分适合。轨迹、活动、通知适合事件；异常带出这类资金相关补偿不应只靠无重试事件 |
| 是否适合命令模式 | 已适合。当前 Command 是协议入口，建议补 `EnterRoomCommandContext` 和结果对象 |
| 是否适合 Builder | 适合。`EnterRoomRespDtoV2` 快照组装应拆为 Builder/Assembler，减少 `handleData` 直接拼字段 |
| 是否适合 Facade | 适合。对外保留 `EnterRoomFacade/Service`，内部封装校验、状态写入、快照、后置动作 |

与 frame 融合：保留 `CommonProto.ENTER_ROOM` 和 `HandlerSpaceService` 注册/路由；在 frame 新增兼容扩展点时只新增接口，不改现有方法签名。与 Master/Action 不替换，只在重连快照读取其结果数据，不在进房中执行游戏动作。

## 7. 改造建议

| 建议编号 | 类型 | 改造建议 | 解决的问题 | 核心理由 | 风险 |
| --- | --- | --- | --- | --- | --- |
| R-01 | 最小安全改造 | 私人房人数上限从“校验即递增”改为“成功加入 WATCH_USER 后按实际集合计算”，失败路径补偿递减，并增加重复请求保护 | P-01/P-04 | 先消除判满和重复计数风险 | 需核对退出/站起/清桌是否仍依赖 `enterRoomUserCount` |
| R-02 | 最小安全改造 | 给钱包、TableHand、RaceGlobalConfig、UserTableItem 增加空值保护和明确业务异常 | P-02 | 避免异常重连直接系统错误 | 需要确认客户端对缺省字段的兼容 |
| R-03 | 最小安全改造 | 收敛日志：进房只打 `tableId/userId/handCode/status/size`，禁止整份 `context/tableHand/handMap` info 日志 | P-09 | 降低高峰 CPU、日志量和敏感数据暴露 | 排障时需补 debug 开关 |
| R-04 | 结构性重构 | 新增 `EnterRoomFlowService`：统一 `initContext -> validate -> applyState -> assembleSnapshot -> afterCommit` | P-05/P-10 | 把主流程从 frame default 的大方法中拆出，Junk 侧可控 | 需要保证旧 Handler 仍走同一入口 |
| R-05 | 结构性重构 | 新增 `EnterRoomValidatorChain`，按空间类型组合设备、账号、代理风控、密码、桌权限、游客、人数上限校验 | P-03/P-10 | 校验可测试、可复用，避免入座/进房校验漂移 | 短期会增加类数量 |
| R-06 | 结构性重构 | 新增 `EnterRoomSnapshotAssembler`，拆分 `UserGameConfigAssembler/TableSnapshotAssembler/HandReconnectAssembler/UserListAssembler` | P-02/P-10 | DTO 字段来源集中，便于协议兼容审计 | 需要完整字段回归 |
| R-07 | 结构性重构 | 将进房状态写入统一提交到 table 串行执行器或增加 table 级状态锁；PLAYER 线程只做请求接入 | P-03/P-04 | 保证和入座、站起、结算同一桌状态线性化 | 改动影响较大，需要灰度 |
| R-08 | 结构性重构 | 后置事件拆分：轨迹/活动异步可失败，异常带出改为可重试补偿任务 | P-06 | 资金相关逻辑不能只靠吞异常事件 | 涉及带出模块联动 |
| R-09 | 是否彻底重写 | 不建议一次性彻底重写协议入口；建议保留协议和 Handler 路由，重写内部 flow/validator/assembler | 全部 | 进房协议字段多、重连场景复杂，一次性推翻风险高 | 需要分阶段迁移 |

## 11. 迁移与灰度

| 维度 | 方案 |
| --- | --- |
| 新旧是否共存 | 保留旧 `EnterTableCommand` 和 Handler；内部通过配置切换旧 `EnterBaseTableHandlerDefault` 与新 `EnterRoomFlowService` |
| 灰度维度 | 按空间类型、俱乐部、牌桌、代理配置开关 |
| 回滚方式 | 配置切回旧 flow；新类不改协议和 DTO |
| 运行中牌局兼容 | 不迁移运行中 TableHand/TableUserHand 字段；只调整读取和组装方式 |
| Redis / 本地缓存旧数据兼容 | 旧本地缓存继续读；新增状态写入先双写或只读校验，不强制 Redis 化 |
| 与其他 Junk 功能同步要求 | 入座、站起、退出必须共享同一用户状态语义；私人房人数和 WATCH_USER 计数不能各自维护 |
| 与 Ceres / Torsen / Duik 接口契约 | Ceres 房主操作只读进房返回按钮权限；Torsen 游戏过程提供 TableHand/TableUserHand 快照；Duik 结算后补码/继续状态供重连读取 |

## 12. 验收场景

| 场景 | 子功能 | 类型 | 前置条件 | 操作 | 期望结果 |
| --- | --- | --- | --- | --- | --- |

## 13. 待确认项

| 待确认项 | 原因 |
| --- | --- |
| `isNewRoomUser` 字段真实语义是否可重命名 | 当前注释和赋值相反，重构建议新增 `alreadyInRoom` 并保留旧字段 |
| 私人房 `enterRoomUserCount` 是否被退出/清桌准确扣减 | 当前扫描只看到进房校验递增和超限递减，需要后续在退出/清桌扫描中闭环 |
| `BaseUserEnterRoomFinishListener` 异常带出是否应属于进房成功前置 | 当前响应前发布事件但事件失败不阻塞，需要和带出模块确认 |
| `TableInfo.raceGlobalConfig` 是否保证所有掼蛋桌非空 | 进房 DTO 直接读取 `getPlayingCardTime()` |
| 进房返回用户配置、好友备注是否仍由 Junk 负责 | 当前进房会读取用户配置和好友备注，但用户要求这些功能暂不处理 |

## 14. 本功能不处理的范围

- 不处理房主指定操作、房主操作、打赏、投屏、大厅换桌。
- 不处理游戏开始、出牌、贡还贡、Master、Action。
- 不处理结算、派彩、注单、回放数据发送。
- 进房链路读取 Ceres / Torsen / Duik 数据时，只记录接口契约，不在 Junk 进房改造中重写对方职责。
