# 实时账单重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/09-实时账单.md`
> 设计日期：2026-06-04
> 工程：dx-game-guandan（3005 为掼蛋协议，全部代码在 guandan；frame 无改动）
> 分支：20260615/junk
> 复杂度：H（P0 高）
> 前置依赖：03-带入、05-带出（账变来源）、20-好友备注（已审，备注名读取走 UserFriendCache 门面）

## 1. 背景与目标

实时账单 3005 `GuandanProto.CONTINUOUS_BILLING`（`@ThreadModel(PLAYER)`）入口 `ContinuousBillingCommand` → `ContinuousBillingHandler.handle`，从桌内缓存 + 多个统计 RPC 拼装展示视图。基于真实代码（`ContinuousBillingHandler.java` / `ContinuousBillingCommand.java`）确认：

- **P0-1：异常被吞 → 半空成功响应 + NPE**。`ContinuousBillingHandler.handle` 整体 `try{...}catch(Exception){log.error}`（L76-78）**不抛出**；`ContinuousBillingCommand.handleCommand` 在 `handle()` 后**无条件** `after(context)`（L52）发送成功响应。且 `handle` L73 `continuousBillingResp.setPlayAttributes(tableHand.getPlayAttributes())` 对 **null `tableHand`** 直接 NPE（L65-66 已知 tableHand 可空，但 L73 未兜底）。结果：组装失败时客户端收到 success + 空/半空数据，监控看不到失败。
- **P0-2：多源数据无一致快照**。`setRealTimeData` 在一次执行内分别独立读 `UserTableItemCache.getAllUserTableItem` / `BaseTableUserCache`（activeDisplayUserIds、quitUserIds）/ `TableInfoCache` / `TableHandCache` + 4 个 RPC（手数 / 局数 / 彩金盈亏 / 输赢分 / 钱包汇总），无版本快照。玩家刚站起/退出/带出时各源时间点不一致，导致"算到的金额"与"展示状态"错位。
- **P1-1：RPC 失败回退口径不统一**。`getUserBonusProfitLossMap` 失败→空 map；`getUserWinLoseScoreMap` 失败→空 map；`getUserRoundCountMap` 失败→**null**（用于区分是否回退手数）；钱包汇总 `getWalletSummaryForGuandanByTableId` 失败→多个金额 map 保持空。客户端无法区分"真实 0"与"统计不可用"。
- **P1-2：踢人列表无空值保护**。`setKickerList` L118-121 直接 `UserBaseInfoCache.getUserBaseInfo(userId)`、`TableInfoCache.getTableInfo(tableId)`、`TableCache.getTable(tableId)` 后 `!userBaseInfo.isTopProxy()` / `tableInfo.getHolderId().equals(userId)`，任一为 null 即 NPE。踢人列表本是附加展示，不应拖垮主账单。
- **P2：海量大对象 info 日志**。`handle`/`setWatchUsers`/`setRealTimeData` 多处 `GsonUtils.toJson(allUserMap/watchUsers/request/各金额 map)`（含金额/用户信息），高峰 CPU + 隐私风险（D12）。

目标：以**最小安全改造**消除两条 P0（异常不再吞成半空成功、关键缓存空值兜底）+ 统一 RPC 失败口径 + 旁观/踢人/账单三块互不拖垮 + 日志脱敏；**保持 3005 请求/响应字段 100% 兼容**（不新增必填字段，内部失败状态只进日志/指标）。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Guandan 主导。3005 全部实现在 guandan，frame 无改动 |
| `handle` 异常处理 | **三块独立 + 主块失败抛出**：`watchUsers`/`kickerList`（附加展示）各自 try/catch 失败返回空列表不拖垮；`realTimeData`（主账单）+ `playAttributes` 构建失败**抛出**交命令层 `catch` 发错误响应，不再"吞异常 + 无条件 after"。命令层已有 BusinessException/系统异常 → 错误响应路径（`ContinuousBillingCommand` L53-63），复用即可 |
| `tableHand`/`tableInfo` 空值 | `playAttributes` 在 tableHand 为 null 时兜底默认（空对象/原 `table.getPlayAttributes` 来源）；`setRealTimeData` 内 `tableInfo`/`raceGlobalConfig` 空值兜底，不 NPE |
| 多源一致性（P0-2） | 引入**内部** `BillingSnapshot`（不进协议）：一次性 capture table/tableInfo/tableHand/allUserItems/activeDisplayUserIds/quitUserIds/friendRemarks + 各 RPC 结果与请求时刻。后续组装只读 snapshot，不重复读 cache |
| RPC 失败口径（P1-1） | 每个统计源封装为 `BillingStatItem<T>{data, status}`，`status ∈ {OK, EMPTY, FAILED, FALLBACK}`。**协议不新增必填字段**；status 仅进日志/指标。保留现有"局数失败回退手数"语义（标 FALLBACK） |
| 踢人列表空保护（P1-2） | `setKickerList` 全程空值保护（userBaseInfo/tableInfo/table 任一空→返回空踢人列表 + warn），独立 try/catch |
| 排序/置灰/总带入为 0 跳过 | **行为保留**：总带入为 0 跳过（SUN-37020）、退出用户置灰、当前请求用户排第一、按盈亏降序——全部保持现有口径不变 |
| 日志收敛 | `GsonUtils.toJson(allUserMap/各金额 map/watchUsers/request)` info → debug；info 仅保留 tableId/userId/handCode/用户数/各源 status 元数据 |
| 灰度策略 | 不做开关，依赖测试 + 影子对比（§11）。新 snapshot 不进协议，可直接切回旧 handler 回滚 |
| 结构性拆分 | 本期可选：把 handler 拆为 `BillingSnapshotBuilder` + `BillingUserDataAssembler` + `BillingWatchUserAssembler` + `BillingKickerAssembler` + `BillingStatsClient`。**最小安全改造优先**：先做 R-01~R-05，结构拆分作为 S-01 一并落地（同包内重构，不改协议） |

## 3. 整体架构

### 改造前

```
ContinuousBillingCommand (PLAYER, 3005)
  ├─ init() — TableCache.getTable 空→TABLE_STATUS_IS_END
  ├─ handle()  ← try{...}catch(Exception){log.error} 不抛出 (P0-1)
  │   ├─ setWatchUsers()      — 旁观 + 备注
  │   ├─ setKickerList()      — 房主/总代可见，无空保护 (P1-2)
  │   ├─ setRealTimeData()    — 多源独立读 + 4 RPC，无快照 (P0-2)，失败口径不一 (P1-1)
  │   └─ setPlayAttributes(tableHand.getPlayAttributes())  ← null NPE (P0-1)
  └─ after() — 无条件发送成功响应 (P0-1)
```

### 改造后

```
ContinuousBillingCommand (PLAYER, 3005, 不变签名)
  ├─ init() — 不变
  ├─ billingFlowService.build(context)   ← 主账单失败抛出，命令层 catch 发错误响应
  │   ├─ BillingSnapshot.capture(tableId, userId)   — 一次性快照 (R-02)
  │   ├─ realTimeData = assembleUserData(snapshot)  — 主块，失败抛出 (R-01)
  │   ├─ watchUsers  = safe(assembleWatchUsers)     — 附加，失败空列表 (R-04)
  │   ├─ kickerList  = safe(assembleKicker)         — 附加，空保护+失败空列表 (R-04/P1-2)
  │   └─ playAttributes = snapshot.tableHand!=null ? ... : 默认 (R-01)
  └─ after() — 发送响应（主块已成功才到这里）

BillingSnapshot (内部, 不进协议)
  └─ table/tableInfo/tableHand/allUserItems/activeDisplayUserIds/quitUserIds
     /friendRemarks/各 RPC 结果(BillingStatItem) + capturedAt

BillingStatItem<T> { T data; SourceStatus status (OK/EMPTY/FAILED/FALLBACK) }  — 仅日志/指标
```

**兼容性原则**：
- 协议号 3005、`ContinuousBillingReq`、`ContinuousBillingResp`（`realTimeData`/`watchUsers`/`kickerList`/`playAttributes`）字段 0 改动。
- `ContinuousBillingCommand` / `ContinuousBillingHandler` public 方法签名保留（内部委托/拆分）。
- `WATCH_USER_CONTINUOUS_BILLING_NOTICE`（入房旁观变动通知）不由本次改动。
- 钱包汇总 / 手数 / 局数 / 彩金盈亏 / 输赢分 RPC 签名不动。

## 4. 安全改造（R-01 ~ R-05）

### R-01：异常不再吞成半空成功 + 关键空值兜底（P0-1）
- `handle`（或新 `build`）：`realTimeData` 主块与 `playAttributes` 构建失败 **抛出**（业务异常/系统异常），由 `ContinuousBillingCommand` 既有 catch 发错误响应；不再"吞异常 + 无条件 after"。
- `playAttributes`：`tableHand == null` 时兜底（取 `context.getTable()` 既有玩法属性或默认空对象），不 NPE。
- `watchUsers`/`kickerList` 为**附加展示**：各自 try/catch，失败返回空列表 + warn + 指标，不影响主账单。

### R-02：`BillingSnapshot` 一次性快照（P0-2）
- 新增内部 `BillingSnapshot`，`capture(tableId, userId)` 一次性读全 table/tableInfo/tableHand/allUserItems/activeDisplayUserIds(DEFAULT+SIT_DOWN_WAIT_*)/quitUserIds/friendRemarks。
- `assembleUserData`/`assembleWatchUsers`/`assembleKicker` 只读 snapshot，不再各自重读 cache。
- snapshot 含 `capturedAt`，日志统一带同一时间戳便于追踪。

### R-03：统计 RPC 失败口径统一（P1-1）
- 封装 `BillingStatsClient`：`bonusProfitLoss / winLoseScore / roundCount / handInfo / walletSummary` 各返回 `BillingStatItem<T>`，`status` 明确 OK/EMPTY/FAILED/FALLBACK。
- **保留现有语义**：局数失败回退手数（status=FALLBACK）；其余失败现状返回 0/空的展示口径不变（status=FAILED，仅日志/指标可区分）。
- 协议不新增必填字段。

### R-04：踢人列表空保护 + 独立化（P1-2）
- `setKickerList`：`userBaseInfo`/`tableInfo`/`table` 任一为 null → 返回空踢人列表 + warn，不 NPE；整体独立 try/catch。
- 权限判断（`isTopProxy` / `holderId.equals(userId)`）在空值兜底后进行。

### R-05：日志收敛 / 脱敏（P2）
- `handle`/`setWatchUsers`/`setRealTimeData`/各 RPC 方法中 `GsonUtils.toJson(allUserMap/金额 map/watchUsers/request)` info → debug。
- info 仅保留：tableId/userId/handCode、各源 status、展示用户数、grayUserIds 数量等元数据。

## 5. 新增组件（S-01，同 guandan 包内）

| 文件 | 包 | 职责 |
|------|-----|------|
| `BillingSnapshot.java` | `com.dx.game.guandan.model.dto.billing` 或 service.biz.billing | 一次性快照（内部，不进协议） |
| `BillingStatItem.java` | 同上 | `{data, status}` 统计源结果包装 |
| `BillingStatsClient.java` | `com.dx.game.guandan.service.biz.billing` | 5 个统计 RPC 统一封装 + 失败状态 |
| `BillingFlowService.java` | 同上 | 编排 snapshot → 主块/附加块 → 组装；主块失败抛出 |
| （可选）`BillingUserDataAssembler` / `BillingWatchUserAssembler` / `BillingKickerAssembler` | 同上 | 三块组装拆分 |

> 最小落地：先在 `ContinuousBillingHandler` 内引入 `BillingSnapshot` + `BillingStatsClient` + 主块/附加块拆分（R-01~R-05），`BillingFlowService`/三 Assembler 作为同包结构性整理同批落地，命令层只改"主块失败抛出"。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | 保留旧 `ContinuousBillingHandler` 逻辑；新 snapshot/assembler 同包并行，命令层切换调用点即可灰度 |
| 灰度维度 | 先切旁观/踢人（附加块），再切主账单；先影子只读对比（用户数/总带入/总带出/盈亏/排序），后全量 |
| 回滚 | snapshot 不进协议；切回旧 handler 调用即可；或 revert |
| 运行中牌局兼容 | 3005 为展示查询，无状态写入；snapshot 只读，无运行中数据兼容问题 |
| 与其他 Junk 同步 | 账变来源（03 带入/05 带出）不动；备注走 20 已审的 `UserFriendCache` 门面 |

## 9. 验收关键场景

| 场景 | 类型 | 期望 |
|------|------|------|
| 主账单组装失败 | 异常 | 返回 3005 错误响应（非半空成功），有指标 |
| tableHand/tableInfo/userBaseInfo 缺失 | 异常 | 不 NPE；playAttributes 兜底；踢人列表空保护 |
| 钱包汇总 RPC 失败 | 异常 | 主块按现状口径，status=FAILED 进日志/指标 |
| 局数 RPC 失败 | 异常 | 回退手数（status=FALLBACK），口径不变 |
| 退出用户 | 正常 | 账单中置灰，旁观/踢人不受影响 |
| 非房主非总代请求 | 正常 | 踢人列表为空，主账单正常 |
| 总带入为 0 用户 | 正常 | 不展示（保持 SUN-37020） |
| 当前请求用户 | 正常 | 排第一；旁观列表中也置顶 |
