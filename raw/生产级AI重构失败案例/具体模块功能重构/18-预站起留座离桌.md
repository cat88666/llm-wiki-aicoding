## 1. 功能边界

本功能覆盖 1045 `CommonProto.PREP_STAND` 的预站起，以及 1034 `CommonProto.USER_HOLD_SEAT` 的留座离桌相关协议边界。

当前代码事实:

- 1045 有 `UserPrepStandCommand` 和 `UserPrepStandCommandDefault`。
- `UserPrepStandCommandDefault` 依赖 `UserStandPrepBootstrapService`。
- 当前两个工程未发现 `UserStandPrepBootstrapService` 的实现类。
- 1034 `USER_HOLD_SEAT` 协议号存在，但未发现对应命令实现。

## 2. 入口与调用链

预站起:

1. `UserPrepStandCommand` 使用 `@ThreadModel(PLAYER)`，`getType` 返回 `CommonProto.PREP_STAND`。
2. `UserPrepStandCommandDefault.handleCommand` 执行 `check -> handle -> after`。
3. `check` 要求用户在坐下玩家列表，`standType` 能映射到 `PrepStandStatusEnum`，并通过 `commonCheckUserBreakLink`。
4. `handle` 从 Spring 获取 `UserStandPrepBootstrapService`，调用 `prepStand(tableId,userId,status)`。
5. `after` 返回 `UserPrepStandRespDto.msg/prepStatus`。

预站起生效:

- `PrepStandLeaveUserVisitor` 在 `UserLeaveForcedEnum.PREPARE_STAND` 场景扫描玩家。
- `BaseUserItem.isPrepStand/isPrepStandHand/isPrepStandRound` 判断立即、本手后、本轮后站起。
- `BaseUserStandFinishedListener` 和 `BaseUserQuitFinishedListener` 会清理 `prepStandStatus`。

留座离桌:

- `CommonProto.USER_HOLD_SEAT=1034` 存在。
- `UserConfig.timeoutHoldSeatSwitch`、`SentinelTypeEnum.HOLD_SEAT`、`UserStandTypeEnum.HOLD_SEAT_COUNTDOWN`、相关业务错误码存在。
- 当前未发现 1034 命令实现。

## 3. 数据流与状态

请求:

- `UserPrepStandReqDto.standType`: 注释写“0 取消，2 本手后，2 本轮后”，实际枚举为 0/1/2/3。

响应:

- `UserPrepStandRespDto.msg`
- `UserPrepStandRespDto.prepStatus`
- `UserPrepStandRespDto.userId` DTO 有字段，但默认 after 未设置。

状态:

- `BaseUserItem.prepStandStatus`
- `PrepStandStatusEnum.NOT_STAND/STAND/STAND_AFTER_HAND/STAND_AFTER_ROUND`
- `UserLeaveForcedEnum.PREPARE_STAND`
- `SpecifyTypeEnum.NEXT_HAND/NEXT_ROUND`

## 4. 现有设计理解

预站起设计上应当是“用户先设置意图，后续在本手或本轮结束时由 visitor 生成离开动作”。现有代码已经有命令、状态枚举和 visitor，但缺少实际写入 `BaseUserItem.prepStandStatus` 的 service 实现。

留座离桌则更像历史德州能力的残留: 协议号、配置、限流、错误码、站起类型存在，但当前工程没有完整命令链路。

## 5. 问题扫描

### P0-1 1045 命令依赖的 service 未发现实现

证据:

- `UserPrepStandCommandDefault.handle` 调用 `SpringContextUtils.getBean(UserStandPrepBootstrapService.class)`。
- `UserStandPrepBootstrapService` 是接口，只定义 `prepStand`。
- 当前两个工程搜索未发现 `implements UserStandPrepBootstrapService`。

影响:

- 1045 在线上可能因找不到 Bean 进入系统异常。
- 预站起状态不会写入 `BaseUserItem.prepStandStatus`，后续 visitor 无法生效。

建议:

- 若 1045 仍使用，必须补默认实现。
- 若不使用，协议文档标注废弃，并让 command 返回明确不支持而不是系统异常。

### P0-2 1034 留座离桌协议号存在但命令缺失

证据:

- `CommonProto.USER_HOLD_SEAT` 定义 1034。
- 搜索未发现 `HoldSeatCommand` 或返回 1034 的 command。
- 仅存在 `timeoutHoldSeatSwitch`、`HOLD_SEAT` 限流、`HOLD_SEAT_COUNTDOWN` 站起类型。

影响:

- 客户端如果发送 1034，无服务端业务处理。
- 配置项可能让产品以为留座离桌已支持。

建议:

- 产品确认留座离桌是否属于掼蛋范围。
- 若实现，单独设计 `HoldSeatFlowService`，不要混入预站起。

### P1-1 `UserPrepStandReqDto.standType` 注释错误

证据:

- DTO 注释写“0:取消预站起 2:本手结束后站起 2:本轮换庄后站起”。
- `PrepStandStatusEnum` 实际为 0 取消、1 立即、2 本手后、3 本轮后。

影响:

- 联调和测试可能传错值。

建议:

- 修正文档和 DTO 注释。
- 明确是否允许 `STAND=1` 通过 1045 立即站起。

### P1-2 1045 校验使用 breakLink 检查，语义不清

证据:

- `UserPrepStandCommandDefault.check` 调用 `UserLeaveBootstrapService.commonCheckUserBreakLink`。
- 04/06 中普通站起和退出分别使用 `commonCheckUserLeave` 或 `commonCheckUserBreakLink`。

影响:

- 预站起不是断线退出，但使用 breakLink 检查可能绕过或误用离开规则。

建议:

- 使用 `LeaveIntent.PREP_STAND` 的专用规则链。

## 6. 架构审视

D5 状态机: 预站起状态写入和生效由不同机制处理，缺少完整 flow。
D7 协议事实: 1045 半实现，1034 只有协议号和枚举残留。
D8 Bean 依赖: command 运行期动态取接口 Bean，缺实现时才失败。
D10 文档/注释: DTO 注释和枚举不一致。

## 7. 改造建议

优先级:

1. P0: 补 `UserStandPrepBootstrapService` 实现或废弃 1045。
2. P0: 明确 1034 留座离桌是否实现。
3. P1: 修正 `standType` 注释和测试。
4. P1: 预站起接入 `LeaveIntent.PREP_STAND` 规则链。

## 8. 落地设计

预站起建议:

- `UserStandPrepBootstrapServiceImpl.prepStand`:
  1. 读取 `BaseUserItem`。
  2. 校验当前状态。
  3. 写入 `prepStandStatus`。
  4. 如果 `STAND=1`，直接调用站起 flow 或返回不支持。
  5. 返回最终枚举。

留座离桌建议:

- `HoldSeatCommand` 返回 1034。
- `HoldSeatFlowService` 校验开关、次数、冷却、用户状态。
- 留座超时后使用 `UserStandTypeEnum.HOLD_SEAT_COUNTDOWN` 站起。
