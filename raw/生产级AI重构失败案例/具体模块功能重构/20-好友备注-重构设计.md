# 好友备注重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/20-好友备注.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame（本功能 dx-game-guandan 侧无业务逻辑，仅消费门面）
> 分支：20260615/junk

## 1. 背景与目标

好友备注流程当前由 `FriendRemarkCommand` 在 PLAYER 线程一把梭，缓存层使用静态 `UserFriendCache`，存在：

- P0：`handleCommand` 中 `addReq.setRemarkUserId(remarkUserId)` 是 `BeanUtils.copyProperties(req, updatedDto)` 后的一行笔误，意图明显是写 `updatedDto`。当前依赖 `BeanUtils` 副本恰好包含 `remarkUserId` 才不出事。
- P1：`getFriendRemarkCache` 对 RPC 未返回的用户写入 `new UserFriendRemarkDto()`，与"备注就是空"无法区分。在用户刚新增备注但 MQ 延迟时，本节点 30 分钟内只能等过期。
- P1：`getFriendRemarkCache` `catch` 后返回 `new HashMap<>()`，玩家服务异常被静默吞掉，调用方完全分不清"无备注"和"备注服务挂了"。
- P1：`SpringContextUtils.getBean(DxPlayerRpcClient.class)` 在静态方法里取 Bean，难以单元测试。
- P2：`log.info("好友备注协议:1033 req: {}", GsonUtils.toJson(req))` 输出用户输入的 `userRemark` 明文；`updateFriendRemarkCache` 每次输出整份 map。
- 死代码：`dx-game-frame-repository/.../redis/cache/UserFriendCache.java` 整个 `getFriendRemarkCache` 被注释，只返回空 Map。

目标：消除 P0/P1 所有问题，把命令处理与缓存策略各自抽成 Spring Bean，保留静态门面让 20+ 调用方零改动，协议与 DTO 完全兼容。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导。好友备注无游戏专属逻辑，guandan 侧只是消费者，不引入扩展点 |
| 调用方迁移 | **不迁**。`UserFriendCache` 静态门面保留，内部委托新 Bean，20+ 调用点零改动 |
| 负缓存策略 | 独立短 TTL（60 秒），用 `FriendRemarkCacheEntry.present=false` 标记，与命中区分 |
| RPC 失败可观测 | RPC 异常时不写入 cache（不留 `present=false`），下次仍会重试；日志升级为 error，并打指标点 |
| 笔误修复方式 | 显式 `updatedDto.setUserId(userId)` / `setRemarkUserId(remarkUserId)`，删除 `BeanUtils` 之后多余的 `addReq.setRemarkUserId(remarkUserId)`，不再依赖 `BeanUtils` 副本 |
| 日志脱敏 | `userRemark` / `userNameRemark` 为用户输入，统一只打长度；req 不再整对象 `toJson` |
| 灰度开关 | 不做开关，直接替换。静态门面签名不变，回滚靠 revert |
| 死代码处理 | 删除 `redis/cache/UserFriendCache.java`（同名重复类，逻辑全注释，仅返回空 Map，无人调用） |

## 3. 整体架构

### 改造前

```
FriendRemarkCommand (PLAYER线程)
  ├─ 校验 remarkUserId
  ├─ BeanUtils 拷贝 → DxPlayerRpcClient.saveFriendRemark
  ├─ 笔误：addReq.setRemarkUserId(...)（本意是 updatedDto）
  ├─ UserFriendCache.updateFriendRemarkCache (static)
  └─ 响应 + 全量 req/resp info 日志

UserFriendCache (static)
  ├─ updateFriendRemarkCache  — info 输出整份 map
  ├─ getFriendRemarkCache     — 命中 + 负缓存写空对象 + catch 静默
  └─ getUserFriendRemarkCache — Guava cache 30min

UserFriendChangeConsumer → UserFriendCache.updateFriendRemarkCache
```

### 改造后

```
FriendRemarkCommand (PLAYER线程)
  └─ FriendRemarkCommandService.saveRemark(req)
       ├─ 校验
       ├─ 显式构造 DxGamePersonalRemarksAddReq / UserFriendRemarkDto
       ├─ DxPlayerRpcClient.saveFriendRemark
       └─ FriendRemarkCacheService.updateRemark

UserFriendCache (static facade, 旧签名保留)
  └─ 委托 FriendRemarkCacheService

FriendRemarkCacheService (Spring Bean)
  ├─ getRemarks(userId, Set<Long>) → FriendRemarkFetchResult
  ├─ updateRemark(userId, dto)
  ├─ positiveCache: LoadingCache<Long, ConcurrentHashMap<Long, FriendRemarkCacheEntry>>  30min
  └─ DxPlayerRpcClient 构造器注入

FriendRemarkCacheEntry { dto, present, fetchedAt }
  ├─ present=true  → 真实备注（dto 字段填充）
  └─ present=false → 负缓存（60秒短TTL，由 fetchedAt 判定过期）

UserFriendChangeConsumer → UserFriendCache.updateFriendRemarkCache（不变）
```

**兼容性原则**：
- 协议号 1033、`FriendRemarkReqDto`、`FriendRemarkRespDto`、`UserFriendRemarkDto` 字段不变
- `UserFriendCache.getFriendRemarkCache(Long, Set<Long>)` / `updateFriendRemarkCache(Long, UserFriendRemarkDto)` 签名保留
- `UserFriendChangeConsumer` 不动
- MQ 消息结构 `GamePersonalRemarksDto` 不变

## 4. 安全改造（F-01 ~ F-05）

### F-01：FriendRemarkCommand 笔误与显式赋值

**文件**：`FriendRemarkCommand.java`

改造点：
1. 删除 `BeanUtils.copyProperties(req, addReq)` 之外多余的一行 `addReq.setRemarkUserId(remarkUserId)`（已经在前面设过）。
2. `UserFriendRemarkDto updatedDto = new UserFriendRemarkDto()` 之后，**不再** `BeanUtils.copyProperties`，改为显式 setter，意图明确：

```java
UserFriendRemarkDto updatedDto = new UserFriendRemarkDto();
updatedDto.setUserId(userId);
updatedDto.setRemarkUserId(remarkUserId);
updatedDto.setUserNameRemark(req.getUserNameRemark());
updatedDto.setUserRemark(req.getUserRemark());
```

3. 删除原 `addReq.setRemarkUserId(remarkUserId);`（在 `updatedDto` 构造之后那一行的死代码）。

### F-02：RPC 失败不写空对象

**文件**：`UserFriendCache.getFriendRemarkCache`（或 `FriendRemarkCacheService.getRemarks`）

改造点：
- `playerRpcService.queryFriendRemarkList(...)` 抛 `BusinessException` 或运行时异常时，**不向 cache 写入 `present=false`**，让下一次请求重试。
- catch 内 `log.warn` 升级为 `log.error`，包含 `userId`/`size`。
- 异常路径返回当前已命中的部分结果，未命中的 key 留给下次。

### F-03：负缓存独立 TTL

**新增**：`FriendRemarkCacheEntry { UserFriendRemarkDto dto; boolean present; long fetchedAt; }`

策略：
- RPC 返回的命中：`present=true`，dto 填充实际值。
- RPC 返回不包含的 key：`present=false`，dto = 空 DTO，`fetchedAt = System.currentTimeMillis()`。
- 读取时：`present=false` 且 `now - fetchedAt > 60_000` 当作未命中，重新 RPC。
- MQ 到达 / 设置备注：覆盖为 `present=true` 的命中 entry。

负缓存 TTL 写死 60 秒（足以撑过 MQ 延迟，又不让删除/新增长时间不可见）。

### F-04：日志脱敏

**文件**：`FriendRemarkCommand`

```java
// 旧
log.info("好友备注协议:1033 req: {}", GsonUtils.toJson(req));
// 新
log.info("好友备注协议:1033 userId:{} remarkUserId:{} nameLen:{} remarkLen:{}",
    req.getUserId(), req.getRemarkUserId(),
    safeLen(req.getUserNameRemark()), safeLen(req.getUserRemark()));
```

resp 日志同理改为字段而非整对象 `toJson`。`safeLen` 用 `StringUtils.length` 或 `Optional.ofNullable(...).map(String::length).orElse(0)`。

### F-05：缓存全量 map 日志

**文件**：`UserFriendCache.updateFriendRemarkCache`

```java
// 旧
log.info("updateFriendRemarkCache userId:{} map:{}", userId, JSON.toJSONString(map));
// 新
log.debug("updateFriendRemarkCache userId:{} remarkUserId:{}", userId, updatedDto.getRemarkUserId());
```

## 5. 结构性重构（S-01 ~ S-02）

### 5.1 `FriendRemarkCacheService`（新增，frame）

**包**：`com.dx.game.frame.repository.local.cache.friendremark`（落在 repository 模块，避免 `UserFriendCache` 静态门面跨模块编译依赖）

```java
@Slf4j
@Component
public class FriendRemarkCacheService {

    private static final long NEGATIVE_TTL_MS = 60_000L;

    @Resource
    private DxPlayerRpcClient dxPlayerRpcClient;

    private final LoadingCache<Long, ConcurrentHashMap<Long, FriendRemarkCacheEntry>> cache =
            CacheBuilder.newBuilder()
                    .expireAfterAccess(30, TimeUnit.MINUTES)
                    .build(CacheLoader.from(k -> new ConcurrentHashMap<>()));

    public Map<Long, UserFriendRemarkDto> getRemarks(Long userId, Set<Long> remarkUserIds) {
        if (CollectionUtils.isEmpty(remarkUserIds)) {
            return new HashMap<>();
        }
        ConcurrentHashMap<Long, FriendRemarkCacheEntry> shard = cache.getUnchecked(userId);
        long now = System.currentTimeMillis();

        Set<Long> toFetch = remarkUserIds.stream()
                .filter(uid -> needFetch(shard.get(uid), now))
                .collect(Collectors.toSet());

        if (!toFetch.isEmpty()) {
            try {
                List<DxGamePersonalRemarksResp> dataList =
                        dxPlayerRpcClient.queryFriendRemarkList(userId, new ArrayList<>(toFetch));
                Set<Long> returned = new HashSet<>();
                for (DxGamePersonalRemarksResp resp : dataList) {
                    UserFriendRemarkDto dto = new UserFriendRemarkDto();
                    BeanUtils.copyProperties(resp, dto);
                    shard.put(resp.getRemarkUserId(), FriendRemarkCacheEntry.present(dto, now));
                    returned.add(resp.getRemarkUserId());
                }
                // 仅在 RPC 成功的前提下，对真实"不存在"的 key 写负缓存
                for (Long uid : toFetch) {
                    if (!returned.contains(uid)) {
                        shard.put(uid, FriendRemarkCacheEntry.absent(now));
                    }
                }
            } catch (Exception e) {
                log.error("获取好友备注RPC异常 userId:{} size:{}", userId, toFetch.size(), e);
                // 不写负缓存，下次重试
            }
        }
        return remarkUserIds.stream()
                .collect(Collectors.toMap(uid -> uid,
                        uid -> Optional.ofNullable(shard.get(uid))
                                .map(FriendRemarkCacheEntry::getDtoOrEmpty)
                                .orElseGet(UserFriendRemarkDto::new)));
    }

    public void updateRemark(Long userId, UserFriendRemarkDto updatedDto) {
        if (updatedDto == null || updatedDto.getRemarkUserId() == null) {
            return;
        }
        ConcurrentHashMap<Long, FriendRemarkCacheEntry> shard = cache.getUnchecked(userId);
        shard.put(updatedDto.getRemarkUserId(),
                FriendRemarkCacheEntry.present(updatedDto, System.currentTimeMillis()));
        log.debug("好友备注缓存更新 userId:{} remarkUserId:{}", userId, updatedDto.getRemarkUserId());
    }

    private boolean needFetch(FriendRemarkCacheEntry entry, long now) {
        if (entry == null) return true;
        if (entry.isPresent()) return false;
        return now - entry.getFetchedAt() > NEGATIVE_TTL_MS;
    }
}
```

### 5.2 `FriendRemarkCacheEntry`（新增，frame）

**包**：`com.dx.game.frame.repository.local.cache.friendremark`（落在 repository 模块，避免 `UserFriendCache` 静态门面跨模块编译依赖）

```java
@Getter
public class FriendRemarkCacheEntry {
    private final UserFriendRemarkDto dto;
    private final boolean present;
    private final long fetchedAt;

    private FriendRemarkCacheEntry(UserFriendRemarkDto dto, boolean present, long fetchedAt) {
        this.dto = dto;
        this.present = present;
        this.fetchedAt = fetchedAt;
    }

    public static FriendRemarkCacheEntry present(UserFriendRemarkDto dto, long now) {
        return new FriendRemarkCacheEntry(dto, true, now);
    }
    public static FriendRemarkCacheEntry absent(long now) {
        return new FriendRemarkCacheEntry(null, false, now);
    }
    public UserFriendRemarkDto getDtoOrEmpty() {
        return dto != null ? dto : new UserFriendRemarkDto();
    }
}
```

### 5.3 `UserFriendCache` 改为静态门面（修改，frame）

**文件**：`dx-game-frame-repository/.../local/cache/UserFriendCache.java`

签名保留，内部委托 Bean。Spring 启动后写入静态引用：

```java
public class UserFriendCache {
    private static FriendRemarkCacheService delegate;

    static FriendRemarkCacheService delegate() {
        if (delegate == null) {
            delegate = SpringContextUtils.getBean(FriendRemarkCacheService.class);
        }
        return delegate;
    }

    public static void updateFriendRemarkCache(Long userId, UserFriendRemarkDto updatedDto) {
        delegate().updateRemark(userId, updatedDto);
    }

    public static Map<Long, UserFriendRemarkDto> getFriendRemarkCache(Long userId, Set<Long> remarkUserIds) {
        return delegate().getRemarks(userId, remarkUserIds);
    }
}
```

> 说明：之所以保留 `SpringContextUtils` 而不是构造器注入，是因为静态门面无法被 Spring 直接注入；做法是首次访问时取一次 Bean 并缓存。20+ 调用方零改动。

### 5.4 `FriendRemarkCommandService`（新增，frame）

**包**：`com.dx.game.frame.repository.local.cache.friendremark`（落在 repository 模块，避免 `UserFriendCache` 静态门面跨模块编译依赖）

```java
@Slf4j
@Component
public class FriendRemarkCommandService {

    @Resource private DxPlayerRpcClient dxPlayerRpcClient;
    @Resource private FriendRemarkCacheService friendRemarkCacheService;

    public FriendRemarkRespDto saveRemark(FriendRemarkReqDto req) {
        Long userId = req.getUserId();
        Long remarkUserId = req.getRemarkUserId();
        if (remarkUserId == null || Objects.equals(remarkUserId, userId)) {
            throw BusinessException.createException(BusinessExceptionTypeEnums.USER_ID_PARAM_ERROR);
        }

        DxGamePersonalRemarksAddReq addReq = new DxGamePersonalRemarksAddReq();
        addReq.setUserId(userId);
        addReq.setRemarkUserId(remarkUserId);
        addReq.setUserNameRemark(req.getUserNameRemark());
        addReq.setUserRemark(req.getUserRemark());
        dxPlayerRpcClient.saveFriendRemark(addReq);

        UserFriendRemarkDto updatedDto = new UserFriendRemarkDto();
        updatedDto.setUserId(userId);
        updatedDto.setRemarkUserId(remarkUserId);
        updatedDto.setUserNameRemark(req.getUserNameRemark());
        updatedDto.setUserRemark(req.getUserRemark());
        friendRemarkCacheService.updateRemark(userId, updatedDto);

        FriendRemarkRespDto resp = new FriendRemarkRespDto();
        resp.setUserId(userId);
        resp.setRemarkUserId(remarkUserId);
        resp.setUserNameRemark(req.getUserNameRemark());
        resp.setUserRemark(req.getUserRemark());
        return resp;
    }
}
```

### 5.5 `FriendRemarkCommand` 简化（修改，frame）

```java
@Slf4j
@Component
@ThreadModel(EThreadModel.PLAYER)
public class FriendRemarkCommand implements RequestCommand<FriendRemarkReqDto> {

    @Resource private FriendRemarkCommandService friendRemarkCommandService;

    @Override
    public ProtoType getType() { return CommonProto.USER_FRIEND_REMARK; }

    @Override
    public void handleCommand(FriendRemarkReqDto req) {
        log.info("好友备注协议:1033 userId:{} remarkUserId:{} nameLen:{} remarkLen:{}",
                req.getUserId(), req.getRemarkUserId(),
                StringUtils.length(req.getUserNameRemark()),
                StringUtils.length(req.getUserRemark()));
        try {
            FriendRemarkRespDto resp = friendRemarkCommandService.saveRemark(req);
            SendUtil.send(NetResponseBuilder.buildResponse(CommonProto.USER_FRIEND_REMARK, resp, req));
            log.info("好友备注协议:1033 成功 userId:{} remarkUserId:{}", req.getUserId(), req.getRemarkUserId());
        } catch (BusinessException e) {
            log.warn("好友备注协议:1033 业务异常 userId:{} remarkUserId:{} code:{}",
                    req.getUserId(), req.getRemarkUserId(), e.getCodeMsg(), e);
            SendUtil.send(NetResponseBuilder.buildErrorResponse(CommonProto.USER_FRIEND_REMARK, e, req));
        } catch (Exception e) {
            log.error("好友备注协议:1033 系统异常 userId:{} remarkUserId:{}",
                    req.getUserId(), req.getRemarkUserId(), e);
            SendUtil.send(NetResponseBuilder.buildErrorResponse(
                    CommonProto.USER_FRIEND_REMARK,
                    BusinessException.createException(BusinessExceptionTypeEnums.SYSTEM_EXCEPTION), req));
        }
    }
}
```

## 10. 验收关键场景

1. **笔误回归**：A 给 B 设置备注，立即查询，本节点 `getFriendRemarkCache` 返回 B 的最新备注。
2. **负缓存**：A 查询从未备注过的 B，第一次走 RPC 并写 `absent`；60 秒内再次查询不再 RPC，返回空 DTO；60 秒后允许重新 RPC。
3. **MQ 覆盖负缓存**：A 第一次查 B 写了 `absent`；30 秒后 A 在其他节点给 B 设了备注，MQ 到本节点，立即覆盖为 `present`；A 下次查询直接看到备注。
4. **RPC 异常**：玩家服务挂掉时 `log.error` 一次，不写 cache；服务恢复后下一次请求重新 RPC 成功；指标可观测。
5. **日志脱敏**：抓 `好友备注协议:1033` info 日志，只能看到 `userId`/`remarkUserId`/字段长度，看不到任何 `userRemark` / `userNameRemark` 明文。
6. **门面零侵入**：`grep getFriendRemarkCache` / `grep updateFriendRemarkCache` 在 dx-game-frame + dx-game-guandan 的调用点行数不变。
