# 预站起 / 留座离桌 重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/18-预站起留座离桌.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame + dx-game-guandan
> 分支：20260615/junk
> 前置依赖：`04-站起.md`、`07-离开状态检查.md`（仅作语义对齐，不在本次实现范围）

## 1. 背景与目标

预站起（1045 `PREP_STAND`）与留座离桌（1034 `USER_HOLD_SEAT`）在当前代码中都存在严重的"半实现/未实现"问题：

- **P0-1**：1045 命令依赖的 `UserStandPrepBootstrapService` 接口存在但**没有任何实现类**。`UserPrepStandCommandDefault.handle` 通过 `SpringContextUtils.getBean(UserStandPrepBootstrapService.class)` 取 Bean，运行期必抛 `NoSuchBeanDefinitionException` → 走 catch (Exception) → 客户端只能收到系统异常。`BaseUserItem.prepStandStatus` 永远不会被写入非 0 值，`PrepStandLeaveUserVisitor` 在 `UserLeaveForcedEnum.PREPARE_STAND` 场景下永远扫不到玩家，整个预站起链路对外**不可用**。
- **P0-2**：1034 `USER_HOLD_SEAT` 协议号、`UserConfig.timeoutHoldSeatSwitch`、`SentinelTypeEnum.HOLD_SEAT`、`UserStandTypeEnum.HOLD_SEAT_COUNTDOWN`、错误码 `TABLE_HOLD_SEAT_SWITCH_CLOSE/4006`、`TABLE_HOLD_SEAT_NOT_ENOUGH/4009`、`HOLD_SEAT_NEXT_POP/4020` 都存在，但**没有 Command 实现**。客户端如果按协议号 1034 发请求，服务端无任何业务处理（连错误响应都没有），网络层只会静默丢包或超时。
- **P1-1**：`UserPrepStandReqDto.standType` 注释为 `"0:取消预站起 2:本手结束后站起 2:本轮换庄后站起"`，与实际枚举 `PrepStandStatusEnum (0/1/2/3)` 不一致（出现两个 2，缺 1 与 3）。
- **P1-2**：`UserPrepStandCommandDefault.check` 使用 `UserLeaveBootstrapService.commonCheckUserBreakLink`（断线退出语义）做预站起前置校验，语义错配。
- **P2-1**：`UserPrepStandRespDto.userId` 字段在 DTO 中存在但 `after` 中未设置。
- **P2-2**：`PrepStandStatusEnum` 类头注释为 `"鱿鱼开关枚举"`，与实际语义不符（明显是从其它枚举复制残留）。
- **P2-3**：1045 命令链路存在 info 级 `GsonUtils.toJson(req)` 大对象日志（成功 / 业务异常 / 系统异常三处）。

目标：以**最小安全改造**消除 P0/P1 问题（让 1045 在线上能返回明确结果而非系统异常、让 1034 不再静默丢包），不引入新的状态机或编排服务，避免与 04-站起 / 06-退出 / 07-离开状态检查 后续重构发生冲突。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 1045 是否保留 | **保留**。补默认实现 `UserStandPrepBootstrapServiceImpl`，写入 `BaseUserItem.prepStandStatus`，依赖既有 `PrepStandLeaveUserVisitor` 生效路径 |
| `STAND=1`（立即站起）如何处理 | **本期不支持**。`UserStandPrepBootstrapServiceImpl` 收到 `STAND=1` 时返回 `PARAM_EXCEPTION` 业务异常。理由：现有 `PrepStandLeaveUserVisitor` 把 `isPrepStand() \|\| isPrepStandHand()` 都映射为 `NEXT_HAND`，与"立即站起"语义不符；真正的立即站起路径是 1004 站起协议，不应混入 1045 |
| 1034 留座离桌是否实现 | **本期不实现**。新增一个 NoSupport 命令明确返回 `NOT_SUPPORT_OPERATION(4069)`，避免客户端发包后无响应。完整留座离桌 flow 待产品确认是否属于掼蛋范围后单独立项（不与预站起混合）|
| 注释纠正 | 直接修正 `UserPrepStandReqDto.standType` 注释与 `PrepStandStatusEnum` 类头注释，无协议字段变更 |
| `commonCheckUserBreakLink` 语义错配 | **本期不动**。等 04-站起 / 06-退出 / 07-离开状态检查 重构统一引入 `LeaveIntent.PREP_STAND` 规则链后一并处理，列入待确认 |
| `UserPrepStandRespDto.userId` 字段 | `after()` 补 `setUserId(request.getUserId())`，DTO 字段不变 |
| 日志收敛 | `UserPrepStandCommandDefault` 的 `GsonUtils.toJson(req)` info/warn/error 日志改为只打关键字段；`PrepStandLeaveUserVisitor` 的 `GsonUtils.toJson(resultList)` info 降为 debug |
| 灰度开关 | 不做开关。1045 当前在线对外不可用（系统异常），新实现只可能改善行为；1034 当前完全无业务，新增 NoSupport 命令也只可能改善行为 |

## 3. 整体架构

### 改造前

```
1045 PREP_STAND ─ UserPrepStandCommand (PLAYER 线程)
                    └─ UserPrepStandCommandDefault.handleCommand
                         ├─ check()   — hasSitDownUser + getByCode + commonCheckUserBreakLink
                         ├─ handle()  — SpringContextUtils.getBean(UserStandPrepBootstrapService.class)
                         │              └─ NoSuchBeanDefinitionException → 系统异常
                         └─ after()   — 不可达
                              ↓ never written
                       BaseUserItem.prepStandStatus 永远 = 0
                              ↓ never matched
                       PrepStandLeaveUserVisitor 永远返回空列表

1034 USER_HOLD_SEAT ─ ❌ 无 Command 实现 ─ 客户端发包静默丢失
```

### 改造后

```
1045 PREP_STAND ─ UserPrepStandCommand (PLAYER 线程, 不变)
                    └─ UserPrepStandCommandDefault.handleCommand
                         ├─ check()  — 修正注释，校验逻辑保留不动
                         ├─ handle() — 通过 SpringContextUtils 取 UserStandPrepBootstrapServiceImpl
                         │              └─ writePrepStandStatus(tableId,userId,enum)
                         │                   ├─ STAND=1   → 抛 PARAM_EXCEPTION（不支持立即）
                         │                   └─ 0/2/3     → 写入 BaseUserItem.prepStandStatus
                         └─ after()  — set msg/prepStatus/userId
                              ↓ written
                       BaseUserItem.prepStandStatus = 0 / 2 / 3
                              ↓ matched by visitor
                       PrepStandLeaveUserVisitor 在本手 / 本轮结束时生成离开动作

1034 USER_HOLD_SEAT ─ UserHoldSeatCommand (PLAYER 线程, 新增)
                       └─ 仅 log.warn + buildErrorResponse(NOT_SUPPORT_OPERATION)
                          客户端收到 4069 明确错误，不再无响应
```

**兼容性原则**：

- 协议号 1045 / 1034 不变。
- `UserPrepStandReqDto` / `UserPrepStandRespDto` 字段不变（注释修正不算协议变更）。
- `UserStandPrepBootstrapService` 接口签名不变；新增实现类不破坏既有接口。
- `PrepStandStatusEnum`、`BaseUserItem.prepStandStatus`、`PrepStandLeaveUserVisitor`、`UserLeaveForcedEnum.PREPARE_STAND`、`SpecifyTypeEnum.NEXT_HAND/NEXT_ROUND`、`BaseUserStandFinishedListener` / `BaseUserQuitFinishedListener` 的清理逻辑全部保留不动。
- 1034 不引入任何留座离桌业务逻辑，只新增一个"明确不支持"的协议占位 Command。

## 4. 安全改造（R-01 ~ R-06）

### R-01：补 `UserStandPrepBootstrapServiceImpl` 实现（P0-1）

**新增文件**：`UserStandPrepBootstrapServiceImpl.java`（dx-game-frame，包 `com.dx.game.frame.service.biz.bootstrap.impl`）

```java
@Slf4j
@Service
public class UserStandPrepBootstrapServiceImpl implements UserStandPrepBootstrapService {

    @Override
    public PrepStandStatusEnum prepStand(Long tableId, Long userId, PrepStandStatusEnum prepStandStatusEnum) {
        BaseUserItem userItem = BaseTableUserCache.getUser(tableId, userId);
        if (userItem == null) {
            log.warn("预站起-用户不在牌桌 tableId:{} userId:{}", tableId, userId);
            throw BusinessException.createException(BusinessExceptionTypeEnums.NO_EXIST_TABLE_USER);
        }
        // STAND=1 立即站起本期不支持：真正的立即站起走 1004 站起协议
        if (prepStandStatusEnum == PrepStandStatusEnum.STAND) {
            log.warn("预站起-暂不支持立即站起（STAND=1），请走 1004 站起 tableId:{} userId:{}", tableId, userId);
            throw BusinessException.createException(BusinessExceptionTypeEnums.PARAM_EXCEPTION);
        }
        Integer oldStatus = userItem.getPrepStandStatus();
        userItem.setPrepStandStatus(prepStandStatusEnum.getCode());
        log.info("预站起-状态写入 tableId:{} userId:{} old:{} new:{}",
                tableId, userId, oldStatus, prepStandStatusEnum.getCode());
        return prepStandStatusEnum;
    }
}
```

> 单纯本地状态写入，不触发任何 visitor / event；生效路径由既有 `PrepStandLeaveUserVisitor` 在本手 / 本轮结束阶段扫描。`BaseUserStandFinishedListener` / `BaseUserQuitFinishedListener` 已经会把 `prepStandStatus` 清回 0（L91/L90），无需额外清理。

## 5. 不在本期范围

以下问题被识别但**不在本期改造**，记录到待确认项 / 后续 Junk 任务：

- **`commonCheckUserBreakLink` 语义错配**：预站起 check 使用断线退出规则。等 04-站起 / 06-退出 / 07-离开状态检查 统一改造 `LeaveIntent` 规则链时一并处理。
- **`STAND=1` 立即站起真实生效**：本期返回 PARAM_EXCEPTION 不支持；如果产品需要让 1045 也支持立即站起，需要 visitor 区分 `isPrepStand()` 与 `isPrepStandHand()` 分别生成 NEXT_HAND vs 直接 STAND，或者在 service 内直接调用 1004 站起 flow。
- **留座离桌完整实现**：1034 当前只返回 NOT_SUPPORT_OPERATION。完整 `HoldSeatFlowService`（校验开关 `timeoutHoldSeatSwitch`、限流 `SentinelTypeEnum.HOLD_SEAT`、超时计数 `UserStandTypeEnum.HOLD_SEAT_COUNTDOWN`、错误码 `HOLD_SEAT_NEXT_POP/NOT_ENOUGH/SWITCH_CLOSE`）待产品确认范围后单独立项。
- **结算补码中的预站起**：扫描第 13 节"待确认项"提出"预站起是否应允许结算补码中的用户设置"，本期 check 仅校验 `hasSitDownUser`，不做补码状态额外判断。
- **PLAYER 线程并发**：1045 在 PLAYER 线程写 `BaseUserItem.prepStandStatus`，与 TABLE 线程上的结算 / 手结束并发存在理论窗口。本期暂不引入表级锁，等 04/06/07 统一线程模型时一并处理。

## 9. 验收关键场景

1. **1045 standType=0（取消）**：在桌入座玩家发送 `standType=0`，返回 `prepStatus=0`，`BaseUserItem.prepStandStatus` 设为 0。
2. **1045 standType=2（本手后）**：返回 `prepStatus=2`，`prepStandStatus=2`；本手结束 visitor 扫描时生成 `NEXT_HAND` 离开动作，玩家被站起。
3. **1045 standType=3（本轮后）**：返回 `prepStatus=3`，`prepStandStatus=3`；本轮换庄 visitor 扫描时生成 `NEXT_ROUND` 离开动作，玩家被站起。
4. **1045 standType=1（暂不支持）**：返回 `PARAM_EXCEPTION(11)` 业务异常，玩家状态不变。
5. **1045 用户不在桌**：返回 `NO_EXIST_TABLE_USER(4016)` 业务异常，且不再有系统异常路径。
6. **1045 service Bean 已就绪**：之前应用启动后第一次 1045 必抛 `NoSuchBeanDefinitionException`，本期之后改为正常返回。
7. **1045 用户站起后清理**：玩家被站起后 `BaseUserStandFinishedListener` 把 `prepStandStatus` 清回 0，再次进入 1045 路径正确。
8. **1045 用户退出后清理**：玩家退出后 `BaseUserQuitFinishedListener` 把 `prepStandStatus` 清回 0。
9. **1034 客户端发包**：客户端发 1034，服务端返回 `NOT_SUPPORT_OPERATION(4069)` 错误响应，不再静默丢失。
10. **预站起日志**：成功 1045 链路只有 `预站起 ...` 入口 info + `预站起-状态写入 ...` 一条 info，无 `GsonUtils.toJson(req)` 大对象输出。

## 10. 迁移与灰度

- 1045 当前线上对外不可用（系统异常），新实现只可能改善客户端体验。
- 1034 当前线上对外完全无业务，NoSupport 命令只可能改善客户端体验（明确错误 vs 静默）。
- 无需灰度，直接上线。
- 回滚：单独 revert R-01（删除 Impl）即可让 1045 回到旧的系统异常状态；单独 revert R-02（删除 UserHoldSeatCommand）即可让 1034 回到静默无响应。
