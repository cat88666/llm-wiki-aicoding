# 放弃补码重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/17-放弃补码.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame（命令骨架）+ dx-game-guandan（审核单失效等扩展）
> 分支：20260615/junk
> 复杂度：H（P0 高，但本质是"协议缺实现"的策略决策）
> 前置依赖：03-带入（占座/补码）、04-站起（TRANSFER 站起）、07-离开（LeaveIntent）、16-私人房固定带入审核（审核单失效）

## 1. 背景与目标

放弃补码理论覆盖 1042 `CommonProto.GIVE_UP_TRANSFER`（`CommonProto.java:44`）、14039 `GIVE_UP_TRANSFER_NOTICE`（`:84`）。基于真实代码确认：

- **P0-1：协议号存在但业务命令缺失**。两工程**未发现** `RequestCommand` 返回 `GIVE_UP_TRANSFER`，也未发现发送 14039 的业务类；仅 `RequestCmdInterceptImpl` 赛事禁用列表引用了 `GIVE_UP_TRANSFER`。客户端若发 1042，服务端**无处理**（静默丢包/超时），协议表会误导联调"放弃补码已实现"。
- **现状替代链路**：放弃补码被折叠进"补码失败/占座超时站起"——`UserStandTypeEnum.TRANSFER(7,"放弃补码/补码失败")`；`OccupyTimeOutTask` 超时按 `TRANSFER` 站起；用户关补码弹窗/补码失败时站起 standType 修正为 TRANSFER。即后端能处理"补码失败导致的站起"，但不能确认客户端**主动** 1042 被正确处理。
- **P0-2：主动放弃与超时/失败未区分**。`UserStandTypeEnum.TRANSFER` 同时描述"放弃补码/补码失败"，`OccupyTimeOutTask` 复用同一 standType，无单独记录主动放弃时间/来源 → 数据分析/风控无法还原真实原因。
- **P1-1：与私人审核等待态冲突风险**。16 `SettleComplementAuditService` 在补码/额度申请保留待审核用户；若 1042 直接站起而不失效审核单，房主稍后审核通过会触发**不该发生的带入**。

目标：**明确 1042 的处理策略**（实现 or 显式不支持），消除"协议存在、实现缺失"的口径不一致；若实现，必须走统一 flow（取消占座超时任务 + 失效待审核单 + TRANSFER 站起），并内部区分放弃原因；**不改 1007/14007 站起通知协议**。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 提供 `GiveUpTransferCommand` + `GiveUpTransferFlowService` 骨架，guandan 提供审核单失效（16）扩展 |
| **1042 实现 vs 废弃（核心待确认）** | **安全默认：实现一个薄命令**，路由到现有 TRANSFER 站起流（行为不新增风险，只是把"无处理"变为"明确处理"）。**是否废弃改走 1007 由产品/客户端确认**（§10）。若产品确认废弃，则退化为 NoSupport 占位（返回 `NOT_SUPPORT_OPERATION`，对齐 18 的 1034 处理） |
| 命令线程模型 | `@ThreadModel(TABLE)`（与站起/带出一致，涉及牌桌状态） |
| 放弃原因区分（P0-2） | 内部新增 `TransferAbortReason { USER_GIVE_UP / TIMEOUT / BALANCE_NOT_ENOUGH / AUDIT_REJECT }`；对外仍复用 1007/14007，仅内部埋点/进出记录区分 |
| 审核单冲突（P1-1） | `GiveUpTransferFlowService` 必须**先取消/失效待审核单**（16 `SettleComplementAuditService`），再执行站起，避免房主后审通过误带入 |
| 14039 通知 | 本期**不强制启用**；最终状态以 1007/14007 站起通知表达。是否需要 14039 待产品确认（§10） |
| 协议兼容 | 若实现 1042，不新增必填请求字段；响应基础成功对象；1007/14007 不动 |
| 灰度策略 | 先确认客户端是否发 1042；若发，灰度实现命令仅记录不改旧行为，再接审核单失效，最后（可选）启 14039 |

## 3. 整体架构

### 改造前

```
1042 GIVE_UP_TRANSFER（协议号存在）── ❌ 无 RequestCommand ── 客户端发包无处理
现状替代：
  关补码弹窗/补码失败 → 站起 standType 修正为 TRANSFER
  OccupyTimeOutTask 超时 → UserLeaveBootstrapService.userStand(..., TRANSFER, KickerReason.TRANSFER)
  最终状态由 1007/14007（站起）表达
```

### 改造后（实现分支，安全默认）

```
1042 GIVE_UP_TRANSFER ── GiveUpTransferCommand(TABLE, 新增)
                          └─ GiveUpTransferFlowService.giveUp(tableId, userId)
                               ├─ 1. 校验用户状态（必须 OCCUPY / 补码等待，否则业务异常）
                               ├─ 2. 取消占座超时任务（OccupyTimeOutTask 对应 event key）
                               ├─ 3. 失效待审核单（guandan SettleComplementAuditService，对齐16）
                               ├─ 4. 走站起 flow：standType=TRANSFER, reason=USER_GIVE_UP（复用04/07）
                               └─ 5. （可选）发送 14039；最终状态 1007/14007
（若产品确认废弃 → GiveUpTransferCommand 退化为 NoSupport：返回 NOT_SUPPORT_OPERATION）
```

**兼容性原则**：
- 1007/14007 站起通知、1005/14008 退出通知不变（最终状态通知）。
- 若实现 1042：不新增必填请求字段；响应为基础成功对象；14039 启用时为**可选**通知。
- `UserStandTypeEnum.TRANSFER` 枚举不动（内部 `TransferAbortReason` 细化原因，不改对外 standType）。
- `OccupyTimeOutTask` / 审核通过链路不改（只在放弃 flow 内取消/失效）。

## 4. 安全改造（R-01 ~ R-03）

### R-01：实现 `GiveUpTransferCommand` + `GiveUpTransferFlowService`（P0-1）
- 新增 `GiveUpTransferCommand`（frame-cmd，`@ThreadModel(TABLE)`，`getType=GIVE_UP_TRANSFER`），委托 `GiveUpTransferFlowService`。
- `GiveUpTransferFlowService.giveUp`：状态校验 → 取消占座超时任务 → 失效待审核单 → TRANSFER 站起（复用 04/07 stand flow）。
- 若产品确认废弃：command 退化为 `log.warn + NOT_SUPPORT_OPERATION`（对齐 18 的 1034），不引入 flow。

### R-02：放弃原因内部区分（P0-2）
- 新增内部 `TransferAbortReason { USER_GIVE_UP / TIMEOUT / BALANCE_NOT_ENOUGH / AUDIT_REJECT }`。
- 1042 主动放弃 = USER_GIVE_UP；`OccupyTimeOutTask` = TIMEOUT；补码失败 = BALANCE_NOT_ENOUGH；审核拒绝 = AUDIT_REJECT。
- 对外仍复用 1007/14007，仅进出记录 / 埋点 / 日志带 reason。

### R-03：放弃前失效待审核单（P1-1）
- `GiveUpTransferFlowService` 在站起前调用 guandan 审核单失效（16 `SettleComplementAuditService`），保证主动放弃后房主后审通过不再误带入。
- 取消占座超时任务（避免放弃后又被超时站起重复处理）。

## 5. 新增组件

| 文件 | 工程 | 职责 |
|------|------|------|
| `GiveUpTransferCommand.java` | frame-cmd | 1042 入口（实现或 NoSupport） |
| `GiveUpTransferReqDto.java` | frame-model | 当前可只依赖基础请求（tableId/userId） |
| `GiveUpTransferRespDto.java` | frame-model | userId/tableId/status/reason |
| `GiveUpTransferFlowService.java` | frame-service | 校验→取消超时→失效审核→TRANSFER站起 |
| `TransferAbortReason`(enum) | frame-constant | 内部放弃原因 |

> guandan 侧：审核单失效（16）作为 flow 的扩展点实现；彩金/私审分支复用既有。

## 6. 协议兼容确认

| 协议号 | 类型 | DTO | 字段是否变更 | 兼容性 |
|--------|------|-----|--------------|--------|

## 7. 可测试性设计

- `GiveUpTransferFlowServiceTest`：OCCUPY 用户主动放弃 → 站起 + 占座超时任务取消 + 审核单失效；非 OCCUPY 用户 1042 → 业务异常；有待审核单时放弃 → 审核单失效。
- 集成：1042 后收到站起通知；放弃后房主审核通过不再上分；放弃后用户进入 WATCH/QUIT 策略符合产品。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 前置 | 先确认客户端是否发 1042（决定实现 vs 废弃 NoSupport） |
| 灰度 | 实现命令先仅记录不改旧行为 → 接审核单失效 → （可选）启 14039 |
| 回滚 | 关闭 1042 command 注册或返回明确不支持；保留旧补码超时站起链路 |
| 与其他 Junk 同步 | TRANSFER 站起复用 04/07；占座超时取消关联 03；审核单失效对齐 16 |

## 9. 验收关键场景

1. 客户端发 1042 → （实现分支）站起成功 + 占座超时任务取消 + 审核单失效；（废弃分支）NOT_SUPPORT_OPERATION。
2. 主动放弃不会再被占座超时重复站起（reason 区分 USER_GIVE_UP vs TIMEOUT）。
3. 有待审核单时放弃 → 审核单失效；房主后审通过不再上分。
4. 非 OCCUPY/非补码等待用户发 1042 → 业务异常。
5. 用户最终状态（WATCH/QUIT）与前端展示一致。

## 10. 待确认项（含阻塞型协议策略）

| 待确认项 | 类型 | 处理 |
|----------|------|------|
| **1042 当前客户端是否仍调用 / 废弃还是实现** | 协议（阻塞型，指导§5.4） | **安全默认：实现薄命令路由 TRANSFER 站起**；待产品确认；若废弃则退 NoSupport |
| 放弃后用户应站起为旁观还是退出房间 | 产品 | 本期默认沿用 TRANSFER 站起（保留桌位语义同现状），待确认 |
| 是否必须发 14039 还是 14007 足够 | 产品/客户端 | 本期不强制 14039，14007 表达最终状态 |
| 放弃补码是否允许在结算补码审核等待中发生 | 产品 | 本期允许并失效审核单，待确认 |

## 11. 本功能不处理的范围

- 不改占座超时任务 `OccupyTimeOutTask` 本体（只在放弃 flow 内取消）。
- 不改审核通过链路（只失效待审核单）。
- 不改 1007/14007 站起通知协议。
- 不改 `UserStandTypeEnum.TRANSFER` 对外枚举值。
