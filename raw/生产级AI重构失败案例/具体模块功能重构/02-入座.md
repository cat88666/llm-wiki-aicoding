## 1. 功能边界

本功能覆盖 `CommonProto.USER_SIT_DOWN` 入座协议，协议号 1001。包含玩家从旁观(WATCH_USER)到入座(PLAYER_USER)的完整流程，含占座(OCCUPY)、占座超时、占座转入座、自动带入、入座风控检测、限流等子功能。

子功能范围：

| 子功能 | 当前覆盖情况 |
| --- | --- |
| 主流程 | `UserSitDownCommand -> GameHandleFactory -> 空间类型 Handler -> *UserSitDownDefault` |
| 入座前状态检查 | `BaseUserSitDownCheckService.checkSitDown()` 含踢出、座位、权限、风控 |
| 座位/椅子校验 | `BaseUserSitDownCheckService.checkSeat()` 含椅号合法、已入座、旁观者、满座、俱乐部代理可见性 |
| 入座风控校验 | `BaseTableRiskService.checkRiskSitDown()` 含视频/设备/GPS/IP 五项检测 |
| 入座限流 | `BaseUserSitDownCheckService.checkSentinel()` 含强制站起、占座、入座三级限流 |
| 占座 | `UserSitDownDefault.userSitDownOccupy()` 设 OCCUPY 状态 + 倒计时 |
| 占座超时 | `OccupyTimeOutTask` 超时触发站起 |
| 占座转入座 | `OccupyToSitDownTask` 带入成功后从 OCCUPY 转 DEFAULT |
| 自动带入 | `UserBringInBootstrapService.autoBringInChipAndFlushCache()` |
| 结算后重入座补码 | `UserSitDownBootstrap.ensureOccupyUserSitDownWhenNoComplement()` + `UserSitDownFinishListener` |

不处理范围：房主操作（Ceres）、游戏开始/出牌（Torsen）、结算/派彩（Duik）、打赏、投屏、大厅换桌。

## 4. 现有设计理解

### 主流程叙事

1. 客户端发送 `USER_SIT_DOWN(1001)` 协议，`UserSitDownCommand` 在 TABLE 线程处理。
2. `GameHandleFactory` 根据牌桌空间类型路由到 CLUB/PRIVATE/HALL/RACE 对应 Handler。HALL 和 RACE 的 handler 是空方法（不支持手动入座）。
3. CLUB/PRIVATE 进入 `*UserSitDownDefault.handle()`，创建 `UserSitDownContext`。
4. 初始化阶段查询用户钱包余额，计算本次需带入金额（`initSitDownScore`），掼蛋私人审核房覆盖为读取 PRIVATE_SCORE 钱包。
5. 校验阶段：踢出→座位合法性→牌桌权限→风控检测→三级限流（强制站起冷却/占座间隔/入座频率）。
6. 筹码校验：私人房跳过余额校验；非私人房余额不足时设置 respDto 并抛异常返回余额信息。
7. 处理入座：先自动带入筹码（`autoBringInChipAndFlushCache`），然后将 WATCH_USER 的 `userItem` 设置 chairId，根据是否需要补码决定走 OCCUPY（占座等待带入）还是直接 DEFAULT/SIT_DOWN_WAIT_ONE_HAND。将用户从 WATCH_USER 迁移到 PLAYER_USER。
8. 后置处理：发布 `UserSitDownFinishEvent` → 发送入座响应 → 广播入座通知(14001) → 如果是 OCCUPY 则发送占座补码通知(14022)并启动超时任务。
9. 掼蛋侧后置扩展：结算补码重发通知、无需补码时兜底占座转入座。

### 条件分支

| 分支 | 触发条件 | 当前处理 |
| --- | --- | --- |
| 需要带入(needTransferIn=true) | 带入金额 > 0 | 走 OCCUPY 占座流程，等待客户端/自动带入 |
| 不需要带入(needTransferIn=false) | 带入金额 = 0 或上次带出足够 | 直接设 DEFAULT/SIT_DOWN_WAIT_ONE_HAND |
| 游戏进行中入座 | `tableHandStatus != WAIT` 且未参与手牌 | 状态设 `SIT_DOWN_WAIT_ONE_HAND`（等一手） |
| 回到原座位 | `tableUserHand != null` 且 chairId 一致 | 状态设 DEFAULT |
| 坐到新座位 | `tableUserHand != null` 且 chairId 不一致 | 状态设 `SIT_DOWN_WAIT_ONE_HAND` |
| 私人审核房 | `tableInfo.isPrivateTransferAudit()` | 覆盖 cashBalance 为 PRIVATE_SCORE，房主 autoBringLimit 为 MAX_VALUE |
| 占座已有审核记录 | `retryBringInReviewRedisService` 有未处理审核 | 复用审核倒计时，不重新申请 |
| 结算后重坐无需补码 | `checkNeedComplement` 返回 false | 清理 needAddChipUsersMap + playerIsContinue，立即触发 `OccupyToSitDownTask` |

### 异常路径

| 异常 | 处理 |
| --- | --- |
| 业务异常（座位已满、风控拒绝、余额不足等） | catch `BusinessException`，返回错误码响应 |
| 系统异常 | catch `Exception`，返回 SYSTEM_EXCEPTION |
| 占座超时 | `OccupyTimeOutTask` 触发 `userStand(TRANSFER)`，玩家被站起 |
| 事件监听异常 | catch 后打日志不阻塞，证据：`BaseUserSitDownFinishListener.java:71`，`UserSitDownFinishListener.java:83` |
| 占座补码通知异常 | catch 后打日志不阻塞，证据：`BaseOccupyUserService.java:235-237` |

### 当前设计中应保留的合理点

- **TABLE 线程模型**：入座协议在 TABLE 线程处理（不同于进房的 PLAYER 线程），保证入座与站起、结算的桌状态变更线性化。
- **三级限流**：强制站起冷却、占座间隔、入座频率三层限流，防止频繁操作。
- **占座机制**：余额不足时不阻塞入座，而是先占座位再等待带入，体验良好。
- **掼蛋侧 `ensureOccupyUserSitDownWhenNoComplement`**：兜底处理重坐后无需补码但仍处于 OCCUPY 状态的边界场景，避免自动开局人数不足。
- **空间类型路由**：HALL/CLUB/PRIVATE/RACE 策略分离方向正确。
- **筹码校验返回余额信息**：余额不足时 respDto 携带 balance/needBringChipScore，客户端可展示充值引导。

### 业务规则隐含知识

- `SIT_DOWN_WAIT_ONE_HAND` 表示玩家入座后需等当前手牌结束才能参与下一手，由游戏开始时统一检查并转为 DEFAULT。
- `firstSitDown` 字段控制是否播放入座特效动画，只在首次入座时为 true（实际在 `userSitDown` 中被设为 false，说明后续通知用 `UserTableItem.isFirstSitDown()` 判断）。
- 占座倒计时区分两种：`occupySitdownCountdown`（首次入座占座）和 `seatCountdown`（结算后补码占座），前者通常更长。
- 私人审核房中，房主的 `autoBringLimit` 设为 `Integer.MAX_VALUE`，意味着房主不受额度限制；非房主受 `PRIVATE_SCORE` 钱包限制。
- HALL 和 RACE 的 `handle()` 方法为空，说明大厅和比赛场的入座由其他机制（如自动分配座位）触发，不走客户端手动入座协议。

### 待理解

- `UserSitDownReqDto.onlyCheck` 字段定义了"仅校验入不入座"，但在 Club/Private/Hall/Race 的 handler 中均未使用该字段做分支判断。可能是预留或已废弃，证据：`UserSitDownReqDto.java`。
- `BaseUserSitDownHandler`（抽象类）与 `BaseUserSitDownDefault`（组件类）存在大量重复代码（`checkChipScore/handler/after` 几乎一致），前者被 HallUserSitDownHandler 继承但未实际调用，后者被 Club/Private/Race 实际使用。两者关系待确认。

## 6. 架构审视

| 问题 | 结论 |
| --- | --- |
| 本功能的本质职责是什么 | 将旁观用户转为桌内玩家，含座位分配、资格校验、筹码准备、状态变更和通知广播 |
| 变化维度是什么 | 空间类型(HALL/CLUB/PRIVATE/RACE)、筹码模式(现金/信用/私人记分)、带入模式(自动/手动/审核)、游戏阶段(等待/进行中)、补码类型(首次入座/结算强制) |
| 核心状态流转是什么 | `WATCH_USER → PLAYER_USER(DEFAULT)` 或 `WATCH_USER → PLAYER_USER(OCCUPY) → PLAYER_USER(DEFAULT/SIT_DOWN_WAIT_ONE_HAND)`；超时则 `OCCUPY → userStand(TRANSFER)` |
| 外部交互模式是什么 | 同步请求响应为主；附带 Spring Event、钱包 RPC 查询、EventManager 定时任务(占座超时)、Redis 状态同步、客户端广播通知 |
| 是否适合状态机 | 适合。`WATCH → OCCUPY → DEFAULT / SIT_DOWN_WAIT_ONE_HAND` 以及 `OCCUPY → (超时) → STAND` 是明确的状态流转，应与 01-进入房间 决策的 UserItem.status FSM 统一 |
| 是否适合策略模式 | 适合。CLUB/PRIVATE 已有独立 Default 类，但 HALL/RACE 是空实现。应统一为策略接口，空间类型决定可入座性和校验规则 |
| 是否适合模板方法 | 适合。`init → checkSitDown → checkSentinel → checkChipScore → handler → after` 已是模板结构，但散布在 Default 类中而非统一模板 |
| 是否适合责任链 | 适合。踢出、座位、权限、风控、限流应拆成可组合校验链，与进房校验链复用共同部分 |
| 是否适合事件驱动 | 部分适合。轨迹推送、活动推送、Redis 同步适合事件；但占座补码通知中注册超时任务等关键副作用不应放在可失败事件中 |
| 是否适合命令模式 | 已适合。当前 Command 是协议入口，`UserSitDownContext` 已是命令上下文 |
| 是否适合 Builder | 部分适合。`SitdownNoticeDto` 组装可提取 Builder，但入座响应 DTO 字段较少不需要 |
| 是否适合 Facade | 适合。应提供 `SitDownFacadeService` 封装校验、状态变更、通知、占座任务的完整编排 |

与 frame 融合：保留 `CommonProto.USER_SIT_DOWN`、`HandlerSpaceService` 注册/路由和 TABLE 线程模型。在 frame 新增校验链扩展点时只新增接口不改现有方法签名。

与 Master/Action 模式：入座不涉及 Master/Action，但 `userSitDown()` 中读取 `TableHandStatus` 和 `TableUserHand` 来判断游戏阶段。重构时保持只读关系，不在入座中修改游戏过程数据。

与德州标杆架构：
- Command-Context-Handler 三层：当前已具备雏形（Command→Handler→Default），但 Default 类承担了编排+校验+处理三重职责，应拆分。
- 空间类型多态路由：已有但 HALL/RACE 空实现需明确处理。
- 事件驱动状态流转：`UserSitDownFinishEvent` 已有，应增强为可靠事件。
- TABLE 线程单线程绑定：已对齐。

| 维度 | 方案 |
| --- | --- |
| 新旧是否共存 | 保留旧 `UserSitDownCommand` 和 Handler；内部通过配置开关切换旧 `*UserSitDownDefault` 与新 `SitDownFlowService` |

## 13. 待确认项

| 待确认项 | 原因 |
| --- | --- |
| `onlyCheck` 字段是否被客户端使用 | 服务端未使用该字段，但可能客户端发送了该字段期望预检 |
| HALL/RACE 入座空 handler 是设计意图还是遗漏 | 大厅和比赛场可能由其他机制触发入座（如匹配后自动分配），需确认是否应显式拒绝手动入座 |
| `BaseUserSitDownHandler` 是否还有其他游戏项目使用 | frame 公共类，掼蛋侧实际走 `BaseUserSitDownDefault`，但麻将侧可能走 Handler 继承体系 |
| 限流 `tryAcquire` 是否设计为包含失败场景 | 当前限流在校验之后、筹码检查之前，失败后令牌已消耗。需确认是防刷设计还是应后移 |
| `OccupyToSitDownTask` 直接调用 `run()` 是否安全 | `UserSitDownBootstrap.java:94` 和 `UserSitDownFinishListener.java:110` 直接同步调用 `.run()` 而非通过 EventManager 调度，跳过了任务调度的线程保证 |
| 占座补码通知失败后是否有补偿机制 | 当前整个 `occupyAddChipNotice` 被 catch 吞掉，失败后无重试也无超时兜底 |

## 14. 本功能不处理的范围

| 范围 | 接口契约 |
| --- | --- |
| 游戏开始(Torsen) | 入座后 `UserItem.status=SIT_DOWN_WAIT_ONE_HAND` 由游戏开始链路检查并转为 DEFAULT。入座只负责设置状态，不触发开局 |
