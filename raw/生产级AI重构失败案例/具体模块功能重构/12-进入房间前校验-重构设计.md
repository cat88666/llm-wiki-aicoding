> 对应分析文档：`docs/Junk-游戏准备重构/12-进入房间前校验.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame（本功能为 frame-only，guandan 无改动）
> 分支：20260615/junk
> 复杂度：M
> 前置依赖：`01-进入房间-重构设计.md`（复用其 `GameHandleFactory` 空间分发模式）、`02-入座-重构设计.md`（同一工厂分发约定）

## 1. 背景与目标

1032 `CommonProto.USER_BEFORE_ENTER_ROOM` 是 1000 进入房间前的轻量预检（确认密码 / 是否已在房间）。入口 `EnterRoomBeforeCommand`（`@ThreadModel(PLAYER)`，dx-game-frame-cmd）委托 `EnterRoomBeforeCommandDefault`。当前存在：

- **P0-1：空间 handler 存在但默认链路从不分发**。`EnterRoomBeforeCommandDefault.check()` 直接调 `BaseUserCheckAuthService.checkUserPwd(...)`，从未通过 `GameHandleFactory` 分发 `ClubEnterRoomBeforeHandler` / `HallEnterRoomBeforeHandler` / `PrivatedEnterRoomBeforeHandler`。后果：`PrivatedEnterRoomBeforeHandler.handle()` 的私人房禁入校验（`privateMajongStatus == 1` 抛 `USER_JOIN_GAME_LOCK_STATUS`，`PrivatedEnterRoomBeforeHandler.java:25-33`）**完全不生效**；1032 与 1000 入房前置口径不一致。对应缺陷 D10（死代码风险）。
- **P1-1：`needPwd` 被计算但无人使用**。`initContext` 设 `context.setNeedPwd(BaseUserCheckAuthService.checkUserisNeedPwd(...))`，但 `check()` 不读它、响应 `EnterRoomBeforeRespDto(tableId)` 也不返回它 → 死计算（`EnterRoomBeforeContext.isNeedPwd`）。
- **P1-2：`roomNo` 请求字段从不参与校验**。`EnterRoomBeforeReqDto.roomNo` 已定义，但 `check()` 只读 `tableId/userId/tableEntryPassword`，`roomNo` 与 `tableId` 不一致时服务端静默以 `tableId` 为准。
- **P2：大对象 info 日志**。`handleCommand` 与 `initContext` 各打一条 `GsonUtils.toJson(context)` info（`EnterRoomBeforeCommandDefault` 为单行压缩文件，方法名定位）。对应缺陷 D12。
- **P2：命名误导**。`PrivatedEnterRoomBeforeHandler.handle()` 日志文案为「玩家账号被禁止入局私人房德州游戏」（`PrivatedEnterRoomBeforeHandler.java:30`），掼蛋工程中「德州」表述误导。

目标：在**完全不改协议、不改请求/响应 DTO 字段**的前提下，修复 P0-1（让空间 handler 真正生效），收敛日志，澄清 `needPwd`/`roomNo` 语义。本功能**仅改 dx-game-frame**，guandan 无需扩展（1032 是入房前预检，无掼蛋特有状态）。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导：在 `GameHandleFactory` 新增 `getEnterRoomBeforeHandler(tableId)`（**additive**，不改既有方法），`EnterRoomBeforeCommandDefault` 改为分发 handler。与 01/02 工厂分发决策一致 |
| **密码校验基线不可丢（核心兼容性决策）** | `PrivatedEnterRoomBeforeHandler.handle()` 当前只校验 `privateMajongStatus`、**不调密码校验**。改造前所有空间（含私人房）都走 `checkUserPwd`。为零回归，Private handler 补 `super.check(req)`（密码校验），再叠加 `privateMajongStatus` 禁入校验 → 行为 = 改造前 + 私人房禁入修复 |
| 无匹配 handler 兜底 | `getEnterRoomBeforeHandler` 找不到 handler 时**回退到只做密码校验**（等价改造前），不复用 `getTypedHandle` 的 `Objects.requireNonNull(handler)` 强制非空（否则未覆盖空间会 NPE）。对照 `getTypedHandle` 对 `ENTER_ROOM` 的 CLUB 兜底 |
| `needPwd` 是否返回客户端 | **不改协议**：响应不新增字段，文档明确「1032 成功仅代表密码校验通过」。是否需要前端密码提示 = 产品待确认；若需要，预留**可选** `isNeedPwd` 字段（老客户端忽略） |
| `roomNo` 处理 | **不新增强制校验**（避免 roomNo/tableId 不一致时误伤现网），保留字段，文档标注待确认是否客户端仍依赖 |
| 日志 | `handleCommand` / `initContext` 的 `GsonUtils.toJson` info 降为 debug；主链路保留一条精简 info（tableId/userId/spaceType） |
| 命名修正 | `PrivatedEnterRoomBeforeHandler` 日志「私人房德州游戏」改为通用文案，仅文案、不改逻辑 |
| 灰度开关 | 不做开关，直接替换，依赖测试与回归（与 01/02 一致） |

## 3. 整体架构

### 改造前

```
EnterRoomBeforeCommand (PLAYER线程, dx-game-frame-cmd)
  └─ EnterRoomBeforeCommandDefault.handleCommand(req)
       ├─ initContext()  — 计算 isInRoomUser / needPwd（needPwd 算了不用, P1-1）
       │     · log.info(GsonUtils.toJson(context))   ← 大对象日志 P2
       └─ check()        — 直接 BaseUserCheckAuthService.checkUserPwd(tableId,userId,pwd)
                            ↑ 从不走 GameHandleFactory（P0-1）

游离未被调用：
  ClubEnterRoomBeforeHandler   (CLUB)        → super.check()=checkUserPwd
  HallEnterRoomBeforeHandler   (HALL/RACE)   → super.check()=checkUserPwd
  PrivatedEnterRoomBeforeHandler (PRIVATE)   → 只校验 privateMajongStatus（不含密码）
```

### 改造后

```
EnterRoomBeforeCommand (PLAYER线程, 不变)
  └─ EnterRoomBeforeCommandDefault.handleCommand(req)
       ├─ initContext()  — 保留 isInRoomUser/needPwd（日志降 debug, R-04）
       └─ check()        — GameHandleFactory.getEnterRoomBeforeHandler(tableId).handle(req)
                            ├─ CLUB        → ClubEnterRoomBeforeHandler  → checkUserPwd
                            ├─ HALL/RACE   → HallEnterRoomBeforeHandler  → checkUserPwd
                            ├─ PRIVATE     → PrivatedEnterRoomBeforeHandler
                            │                  → super.check()=checkUserPwd（R-02 补）
                            │                  → privateMajongStatus 禁入校验（原本失效, 现生效）
                            └─ 无匹配      → 兜底 checkUserPwd（等价改造前）

GameHandleFactory 新增 getEnterRoomBeforeHandler(tableId)（R-01, additive）
```

**兼容性原则**：

- 协议号 1032、`EnterRoomBeforeReqDto`（`tableEntryPassword`/`roomNo`）、`EnterRoomBeforeRespDto`（`tableId`）字段**全部不变**
- `EnterRoomBeforeCommand`、`@ThreadModel(PLAYER)`、`HandlerSpaceService` 注册不变
- 密码错误仍走现有 `BaseUserCheckAuthService.checkUserPwd` 的业务异常，错误响应路径不变
- `GameHandleFactory` 既有方法（`getEnterRoomHandler`/`getUserSitDownHandler`/`getTypedHandle`）不改

## 4. 改造任务（R-01 ~ R-05）

### R-01（P0）：`GameHandleFactory` 新增 `getEnterRoomBeforeHandler` 分发

**文件**：`GameHandleFactory.java`（frame，`com.dx.game.frame.service.factory`）

新增方法（additive，不动既有方法）。因 1032 不要求所有空间都注册 handler，**不使用** `getTypedHandle` 的 `requireNonNull`，无匹配时返回 `null` 交由调用方兜底：

```java
public static IEnterRoomBeforeHandler getEnterRoomBeforeHandler(Long tableId) {
    Objects.requireNonNull(tableId, "tableId cannot be null");
    BaseTableInfo tableInfo = BaseTableInfoCache.getTableInfo(tableId);
    GameSpaceTypeEnums gameSpaceType = GameSpaceTypeEnums.getEnum(tableInfo.getSpaceType());
    if (gameSpaceType == null) {
        gameSpaceType = GameSpaceTypeEnums.CLUB;
    }
    return (IEnterRoomBeforeHandler) findHandler(gameSpaceType, CommonProto.USER_BEFORE_ENTER_ROOM)
            .orElse(null);
}
```

> 复用既有 private `findHandler(gameSpaceType, protoType)`，无需改动。

### R-02（P0）：`EnterRoomBeforeCommandDefault.check()` 走 handler 分发 + Private 补密码基线

**文件**：`EnterRoomBeforeCommandDefault.java`、`PrivatedEnterRoomBeforeHandler.java`（frame）

`check()` 改为：

```java
public void check(EnterRoomBeforeContext context) {
    EnterRoomBeforeReqDto req = context.getRequest();
    IEnterRoomBeforeHandler handler = GameHandleFactory.getEnterRoomBeforeHandler(req.getTableId());
    if (handler != null) {
        handler.handle(req);            // 含密码校验 + 空间差异化校验
    } else {
        // 兜底：未注册空间 handler 时等价改造前，仅密码校验
        BaseUserCheckAuthService.checkUserPwd(req.getTableId(), req.getUserId(), req.getTableEntryPassword());
    }
}
```

`PrivatedEnterRoomBeforeHandler.handle()` 补密码基线（保持与改造前一致，再叠加禁入校验）：

```java
@Override
public void handle(EnterRoomBeforeReqDto req) {
    super.check(req);   // R-02: 密码校验基线，等价改造前所有房都校验密码
    Long userId = req.getUserId();
    UserBaseInfo userBaseInfo = UserBaseInfoCache.getUserBaseInfo(userId);
    if (Objects.equals(1, userBaseInfo.getPrivateMajongStatus())) {
        log.info("玩家账号被禁止进入私人房，请联系代理 userId:{}", userId);   // R-05 文案
        throw BusinessException.createException(BusinessExceptionTypeEnums.USER_JOIN_GAME_LOCK_STATUS);
    }
}
```

> Club/Hall handler 的 `handle()=super.check()` 已等价改造前的密码校验，无需改动。

### R-03（P1）：`needPwd` / `roomNo` 语义澄清（不改协议）

- `EnterRoomBeforeContext` 的 `isInRoomUser` / `isNeedPwd` 为内部上下文，**保留**计算（其它链路或后续可选字段可能用到），不删字段。
- 响应 DTO **不新增字段**；在代码注释与本文档明确：1032 成功仅代表「密码校验通过且空间禁入校验通过」。
- `roomNo` 不新增强制校验，保留请求字段；是否仍被客户端使用列入待确认。

> R-03 主要是注释 + 文档澄清，代码改动极小（仅去除对 `needPwd` 的误导性使用预期）。

### R-04（P2）：日志降级

**文件**：`EnterRoomBeforeCommandDefault.java`

- `handleCommand` 的 `log.info("进入房间前-请求协议,context:{}", GsonUtils.toJson(context))` → `log.debug(...)`
- `initContext` 的 `log.info("进入房间前-初始化完成,context:{}", GsonUtils.toJson(context))` → `log.debug(...)`
- 主链路保留一条精简 info：`log.info("进入房间前 tableId:{} userId:{} inRoom:{}", tableId, userId, inRoomUser)`
- 业务异常 `log.warn` / 系统异常 `log.error` 保留。

### R-05（P2）：`PrivatedEnterRoomBeforeHandler` 日志文案修正

见 R-02 代码：「私人房德州游戏」→「私人房」，仅文案，不改逻辑、不改异常码。

## 7. 协议兼容确认

| 协议号 | 类型 | DTO | 字段是否变更 | 兼容性 |
|--------|------|-----|--------------|--------|
| 1032 | 请求 | `EnterRoomBeforeReqDto` | 不变（`tableEntryPassword`/`roomNo`） | 完全兼容 |
| 1032 | 响应 | `EnterRoomBeforeRespDto` | 不变（`tableId`） | 完全兼容 |
| 1032 | 错误 | — | 密码错误/禁入 仍走既有业务异常 | 完全兼容 |

兼容性待确认：

- **私人房是否本就要求密码**：R-02 让私人房显式走 `checkUserPwd`。因改造前默认链路对所有房都已调 `checkUserPwd`，此为零回归；但需确认私人房是否真有密码配置（`checkUserPwd` 在无密码时应为 no-op）。
- 私人房禁入校验从「不生效」变为「生效」是预期修复，但需确认现网是否有依赖该校验**不生效**的灰度玩家。

## 9. 验收关键场景

| 场景 | 类型 | 前置 | 操作 | 期望 |
|------|------|------|------|------|
| 俱乐部密码房正确密码 | 正常 | CLUB 桌设密码 | 发 1032 带正确密码 | 返回 tableId，行为同改造前 |
| 俱乐部密码房错误密码 | 异常 | CLUB 桌设密码 | 发 1032 错误密码 | 密码业务异常，同改造前 |
| 私人房禁入用户 | 兼容性/修复 | PRIVATE 桌，用户 `privateMajongStatus=1` | 发 1032 | 抛 `USER_JOIN_GAME_LOCK_STATUS`（改造前**漏拦**，现拦截） |
| 私人房正常用户 | 正常 | PRIVATE 桌，`privateMajongStatus≠1` | 发 1032 | 密码校验通过则返回 tableId |
| 大厅/比赛桌 | 正常 | HALL/RACE 桌 | 发 1032 | 走 Hall handler 密码校验，行为同改造前 |
| 1032 成功后 1000 入房 | 阶段交叉 | 任意桌 | 1032 成功 → 1000 | 1000 正常进入 |

| 维度 | 方案 |
|------|------|
| 新旧共存 | 不共存，直接替换；改造后行为 = 改造前 + 私人房禁入修复 |

## 11. 待确认项

| 待确认项 | 原因 |
|----------|------|
| 私人房是否要求密码 | R-02 让私人房显式走 `checkUserPwd`；需确认私人房密码配置语义，确保零回归 |
| 私人房禁入校验「由失效转生效」是否有灰度依赖 | 修复后 `privateMajongStatus=1` 玩家将被 1032 拦截，需产品/风控确认无依赖旧行为 |
| `needPwd` 是否需要回传客户端 | 决定是否新增可选 `isNeedPwd` 响应字段（产品） |

| `GameSpaceTypeEnums` 是否还有 CLUB/HALL/RACE/PRIVATE 之外的空间类型 | 决定兜底分支覆盖面 |
