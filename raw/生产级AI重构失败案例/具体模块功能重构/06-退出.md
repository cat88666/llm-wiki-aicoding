## 1. 功能边界

退出覆盖 1005 `CommonProto.USER_QUIT`、14008 `USER_QUIT_NOTICE`，以及强制踢出、断线/破产/现金不足退出、退出后自动带出和用户清理。

本功能负责:

- 用户主动退出房间。
- 断线退出 `handType=1/2` 的特殊链路。
- 房主/风控/系统踢出后的退出任务。
- 退出后进入 `QUIT_USER`，并发送大厅在局状态变更。
- 掼蛋退出后的自动带出、过级彩金扣除、轮服务费处理。

本功能不负责:

- 站起但保留房间旁观，归属 04。
- 可带出金额和带出钱包执行细节，归属 05。
- 入房重连，归属 14。

## 2. 入口与调用链

主动退出入口:

1. `UserQuitCommand` 使用 `@ThreadModel(PLAYER)`，`getType` 返回 `CommonProto.USER_QUIT`。
2. `UserQuitCommandDefault.handleCommand` 执行 `init -> check -> handle`。
3. `check` 对 `handType=1/2` 调用 `commonCheckUserBreakLink`，否则调用 `commonCheckUserLeave`。
4. `handle` 如果用户不在任何用户列表，直接发送退出通知；断线链路调用 `OtherQuitBootstrapService.breakLinkForTable`；普通退出发布 `UserQuitTask`。
5. `UserQuitTask.run` 调用 `UserLeaveBootstrapService.playerQuit(context)`。

掼蛋实际执行:

- `GdUserLeaveBootstrapService.playerQuit` 做重复退出保护、共用离开检查、占座限流、保存 `QUIT_USER`、过级彩金结算、轮服务费、自动带出、退出事件、退出通知、阶段 MQ。
- `UserQuitNotice.notice` 组装 1005 响应和 14008 广播，并移除网关状态。

## 3. 数据流与状态

请求:

- `UserQuitReqDto.handType` 区分普通退出和断线退出。
- `UserQuitContext.quitType`、`kickerReason` 表达主动、强制、断线等退出原因。

状态:

- 普通退出核心转移是 `PLAYER_USER/WATCH_USER/OCCUPY_USER -> QUIT_USER`。
- 退出用户不应继续参与重连、补码倒计时、开局检查。
- 掼蛋退出会触发 `bringOutByQuit` 或彩金自动带出。

通知:

- 1005 响应本人。
- 14008 广播桌内用户。
- 大厅当前牌桌 MQ、用户阶段 MQ 后续刷新大厅态。

## 4. 现有设计理解

退出和站起共用“是否允许离开”判断，但退出比站起更重:

- 退出会离开房间，进入 `QUIT_USER`。
- 退出后桌面筹码需要收尾，掼蛋中包括主钱包和彩金。
- 已退出用户在实时账单中可能仍作为历史玩家展示，但桌内活跃状态应置灰。

## 5. 问题扫描

### P0-1 退出入口在线程模型上先走 PLAYER，再发布 TABLE 任务

证据:

- `UserQuitCommand` 标注 `@ThreadModel(PLAYER)`。
- 普通退出在 `UserQuitCommandDefault.handle` 中发布 `UserQuitTask`，任务 `groupId` 为 `tableId`。
- 断线退出 `handType=1/2` 在 check 阶段走 `commonCheckUserBreakLink`，随后可能直接调用 `OtherQuitBootstrapService.breakLinkForTable`。

影响:

- 部分校验在玩家线程读取桌级缓存，和桌线程内的入座、带入、站起可能存在时序差。
- 断线退出和普通退出的线程边界不一致。

建议:

- 入口只做 DTO 解析，所有桌内状态读取和校验都收口到 table 任务。
- 断线退出也建 `QuitIntent.BREAK_LINK`，由同一个 quit flow 决策。

### P0-2 退出与站起共享 `commonCheckUserLeave`，但退出后置动作完全不同

证据:

- `UserQuitCommandDefault.check` 普通退出调用 `commonCheckUserLeave`。
- `GdUserLeaveBootstrapService.playerQuit` 会保存 `QUIT_USER`、自动带出、发送阶段 MQ。
- `GdUserLeaveBootstrapService.playerStand` 保存 `WATCH_USER` 且明确注释站起不自动带出。

影响:

- 校验结果无法表达“允许站起但不允许退出”或“允许退出但需要先结算带出”。
- 后置动作靠调用方分支保证，缺少统一意图模型。

建议:

- 和 04 共用 `LeaveIntent`，退出意图输出 `QuitDecision`，包含是否自动带出、是否立即移出房间、是否保留账单展示。

### P1-1 退出通知组装直接读取钱包缓存

证据:

- `UserQuitNotice.notice` 使用 `TableUserWalletCache.getChipScore` 构造通知。
- `TableUserWalletCache.getWallet` 对 `UserTableItemCache` 缺失敏感。

影响:

- 用户状态已经退出，但通知因钱包缓存异常失败，客户端房间状态不一致。
- 自动带出和退出通知同时读取钱包，容易出现金额口径前后不一致。

建议:

- 退出 flow 先生成 `QuitSnapshot`，带出前、带出后金额都在 snapshot 中记录。
- 通知只读 snapshot，不重新读取钱包缓存。

### P1-2 重复退出保护分散

证据:

- `UserLeaveDefault.playerQuit` 和 `GdUserLeaveBootstrapService.playerQuit` 都有 `BaseTableUserCache.hasQuitUser` 判断。
- 不在任何用户列表时，`UserQuitCommandDefault.handle` 直接发送退出通知。

影响:

- 幂等成功和真正异常的边界不清晰。
- 重复退出时是否需要再次自动带出依赖不同入口位置。

建议:

- `QuitStateService` 对重复退出返回 `QuitResult.alreadyQuit`，后续通知和带出策略显式选择。

## 6. 架构审视

D1 单节点缓存: 退出依赖 `BaseTableUserCache` 和钱包缓存，状态移出后要保证大厅/Redis/MQ 补偿。
D5 状态机: `PLAYER/WATCH/OCCUPY -> QUIT` 和 `PLAYER/OCCUPY -> WATCH` 未统一建模。
D6 事务边界: 自动带出、保存 `QUIT_USER`、MQ 通知分散执行，失败补偿不集中。
D8 线程边界: PLAYER 入口读取桌状态后再 TABLE 任务执行，存在时序差。
D12 日志: 退出和钱包日志包含用户与资金快照，需要脱敏规则。

## 7. 改造建议

优先级:

1. P0: 建 `QuitFlowService`，所有退出意图进入 table 串行执行。
2. P0: 将 `commonCheckUserLeave` 升级为 `LeaveRuleChain`，按 `LeaveIntent.QUIT` 输出决策。
3. P1: 退出状态、自动带出、通知使用同一 `QuitSnapshot`。
4. P1: 重复退出统一幂等结果。

## 8. 落地设计

建议类:

- `QuitCommandAdapter`: 解析 1005 和 `handType`。
- `QuitIntentResolver`: 主动、断线、强制、风控、现金不足。
- `QuitRuleChain`: 只做退出前置判断。
- `QuitStateService`: 执行用户区域迁移和任务清理。
- `QuitBringOutCoordinator`: 调用自动带出并记录结果。
- `QuitNoticeService`: 发送 1005/14008 和大厅 MQ。

关键字段:

- `QuitSnapshot.beforeArea/afterArea`.
- `QuitSnapshot.bringOutResult`.
- `QuitSnapshot.quitReason`.
- `QuitSnapshot.idempotent`.

## 9. 协议兼容确认

请求兼容:

- 1005 请求字段保持，`handType` 语义不变。

响应兼容:

- 1005 响应 DTO 保持。
- 重复退出仍可按成功响应处理，但服务端日志标记幂等。

通知兼容:

- 14008 `USER_QUIT_NOTICE` 不删除字段。
- 自动带出产生的 14009 保持现有 DTO。

## 10. 可测试性设计

单元测试:

- `QuitIntentResolverTest`: 普通退出、断线退出、强制踢出。
- `QuitStateServiceTest`: 玩家、旁观、占座、已退出四类状态。
- `QuitBringOutCoordinatorTest`: 主钱包、彩金、空钱包、带出失败。

集成测试:

- 主动退出后用户进入 `QUIT_USER`，不再重连为坐下用户。
- 占座用户退出会清占座任务并自动收尾。
- 断线退出不绕过桌线程状态机。
- 重复退出不会重复带出。

## 11. 迁移与灰度

迁移步骤:

1. 新增 quit flow，旧入口只调用新 flow，保留旧行为日志。
2. table 线程执行开关灰度，先覆盖普通退出，再覆盖断线退出。
3. 比对 `QUIT_USER`、钱包下分、14008、14009 和大厅 MQ。
4. 收口重复退出幂等逻辑。

回滚:

- 入口保留旧 `UserQuitCommandDefault` 分支。
- 新 snapshot 不进入协议，回滚不会影响客户端。

## 12. 验收场景

- 玩家主动退出，1005 成功、14008 广播、用户进入 `QUIT_USER`。
- 玩家已退出再次请求 1005，不重复带出。
- 退出时有主钱包筹码，自动下分并通知。
- 退出时有 `GD_BONUS`，彩金带出或结算链路正确衔接。
- 过级/结算限制命中时，返回明确业务异常。
- 断线退出和普通退出均在桌维度串行处理。

## 13. 待确认项

- `handType=1/2` 的准确客户端语义和是否仍在使用。
- 退出后实时账单是否需要保留用户历史数据，保留多久。
- 自动带出失败时，用户是否仍允许进入 `QUIT_USER`。
- 强制踢出是否必须先发站起通知再发退出通知。

## 14. 本功能不处理的范围

- 不改变带出金额规则。
- 不新增退出协议字段。
- 不处理在线状态心跳，归属 15。
