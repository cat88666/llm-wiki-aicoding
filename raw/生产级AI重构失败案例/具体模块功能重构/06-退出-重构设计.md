## 1. 背景与目标

当前退出协议 1005 由 `UserQuitCommand → UserQuitCommandDefault → UserQuitTask → UserLeaveBootstrapService.playerQuit → GdUserLeaveBootstrapService.playerQuit → UserQuitNotice` 串联，存在两类问题：

**实际稳定性问题**：

- **P0**：`UserQuitNotice.notice` 第 35 行 `BaseUserItem userItem = BaseTableUserCache.getUser(tableId, userId)` 对**不在 cache 中**的用户返回 `null`，但后续 41/42 行直接 `userItem.getChairId()` / `userItem.getStatus()` —— **必然 NPE**。触发场景：`UserQuitCommandDefault.handle` 在 L86-88 对"不在 uidList"的用户（如已退出再 1005、断线超时清理后再 1005、跨表错误请求等）调用 `notice(context, true, new UserBringOutResult())`，立即栈崩。
- **P1**：`UserQuitNotice.notice` L43 `TableUserWalletCache.getChipScore(tableId, userId)` 在 `UserTableItem / TableUserWallet` 缓存缺失时抛异常 —— 退出主流程已经完成，但 14008 通知失败，桌内其他人看不到状态更新。与 04 R-02 完全同型问题。
- **P2**：1005 链路 `UserQuitCommandDefault.handleCommand` / `UserQuitTask.run` / `UserQuitNotice.notice` 大量 `info` 级 `GsonUtils.toJson(req/context/userItem/userQuitNoticeDto)`，含设备 / 用户上下文，高峰 CPU 与日志量放大。

**结构性问题（不在本期）**：

- `LeaveIntent` + `LeaveRuleChain` + `QuitFlowService` —— 跨 04/05/06/07 的整体重构，单独立项做，不在本期承诺。

目标：消除 `notice` 的 P0 NPE，加固钱包读取兜底，收敛 1005 链路含敏感 payload 的 info 日志。保持 1005/14008 协议字段 100% 兼容，`UserQuitNoticeDto` 字段不变。

## 4. 安全改造

### R-01：`UserQuitNotice` userItem 空值保护（P0 NPE 修复）

**文件**：`UserQuitNotice.java`

当 `BaseTableUserCache.getUser(tableId, userId)` 返回 null（用户已不在任何 cache 列表中）时，仍能正常发送 1005/14008，保持"未在房间的用户 1005 仍返回成功"的现有幂等语义。

```java
public void notice(UserQuitContext context, boolean isWatch, UserBringOutResult userBringOutResult) {
    Long tableId = context.getRequest().getTableId();
    Long userId = context.getRequest().getUserId();
    BaseUserItem userItem = BaseTableUserCache.getUser(tableId, userId);
    log.info("用户退出通知开始 tableId:{} userId:{} quitType:{} kickerReason:{} userItemPresent:{}",
            tableId, userId, context.getQuitType(), context.getKickerReason(), userItem != null);
    log.debug("用户退出通知开始-context tableId:{} userId:{} userItem:{} context:{} bringOut:{}",
            tableId, userId, GsonUtils.toJson(userItem), GsonUtils.toJson(context), GsonUtils.toJson(userBringOutResult));

    UserQuitNoticeDto userQuitNoticeDto = new UserQuitNoticeDto();
    userQuitNoticeDto.setUserId(userId);
    userQuitNoticeDto.setTableId(tableId);
    // R-01: userItem 缺失时降级为 chairId=null / status=null，避免直接 NPE
    Integer chairId = isWatch || userItem == null ? null : userItem.getChairId();
    userQuitNoticeDto.setChairId(chairId);
    userQuitNoticeDto.setStatus(userItem == null ? null : userItem.getStatus());
    userQuitNoticeDto.setChipScore(safeFetchChipScore(tableId, userId));   // R-02
    // ... 其余字段不变
}
```

### R-02：`UserQuitNotice` 钱包读取兜底（P1）

新增 private helper（与 04 `UserStandNotice.safeFetchChipScore` 同模式）：

```java
private BigDecimal safeFetchChipScore(Long tableId, Long userId) {
    try {
        BigDecimal chipScore = TableUserWalletCache.getChipScore(tableId, userId);
        return chipScore == null ? BigDecimal.ZERO : chipScore;
    } catch (Exception e) {
        log.warn("用户退出通知-读取桌面筹码失败,使用0兜底 tableId:{} userId:{}", tableId, userId, e);
        return BigDecimal.ZERO;
    }
}
```

## 10. 验收关键场景

1. **userItem 缺失场景**：mock `BaseTableUserCache.getUser(...)` 返回 null，1005 链路仍发出 1005 响应 + 14008 通知，chairId=null、status=null、chipScore=0；warn 日志能定位 tableId+userId。
2. **钱包缺失场景**：mock `TableUserWalletCache.getChipScore` 抛 NPE，1005 仍发出通知，chipScore=0，warn 日志能定位 userId。
3. **正常退出**：玩家在 PLAYER_USER → QUIT_USER，userItem 正常，chairId 正常，chipScore 正常；行为完全不变。
4. **重复退出**：同一用户连续 2 次 1005，第二次走"不在 uidList → notice"分支，因 R-01 兜底不再 NPE，返回 1005 成功。
5. **断线退出 handType=1/2**：行为完全不变（走 `breakLinkForTable`），不进入 notice。
6. **日志体积**：1 次正常 1005 产生 ≤ 4 条 info（请求 / 任务开始 / 通知开始 / 通知完成），全部 tableId+userId+quitType 元数据级别，无 toJson 大对象。
