# 离开状态检查重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/07-离开状态检查.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame
> 分支：20260615/junk
> 前置依赖：04 站起、06 退出（已完成；本任务提供结构化 API 为它们的后续大改打基础）
> 后续联动：05 带出、17 放弃补码、未来的 LeaveRuleChain 专项

## 1. 背景与目标

离开状态检查是入座 / 站起 / 退出 / 预站起 / 风控站起 / 房主强制踢出 等多条链路的公共判定能力，当前由 `UserLeaveDefault.commonCheckUserLeave` 和 `commonCheckUserBreakLink` 承担。

**实际问题**：

- **P0**：`UserLeaveDefault.userStand`（系统站起入口，被 08 风控 / 17 补码失败 / 房主强制等调用）第 179 行存在 **与 04 R-01 同型 bug**：`EventUtils.getKey(OnceEventTypeEnum.USER_QUIT_TASK, ...)` 用于发布 `UserStandTask`。04 R-01 只修了客户端 1007 入口 `UserStandCommandDefault.handle`；本系统入口未修 —— 08 风控触发的实时站起仍然走错 event key，并发 1006/系统站起场景 P0 bug 复发。
- **P0（结构）**：当前所有检查 API 返回的都是裸 `BusinessExceptionTypeEnums`，调用方无法区分"无错误（允许）/ 业务错误（拒绝）/ 业务延迟（允许但延后）"等不同语义；新增 04/05/06/17 需要的"允许站起但不允许退出"或"允许退出但需带出"等表达力时只能塞错误码，进一步把语义压扁。
- **P1**：调用方都要自己读 `BaseTableUserCache.hasWatchUser/hasQuitUser/hasOccupyUser` 来判断当前区域，与 `commonCheckUserLeave` 内的逻辑重复 —— 一份"用户当前区域"的事实读取分散在 5 个调用方。
- **P2**：`commonCheckUserLeave` / `userStand` / `pushQuitTask` 多处 `info` 级 `GsonUtils.toJson(context)`，含用户/桌子状态快照。

**结构性问题（不在本期）**：

- 完整的 `LeaveRule` 链 + 每条规则独立测试 + `GdUserLeaveBootstrapService.checkSitDownUserRealCanLeave` 内部规则拆分（过级 / 彩金 / 补码 / 风控 / 现金不足 5 类规则）—— 涉及 guandan 端复杂业务，等专项立项。

目标：消除 04 R-01 的姊妹 P0（系统站起 event key）；为离开链路引入结构化 API 雏形（`LeaveDecision` + `LeaveIntent` + `LeaveDecisionService`），暴露"当前用户区域"等公共事实；保留原 `commonCheckUserLeave` / `commonCheckUserBreakLink` 行为不变，旧调用方零感知。

## 3. 整体架构

### 改造前

```
UserStandCommandDefault.check (协议入口)  → UserLeaveBootstrapService.commonCheckUserLeave → BusinessExceptionTypeEnums
UserQuitCommandDefault.check  (协议入口)  → UserLeaveBootstrapService.commonCheckUserLeave / commonCheckUserBreakLink
UserPrepStandCommandDefault.check         → UserLeaveBootstrapService.commonCheckUserBreakLink
UserLeaveDefault.userStand (系统入口)
  └─ 内部直接调度，不走检查
  └─ EventUtils.getKey(USER_QUIT_TASK, ...) ← P0 sibling bug (04 R-01 残留)
  └─ eventManager.publish(new UserStandTask(...))
UserLeaveDefault.pushQuitTask (系统入口) → eventManager.publish(new UserQuitTask(USER_QUIT_TASK key))   ✓ key 正确

调用方各自再读 hasWatchUser/hasQuitUser/hasOccupyUser 判断区域，逻辑重复
```

### 改造后

```
新平行 API（结构化）
  LeaveDecisionService.evaluate(LeaveIntent, tableId, userId) → LeaveDecision
    ├─ 计算 currentArea（一次性读取 BaseTableUserCache）
    ├─ 委托 commonCheckUserLeave 或 commonCheckUserBreakLink 取 errorCode
    └─ 装配 LeaveDecision

旧 API（行为不变）
  UserLeaveBootstrapService.commonCheckUserLeave(...)   保留
  UserLeaveBootstrapService.commonCheckUserBreakLink(...) 保留

UserLeaveDefault.userStand
  ├─ R-01: USER_QUIT_TASK → USER_STAND_TASK 修复 P0 sibling
  └─ 其余不变

UserLeaveDefault.commonCheckUserLeave / userStand / pushQuitTask
  └─ R-04: 含 context toJson 的 info 日志降级 debug；info 保留 tableId/userId/状态元数据
```

**兼容性原则**：
- `UserLeaveBootstrapService.commonCheckUserLeave / commonCheckUserBreakLink` 签名、返回类型、行为 100% 兼容
- `UserLeaveDefault.userStand / pushQuitTask / playerQuit / playerStand` public 方法签名不变
- 1001/1005/1007/1045 协议字段不变
- `BaseUserLeaveService.OK_LEAVE_LIST` 不变（仍只含 SUCCESS）
- 新增 API 是 **opt-in**：现有 5 处调用方完全不变；新代码可选择走 `LeaveDecisionService` 拿到 `currentArea` 等公共事实

### R-03：新增 `LeaveDecisionService`（P1 结构）

**文件**：`LeaveDecisionService.java`（frame-service，`com.dx.game.frame.service.biz.leave`）

```java
@Component
public class LeaveDecisionService {

    public LeaveDecision evaluate(LeaveIntent intent, Long tableId, Long userId) {
        UserArea area = detectArea(tableId, userId);
        BusinessExceptionTypeEnums errorCode = invokeUnderlyingCheck(intent, tableId, userId);
        boolean allowed = BusinessExceptionTypeEnums.SUCCESS == errorCode;
        return LeaveDecision.builder()
                .intent(intent)
                .allowed(allowed)
                .errorCode(errorCode)
                .currentArea(area)
                .evidence(allowed ? null : "underlyingCheck=" + errorCode.name())
                .build();
    }

    /** 用户当前所在区域（一次性快照读取，避免调用方重复读 cache） */
    private UserArea detectArea(Long tableId, Long userId) {
        if (BaseTableCache.checkDisbandTable(tableId)) return UserArea.DISBAND;
        if (BaseTableUserCache.hasWatchUser(tableId, userId)) return UserArea.WATCH;
        if (BaseTableUserCache.hasQuitUser(tableId, userId)) return UserArea.QUIT;
        if (BaseTableUserCache.hasOccupyUser(tableId, userId)) return UserArea.OCCUPY;
        if (BaseTableUserCache.fetchTablePlayerUserIds(tableId,
                UserStatusEnum.SIT_DOWN_WAIT_ONE_HAND,
                UserStatusEnum.SIT_DOWN_WAIT_TWO_HAND,
                UserStatusEnum.SIT_DOWN_WAIT_ONE_ROUND).contains(userId)) {
            return UserArea.SIT_DOWN_WAITING;
        }
        if (BaseTableUserCache.hasPlayerUser(tableId, userId)) return UserArea.PLAYER;
        return UserArea.UNKNOWN;
    }

    private BusinessExceptionTypeEnums invokeUnderlyingCheck(LeaveIntent intent, Long tableId, Long userId) {
        switch (intent) {
            case STAND:
            case QUIT:
                return UserLeaveBootstrapService.commonCheckUserLeave(tableId, userId);
            case BREAK_LINK:
            case PREP_STAND:
                return UserLeaveBootstrapService.commonCheckUserBreakLink(tableId, userId);
            default:
                return BusinessExceptionTypeEnums.SYSTEM_EXCEPTION;
        }
    }
}
```

**消费方式**（demonstrative，本期不强制迁移）：
```java
// 老写法（继续工作）
BusinessExceptionTypeEnums type = UserLeaveBootstrapService.commonCheckUserLeave(tableId, userId);
if (!BaseUserLeaveService.OK_LEAVE_LIST.contains(type)) {

```

1. **R-01 event key 修复**：08 风控触发 `realtimeCheckRisk` → `UserLeaveBootstrapService.userStand` → `UserLeaveDefault.userStand`，event key 是 `TABLE_USER_STAND_<tableId>_UID_<userId>`（不再是 quit）；并发系统站起 + 客户端 1006 不再互相覆盖。
