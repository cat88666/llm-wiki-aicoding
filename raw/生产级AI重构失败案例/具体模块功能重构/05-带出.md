## 1. 功能边界

带出覆盖 1023 `CommonProto.USER_BRING_OUT_VIEW`、1024 `CommonProto.USER_BRING_OUT`、14009 `USER_SCORE_CHANGE_NOTICE`，以及站起/退出链路中的自动带出。

本功能负责:

- 可带出金额查询: 保留筹码、最大可带出、当前手/下一手带出类型。
- 手动带出: 当前手立即下分、游戏中预约下一手带出。
- 自动带出: 退出、特殊站起、结算补码、房主强制、风控等场景。
- 彩金带出: `GD_BONUS` 带出和过级彩金结算协同。
- 带出结果响应与分数变化通知。

本功能不负责:

- 是否允许站起/退出的离开校验，归属 04/06/07。
- 带入和补码，归属 03。
- 单手/局结算派彩，归属结算模块。

## 2. 入口与调用链

可带出查询:

1. `UserBringOutViewCommand` 使用 `@ThreadModel(PLAYER)`，返回 `CommonProto.USER_BRING_OUT_VIEW`。
2. `UserBringOutViewCommandDefault.handle` 执行 `check -> handle -> after`。
3. `handle` 调用 `UserBringOutBootstrapService.getCanBringOutChipScore` 和 `checkCanLeaveResult`。

手动带出:

1. `UserBringOutCommand` 使用 `@ThreadModel(TABLE)`，返回 `CommonProto.USER_BRING_OUT`。
2. `UserBringOutCommandDefault.check` 校验玩家、钱包、金额、最小保留额。
3. `UserBringOutBootstrap.handleManual` 执行当前手下分或下一手预约。

自动带出:

- `UserLeaveDefault.playerQuit` 默认退出后调用 `bringOUtByQuit`。
- `GdUserLeaveBootstrapService.playerQuit` 在用户退出时，如果桌面有普通筹码或彩金，调用 `UserBringOutBootstrap.autoBringOut`。
- 掼蛋 `UserBringOutBootstrap.autoBringOut` 根据结算、彩金、私人审核、房主强制、轮服务费等分支决定立即带出或下一手带出。

## 3. 数据流与状态

请求:

- `UserBringOutReqDto.bringOutScore`: 旧单金额。
- `UserBringOutReqDto.walletList`: 钱包列表。

查询响应:

- `UserBringOutViewRespDto.minBringOutScore/maxBringOutScore/keepChipScore/chipScore/bringOutType/bureauServiceFee/remainingHands/remainingSeconds`。

执行响应:

- `UserBringOutRespDto.bringOutScore/chipScore/nextBringOutChipScore/bringOutStatus`。

缓存与账变:

- `TableUserWalletCache.getChipScore` 读取桌面筹码。
- `TableUserWalletCache.getTableDepositAmount` 参与 `DownScoreReq` 组装。
- `BaseUserWalletService.downScore` 完成钱包侧下分。
- `nextBringOutAmount` 表示游戏中参与本手后的预约带出金额。

## 4. 现有设计理解

框架默认带出以“桌面筹码必须保留最低带入”为核心规则。掼蛋扩展增加了过级彩金和结算特殊离场，因此并非所有站起都会立即带出:

- 主动站起通常保留桌面筹码。
- 退出通常自动带出。
- 游戏中参与本手的手动带出预约到下一手。
- 结算特殊离场、房主强制、私人审核结算离场可能允许立即带出。

## 5. 问题扫描

### P0-1 自动带出分支过多，规则隐含在实现顺序中

证据:

- `UserBringOutBootstrap.autoBringOut` 同时判断 `bonusFlag`、结算站起、私人审核结算离场、房主强制、是否需要彩金结算、私人房等。
- `shouldCurrentBringOutForSettleStand`、`shouldCurrentBringOutForOwnerForcedLeave`、`shouldCurrentBringOutForPrivateAuditSettleLeave`、`shouldCurrentBringOutForRoundFeeActiveLeave` 分散在同一类。

影响:

- 分支顺序即业务优先级，但没有显式规则表。
- 新增离场原因时，容易把下一手预约和立即下分条件放错位置。

建议:

- 建 `BringOutPolicyResolver`，输入 `LeaveIntent`、`TableHandStatus`、`walletSnapshot`、`bonusState`，输出 `CURRENT / NEXT_HAND / RETAIN / SKIP`。
- 用规则优先级配置代替长 if 链。

### P0-2 带出命令中存在空指针风险

证据:

- `UserBringOutCommandDefault.handleCommand` 在 try 前读取 `BaseTableHandCache.getTableHand(req.getTableId())`，catch 日志又读取 `tableHand.getHandCode()`。
- `UserBringOutViewCommandDefault.handleCommand` 同样在 try 前读取 `BaseTableHandCache.getTableHand`。

影响:

- 已解散、刚初始化或缓存未命中时，本应返回业务异常，却可能直接 NPE。
- NPE 发生在 try 前时，协议错误响应可能无法发送。

建议:

- 将 `tableHand` 读取移入 `initContext`，统一空值兜底为 `"-"`。
- 先检查 `BaseTableCache.checkDisbandTable` 和 `BaseTableInfoCache`，再组装日志。

### P1-1 掼蛋带出覆盖没有复用框架 `checkBringOutTableLimit`

证据:

- `UserBringOutCommandDefault.check` 在默认链路末尾调用 `checkBringOutTableLimit`。
- `UserBringOutCommandExtend` 重写校验后只处理私人固定房保留筹码与金额，没有继续调用表级限制校验。

影响:

- 如果框架后续扩展表级带出限制，掼蛋覆盖类可能绕过。
- 私人固定房和普通房的限制来源不一致。

建议:

- 校验拆为 `CommonBringOutAmountRule`、`PrivateFixedKeepRule`、`TableLimitRule`，覆盖类只追加规则，不替换整段校验。

### P1-2 成功响应使用 error response builder

证据:

- `UserBringOutDefault.sendBringOutResp` 使用 `NetResponseBuilder.buildErrorResponse` 构造成功响应。
- 掼蛋 `UserBringOutBootstrap.sendBringOutResp` 保持同一模式。

影响:

- 如果 builder 内部语义或客户端埋点按 error response 分类，成功带出会被错误统计。
- 未来统一协议网关时可能被错误拦截。

建议:

- 短期确认 builder 实际输出结构，补测试锁定兼容。
- 中期改为正常 response builder，同时通过灰度比较客户端解析。

## 6. 架构审视

D1 单节点缓存: 带出依赖桌面钱包本地缓存和钱包 RPC，下分成功后的本地同步必须可重试。
D5 状态机: 带出策略依赖站起/退出意图，但当前通过方法名和分支推断。
D6 事务边界: `downScore` 成功、`syncWalletCacheAfterBringOutSuccess`、通知发送不是一个显式事务。
D11 复杂状态: 结算、彩金、强制离场、轮费状态都集中在 `UserBringOutBootstrap`。
D12 日志: 带出金额、桌面筹码、钱包列表均属于资金敏感日志。

## 7. 改造建议

优先级:

1. P0: 抽 `BringOutPolicyResolver`，把立即/下一手/保留/跳过做成可测试决策。
2. P0: 修复命令入口空指针风险，所有日志 handCode 空值兜底。
3. P1: 校验规则链化，掼蛋只注册新增规则。
4. P1: 统一成功响应 builder，先补兼容测试。

## 8. 落地设计

建议类:

- `BringOutViewService`: 查询可带出金额，不修改状态。
- `BringOutCommandService`: 执行手动带出。
- `AutoBringOutService`: 执行离场触发的自动带出。
- `BringOutPolicyResolver`: 输出 `BringOutPolicy`。
- `BringOutWalletService`: 封装 `DownScoreReq`、主钱包和彩金金额列表、本地缓存同步。
- `BringOutNoticeService`: 统一 1024 响应和 14009 通知。

`BringOutPolicy` 字段:

- `applyTiming`: `CURRENT` 或 `NEXT_HAND`。
- `walletAmounts`: 主钱包和彩金金额。
- `retainAmount`: 保留筹码。
- `reason`: `MANUAL / QUIT / STAND / SETTLEMENT / BONUS / FORCED / ROUND_FEE`。

## 9. 协议兼容确认

请求兼容:

- 1023 请求不变。
- 1024 `bringOutScore` 和 `walletList` 保留，旧客户端仍可只传单金额。

响应兼容:

- 1023 `UserBringOutViewRespDto` 字段不删除。
- 1024 `UserBringOutRespDto` 字段不删除，`bringOutStatus` 枚举语义保持。

通知兼容:

- 14009 仍由 `AmountChangeNotice` 下发。
- 自动带出失败通知如果已有协议，保持现有协议号，不在本功能新增必填字段。

## 10. 可测试性设计

单元测试:

- `BringOutPolicyResolverTest`: 覆盖当前手、下一手、退出自动带出、结算站起、私人审核、房主强制、彩金。
- `BringOutAmountRuleTest`: 覆盖最小保留、最大带出、金额精度、钱包类型。
- `BringOutCommandNullTableHandTest`: 覆盖无 `tableHand` 时仍返回协议错误。

集成测试:

- 手动带出当前手成功后桌面筹码减少，钱包 RPC 请求金额正确。
- 游戏中参与本手带出只写 `nextBringOutChipScore`。
- 退出触发自动带出，主钱包和彩金均下分。
- 结算特殊离场不误判为下一手预约。

## 11. 迁移与灰度

迁移步骤:

1. 先引入 policy resolver，只做影子决策日志。
2. 对比旧分支与新 resolver 输出，不一致时打印完整原因。
3. 灰度启用手动带出，再启用自动带出。
4. 最后收口彩金与私人审核特殊分支。

回滚:

- 通过配置恢复旧 `UserBringOutBootstrap` 分支。
- 新 policy 不进入外部协议，回滚不影响客户端。

## 12. 验收场景

- 查询可带出金额时返回最小保留和最大可带出。
- 金额小于等于 0 或超过最大可带出时返回业务异常。
- 参与本手的玩家手动带出，1024 返回下一手成功状态。
- 未参与本手或结算特殊离场，带出立即下分。
- 退出时主钱包和彩金都被自动带出。
- 下分成功后发送 14009，客户端桌面筹码刷新。

## 13. 待确认项

- `buildErrorResponse` 成功响应是否是历史协议兼容要求。
- 私人固定带入保留筹码是否必须覆盖所有自动带出场景。
- 彩金结算和自动带出的先后顺序是否有账变审计要求。
- 轮服务费场景下 `NEXT_HAND_SUCCESS_ROUND_FEE` 是否仍被客户端使用。

## 14. 本功能不处理的范围

- 不改站起/退出是否允许的规则。
- 不改钱包 RPC。
- 不改结算派彩和注单。
- 不处理实时账单统计口径，归属 09。
