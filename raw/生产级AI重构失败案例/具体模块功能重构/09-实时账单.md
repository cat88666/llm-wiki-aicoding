## 1. 功能边界

实时账单覆盖掼蛋 3005 `GuandanProto.CONTINUOUS_BILLING`，用于桌内实时展示玩家账单、旁观列表、踢人列表和玩法属性。

本功能负责:

- 展示桌内玩家总带入、总带出、打牌盈亏、彩金盈亏、服务费、手数、局数。
- 展示旁观用户列表。
- 展示房主/总代可见的踢人列表。
- 展示好友备注名。
- 展示玩法属性 `playAttributes`。

本功能不负责:

- 账变产生，归属带入、带出、结算。
- 好友备注修改，归属 20。
- 大厅场实时盈亏 1056，不在本掼蛋 3005 范围。

## 2. 入口与调用链

入口:

1. `ContinuousBillingCommand` 使用 `@ThreadModel(PLAYER)`，`getType` 返回 `GuandanProto.CONTINUOUS_BILLING`。
2. `init` 从 `TableCache.getTable` 读取桌子，缺失返回 `TABLE_STATUS_IS_END`。
3. `ContinuousBillingHandler.handle` 组装响应。
4. `after` 发送 3005 响应。

处理链路:

- `setWatchUsers` 从 `BaseTableUserCache.WATCH_USER` 读取旁观用户，并用 `UserFriendCache` 补备注名。
- `setKickerList` 读取 `Table.forcedMap` 中已执行踢人列表。
- `setRealTimeData` 读取 `UserTableItemCache.getAllUserTableItem`、手记录 RPC、局数 RPC、钱包汇总 RPC、彩金盈亏 RPC、输赢分 RPC。

## 3. 数据流与状态

请求:

- `ContinuousBillingReq` 无额外字段，依赖基础请求的 `tableId/userId`。

响应:

- `ContinuousBillingResp.realTimeData`: `RealTimeDataDto` 列表。
- `ContinuousBillingResp.watchUsers`: 旁观用户。
- `ContinuousBillingResp.kickerList`: 踢人列表。
- `ContinuousBillingResp.playAttributes`: 玩法参数。

关键口径:

- 总带入来自钱包账变汇总 RPC，当前注释说明主钱包 `walletType=7`、带入 `eventType=0`。
- 总带出来自钱包账变汇总 RPC，带出 `eventType=1`。
- 彩金盈亏优先用 `getPlayerBonusProfitLossMap`，失败退回钱包汇总中的彩金字段。
- 局数优先用 `getPlayerRoundCountMap`，失败回退到手数。

## 4. 现有设计理解

实时账单不是“实时计算账变”，而是从桌内缓存和多个外部统计服务拼一个展示视图。当前实现已经针对历史问题增加了:

- 离桌置灰: 只把 `DEFAULT` 和等待下一手/下一轮状态作为活跃展示。
- 总带入为 0 的玩家跳过，避免从未入座用户展示。
- 局数 RPC 失败时回退手数。
- 好友备注按当前请求用户维度查询。

## 5. 问题扫描

### P0-1 `handle` 吞异常后仍可能返回半空响应

证据:

- `ContinuousBillingHandler.handle` 内部 catch `Exception` 后只打印日志，没有向命令层抛出。
- `ContinuousBillingCommand.handleCommand` 在 `continuousBillingHandler.handle(context)` 后无条件 `after(context)`。
- `handle` 中 `continuousBillingResp.setPlayAttributes(tableHand.getPlayAttributes())` 没有对 `tableHand` 为空做兜底。

影响:

- 组装失败时客户端收到成功响应但数据为空或部分为空。
- 监控上看不到协议失败，只能靠服务日志。

建议:

- `ContinuousBillingHandler` 返回 `BillingBuildResult`，失败由命令层发送业务错误或带 `partial=true` 的兼容响应。
- `playAttributes` 空值兜底为默认对象，并记录指标。

### P0-2 响应依赖多源数据但没有快照版本

证据:

- `setRealTimeData` 同时读取本地 `UserTableItemCache`、`BaseTableUserCache`、`TableInfoCache`、`TableHandCache` 和多个 RPC。
- `activeDisplayUserIds`、`quitUserIds`、`allUserMap` 分别独立读取。

影响:

- 用户刚站起/退出/带出时，不同数据源可能处于不同时间点。
- 总带出已刷新但用户状态还未置灰，或相反。

建议:

- 引入 `BillingSnapshot`，一次性记录本地用户、桌状态、手牌状态、钱包汇总请求时间。
- 响应增加内部日志字段，不进入协议，用于追踪同一快照。

### P1-1 RPC 失败回退口径不统一

证据:

- `getUserBonusProfitLossMap` RPC 失败返回 empty map。
- `getUserWinLoseScoreMap` RPC 失败返回 empty map。
- `getUserRoundCountMap` RPC 失败返回 null，用于区分是否回退手数。
- 钱包汇总 RPC 失败时多个金额 map 保持空。

影响:

- 有的失败显示 0，有的失败回退，有的失败跳过，客户端无法区分真实 0 和统计不可用。

建议:

- 为每个外部统计结果建立 `DataSourceStatus`，日志和指标中明确 `OK / EMPTY / FAILED / FALLBACK`。
- 协议不新增必填字段，但内部监控必须区分。

### P1-2 房主/总代判断缺少空值保护

证据:

- `setKickerList` 直接读取 `UserBaseInfoCache.getUserBaseInfo(userId)`、`TableInfoCache.getTableInfo(tableId)`、`TableCache.getTable(tableId)`。
- 直接判断 `!userBaseInfo.isTopProxy()` 和 `!tableInfo.getHolderId().equals(userId)`。

影响:

- 基础信息缺失时账单整体异常。
- 踢人列表本应是附加展示，不应影响实时账单主列表。

建议:

- 踢人列表单独 try/catch，失败只返回空踢人列表并打指标。

## 6. 架构审视

D1 单节点缓存: 实时账单直接读本地桌缓存，跨节点进入时需要保证路由到桌节点。
D6 外部依赖: 多个 RPC 没有统一超时、失败状态和降级口径。
D11 大 DTO 聚合: 一个 handler 同时拼旁观、踢人、账单、玩法，职责过宽。
D12 日志: 输出账变汇总和玩家信息，需脱敏和采样。

## 7. 改造建议

优先级:

1. P0: 拆 `BillingSnapshotBuilder` 和 `BillingResponseAssembler`，异常不能被静默吞掉。
2. P0: `playAttributes`、`tableHand`、`tableInfo` 空值统一兜底。
3. P1: 外部统计封装 `BillingDataSourceClient`，统一失败状态。
4. P1: 旁观、踢人、账单三块独立构建，任一失败不拖垮其他模块。

## 8. 落地设计

## 9. 协议兼容确认

请求兼容:

- 3005 请求不新增字段。

响应兼容:

- `ContinuousBillingResp.realTimeData/watchUsers/kickerList/playAttributes` 保持。
- 不新增必填字段；内部失败状态只进日志和指标。

通知兼容:

- 本功能不发送通知。
- `WATCH_USER_CONTINUOUS_BILLING_NOTICE` 是入房旁观变动通知，不由 3005 改造变更。

## 10. 可测试性设计

单元测试:

- RPC 成功、空数据、失败、超时四种统计源状态。
- 总带入为 0 玩家跳过。
- 退出用户置灰。
- 当前请求用户排序到第一。
- 备注名按请求用户维度展示。

集成测试:

- 带入后 3005 总带入刷新。
- 带出后 3005 总带出刷新。
- 彩金桌结算后彩金盈亏口径正确。
- 手数 RPC 失败时局数回退逻辑可控。

## 11. 迁移与灰度

迁移步骤:

1. 新 assembler 只读生成影子响应摘要。
2. 对比旧响应用户数、总带入、总带出、盈亏、排序。
3. 先切旁观和踢人列表，再切实时账单主列表。
4. 增加 RPC 失败指标后全量。

回滚:

- 保留旧 `ContinuousBillingHandler`。
- 新 snapshot 不进入协议，可直接切回旧 handler。

## 12. 验收场景

- 空旁观用户返回空列表，不影响账单。
- 非房主非总代请求时踢人列表为空。
- 退出用户在账单中置灰。
- RPC 失败时协议行为明确，不返回不带告警的半空成功。
- 彩金盈亏优先使用准确 RPC。
- 总带入为 0 的用户不展示。

## 13. 待确认项

- 3005 失败时客户端期望错误响应还是空成功响应。
- 钱包汇总 RPC 失败时金额是否应显示 0、隐藏用户还是展示上一次快照。
- 总带入为 0 跳过是否适用于房主和旁观用户。
- 局数回退手数是否仍符合产品口径。

## 14. 本功能不处理的范围

- 不改账变事件生成。
- 不改钱包汇总 RPC。
- 不改好友备注协议。
- 不改入房旁观通知。
