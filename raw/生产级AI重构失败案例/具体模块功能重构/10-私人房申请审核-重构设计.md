# 私人房申请审核重构设计文档

> 对应分析文档：`docs/Junk-游戏准备重构/10-私人房申请审核.md`
> 设计日期：2026-06-04
> 工程：dx-game-frame（命令骨架在 frame-service-core）+ dx-game-guandan（彩金扩展）
> 分支：20260615/junk
> 复杂度：H（P0 高）
> 前置依赖：03-带入（钱包/彩金、ReviewWalletAmount 视图）、16-私人房固定带入审核（共享审核单）、11-私人房审核通过（审核结果上分，本功能只建单）
> 跨负责人契约：房主审核动作归 Ceres——本功能只建审核单 + 发审核 MQ，记录与 Ceres 的 MQ/审核单接口契约，不重构房主审核

## 1. 背景与目标

私人房申请审核覆盖 1078 `CommonProto.AMOUNT_APPLY`（PROTO_START_1000+78，`@ThreadModel(TABLE)`），玩家申请额度并生成待审核 `BringInReviewDto`。链路 `AmountApplyCommand → AmountApplyDefault`（frame-service-core）`handleCommand: check → amountApplyStore → dataHandler → sendMq → after → notice`，guandan `AmountApplyBootstrap extends AmountApplyImpl` 重写 `amountApplyStore` 记录主筹码 + 彩金。基于真实代码确认：

- **P0-1：无显式申请资格校验**。`AmountApplyDefault.amountApplyStore`（`:105`）直接 `BaseTableUserCache.getUser(...)`（`:111`）后建单；`check`（`:91`）只查重复审核单（`selectTableUserAuditIng`→`COMMON_TRANSFER_AUDIT_DUPLICATE` `:102`），**未显式确认申请人仍在房间/仍 OCCUPY 或补码流程**。用户退出/站起/缓存缺失时仍尝试建单 → NPE/系统异常，审核单与桌内状态不一致。
- **P0-2：建单/事件/MQ/通知非原子**。`handleCommand` 顺序 `amountApplyStore(:74)→dataHandler(:76 发 BringInReviewEvent)→sendMq(:78)→after(:80)→notice(:82)`。Redis 写成功后 event/MQ/通知失败则返回错误，但**审核单已存在**：客户端看到失败，房主侧可能稍后收到审核单、或下次申请因已有待审核单被重复拦截；MQ 成功但通知失败时占座弹窗与审核状态不一致。
- **P1-1：`amountApply` 服务端覆盖规则不透明**。`getAmountApply` 对非房主按固定/非固定带入覆盖申请额度，房主按 walletList 求和，`AmountApplyImpl.check` 再用 `userCashBalance+amountApply<totalBring` 校验。客户端传入 amountApply 不一定生效，主钱包+彩金一起申请时关系不直观。
- **P1-2：彩金沿用 `bringInSquidAmount`**。`AmountApplyBootstrap.amountApplyStore`（`AmountApplyBootstrap.java:56`）将 `GD_BONUS` 写入 `bringInSquidAmount`（与 03 同问题）。

目标：**保持 1078 `AmountApplyReqVO`、审核单 `BringInReviewDto`、审核 MQ 结构 100% 兼容**前提下，新增申请资格校验、为建单后失败引入补偿/补发状态、把额度覆盖规则显式化（calc result）、彩金内部用 ReviewWalletAmount 视图（复用 03）。**最小安全改造优先**，与 Ceres 房主审核保持接口契约。

## 2. 决策记录

| 决策项 | 结论 |
|--------|------|
| 主导方案 | Frame 主导骨架（frame-service-core 的 AmountApply* 链），guandan 提供彩金扩展 |
| 协议兼容 | 1078 `AmountApplyReqVO(amountApply/walletList)`、响应 `BringInReviewDto`、审核 MQ 消息结构、`OCCUPY_ADD_CHIPS_NOTICE` 字段不变 |
| 申请资格校验（P0-1） | 新增 `AmountApplyEligibilityValidator`：校验 tableInfo/userItem/user area/user status/wallet list；不满足返回明确业务异常，不进入建单 |
| 建单后失败补偿（P0-2） | 引入 `AmountApplyTransactionRecord{stored, eventPublished, mqSent, noticeSent}`；建单成功后 event/MQ/通知失败 → 按审核单 id 补偿任务/补发，对外明确"申请已受理，通知补发中"（不再"建单成功却返回失败"） |
| 额度覆盖透明（P1-1） | 内部 `AmountApplyCalcResult{clientAmount, serverAmount, calcReason, totalBring, mainAmount, bonusAmount}`；响应审核单保持兼容，日志/测试用 calc result |
| 彩金字段（P1-2） | 外部 `bringInSquidAmount` 不动；内部 `ReviewWalletAmount` 视图（复用 03 R-04），`GD_BONUS` 映射在一处 |
| 与 Ceres 契约 | 房主审核动作不重构；记录"建单→审核 MQ→Ceres 房主审核→审核结果回调（11 消费）"的接口契约（MQ topic/审核单字段/状态机）|
| 灰度策略 | 新资格校验先只记录不拦截观察命中 → 开建单补偿记录 → 灰度启用拦截 → 最后切金额计算器 |

## 3. 整体架构

### 改造前

```
1078 AmountApplyCommand(TABLE) → AmountApplyDefault.handleCommand
  ├─ check :91 selectTableUserAuditIng → COMMON_TRANSFER_AUDIT_DUPLICATE :102 (仅查重复, 无资格校验 P0-1)
  ├─ amountApplyStore :105 getUser:111 + 建 BringInReviewDto + Redis addOrUpdate
  ├─ dataHandler :76 发 BringInReviewEvent
  ├─ sendMq :78 sendPrivateTableAuditMq
  ├─ after :80
  └─ notice :82 发 AmountApplyUserSitDownNoticeEvent
  （store→event→mq→notice 非原子, 建单后失败客户端见失败但单已存在 P0-2）
guandan AmountApplyBootstrap.amountApplyStore: 主筹码 bringInAmount + 彩金 bringInSquidAmount:56 (P1-2)
```

### 改造后

```
1078 AmountApplyCommand(TABLE, 不变) → AmountApplyDefault(入口保留)
  └─ AmountApplyFlowService.apply(context)
       ├─ AmountApplyEligibilityValidator  — tableInfo/userItem/area/status/wallet 资格校验(R-01)
       ├─ check 重复审核单(保留)
       ├─ AmountApplyCalculator → AmountApplyCalcResult(额度覆盖透明 R-03)
       ├─ BringInReviewFactory → BringInReviewDto(ReviewWalletAmount 视图, R-04)
       ├─ Redis addOrUpdate + AmountApplyTransactionRecord(stored=true)
       ├─ AmountApplyEventPublisher(eventPublished) / sendMq(mqSent)
       └─ AmountApplyNoticeCoordinator(noticeSent) — 失败补发, 不回退已建单(R-02)
```

**兼容性原则**：
- 1078 请求/响应、`BringInReviewDto`、审核 MQ、`OCCUPY_ADD_CHIPS_NOTICE` 字段不变。
- `AmountApplyCommand`/`AmountApplyDefault`/`AmountApplyImpl` public 签名保留（内部委托）。
- `bringInSquidAmount` 字段保留（内部 ReviewWalletAmount 视图）。
- 房主审核（Ceres）/审核通过上分（11）链路不动。

## 4. 安全改造（R-01 ~ R-04）

### R-01：申请资格校验器（P0-1）
- 新增 `AmountApplyEligibilityValidator`：校验 `tableInfo!=null`/`userItem!=null`/user area(在房间)/user status(OCCUPY 或补码等待)/wallet list 合法；不满足返回明确业务异常，不进入 `amountApplyStore`。

### R-02：建单后失败补偿（P0-2）
- 新增 `AmountApplyTransactionRecord{stored,eventPublished,mqSent,noticeSent}`。
- 建单成功（stored=true）后 event/MQ/通知任一失败 → 按审核单 id 发补偿任务/补发；对外返回"申请已受理"（不再建单成功却返回系统异常）。
- MQ 失败补偿策略与 Ceres 审核侧约定（重试/补发）。

### R-03：额度覆盖透明（P1-1）
- 新增内部 `AmountApplyCalculator` 产出 `AmountApplyCalcResult`（clientAmount/serverAmount/calcReason/totalBring/mainAmount/bonusAmount）；响应审核单字段不变，日志/测试用 calc result 可追溯覆盖原因。

### R-04：彩金 ReviewWalletAmount 视图（P1-2，复用 03）
- 复用 03 的 `ReviewWalletAmount` 视图：`bringInAmount→GUANDAN`、`bringInSquidAmount→GD_BONUS`；`AmountApplyBootstrap.amountApplyStore` 改为构造视图，外部字段不变。

## 8. 迁移与灰度

| 维度 | 方案 |
|------|------|
| 新旧共存 | 资格校验先只记录不拦截观察命中；建单补偿记录开启 |
| 灰度 | 资格拦截灰度 → 金额计算器切换 |
| 回滚 | 关闭新校验器，保留旧 `AmountApplyDefault` 步骤顺序 |
| 与其他 Junk/Ceres 同步 | 彩金视图复用 03；审核通过上分见 11；固定带入见 16；房主审核 MQ 契约与 Ceres 对齐 |

## 9. 验收关键场景

1. 非房主余额不足 → 1078 返回参数/额度异常。
2. 已有未过期审核单 → 重复申请错误（保留）。
3. 审核单过期 → 可重新申请。
4. 主钱包 + 彩金都进审核单（ReviewWalletAmount 视图）。
5. 申请成功 → Redis INIT 审核单 + 审核 MQ 发出；MQ 失败 → 补偿/补发，不返回系统异常。
6. 用户已退出/非 OCCUPY → 资格校验返回明确业务异常，不 NPE、不建单。

## 11. 本功能不处理的范围

- 不处理审核通过后上分（归 11）。
- 不处理固定带入首次入座自动触发审核（归 16）。
- 不重构房主审核动作（归 Ceres，仅记 MQ/审核单契约）。
- 不改审核 MQ 协议、钱包 RPC。
